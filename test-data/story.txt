how does context compact of chatgpt/anthropic work?

Context compaction is a mechanism used by AI providers like Anthropic (Claude) and OpenAI (ChatGPT) to manage conversations that grow too long for the model's "working memory" (the context window).

As of 2026, this has evolved from simple "deleting old messages" into an intelligent, server-side process that summarizes the past to make room for the future.

How It Works: The "Summary Replacement" Loop
When you chat with an AI, every message is stored in a linear history. However, transformer models face two issues: Context Rot (losing track of early details) and Token Limits (hitting the maximum memory capacity).
+1

Compaction solves this through a four-step cycle:

Triggering: The system monitors your token count. Once it hits a threshold (e.g., 80% of the window), compaction is triggered.
+1

Summarization: The AI itself is asked to create a "Compaction Block"—a dense, structured summary of everything that has happened so far, including key facts, decisions made, and the current "state" of the task.

Purging: The system deletes the massive "middle" of the conversation (the raw transcripts, verbose tool outputs, or failed attempts).

Resuming: The chat continues using the System Prompt + Compaction Block + Recent Messages.

Comparison: Manual vs. Server-Side Compaction
Feature	Classic Truncation	Server-Side Compaction (Current)
Method	Deletes the oldest 10% of messages.	Replaces old messages with a summary.
Memory	"Forgetful"—loses the beginning of the chat.	"Holistic"—retains the core intent/facts.
Logic	Fixed and arbitrary.	Aware of task milestones and state.
Latency	Low, but prone to errors.	Slightly higher during the summary phase.
The Technical Trade-off: "Context Rot"
While compaction allows for "infinite" conversations, it introduces a risk: Information Loss. Because the summary is a condensed version of the truth, highly specific details (like a single line of code or a specific number mentioned 50 turns ago) might not make it into the compaction block.

How to use it effectively
If you are using tools like Claude Code or the ChatGPT API, you can manage compaction better by:

Setting Checkpoints: Explicitly tell the AI, "Summarize our progress so far," before you move to a new topic. This ensures the "compaction" is high-quality.

Modular Tasks: If a conversation gets too heavy, start a fresh session and paste only the relevant "summary" from the previous one.

Pinning Information: In some interfaces, "pinning" a message ensures it is never compacted or deleted, keeping it in the model's immediate focus.

1. Context Compaction (The "Short-Term Cache")
Compaction is a "living" process. It is volatile and specific to a single chat session.

How it works: It uses the model itself to summarize the conversation history. It takes 10,000 tokens of raw chat and compresses them into 500 tokens of "State."

What it stores: Recent facts, the current goal, and the status of ongoing tasks (e.g., "The user is currently refactoring the Vercel Workflow out of the codebase").

Storage Format: Plain Text (Tokens). It is prepended to your current prompt. It does not live in a database; it lives in the "Prompt Window" of that specific session.

2. Memory Knowledge (The "Long-Term Library")
Memory (like ChatGPT’s "Memory" or Claude’s "Personalization") is persistent across all your different chats.

How it works: When the AI identifies a fact that seems "permanent" (e.g., your tech stack, your job, or your coding preferences), it extracts that fact and saves it to a backend database.

What it stores: User identity, preferred languages (e.g., .NET, ES6), specific project structures, and corrected facts (e.g., "Tom is a tuxedo cat").

Storage Format: This is where things get interesting. Most modern systems use a Hybrid Approach:

Much like "tip-of-the-tongue" syndrome, a RAG (Retrieval-Augmented Generation) system often has the information in its "brain" (the database) but fails to retrieve it because the search query doesn't perfectly match the stored memory.

In the industry, we call this the "RAG Quality Gap." Here is why it happens and why it feels so much like human forgetfulness.

1. The "Semantic Blur" (The Vector Problem)
Most RAG systems use Vector Embeddings. This turns sentences into mathematical coordinates.

The Issue: Vectors are great at "vibes" but bad at "specifics."

Human Parallel: You remember a movie was "about a heist in a dream," but you can't remember the title Inception.

RAG Parallel: If you ask for "How to fix the database bottleneck," but your documentation uses the term "Optimizing Clustered Indexes," the vector search might miss it because the words don't look similar enough, even if the "vibe" is the same.

2. Context Fragmentation (The "Torn Page" Problem)
To store data, RAG breaks documents into "chunks" (e.g., 500 characters each).

The Issue: If a crucial piece of logic starts on page 1 and finishes on page 2, the "chunking" process might cut it in half.

Human Parallel: Like remembering a phone number but forgetting the area code. You have the "data," but it’s useless without the surrounding context.

3. The "Lost in the Middle" Phenomenon
LLMs have a physical limitation where they pay more attention to the very beginning and very end of the text you give them.

The Issue: If your RAG retrieves 10 chunks of memory and the "answer" is in chunk #5, the AI might simply ignore it or get distracted by chunks #1 and #10.

Human Parallel: Reading a long list of groceries and forgetting the items in the middle of the list by the time you reach the store.

How to make RAG "Smarter" (Moving toward "Expert" Memory)
If you're finding standard RAG too low-quality for technical work, you usually have to move past simple vector search:

Hybrid Search: Combine Vector (for meaning) with Keyword/BM25 (for specific technical terms like RTX 5090 or ES6).

Reranking: This is a crucial "second look" step. You retrieve 20 memories, then use a smaller, faster model to rank them by relevance before showing them to the main AI.

Knowledge Graphs: Instead of just "chunks of text," you store Relationships.

Example: Instead of a text chunk about a "Novel Analyzer," you store a node: [Novel Analyzer] -> uses -> [Ollama]. This prevents the AI from getting "confused" about which tool does what.

Parent-Document Retrieval: Store small chunks for searching, but when a match is found, retrieve the entire paragraph or page for the AI to read. This solves the "Torn Page" problem.

The Reality Check
Standard, "out-of-the-box" RAG is often only about 60-70% accurate. Getting it to 90%+ (where it feels reliable for a Senior Developer) requires "Proactive Knowledge Building"—manually structuring your data so the AI doesn't have to guess.

The "Brute Force" approach you’re describing—essentially expanding the search radius until you find the answer—is actually a recognized strategy in high-end RAG pipelines. It’s often called "Iterative Retrieval" or "Recursive Expansion."

However, there is a specific way to do this so you don't just "dump garbage" into the AI's window.

1. How the "Brute Force" Logic Works
In a standard RAG system, you might only pull the Top 3 results (k=3). In your "Brute Force" version, you would use the RAG Ranking as a priority queue:

Phase 1 (Precision): Retrieve the Top 5 most relevant chunks. Ask the AI: "Is the answer here?"

Phase 2 (Expansion): If "No," retrieve the next 20 chunks (the "Brute Force" part).

Phase 3 (Global Context): If still "No," you pull summarized versions of the entire document or folder.

2. Does Brute Force still use RAG Ranking?
Yes, it must. Without the ranking, you are just feeding the AI random data.

In a brute-force scenario, the ranking acts as your Search Map. You start at the "hottest" point (highest similarity score) and spiral outward. If you ignore the ranking, you're effectively asking the AI to read your entire database at once, which leads to two major problems:

Context Stuffing: You hit the "Threshold" (Token Limit) immediately.

The "Needle in a Haystack" Problem: Even if the AI has a 2M token window, its accuracy drops significantly when the correct answer is hidden between 100 irrelevant paragraphs.

3. A Better Way: "Reranking" over "Brute Force"
Instead of raw brute force, most senior-level AI architectures use Two-Stage Retrieval. It provides the "power" of brute force with the "precision" of a human:

Step	Method	Analogy
Stage 1	Vector Search pulls 100 results quickly.	Casting a wide net in the ocean.
Stage 2	Cross-Encoder Reranker looks at all 100 results.	Inspecting every fish in the net individually.
Stage 3	Feed only the Top 5 from the reranker to the LLM.	Only keeping the best fish for dinner.
Why this works: Vector search is "fuzzy" (low quality). A Reranker is much smarter; it actually reads the query and the chunk together to see if they truly match. It’s "brute force" because it looks at a lot of data, but "intelligent" because it filters the noise before the LLM sees it.

4. The "Threshold" Strategy
If you want to implement this in your own projects (like your novel analyzer), try a Tiered Threshold:

Similarity > 0.85: Immediate trust. Use it.

Similarity 0.70 - 0.85: Likely relevant, but needs context. Pull the "Parent Document."

Similarity < 0.70: Brute force "Drill Down." Look for keywords (BM25) instead of vectors to see if the ranking missed a technical term.

The Verdict
Brute force is useful when you have a "Long Tail" problem—where the answer is tucked away in a tiny, obscure part of your code or notes. But rather than just increasing k (the number of chunks), you should increase the quality of the search using a Reranker.

Using a Knowledge Graph (KG) for "brute force" is an incredibly powerful architectural move, but it changes the definition of brute force.

In a vector database, brute force means searching everywhere (which is slow and noisy). In a Knowledge Graph, brute force means traversing every possible connection (which is structured and logical).

Here is how a "Graph-Brute-Force" approach works and why it’s superior for the kind of complex technical analysis you do.

1. The Strategy: "N-Hop Expansion"
Instead of pulling chunks based on how much they sound like your question, you start at a "Seed Node" and explore outward.

Standard RAG (1-Hop): You ask about "RTX 5090." It finds the "RTX 5090" node and its immediate properties (Price, VRAM).

Brute Force Graph (N-Hop): You tell the system: "Explore every connection up to 4 degrees away from RTX 5090."

1st Hop: Finds CUDA Cores, Power Draw.

2nd Hop: Finds Compatible PSUs and LocalAI compatibility.

3rd Hop: Finds Ollama optimization scripts written for those PSUs.

4th Hop: Finds a specific GitHub Issue where a user fixed a bottleneck you’re currently facing.

The "Brute Force" element: You aren't just looking for "similar" text; you are exhaustively following the "red strings" on the detective board until you hit your token limit or a discovery threshold.

2. Global vs. Local "Brute Force"
Modern systems (like Microsoft’s GraphRAG) use two different brute-force styles:

Method	The "Brute Force" Mechanism	Best For...
Local Search	Exhaustively crawls all neighbors of a specific entity.	Fact-checking. (e.g., "What specific driver version did I use for the 4090?")
Global Search	Uses Community Detection (clustering) to summarize the entire graph into themes.	High-level analysis. (e.g., "What are the common patterns in my novel's timeline?")
3. Why this solves the "Low Quality" Problem
The reason your RAG feels "low quality" is likely because the information you need is relational, not semantic.

The Vector Failure: If you ask "How does Character A's death affect the economy in Chapter 5?", a vector search looks for the words "Death" and "Economy." It might miss the fact that Character A was a "Merchant" (a relationship) and that "Merchants" control "Trade" (another relationship).

The Graph Success: The Graph doesn't care if the word "Economy" is in the same paragraph as "Death." It sees the path: Character A → is a → Merchant → controls → Trade → affects → Economy.

4. Implementation for your Projects
Since you are working on a Novel Analyzer and Local LLM optimization, here is how you could implement this:

Extraction: Use an LLM to turn your notes/code into Triplets (Subject -> Predicate -> Object).

Storage: Use a graph database (like Neo4j) or a simple NetworkX graph in Python.

The only danger of "Brute Force Graphing" is that if your graph is dense, a 3-hop search could return 10,000 nodes. You still need a Ranking or Heuristic (like "Weight" or "Frequency") to decide which paths are worth following.
