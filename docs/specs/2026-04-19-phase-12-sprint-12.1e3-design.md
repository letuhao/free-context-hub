# Design — Phase 12 Sprint 12.1e3

Resolves 3 questions deferred from CLARIFY.

## 1. Helper name

**Decision: `isSalienceWriteDisabled()`**.

Matches existing `isSalienceDisabled()` pattern. Adjective-first naming (`Write-Disabled`) mirrors `is<Domain><State>` convention. Alternative `isSalienceLogDisabled` or `isSalienceNoWrite` would read less idiomatically; `isSalienceWriteDisabled` says "writes are disabled" — unambiguous.

## 2. Composition semantics — DISABLED vs NO_WRITE

Both flags are **independent boolean gates**. Behavior matrix:

| `LESSONS_SALIENCE_DISABLED` | `LESSONS_SALIENCE_NO_WRITE` | Reads | Writes |
|---:|---:|---|---|
| false | false | ✅ active (boost ranking) | ✅ active (log events) |
| true | false | ❌ skipped (ranking = pure hybrid) | ❌ skipped (nothing to boost from) |
| false | **true** | ✅ active (uses existing log state) | ❌ skipped |
| true | true | ❌ skipped | ❌ skipped (redundant; same as DISABLED=true) |

**Key insight:** `DISABLED=true` already implies no writes (because search paths check `isSalienceDisabled()` BEFORE calling `logLessonAccess`). So when `DISABLED=true`, the new NO_WRITE flag is redundant but not harmful.

**Documentation in salience.ts:** the docstring on `isSalienceWriteDisabled` will note that operators typically use this flag for **measurement isolation** — they WANT reads to happen (to measure ranking) but NOT writes (to keep the access-log state pristine for repeated A/B). Use cases:
- QC baseline runs where pollution would confound subsequent measurements
- Debugging salience boost for a single query without perturbing the log
- Pre-production staging where the operator wants to measure behavior on a fixed-state log

## 3. Gate placement inside `logLessonAccess`

**Decision: early-return at function top, BEFORE the empty-batch check.**

```typescript
export async function logLessonAccess(pool: Pool, entries: AccessLogEntry[]): Promise<void> {
  if (isSalienceWriteDisabled()) return;  // ← NEW
  if (entries.length === 0) return;
  // ... existing code unchanged
}
```

Rationale:
- Placing the check BEFORE `entries.length === 0` means even an empty-batch call short-circuits on the flag (no behavioral difference, but cleaner reasoning: "NO_WRITE → nothing happens").
- Placing the check INSIDE the function (not at call sites) means the 6 call sites don't need updating. One edit → six call sites respected.
- Function remains `async`; the return type is `Promise<void>`. Early-return from an async function resolves immediately.

## 4. Unit test design

**Two new tests in `src/services/salience.test.ts`:**

```typescript
test('logLessonAccess respects LESSONS_SALIENCE_NO_WRITE', async (t) => {
  await t.test('flag=false (default) → INSERT happens', async () => {
    const { pool, calls } = mockPool({ rows: [] });
    const entries: AccessLogEntry[] = [
      { lesson_id: 'aaa', project_id: 'p', context: 'consumption-reflect' },
    ];
    await logLessonAccess(pool, entries);
    assert.equal(calls.length, 1);
  });

  await t.test('flag=true → no SQL call', async () => {
    const originalEnv = process.env.LESSONS_SALIENCE_NO_WRITE;
    process.env.LESSONS_SALIENCE_NO_WRITE = 'true';
    try {
      // Force env cache bust. getEnv() in salience.ts re-parses on each call per 12.1c design,
      // so setting process.env is enough.
      const { pool, calls } = mockPool({ rows: [] });
      const entries: AccessLogEntry[] = [
        { lesson_id: 'aaa', project_id: 'p', context: 'consumption-reflect' },
      ];
      await logLessonAccess(pool, entries);
      assert.equal(calls.length, 0, 'no SQL call expected when NO_WRITE=true');
    } finally {
      if (originalEnv === undefined) delete process.env.LESSONS_SALIENCE_NO_WRITE;
      else process.env.LESSONS_SALIENCE_NO_WRITE = originalEnv;
    }
  });
});
```

**Verification before commit:** run the new test in isolation to confirm env-var setup/teardown doesn't leak into other tests. `npm test` full suite must remain green.

**Caveat — env.ts caching.** If `getEnv()` caches its parsed env on first call (memoization), toggling `process.env.LESSONS_SALIENCE_NO_WRITE` at test-time won't change the cached value. Need to verify whether env.ts caches. Check in BUILD; if it caches, the test needs an alternate approach (maybe export a `__setSalienceWriteDisabledForTests__` helper, or instrument via vi.mock / sinon stub).

## 6. Validation baseline

**Procedure:**

1. Pre-run: `docker compose exec db psql -c "SELECT COUNT(*) FROM lesson_access_log;"` → record `N_BEFORE`.
2. Recreate mcp with flag: `LESSONS_SALIENCE_NO_WRITE=true docker compose up -d --force-recreate mcp`.
3. Verify env: `docker compose exec mcp node -e "console.log(process.env.LESSONS_SALIENCE_NO_WRITE)"` → `true`.
4. Run baseline: `npm run qc:baseline -- --tag sprint-12.1e3-validate --samples 1 --surfaces lessons`.
5. Post-run: same `COUNT(*)` → must equal `N_BEFORE`.
6. Restore default: `docker compose up -d --force-recreate mcp` (no env override; picks up .env / compose defaults).

**Assertion in sprint session:** N_BEFORE == N_AFTER. If they differ by even 1 row, the flag is broken — halt + debug.

## 7. Friction-class entry structure

New section in `docs/qc/friction-classes.md`:

```markdown
### goldenset-pollution

**Definition.** Measurement infrastructure that writes to the same state
it reads from can distort subsequent measurements. For free-context-hub,
every `qc:baseline` run accumulates `consideration-search` rows in
`lesson_access_log`; accumulated over many runs, these inflate the
salience of goldenset-target lessons and shift subsequent ranking
measurements.

**Why it happens.** `searchLessons` and `searchLessonsMulti` call
`logLessonAccess` fire-and-forget for every top-K match (per Sprint
12.1c write paths). Baseline runs exercise these paths. A 40-query
goldenset × K=10 × samples=3 writes ~1200 rows per run. Duplicate-trap
queries with 9-20 UUID targets each write 9-20 rows per query sample,
disproportionately inflating fixture-lesson salience.

**Diagnostic signal.** A baseline run shows top-K results dominated by
lessons whose titles look like test fixtures (`impexp-*`,
`agent-bootstrap-e2e-*`, `gui-filter-*`) or duplicate-group cluster
members, especially for abstract or semantically-distant queries.
Numerically: the change shows up at longer half-lives (≥ 14d) where
old access-log rows have enough weight to compete with hybrid-score.

**Example.** Sprint 12.1e2 polluted-sweep HL=30: the undici paraphrase
query `why does swapping http agents between the default and an
installed one break streaming bodies` dropped rank-1 → rank-7 because
6 fixture lessons outranked the true target via accumulated salience.
On cleaned state (90 audit-bootstrap rows only), the same query hits
rank-2.

**Mitigation paths.**
1. **Use `LESSONS_SALIENCE_NO_WRITE=true`** (Sprint 12.1e3) before
   salience-sensitive baseline runs. Reads still compute salience;
   writes are suppressed. One env flag, no state mutation.
2. **Truncate before/after** (manual, error-prone): `DELETE FROM
   lesson_access_log WHERE context='consideration-search'` after
   backing up. Used by Sprint 12.1e2 before NO_WRITE existed.
3. **Isolated measurement database** (long-term): a separate
   `lesson_access_log_baseline` table. Higher complexity; unnecessary
   given NO_WRITE.

**Cross-ref.** Sprint 12.1e2 POST-REVIEW (commit 301d3e0) discovered
the pattern; Sprint 12.1e3 (pending commit) lands the NO_WRITE flag.
```
