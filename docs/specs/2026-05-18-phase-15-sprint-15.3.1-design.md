## 0.5 What F1 delivers — and what it does NOT (the honest scope)

Audit Finding 1 (CRITICAL) is: a caller forges the no-self-approval invariant by **picking
two identity strings** in JSON request bodies (`submitted_by:"alice"`, `actor_id:"bob"`).
F1 closes the *string-picking* vector. It is **not**, by itself, the whole CRITICAL.

**F1 binds the REST acting identity by caller mode:**

| Caller | `req.apiKeyName`? | F1 behaviour | String-forgery status |
|--------|-------------------|--------------|-----------------------|
| `MCP_AUTH_ENABLED=true` + DB api key | set (`api_keys.name`) | `submitted_by`/`actor_id` forced to `apiKeyName`; a differing body value → 403 | **CLOSED** — a DB-keyed caller can only act as its own key; it cannot be two actors |
| `MCP_AUTH_ENABLED=true` + env token (`CONTEXT_HUB_WORKSPACE_TOKEN`) | not set | body value stands | open — but the env token is **one shared super-admin secret**: a single principal by construction. Justified on its own merits, independent of the auth-off case. |
| `MCP_AUTH_ENABLED=false` | not set | body value stands | **open** — there is no authentication at all. Accepted *only* as the single-trusted-operator dev posture. DEFERRED-009/015 carry the HARD trigger: the coordination authz model must not be relied on with auth off in any multi-actor / production deployment. |

**What F1 does NOT do (explicitly out of 15.3.1 scope):**

1. **It does not make one human = one principal.** Nothing bounds how many keys one operator
   may mint. A human who can create api keys can create N keys named `submitter`,
   `approver-L1`, `approver-L2`; F1 stamps each request/step with that key's name; the
   self-decision guard (`actor_id !== submitted_by`) sees distinct strings and passes. F1
   raises the forgery cost from "edit a JSON field" to "mint N api keys" — a real
   improvement, but **not** a per-human guarantee. Bounding key multiplicity is the
   key-provisioning authorization model → **DEFERRED-016** (filed at REVIEW-DESIGN r2; same
   HARD trigger class as DEFERRED-015).
2. **It does not make `level` authoritative.** Even a genuinely distinct principal still
   self-declares its `topic_participants.level` via `joinTopic` (audit F2). The officeholder
   check (`level === target_office`) is only as trustworthy as that self-assertion. **F2 →
   DEFERRED-015.** The audit's CRITICAL Finding 1 is *fully* closed only by **F1 + F2 +
   DEFERRED-016 together**; 15.3.1 ships the F1 (identity-binding) piece — exactly the
   user's chosen disposition.

**Why binding to `name` (not the unique `key_id`) is acceptable.** Two keys that share a
`name` both resolve to the *same* coordination identity — so the self-decision guard treats
them as one actor (one cannot be both submitter and decider). Same-named keys therefore
**cannot** collude; the worst they cause is audit-trail ambiguity ("which physical key acted
as `alice`?"), a known, documented limitation — not a forgery vector. The forgery needs
*distinct* identities, i.e. distinctly-*named* keys, which `key_id`-binding would not prevent
either (N keys = N `key_id`s just as much as N names). Binding to `name` keeps coordination
`actor_id`s human-readable and consistent with the Phase-15 participant model (`joinTopic`
actor_ids are readable names), at no security cost relative to `key_id`.

**Net:** F1's deliverable is precise — *on the `MCP_AUTH_ENABLED=true` DB-key path, the
acting identity is the authenticated key and cannot be forged by editing the request body.*
The design makes no claim beyond that.

## 1. F1 — bind the acting identity to the authenticated key (REST)

**File:** `src/api/routes/requests.ts`. **Service layer unchanged** — `submitRequest` /
`decideStep` keep their `submitted_by` / `actor_id` params; they receive a token-bound value
from the route.

**New route-local helper:**

```ts
/**
 * F1 (Sprint 15.3.1) — resolve the acting coordination identity.
 * DB-keyed caller (apiKeyName set): identity IS the key name; a body value naming a
 * different actor is rejected. Env-token admin / auth-disabled (no apiKeyName): the
 * body value stands (see DESIGN §0.5 — single-trusted-principal / dev posture).
 */
function resolveActorIdentity(
  req: Request,
  bodyValue: string,
): { ok: true; actor: string } | { ok: false } {
  const authedName = (req as Request & { apiKeyName?: string }).apiKeyName;
  if (typeof authedName === 'string' && authedName.length > 0) {
    if (bodyValue && bodyValue !== authedName) return { ok: false };
    return { ok: true, actor: authedName };
  }
  return { ok: true, actor: bodyValue };
}
```

> **Precondition — F1 depends on the writer-role POST gate** (REVIEW-DESIGN r2, NEW FINDING 2).
> `bearerAuth` sets `req.apiKeyName` for *every* valid DB key regardless of role, including
> `reader` keys. F1's "a DB-keyed caller can only act as its own key" guarantee assumes the
> caller has already passed `requireRole('writer')` — the middleware on both POST routes
> (`routes/requests.ts:87,129`). **15.3.1 does not change that gate**; a `reader` key must
> never reach `submitRequest`/`decideStep`. AC1's route test asserts a `reader`-key POST is
> still 403, making this precondition executable. (F4 adds `requireRole('reader')` to the
> GET routes only — it does not touch the POST gate.)

**`POST /api/topics/:id/requests`** — before calling `submitRequest`:

```ts
const id = resolveActorIdentity(req, asString(body.submitted_by));
if (!id.ok) {
  res.status(403).json({ status: 'error',
    error: 'submitted_by does not match the authenticated key', code: 'IDENTITY_MISMATCH' });
  return;
}
// … submitRequest({ …, submitted_by: id.actor })
```

**`POST /api/requests/:id/steps/:n/decide`** — identical, with `body.actor_id`:

```ts
const id = resolveActorIdentity(req, asString(body.actor_id));
if (!id.ok) {
  res.status(403).json({ status: 'error',
    error: 'actor_id does not match the authenticated key', code: 'IDENTITY_MISMATCH' });
  return;
}
// … decideStep({ …, actor_id: id.actor })
```

The direct `res.status(403)` mirrors the existing `parseInt` NaN → 400 branch (no new
`ContextHubError` code; `CODE_TO_STATUS` has no `FORBIDDEN`, and `UNAUTHORIZED` maps to 401).

**MCP — out of scope, by design** (CLARIFY §4, §0.5): the MCP transport authenticates a
*workspace* (one shared `workspace_token`), not a per-caller identity. A workspace-token
holder is a single trusted principal — the same trust level as REST's env-token admin. A
doc comment (§6) records this; real MCP per-caller identity is the Phase 15 authz model
(DEFERRED-009/015).

## 2. F3a — tie request, topic, and subject artifact together

```ts
  const upd = await client.query<{ version: number; content_ref: string | null; topic_id: string }>(

```

The `topic_id` in `RETURNING` is a **locked, consistent read** — it comes from the very row
the `UPDATE` write-locks; it is not an unlocked side-read. The two `decideStep` call sites
drop the `topicId` argument: `resolveArtifact(client, 'approve', artifactId, actorId)` and
`…'return'…`. Mirrors `src/services/artifacts.ts:topicIdForArtifact` (line 394) — 15.2's
`writeArtifact` derives the topic from the artifact identically. The topic is read inside the
same guarded UPDATE → **no extra query, no new lock**; the canonical order
`request → request_step → artifact → topics` (design §0.2) is unchanged.

`decideStep` keeps its own `topicId` (`req.topic_id`) for the `request.*` events — those
correctly belong to the request's topic.

## 2.5 Invariant F3a depends on — `artifacts.topic_id` is immutable

F3a's 2a check is at submit time; `resolveArtifact` runs later at decide time. Their
agreement requires the artifact's `topic_id` not to change in between.

**Verified (REVIEW-DESIGN r1+r2, Finding 3):** `grep -rn "UPDATE artifacts SET" src` returns
exactly 3 sites — `board.ts:627` (sets `state`), `requests.ts:281` (`resolveArtifact`, sets
`state`+`version`), `requests.test.ts:551` (test helper, sets `state`). **None mutates
`topic_id`.** Reinforced structurally: `artifact_id` is the derived key
`<topic_id>:<task_id>:<slot>` (migration 0054), `topics` rows are permanent (migration 0054
FK note — Phase 15 has no delete-topic / re-parent operation), and there is no
`re-parent-artifact` service.

**Invariant (15.3.1):** `artifacts.topic_id` is immutable post-creation. F3a's 2a check is
therefore stable across the submit→decide gap. *If a future sprint adds an
`UPDATE artifacts SET topic_id` path, this invariant — and F3a's 2a check — must be
revisited* (recorded for the SESSION-phase deferred scan).

## 7. Regression surface (what could break)

- **F1 + existing route tests** — `routes/requests.test.ts` runs with `MCP_AUTH_ENABLED=false`
  → no `apiKeyName` → `resolveActorIdentity` returns the body value → behaviour identical.
  Existing 6 route tests stay green (AC2).
- **F3a + existing service tests** — `requests.test.ts` fixtures create the artifact in the
  *same* topic as the request → the 2a check passes. `resolveArtifact`'s signature change is
  internal (not exported) — only the 2 in-file call sites change. Existing 19 service tests
  stay green.
- **F3a `resolveArtifact` 0-row path** — the REVIEW-CODE MED-1 test covers the
  artifact-not-in-`for_review` path; it returns early *before* destructuring the `RETURNING`
  row → still valid (no row → no `topic_id` read).
- **F3a cross-topic — now closed, given §2.5** — 2a rejects a wrong-topic artifact at submit;
  given `topic_id` immutability (§2.5) the artifact cannot move topics afterward, so
  `resolveArtifact` always emits on the artifact's own topic, which equals the request's
  topic. The AC11 cross-topic break is **unreachable** for any request that passed 2a —
  *conditional on the §2.5 invariant*, which is grep-verified for the current codebase.
- **F5/F7** — existing tests use a valid `step_index` (0) and short `kind`/`subject_id` →
  unaffected.
- **Lock order** — F3a adds one column to an existing `RETURNING`; no new query, no new lock.
  Canonical order `request → request_step → artifact → topics` (§0.2) unchanged.
