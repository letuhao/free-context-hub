This is design **rev 3**. Per the longrun calibration note, the per-transaction lock-order
table (§10) is **derived up front** — 15.3 adds a second concurrent sweep, so the lock graph
is established before BUILD, not across review rounds.

**Rev history — rev 1 → rev 2.** REVIEW-DESIGN round 1 (cold-start Adversary) returned 3
BLOCK findings, all resolved here:
- **B1 — self-approval (authorization).** Both route shapes can derive a step whose
  `target_office` is the submitter's own level (`escalate_to_authority` always; `counter_sign`
  via the empty-ladder fallback), and `decideStep` authorized by level alone — so a submitter
  could endorse their own request. Fixed: `decideStep` rejects `actor_id == submitted_by`
  with `self_decision_forbidden` (D5, §3.2, inv. 8); §11.7 documents the lone-officeholder
  consequence (escalation handles it).
- **B2 — `artifact_versions` INSERT incomplete (schema).** §3.3's history-row INSERT omitted
  the NOT NULL `created_by` (no actor was in scope). Fixed: `resolveArtifact` gains an
  `actorId` argument; §3.3 now specifies every column.
- **B3 — closed-topic contract hole (cross-sprint drift).** `submitRequest`/`decideStep`
  relied on `appendEvent`'s seal, surfacing a raw `BAD_REQUEST`/400 — the 15.2 rev-6 MED-2
  defect. Fixed: both plain-read `topics.status` and return a clean `topic_closed`/409.

**Rev history — rev 2 → rev 3.** REVIEW-DESIGN round 2 (cold-start Adversary) verified all 3
round-1 fixes RESOLVED with no regression and walked all 14 ACs; it returned 1 new BLOCK +
1 WARN, both resolved here:
- **B4 — `weight` unbounded above (input-trust).** §3.1 validated `weight ≥ 0` but not the
  upper `INT` bound; a value > 2147483647 reached the `INT` column / matrix comparison as an
  unhandled `22003` → 500 (violating CLARIFY assumption A5 — "`submitRequest` fails CLEAR").
  Fixed: §3.1 bounds `weight` to `[0, 2147483647]` and throws `BAD_REQUEST` above it; a
  boundary test is added (§8 T1–T12).
- **W1 — escalation collapses the counter-sign distinct-endorser guarantee.** When an
  escalated `counter_sign` step climbs to a level a later step also targets, one officeholder
  can satisfy both steps — the multi-party guarantee is lost, not just the step count. §11.2
  now states this explicitly; invariant 3 carries the caveat; it is logged as **DEFERRED-013**
  (distinct-endorser enforcement / step-collapse; trigger Sprint 15.4/15.5).

## 0. Instantiation decisions (CLARIFY → formalized)

| D | Decision |
|---|---|
| D1 | **The DoA matrix is a per-project table `doa_matrix`** keyed by `(project_id, topic_id?, kind, weight_min, weight_max)` → `required_level` + `route_shape`. A `topic_id`-bearing row is a **topic override**; a `topic_id IS NULL` row is project-level. A reserved `project_id='__default__'` row set is seeded by migration 0056 so every project resolves a route without per-project setup. |
| D2 | **Resolution precedence:** topic-override rows → the request's project rows → `__default__` rows. Within a tier, the **narrowest weight span** wins; ties broken by `matrix_id` (deterministic). No match → `no_route`. |
| D3 | **Two route shapes.** `counter_sign` — the route is every level strictly above the submitter up to `required_level`; **all** endorse. `escalate_to_authority` — the route is the single level `required_level`; one covering endorsement resolves it. Both yield ≥1 step (a `counter_sign` ladder that would be empty falls back to a single `[required_level]` step). |
| D4 | **`doa_snapshot`** = the matched `matrix_id` (a UUID) + the resolution tier — frozen onto every `request_steps` row at submission. A later matrix/override edit never re-targets an in-flight request: its steps were materialized at submission. (B.7 "snapshot the rules.") |
| D5 | **Officeholder = any topic participant at the step's `target_office` level — except the request's own `submitted_by`.** Self-select, no assignment (office ≠ officeholder; resolved at decision time). A submitter is **never** an officeholder for their own request — `decideStep` rejects `actor_id == submitted_by` (`self_decision_forbidden`; §3.2, inv. 8). If the submitter is the *only* participant at a step's level the step is undecidable until another joins, or the escalation sweep (§4) carries it forward — liveness holds (§11.7). |
| D6 | **Procedure: unilateral only.** `submitRequest` rejects `procedure='collective'` with `BAD_REQUEST` (decision bodies are Sprint 15.4). |
| D7 | **Subject: `artifact` only** in 15.3. `submitRequest` rejects other `subject_type`s — a request approves an artifact (the Phase-13-F2 evolution). |
| D8 | **Approval outcome.** An `approved` request advances its `for_review` subject artifact `→ final`; a `returned` request sends it `→ working`; a `rejected` request leaves the artifact untouched (the request is killed; the artifact's next move is out of 15.3 scope). |
| D9 | **Escalation re-targets the current step in place, climbing one level per sweep tick.** A timed-out `pending` step below `authority` has its `target_office` bumped up one level with a fresh deadline; at `authority` it is `escalated` and the request resolves `escalation_exhausted` (Sprint 15.5 wires this to a dispute instead). |
| D10 | **Timeouts:** `STEP_DEADLINE_MINUTES = 60` (a step's clock starts when it becomes active); the escalation sweep runs on the existing 5-minute coordination-sweep cycle. |

### 0.1 Transaction & connection contract

Verbatim the Phase 13 / 15.1 / 15.2 pattern: every transactional fn does
`const c = await pool.connect(); try { BEGIN … COMMIT } catch (e) { await
c.query('ROLLBACK').catch(()=>{}); logger.error(...); throw e } finally { c.release() }`,
with an explicit `ROLLBACK` before every early return. The escalation sweep's per-step loop
uses the **§0.1-loop** variant — the catch logs and `continue`s, never rethrows.

### 0.2 Canonical lock order

15.3 extends the global canonical row-lock order to:

> **`task → claim → request → request_step → artifact → topics`**

Every transaction (15.2's and 15.3's) acquires its row locks as a **prefix-consistent
subsequence** of this order. 15.2's Board transactions use the `task → claim → artifact →
topics` subsequence; 15.3's Request transactions use `request → request_step → artifact →
topics`. The two primitives' lock sets are **disjoint except `artifact` and `topics`**, and
both acquire `artifact` before `topics` — so no Board↔Request ABBA is possible. `appendEvent`
does `UPDATE topics SET next_seq…` — it locks the `topics` row — so it is always the final
lock. Derived exhaustively in §10.

## 1. Migration `0056_request_approval.sql`

All `CREATE … IF NOT EXISTS`; the seed is `WHERE NOT EXISTS`-guarded (idempotent, additive).

```sql
-- 0056_request_approval.sql — Phase 15 Sprint 15.3
-- Request-Approval: requests + request_steps multi-level routing + the DoA matrix.

-- The Delegation-of-Authority matrix (D1). topic_id NULL = a project-level row;
-- topic_id set = a topic override. project_id='__default__' = the seeded fallback.
CREATE TABLE IF NOT EXISTS doa_matrix (
  matrix_id      UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id     TEXT        NOT NULL,
  topic_id       TEXT,                              -- NULL = project-level; set = topic override
  kind           TEXT        NOT NULL,
  weight_min     INT         NOT NULL DEFAULT 0,
  weight_max     INT         NOT NULL DEFAULT 2147483647,
  required_level TEXT        NOT NULL CHECK (required_level IN ('execution','coordination','authority')),
  route_shape    TEXT        NOT NULL CHECK (route_shape IN ('counter_sign','escalate_to_authority')),
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (weight_min <= weight_max)
);
CREATE INDEX IF NOT EXISTS doa_matrix_lookup_idx
  ON doa_matrix (project_id, kind, weight_min, weight_max);

-- requests — evolves review_requests (the Request-Approval primitive).
CREATE TABLE IF NOT EXISTS requests (
  request_id   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id     TEXT        NOT NULL REFERENCES topics(topic_id),
  subject_type TEXT        NOT NULL,                -- 'artifact' in 15.3 (D7)
  subject_id   TEXT        NOT NULL,
  kind         TEXT        NOT NULL,
  weight       INT         NOT NULL,
  procedure    TEXT        NOT NULL CHECK (procedure IN ('unilateral','collective')),
  route_shape  TEXT        NOT NULL CHECK (route_shape IN ('counter_sign','escalate_to_authority')),
  status       TEXT        NOT NULL DEFAULT 'open'
                 CHECK (status IN ('open','approved','returned','rejected','escalation_exhausted')),
  current_step INT         NOT NULL DEFAULT 0,      -- the active request_steps.step_index
  submitted_by TEXT        NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS requests_topic_status_idx ON requests (topic_id, status);

-- request_steps — the materialized, frozen route (B.7).
CREATE TABLE IF NOT EXISTS request_steps (
  request_id    UUID        NOT NULL REFERENCES requests(request_id),
  step_index    INT         NOT NULL,
  target_office TEXT        NOT NULL CHECK (target_office IN ('execution','coordination','authority')),
  doa_snapshot  TEXT        NOT NULL,                -- D4 — frozen matrix_id:tier
  procedure     TEXT        NOT NULL DEFAULT 'unilateral',
  deadline      TIMESTAMPTZ NOT NULL,                -- meaningful while status='pending'; reset on activation
  status        TEXT        NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending','endorsed','returned','rejected','escalated')),
  decided_by    TEXT,
  decided_at    TIMESTAMPTZ,
  PRIMARY KEY (request_id, step_index)
);
-- the escalation-sweep scan predicate
CREATE INDEX IF NOT EXISTS request_steps_sweep_idx ON request_steps (status, deadline);

SELECT '__default__', NULL, 'artifact_review', 0,  49,         'coordination', 'counter_sign'
 WHERE NOT EXISTS (SELECT 1 FROM doa_matrix WHERE project_id='__default__' AND kind='artifact_review');
INSERT INTO doa_matrix (project_id, topic_id, kind, weight_min, weight_max, required_level, route_shape)
SELECT '__default__', NULL, 'artifact_review', 50, 2147483647, 'authority',    'escalate_to_authority'

```

`requests`/`request_steps` reference `topics` with **no `ON DELETE`** (the §0.3 topics-permanence

references; a deleted topic simply leaves dead override rows (harmless).

**No ALTER to any existing table.** Migration 0056 only `CREATE`s — 15.1 already

## 2. `src/services/doaMatrix.ts` (new) — matrix resolution + route derivation

`LEVEL_RANK = { execution: 0, coordination: 1, authority: 2 }`; `LEVELS_ASC` ordered.
`STEP_DEADLINE_MINUTES = 60`.

### 2.1 `resolveMatrixRow(client, { project_id, topic_id, kind, weight }) → MatrixRow | null`

A plain read (no lock). One query with a tier-ranked `ORDER BY`:
```sql
SELECT matrix_id, required_level, route_shape,
       (weight_max - weight_min) AS span,
       CASE WHEN topic_id = $topic                       THEN 0   -- topic override
            WHEN topic_id IS NULL AND project_id = $proj  THEN 1   -- project row
            ELSE 2 END                                   AS tier  -- __default__
  FROM doa_matrix
 WHERE kind = $kind AND weight_min <= $weight AND weight_max >= $weight
   AND ( topic_id = $topic
      OR (topic_id IS NULL AND project_id = $proj)
      OR (topic_id IS NULL AND project_id = '__default__') )
 ORDER BY tier ASC, span ASC, matrix_id ASC
 LIMIT 1
```
0 rows → `null`. `doa_snapshot` is composed as `<matrix_id>:t<tier>`.

### 2.2 `deriveRoute(submitterLevel, requiredLevel, routeShape) → string[]`

- `escalate_to_authority` → `[requiredLevel]` (one step at the covering level).
- `counter_sign` → `[ L for L in LEVELS_ASC if RANK(submitter) < RANK(L) ≤ RANK(required) ]`;
  if that list is empty (the submitter already outranks `required`) → `[requiredLevel]`.

Both shapes therefore return **≥1** target office, ordered ascending by rank.

## 3. `src/services/requests.ts` (new) — the request lifecycle

### 3.1 `submitRequest({ topic_id, subject_type, subject_id, kind, weight, procedure, submitted_by }) → SubmitResult`

Validate: all string fields non-empty; `weight` an **integer in `[0, 2147483647]`** — the
`INT` domain of `requests.weight` and the `doa_matrix.weight_max` ceiling; a value outside
that range is a `ContextHubError('BAD_REQUEST', 'weight out of range')`, never an unhandled
`22003`/500 (CLARIFY A5 — `submitRequest` fails CLEAR). `procedure ∈ {unilateral,collective}`.
**`procedure='collective'` → `ContextHubError('BAD_REQUEST',
'collective steps are Sprint 15.4')`** (D6). **`subject_type ≠ 'artifact'` → `BAD_REQUEST`**
(D7).

Pre-`BEGIN` plain reads (no lock; topics/tasks/artifacts are permanent — but a topic's
*status* is not, so it is checked here too — the 15.2 rev-6 MED-2 pattern):
```
(project_id, topic_status)
   ← SELECT project_id, status FROM topics WHERE topic_id=$1      -- none → NOT_FOUND (404)
                                          -- status='closed' → return {status:'topic_closed'}
     SELECT 1 FROM artifacts WHERE artifact_id=$subject           -- none → NOT_FOUND (404)
submitterLevel ← SELECT level FROM topic_participants WHERE topic_id=$1 AND actor_id=$by
                                                                 -- none → return {status:'not_participant'}
row ← resolveMatrixRow({project_id, topic_id, kind, weight})     -- null → return {status:'no_route'}
route ← deriveRoute(submitterLevel, row.required_level, row.route_shape)   -- ≥1 office
```
The `topic_closed` pre-check is a plain non-locking read — a clean status for the common
already-closed case; `appendEvent`'s close-seal (below) is the authoritative guard for a
topic that closes mid-transaction.
```
BEGIN
  INSERT INTO requests (request_id, topic_id, subject_type, subject_id, kind, weight,
                        procedure, route_shape, status, current_step, submitted_by)
    VALUES ($req, $1, 'artifact', $subject, $kind, $weight, 'unilateral', $shape, 'open', 0, $by)
  for i, office in route:
    INSERT INTO request_steps (request_id, step_index, target_office, doa_snapshot,
                               procedure, deadline, status)
      VALUES ($req, $i, $office, $snapshot, 'unilateral', now() + 60min, 'pending')
  appendEvent(client, { topic_id, actor_id:$by, type:'request.submitted',
                        subject_type:'request', subject_id:$req,
                        payload:{ subject_id, kind, weight, route_shape:$shape, route } })   -- (topics)
COMMIT
return { status:'submitted', request_id:$req, route, current_step:0 }
```
All steps are inserted `pending`; only step 0 is *active* (`requests.current_step=0`). A
non-active step's `deadline` is a placeholder reset to `now()+60min` when it becomes active
(§3.2). Lock order: **`topics` only** (the INSERTs are fresh rows; the reads are unlocked).

### 3.2 `decideStep({ request_id, step_index, actor_id, decision }) → DecideResult` — *the core*

`decision ∈ {endorse, return, reject}`. Lock order **`request → request_step → artifact →
topics`** (§0.2):
```
BEGIN
  SELECT topic_id, status, current_step, subject_id, submitted_by
    FROM requests WHERE request_id=$1 FOR UPDATE                     -- (request)
    none                       → ROLLBACK; { status:'not_found' }
    status ≠ 'open'            → ROLLBACK; { status:'already_resolved' }
    current_step ≠ step_index  → ROLLBACK; { status:'not_current_step' }
  SELECT status FROM topics WHERE topic_id=$topic                    -- plain read, no lock
    status = 'closed'          → ROLLBACK; { status:'topic_closed' } -- 15.2 MED-2 pattern
  SELECT target_office, status FROM request_steps
    WHERE request_id=$1 AND step_index=$2 FOR UPDATE                 -- (request_step)
    status ≠ 'pending'         → ROLLBACK; { status:'conflict' }     -- defensive
  -- authorization (D5) — office = level; officeholder = any participant at that
  -- level EXCEPT the request's own submitter (no self-approval — inv. 8)
  SELECT level FROM topic_participants WHERE topic_id=$topic AND actor_id=$actor
    none                       → ROLLBACK; { status:'not_participant' }
    actor = submitted_by       → ROLLBACK; { status:'self_decision_forbidden' }  -- D5
    level ≠ target_office      → ROLLBACK; { status:'not_authorized' }
  UPDATE request_steps SET status=<decided>, decided_by=$actor, decided_at=now()
    WHERE request_id=$1 AND step_index=$2
  ── decision = endorse ──
    hasNext ← SELECT 1 FROM request_steps WHERE request_id=$1 AND step_index=$2+1
    if hasNext:
      UPDATE request_steps SET deadline = now()+60min               -- activate the next step
        WHERE request_id=$1 AND step_index=$2+1
      UPDATE requests SET current_step=$2+1
      appendEvent(request.step_decided, subject request, payload:{ step_index, decision, decided_by })  -- (topics)
      COMMIT; return { status:'step_endorsed', current_step:$2+1 }
    else:                                                            -- last step → approved
      UPDATE requests SET status='approved'
      resolveArtifact(client, 'approve', subject_id, $actor)         -- (artifact) — §3.3
      appendEvent(request.step_decided, …) ; appendEvent(request.resolved, payload:{ outcome:'approved' })  -- (topics)
      COMMIT; return { status:'approved' }
  ── decision = return ──
    UPDATE requests SET status='returned'
    resolveArtifact(client, 'return', subject_id, $actor)            -- (artifact)
    appendEvent(request.step_decided, …) ; appendEvent(request.resolved, payload:{ outcome:'returned' })
    COMMIT; return { status:'returned' }
  ── decision = reject ──
    UPDATE requests SET status='rejected'
    appendEvent(request.step_decided, …) ; appendEvent(request.resolved, payload:{ outcome:'rejected' })
    COMMIT; return { status:'rejected' }                             -- artifact untouched (D8)
```

### 3.3 internal — `resolveArtifact(client, outcome, artifactId, actorId)`

The cross-primitive artifact advance (D8). A **guarded `UPDATE`** — the `from` state is
pinned by the `WHERE`, so the `artifact.state_changed` payload's `from` is the constant
`'for_review'` (no pre-state read needed — unlike 15.2's writeArtifact). It `RETURNING`s
both `version` **and** `content_ref`, so the history row needs no extra read:
```
outcome='approve':  UPDATE artifacts SET state='final',   version=version+1
                      WHERE artifact_id=$ AND state='for_review'
                      RETURNING version, content_ref
outcome='return':   UPDATE artifacts SET state='working', version=version+1
                      WHERE artifact_id=$ AND state='for_review'
                      RETURNING version, content_ref
```
On **1 row** — append the `artifact_versions` history row with **every column explicit**
(mirroring 15.2's `writeArtifact`/`baselineArtifact`; migration 0054 makes
`version,state,note,created_by` NOT NULL):
```
INSERT INTO artifact_versions
  (artifact_id, version, state, content_ref, fencing_token, note, created_by)
VALUES ($artifactId, <returned version>, <new state>, <returned content_ref>,
        NULL, 'request <outcome>', $actorId)
```
`created_by` is the **deciding actor** (`$actor`, threaded from `decideStep`) — real
provenance, not a system literal. `fencing_token` is `NULL` — a request-driven state
advance is not a fencing write (the 15.2 baseline/revert convention). `content_ref` is the
value `RETURNING`ed from the `UPDATE` (carried forward unchanged — the advance does not edit
content). Then emit `artifact.versioned` (`payload:{ version }`) + `artifact.state_changed`
(`payload:{ from:'for_review', to:<new> }`). On **0 rows** (the artifact was not in
`for_review` — already advanced, or never reached review): the request still resolves; **no
`artifact_versions` row and no artifact event**; the `request.resolved` payload carries
`artifact_advanced:false`. The request outcome is authoritative; the artifact advance is
best-effort and idempotent. `accepted_fencing_token` is untouched (monotonic — 15.2 invariant).

### 3.4 `getRequest({ request_id })` / `listRequests({ topic_id, status? })`

Plain reads — the request row + its `request_steps` ordered by `step_index`; `listRequests`
filters `requests` by `topic_id` (+ optional `status`).

## 4. `src/services/coordinationSweep.ts` (modified) — the stalled-step escalation sweep

A **second swept entity** alongside `sweepAbandonedClaims` (CLARIFY Q9). The scheduler
cycle calls both: `sweepAbandonedClaims()` then `sweepStalledSteps()`.

### 4.1 `sweepStalledSteps({ grace_minutes? }) → { escalated: number, swept_at }`

Scan the *active* stalled steps — the `current_step` join ensures only a request's active
step is eligible:
```
SELECT s.request_id, s.step_index, s.target_office, r.topic_id
  FROM request_steps s
  JOIN requests r ON r.request_id = s.request_id AND r.current_step = s.step_index
 WHERE s.status='pending' AND r.status='open'
   AND s.deadline < now() - make_interval(mins => $grace)
```
Each is processed in its **own §0.1-loop transaction** (catch → log → `continue`).
Lock order **`request → request_step → topics`** (§0.2 — no artifact; escalation never
touches artifacts):
```
BEGIN
  SELECT status, current_step FROM requests WHERE request_id=$ FOR UPDATE      -- (request)
    status≠'open' OR current_step≠step_index → ROLLBACK; continue              -- decided since the scan
  SELECT target_office, status, deadline FROM request_steps
    WHERE request_id=$ AND step_index=$ FOR UPDATE                             -- (request_step)
    status≠'pending' OR deadline ≥ now() → ROLLBACK; continue
  SELECT status FROM topics WHERE topic_id=$ FOR UPDATE                        -- (topics) — closeTopic serialized
    status='closed' → ROLLBACK; continue                                      -- closed topic: skip (frozen; no events)
  if target_office < authority:                                               -- climb one level in place (D9)
    UPDATE request_steps SET target_office = up(target_office), deadline = now()+60min
      WHERE request_id=$ AND step_index=$                                      -- status stays 'pending'
    appendEvent(request.step_escalated, subject request,
                payload:{ step_index, from_office, to_office:up(target_office) })   -- (topics)
  else:                                                                       -- already authority → terminal
    UPDATE request_steps SET status='escalated', decided_by='system:sweep', decided_at=now()
      WHERE request_id=$ AND step_index=$
    UPDATE requests SET status='escalation_exhausted'
    appendEvent(request.step_escalated, payload:{ step_index, exhausted:true })
    appendEvent(request.resolved,      payload:{ outcome:'escalation_exhausted' })
  COMMIT; escalated++
```
**Closed-topic branch** — skip (no mutation, no events): a closed topic's requests are
frozen, and the sealed log faithfully shows the request mid-flight. (Unlike a dangling
*claim*, a stalled *step* is not a lock — nothing needs cleanup.)
**Convergence** — a step climbs at most `execution → coordination → authority` (≤ 2 ticks),
then terminates. It cannot loop.

### 4.2 Scheduler

`startClaimsSweepScheduler` is generalized — its `setTimeout`-chained cycle (advisory-locked,
`SWEEP_INTERVAL_MS = 5 min`) calls **both** sweeps in sequence inside the one advisory-lock
hold. No new scheduler, no new advisory key.

## 9. Key invariants & contracts

1. **Route frozen at submission** — `request_steps` rows + `doa_snapshot` are materialized at
   submit; no later `doa_matrix` edit re-targets an in-flight request (D4).
2. **Only the active step is decidable** — `decideStep` rejects any `step_index ≠
   requests.current_step`; the escalation sweep only ever touches the `current_step` (the
   `current_step` join).
3. **One officeholder = one level** — a step is decided only by a participant whose level
   equals `target_office` and who is not the request's submitter (D5, inv. 8). 15.3 does
   **not** enforce *distinct* endorsers across a request's steps — after an escalation a
   single officeholder may satisfy two same-level steps (§11.2, DEFERRED-013).
4. **Escalation converges** — `target_office` climbs monotonically, ≤ 2 ticks, terminal at
   `authority`; it cannot loop or wedge.
5. **The artifact advance is idempotent + best-effort** — guarded by `state='for_review'`; a
   0-row advance does not fail the request (§3.3).
6. **Lock order** — every transaction is a prefix-consistent subsequence of `task → claim →
   request → request_step → artifact → topics` (§10); no ABBA, in-primitive or cross-primitive.
7. **Sealed-topic safety** — `submitRequest`/`decideStep` plain-read `topics.status` and
   return a clean `topic_closed` for the common already-closed case (15.2 rev-6 MED-2
   pattern); `appendEvent`'s close-seal is the authoritative guard for a topic that closes
   mid-transaction (a rare race → a thrown `BAD_REQUEST`, rolled back, no corruption). The
   escalation sweep skips closed topics.
8. **No self-approval** — `decideStep` rejects `actor_id == requests.submitted_by`
   (`self_decision_forbidden`); a request's submitter is never an officeholder for it (D5).
   A step whose `target_office` level has the submitter as its only participant is
   undecidable until another such participant joins or the escalation sweep carries it
   forward — safety by this rule, liveness by §4 (see §11.7).

## 10. Lock-order derivation — *front-loaded (calibration note)*

Global canonical order: **`task → claim → request → request_step → artifact → topics`**.
Each row-lock-acquiring transaction, in acquisition order:

| Transaction | Lock acquisition sequence | Prefix-consistent? |
|---|---|---|
| `submitRequest` | `topics` (appendEvent) | ✓ — single lock |
| `decideStep` — endorse, non-final | `request` → `request_step` → `topics` | ✓ |
| `decideStep` — endorse-final / return | `request` → `request_step` → `artifact` → `topics` | ✓ |
| `decideStep` — reject | `request` → `request_step` → `topics` | ✓ |
| `sweepStalledSteps` (per step) | `request` → `request_step` → `topics` | ✓ |

15.2's transactions (`postTask`, `claimTask`, `releaseTask`, `completeTask`, `writeArtifact`,
`baselineArtifact`, `sweepAbandonedClaims`) lock only within `{task, claim, artifact, topics}`
— each a subsequence of the global order, **disjoint from `{request, request_step}`**.
**Cross-primitive check:** a Board transaction and a Request transaction share at most
`artifact` and `topics`; both acquire `artifact` before `topics`. No two transactions acquire
a shared lock pair in opposed order ⇒ **no ABBA cycle**, in-primitive or cross-primitive.
`appendEvent`'s `topics` lock is the final lock in every transaction. The closed-topic
pre-checks added in rev 2 (§3.1, §3.2) are **plain non-locking `SELECT`s** — they take no
row lock, so they neither appear in nor perturb this table.

## 11. Pre-emptive self-review — anticipated Adversary challenges

1. **`escalate_to_authority` is a 1-step route** (`[required_level]`); its only escalation is
   the sweep climbing that step to `authority` then terminating. Confirmed convergent.
2. **Counter-sign collapse under escalation** — when an escalated step climbs to a level a
   *later* step also targets (e.g. `[coordination, authority]` where the coordination step
   times out and climbs to `authority`), the route then has two steps at `authority`.
   `decideStep` authorizes by `level == target_office` (+ `actor ≠ submitted_by`) and does
   **not** track which actors decided earlier steps — so a *single* authority officeholder
   can endorse both steps. This loses not just the step count but the **distinct-endorser**
   property that is the whole point of a `counter_sign` route. **Accepted for 15.3** — it
   arises only on the post-deadline escalation path (already abnormal), it is fully recorded
   in the event log, and the request still terminates correctly. Restoring the distinct-endorser
   guarantee (`decideStep` rejecting an actor who decided an earlier step of the same request)
   and/or same-level step-collapse is **DEFERRED-013** (trigger: Sprint 15.4/15.5).
3. **`doa_snapshot` freeze** — the route is materialized into `request_steps` at submission;
   `decideStep`/`sweepStalledSteps` never re-read `doa_matrix`. A matrix edit cannot affect a
   live request. Tested (T13).
4. **The artifact advance crosses into 15.2's `artifacts`** — it is a guarded `UPDATE` pinned
   by `state='for_review'`, at the canonical `artifact` lock position (after `request_step`,
   before `topics`); a 0-row advance is non-fatal (§3.3). The `from` state is constant — no
   pre-state read, so the 15.2.1 code-r1 CTE pitfall does not arise here.
5. **`current_step` integrity** — `decideStep` advances it under the `request` `FOR UPDATE`
   lock; the sweep re-checks it under the same lock. No two actors can both decide.
6. **Closed-topic escalation** — skipped (not dropped): a stalled step is not a lease, so
   nothing leaks; the sealed log shows the request frozen mid-flight (consistent with the
   15.2 closed-topic posture, and with DEFERRED-012 — the `closing` drain — still open).
7. **Self-decision & the lone officeholder** — `decideStep` forbids the submitter deciding
   their own request (inv. 8). Consequence: if a step's `target_office` level has *no
   participant other than the submitter*, no one can endorse that step. This is intentional
   and safe — the escalation sweep (§4) climbs the stalled step one level per tick; at
   `authority` it terminates the request `escalation_exhausted`. The request never wedges
   (liveness) and is never self-approved (safety). The `escalate_to_authority` self-submit
   path — an `authority` participant submitting a `weight≥50` `artifact_review` (the seeded
   default) — resolves exactly this way: the lone `authority` step escalates and exhausts
   rather than self-approving. Sprint 15.5 converts `escalation_exhausted` into a dispute.
