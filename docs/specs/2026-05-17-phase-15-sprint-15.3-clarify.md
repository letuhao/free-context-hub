## Intent

Build **the Request-Approval primitive** — primitive A of the Phase 15 design — evolving
Phase 13's F2 `review_requests` into **multi-level approval routing**. A *request* targets a
subject (an artifact), carries a **route** of ordered *steps* derived from a
**Delegation-of-Authority (DoA) matrix**, and climbs that route; each step is decided —
**endorse-and-forward / return / reject** — by an officeholder at a target level; a step has
a **deadline**, and a stalled step is **escalated** by a sweep. An approved request advances
its subject artifact `for_review → final`. It is the mirror of 15.2's abandoned-claim sweep —
no multi-level request ever wedges on a vanished officeholder.

## Why now

15.1 (substrate) and 15.2 (the Board) are shipped. 15.3 is the **Request-Approval** half of
the two-primitive protocol — the sign-off / escalation mechanism — and the natural next
sprint in the design's sprint sketch. It also gives the Board's `for_review` artifacts
(15.2's `completeTask` parks them there) their next transition: approval → `final`.

## Out of scope (deferred)

- **Collective-procedure steps** — a step whose `procedure='collective'` is decided by a
  decision body (B.6), which needs `decision_bodies`/`motions`/`votes` → **Sprint 15.4**.
  15.3 is **unilateral-only**; a `collective` procedure is rejected at submission with a
  clear "not until 15.4" error.
- **Dispute conversion** — the design (C.4) converts an escalation-exhausted step to a
  *dispute*; disputes are **Sprint 15.5**. In 15.3, an `authority`-level step that times out
  goes to a **terminal status** recorded in the event log; the dispute wiring lands in 15.5.
- **The generic "resolved request posts a new board task" chaining** (C.4) — underspecified
  ("what task?") and it interlocks with the topic `closing`-drain, which is **DEFERRED-012**.
  15.3's request outcome = the artifact advance + the `request.resolved` event.
- **DoA-matrix management API / GUI** — 15.3 *seeds* a default matrix and *consumes* it;
  matrix CRUD is deferred to the 15.6 GUI (or a later sprint).
- **Phase 13 `review_requests` row migration** — master-design Open Q4; 0056 is additive,
  `review_requests` is untouched and continues to serve the Phase 13 API.

## Open questions — decided at CLARIFY (human-in-loop, 2026-05-17)

| # | Question | Decision |
|---|---|---|
| Q1 | Routing — explicit route vs the DoA matrix? | **Build the DoA matrix** — a `doa_matrix` table + route *derivation*, supporting both route shapes. |
| Q2 | Where does the matrix live? (master Open Q3) | **Per-project DB table**, keyed by `project_id`. Seeded with a default at migration time. |
| Q3 | How is the matrix indexed? | **Both `kind` + `weight`** — a request carries a decision `kind` (a string) and a numeric `weight`; matrix rows are keyed by `(project_id, kind, weight_min, weight_max)` → `required_level` + `route_shape`. |
| Q4 | May a topic override the project matrix? (master Open Q3) | **Yes** — a topic may store its own matrix rows; resolution = **topic rows take precedence over project rows**. The route-rules snapshot (`doa_snapshot`) isolates any in-flight request from a later matrix change. |
| Q5 | What does an approved request do? | An approved request **advances its subject `for_review` artifact → `final`**; a `returned` request sends it back to `working`; `request.resolved` is emitted either way. |
| Q6 | Step decision procedure | **Unilateral only** in 15.3; `collective` rejected at submission (→ 15.4). |
| Q7 | Who decides a step? | A step targets a **level** (`target_office` ∈ `authority`/`coordination`/`execution`); it is decidable by **any topic participant whose level matches** — self-select, no assignment (office ≠ officeholder; resolved at decision time). |
| Q8 | Escalation terminal | An `authority`-level step timed out with nowhere to climb → a **terminal status** (recorded); dispute conversion is wired in 15.5. |
| Q9 | The escalation sweep | A **second swept entity inside `coordinationSweep.ts`** — one scheduler cycle sweeps both abandoned claims and stalled request-steps. |
| Q10 | Timeouts (master Open Q7) | Per-step `deadline`, default **60 minutes** from when the step becomes active; an escalated step receives a **fresh 60 min**; sweep interval **5 min** (the 15.2 cadence). |
| Q11 | Migration | **Additive** — `requests` / `request_steps` / `doa_matrix` are new tables; `review_requests` is untouched. |

## Approaches considered

### Approach A — three concerns: `doaMatrix.ts` + `requests.ts` + sweep extension (RECOMMENDED)

`doaMatrix.ts` (matrix resolution + route derivation — the novel policy logic), `requests.ts`
(the request lifecycle: submit/decide + the artifact-advance), and the stalled-step sweep
**added to the existing `coordinationSweep.ts`** (one scheduler, two swept entities, per Q9).
All write ops emit events via 15.1's `appendEvent` within their transaction.
- **Pros:** the matrix/derivation logic is isolated and unit-testable on its own; the sweep
  reuses the proven 15.2 scheduler; concern-split scales.
- **This is the recommendation.**

### Approach B — one `requests.ts` covering matrix + lifecycle + sweep — rejected (one oversized module; the sweep wants the shared scheduler).

## Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Route-derivation correctness — matrix lookup over `(kind, weight)` ranges + topic-override precedence + the two route shapes. | HIGH | An explicit derivation algorithm in DESIGN; a derivation unit-test matrix (per kind, weight boundary, shape, override-vs-not). |

| R3 | A topic-override or project-matrix change races an in-flight request. | MED | The route is *frozen* at submission (`doa_snapshot` + the materialized `request_steps`); a later matrix change never re-targets an in-flight request. Test it. |
| R4 | The approved-request → `artifact for_review→final` advance crosses into 15.2's `artifacts` surface. | MED | A guarded `UPDATE` (state IN ('for_review') → 'final'); respects the artifact state machine + the canonical lock order; conflict if the artifact is not `for_review`. |
| R5 | `weight`-range rows overlap or leave a gap → an ambiguous or absent route. | MED | Seed a total, non-overlapping default matrix; `submitRequest` resolves deterministically (most-specific row; documented tie-break) and returns `NO_ROUTE` cleanly if unmatched. |
| R6 | MCP `outputSchema` `z.discriminatedUnion` crash. | LOW | Flat `z.object` only (DEFERRED-007) — carried from 15.1/15.2. |

## Acceptance criteria (Sprint 15.3)

- [ ] **AC1** — Migration 0056 applies cleanly + idempotently; `requests`, `request_steps`,
  `doa_matrix` tables + indexes; a default per-project matrix is seeded.
- [ ] **AC2** — `doa_matrix` is per-project; a topic may store override rows; resolution
  returns the topic rows when present, else the project rows.
- [ ] **AC3** — `submit_request` / `POST /requests` carries a `(subject, kind, weight)`;
  consults the resolved matrix; derives the ordered `request_steps` (submitter level →
  `required_level`); **freezes** each step's `target_office` + `doa_snapshot`; emits
  `request.submitted`.
- [ ] **AC4** — Both route shapes work — `counter_sign` (every step must endorse) and
  `escalate_to_authority` (climbs until a covering level endorses).
- [ ] **AC5** — `decide_request_step` / `POST /requests/:id/steps/:n/decide`: **endorse** →
  the next step activates, or the request → `approved` if it was the last; **return** →
  request → `returned`; **reject** → request → `rejected`. Emits `request.step_decided`.
- [ ] **AC6** — A step is decidable only by a topic participant whose level equals the step's
  `target_office`; a non-matching actor is rejected (`not_authorized`).
- [ ] **AC7** — A `collective` procedure is rejected at submission with a clear status (15.4).
- [ ] **AC8** — An `approved` request advances its `for_review` subject artifact → `final`;
  a `returned` request sends it → `working`; `request.resolved` is emitted. The artifact
  advance is a guarded `UPDATE` (conflict if the artifact is not in the expected state).
- [ ] **AC9** — The escalation sweep detects a `request_steps` row `pending` past `deadline`,
  sets it `escalated`, re-targets one level up with a **fresh** deadline, and emits
  `request.step_escalated`.
- [ ] **AC10** — A step already at `authority` that times out goes to a **terminal status**
  (the request resolves terminally); no dispute is created in 15.3 (→ 15.5).
- [ ] **AC11** — Every state change emits a `coordination_events` row; nothing is destructively
  deleted; the request lifecycle is fully reconstructable from the log.
- [ ] **AC12** — REST mirrors the MCP tools 1:1; one envelope `{status, data?, error?}`.
- [ ] **AC13** — `tsc --noEmit` clean; new unit tests pass; the existing suite stays green.
- [ ] **AC14** — Live smoke on the deployed stack: seed/charter → submit a ≥2-level request →
  decide step 1 (endorse) → decide step 2 (endorse) → request `approved` → subject artifact
  `final`; then a second scenario where a step is left to expire and the sweep escalates it.

## Pre-emptive self-review (for the cold-start Adversary)

1. **Route derivation is the novel core.** DESIGN must give the explicit algorithm: given
   `(project, topic, kind, weight)`, resolve the matrix (topic > project), select the row,
   produce the ordered `request_steps` + the shape — and define the tie-break / `NO_ROUTE`
   behaviour. The `weight`-range rows must be total and non-overlapping in the seed.
2. **The escalation sweep is a second concurrent sweep.** Per the calibration note, the
   §10 per-transaction lock-order table must be derived **in DESIGN** — covering the new
   stalled-step transaction alongside the existing claims sweep. No ABBA.
3. **"Snapshot the rules, not the people."** The `doa_snapshot` + the materialized
   `request_steps` must genuinely freeze the route at submission; a later matrix/override
   change must not re-target an in-flight request. Needs an explicit test.
4. **The artifact advance crosses primitives.** An approved request doing
   `artifact for_review → final` touches 15.2's `artifacts` table — DESIGN must show the
   guarded `UPDATE`, its lock position in the canonical order, and the conflict path when
   the artifact is not `for_review`.
5. **`escalate_to_authority` vs `counter_sign` terminal conditions** must be precisely
   defined — when is the request `approved`? (counter_sign: after the last step; escalate:
   when a covering level endorses). Ambiguity here is a likely Adversary finding.
6. **Escalation must converge.** A stalled chain escalates one level per sweep tick with a
   fresh deadline; at `authority` it terminates. DESIGN must show it cannot loop forever.
