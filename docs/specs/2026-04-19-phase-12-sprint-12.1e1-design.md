# Design — Phase 12 Sprint 12.1e1

Translates the spec into concrete data/procedure decisions. No code change; this doc is the authoritative reference for BUILD.

## 1. Query JSON schema (additions)

Existing fields (unchanged): `id`, `group`, `query`, `target_lesson_ids`, `must_keywords`.

**New optional field (JSON-only, NOT added to `GoldenQuery` TS type):**

```json
{
  "reasoning": "string — 1-3 sentences explaining why this query is in the set and why these specific target IDs were chosen."
}
```

**Rationale:** `runBaseline` consumes queries via `JSON.parse` + structural cast. Unrecognized fields are silently ignored. Keeping this field JSON-only:
- Preserves the zero-code-change constraint.
- Leaves a traceable audit of target-selection reasoning next to each query.
- Future sprints can promote it to a real type field if tooling ever consumes it.

**Field ordering convention (stylistic, for the 20 new queries):**
```json
{
  "id": "...",
  "group": "...",
  "query": "...",
  "target_lesson_ids": [...],
  "must_keywords": [...],
  "reasoning": "..."
}
```

## 2. New groups

Three new group values in the `group` field:

| Group | Count | Target cardinality | Must-keywords | Reasoning required |
|---|---:|---|---|---|
| `ambiguous-multi-target` | 12 | 2-4 plausible IDs | Usually `[]` (keyword hits would over-constrain) | Yes |
| `semantic-paraphrase` | 5 | Exactly 1 ID | Usually 1-2 (strong keyword for the target) | Yes |
| `real-dogfood` | 3 | 1-2 IDs (depends on what the real query hit) | Inherit from query text (the tokens that would have mattered) | Yes, plus source provenance ("mined from access log on 2026-04-19, DISTINCT query text after filter X") |

**Group-name `id` prefix convention** (mirrors existing `lesson-cross-*`, `lesson-miss-*`):
- `ambiguous-multi-target` → `lesson-ambig-<slug>`
- `semantic-paraphrase` → `lesson-para-<slug>`
- `real-dogfood` → `lesson-real-<slug>`

## 3. Target-ID selection procedure

For each synthesized query (`ambiguous-multi-target` + `semantic-paraphrase`):

1. **Candidate gather:** issue `curl 'http://localhost:3001/api/lessons?q=<query>&project_id=free-context-hub&limit=10'` and collect the top 10 hits. This uses the current (salience-ON, α=0.10, half-life=7d) ranking — i.e., production today.
2. **For `ambiguous-multi-target`:** skim titles + content. Mark as target each lesson a reasonable reader would consider genuinely on-topic (2-4 IDs). If only 1 is genuinely on-topic, **reject the query** and rewrite until 2+ plausible hits exist, OR reclassify to `semantic-paraphrase`.
3. **For `semantic-paraphrase`:** pick the ONE intended target. Verify it is the clearly-correct hit (re-reading the content makes it obvious). If >1 lesson is equally plausible, rewrite the query sharper until only one remains.
4. **Record reasoning.** Include: (a) why this topic matters for measurement, (b) why these specific IDs, (c) for ambiguous: why they are alternatives rather than one being obviously right.

**Self-check before committing a query:** does the rank-1 result for this query in current production look obviously-correct, or does it feel like a coin flip? For ambiguous queries we WANT the coin-flip feel.

## 4. Mining SQL (from `lesson_access_log`)

```sql
SELECT
  DISTINCT metadata->>'query' AS query_text,
  COUNT(*) AS hit_count,
  MIN(accessed_at) AS first_seen,
  MAX(accessed_at) AS last_seen
FROM lesson_access_log
WHERE context = 'consideration-search'
  AND project_id = 'free-context-hub'
  AND metadata->>'query' IS NOT NULL
  AND accessed_at > NOW() - INTERVAL '180 days'
GROUP BY metadata->>'query'
ORDER BY hit_count DESC, last_seen DESC
LIMIT 100;
```

**Issued via:** `docker compose exec db psql -U contexthub -d contexthub -c "..."`.

## 5. Goldenset-adjacent filter (for mined queries)

A mined query text `T` is "goldenset-adjacent" and therefore REJECTED if any of:

- **Substring test:** `T.toLowerCase()` is a substring of any existing goldenset `query` (or vice versa), after normalizing whitespace.
- **Token-overlap test:** tokenize (split on whitespace + lowercase), strip stopwords `['the','a','an','is','to','of','in','on','for','with','how','what','why']`. If `|T_tokens ∩ G_tokens| / |T_tokens| >= 0.7` for any existing goldenset query `G`, reject.
- **Manual review:** I eyeball the final candidate list for obvious rephrasings of existing queries that the heuristics missed.

**Decision rule:** apply both heuristics + manual review. If fewer than 3 survive, fall back to synthesizing the shortfall (noted in the spec's D4).

## 6. Baseline run command + file layout

Use `--control` mode for free noise-floor data:

```bash
npm run qc:baseline -- \
  --tag sprint-12.1e1 \
  --samples 3 \
  --control \
  --surfaces lessons
```

**Why `--control` even though this isn't A/B:** the mode runs the goldenset back-to-back twice and computes `|run2 − run1|` per metric as the noise floor. Both runs are the SAME system state (we change nothing between them), so the delta is pure measurement jitter. Reusing this noise floor in 12.1e2 avoids re-establishing it there.

**Why `--surfaces lessons`:** only the lessons goldenset changed. Running chunks/global/code is wasted effort.

**Output files (per runBaseline convention):**
- `docs/qc/baselines/2026-04-19-sprint-12.1e1-control.{json,md}` — first run, labeled "control" by runBaseline
- `docs/qc/baselines/2026-04-19-sprint-12.1e1-new.{json,md}` — second run, canonical reference baseline
- `docs/qc/baselines/2026-04-19-sprint-12.1e1.diff.md` — noise-floor-aware diff (auto-generated)

**The `-new` file is the reference baseline for 12.1e2.** Its metric values are "current production on the broader goldenset".

**Update to spec A6:** revise to `2026-04-19-sprint-12.1e1-new.{json,md}` as the canonical archive, with `-control` + `.diff.md` as partners. (Spec said `-baseline` — that name isn't used elsewhere. Matching existing pattern.)

## 10. Risks introduced at design time

**R-D1 — Per-group metric reporting requires reading JSON by hand.** runBaseline doesn't break out by group today. My manual breakdown in the `.diff.md` is subject to human error (arithmetic mistakes). Mitigation: show the raw per-query foundRanks + groups in an appendix so the breakdown is reconstructible.

**R-D2 — Target-ID selection self-confirmation bias.** I use the current hybrid search to gather candidates; if the search is weak in some area, my "ambiguous" queries may inherit that weakness (e.g., both my "alternative" targets are in fact both-wrong, and the true target is at rank 15). Mitigation: for each query, I also check lesson titles via `/api/lessons?project_id=...&limit=100` and scan for obvious missed candidates.

**R-D3 — Mining returns stale / noisy queries.** Access log may contain internal debug queries (e.g., from smoke tests). Mitigation: the `hit_count` column in the SQL ranks by frequency; top-100 by count will surface the meaningful queries. Manual review (step 3 of §5) catches smoke-test residue.
