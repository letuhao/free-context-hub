## Intent

Build **the Board** — primitive B of the Phase 15 design (post / claim / produce). On top of
the Sprint 15.1 substrate, this sprint adds: **tasks** posted to a topic's board, **artifacts**
with derived identity + versioned states, **claims** (the evolution of Phase 13 artifact
leasing — now with a fencing token and a claim-liveness check), and the **abandoned-claim
sweep** that recovers work when an ephemeral actor disbands mid-task.

After this sprint a coordinator can post a task to a topic; an execution actor can claim it
(getting an exclusive, fenced working lease on its output artifact), write versioned content,
baseline it, and complete the task — and if the actor vanishes, the sweep returns the task to
the board and reverts the artifact to its last safe version.

## Why now

Sprint 15.1 shipped the substrate (the event log + Topic/Actor model). The design's sprint
sequence puts the Board next (15.2). The Board is what makes a topic *do work*: 15.1 lets
actors join a topic and replay its events; 15.2 lets them claim and produce. It also closes
**Run 1's #1 gap** — free-text artifact identity — structurally: an artifact's id is
*derived* from `(topic, task, slot)`, so two actors cannot diverge on the name.

## Out of scope (deferred)

- **Request-Approval / voting / intake / dispute** → Sprints 15.3–15.5.
- **GUI** (Board view, live stream) → Sprint 15.6.
- **The `closing` drain-state machinery** — `closeTopic` stays **atomic** (`chartered|active →
  closed`, as Sprint 15.1 shipped). A closed topic's active claims are cleaned up by the
  abandoned-claim sweep (it expires claims past `expires_at`; a closed topic's tasks simply do
  not progress — `appendEvent` is sealed). The explicit `closing` drain-state + force-lapse
  belongs with the *full* in-flight item set (claims + requests + motions + disputes) and is
  built no earlier than 15.5. *(This re-defers Sprint 15.1 design decision D4's "15.2 adds the
  drain"; rationale: the sweep already recovers closed-topic claims, and a partial drain now
  would be rebuilt when requests/motions/disputes exist. Flagged for the Adversary.)*
- **Active topology-ordering enforcement** — the `topology` (`parallel|sequential|rolling`) and
  `depends_on` columns ship, and `baselineArtifact` ships (the rolling-handoff primitive). But
  *enforcing* the wiring (blocking a `sequential` task's claim until `depends_on` tasks are
  `completed`; gating a rolling consumer on a `baselined` upstream) is **not** built in 15.2 —
  the columns are recorded; enforcement is a follow-up. *(Flagged — DEFERRED candidate.)*
- **Multi-output-artifact tasks** — a task has **one** output artifact in 15.2 (one declared
  `slot`). The `slot` column + the derived-id scheme already support multiple; multi-output is
  a noted future extension.
- **Phase 13 `artifact_leases` row migration** — design Open Q4; 0054 is purely additive,
  `artifact_leases` is untouched and continues to serve the Phase 13 leasing API.

## Open questions — decided inline at CLARIFY

| # | Question | Decision |
|---|---|---|
| Q1 | Derived `artifact_id` scheme (design Open Q1) | **`<topic_id>:<task_id>:<slot>`** — colon-separated. `topic_id`/`task_id` are UUIDs (URL-safe); `slot` is constrained to a lowercase-kebab slug (`^[a-z0-9][a-z0-9-]*$`). Colon is RFC-3986-legal inside a path segment, so `PUT /artifacts/<id>` routes cleanly (no slashes in the id). Max length ≈ 36+36+2+slot ≈ 110 chars — fine for a TEXT PK and a URL. Derived by the service from `(topic, task, slot)` — actors never supply it (closes Run 1's #1 gap). |
| Q2 | When are `artifacts` rows created? | At **task-post time**. `POST /topics/:id/tasks` declares the output slot (`slot` + `kind`); `postTask` inserts the `tasks` row **and** the one `artifacts` row (`state='draft'`, `version=1`, `accepted_fencing_token=0`) in one transaction. The derived `artifact_id` is therefore known and claimable immediately. |
| Q3 | What does a task claim cover? | A task has **exactly one output artifact** in 15.2. `POST /tasks/:id/claim` creates **one** `claims` row on that artifact and returns `{claim_id, fencing_token, expires_at}`. A task's *input* artifacts are the *output* artifacts of its `depends_on` predecessors (read, not claimed). Multi-output tasks are deferred. |
| Q4 | `completeTask` — what state does the artifact reach? | `POST /tasks/:id/complete` transitions the task `→ completed` and the output artifact `→ for_review`, and **releases the claim** (per design B.5 — a claim covers only the writing states `draft→working→baselined`; it is released at `for_review`). In 15.2 a `for_review` artifact simply rests (the review primitive is 15.3). Emits `task.completed` + `artifact.state_changed`. |
| Q5 | `fencing_token` allocation | A Postgres **`SEQUENCE`** — `coordination_fencing_seq`. `claimTask` allocates `nextval('coordination_fencing_seq')`; sequences are non-transactional and strictly monotonic, so any later claim's token is strictly greater (design C.2). |
| Q6 | `claims` uniqueness | The **shipped Phase 13 pattern** — a plain `UNIQUE INDEX claims_active_uniq ON claims(artifact_id)` + the service-layer atomic claim transaction (step 1 DELETE expired claims for this artifact; step 2 INSERT; step 3 on `23505` re-SELECT to classify still-active-conflict vs expired-race). **Not** a `WHERE now()` partial index (invalid — migration 0048 documents this). Verbatim the `artifactLeases.ts` claim algorithm. |
| Q7 | The sweep — job or in-process scheduler? | **In-process scheduler**, reusing the Phase 13 `sweepScheduler.ts` pattern (a `setTimeout`/interval loop guarded by a Postgres advisory lock so only one replica sweeps). 15.2 may add a sibling scheduler entry or extend the existing one; **no new `async_jobs` job type** unless DESIGN finds it necessary — avoids an `async_jobs` CHECK migration. |

## Approaches considered

### Approach A — Three service modules + the shipped Phase-13 claim pattern (RECOMMENDED)

`board.ts` (tasks + the claim/release/complete lifecycle), `artifacts.ts` (the write/version
+ baseline + the derived-id + revert helpers), `coordinationSweep.ts` (the abandoned-claim
sweep). Claims live inside `board.ts` (claim is a task operation). The claim transaction is
**verbatim the `artifactLeases.ts` `_claimArtifactOnce` algorithm** (DELETE-expired → INSERT
→ 23505 re-SELECT), plus the fencing-token `nextval`. All write ops emit events via the 15.1
`appendEvent` within their transaction. Thin MCP + REST handlers.

- **Pros:** the claim path is a proven, audited Phase 13 algorithm; concern-split scales (15.3+
  add more services); event/state atomicity is structural (15.1's `appendEvent` contract).
- **Cons:** `board.ts` is the largest module (tasks + claims + the lifecycle).
- **This is the recommendation.**

### Approach B — One `board.ts` covering tasks + artifacts + claims + sweep

- **Cons:** one ~600-line module; the sweep is a background concern that wants its own file
  (testability, the scheduler import). **Rejected.**

### Approach C — Reuse `artifactLeases.ts` directly for claims (alias the table)

Point the Board's claims at the existing `artifact_leases` table.
- **Cons:** the design (Part D) is explicit — `claims` is a *new* table that *evolves* leasing
  (+ derived identity, + `fencing_token`, + `task_id`, + events); `artifact_leases` stays for
  the Phase 13 API. Conflating them couples two live surfaces. **Rejected** — additive migration.

## Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Concurrent `claimTask` on one artifact → both INSERT. | HIGH | `claims_active_uniq` + the 23505 re-SELECT (the proven Phase 13 path); concurrent-claim unit test. |
| R2 | A stale claim-holder's `PUT /artifacts/:id` corrupts a newer holder's content. | HIGH | The artifact write checks **both**, atomically: (a) `claim_id` is a *live* claim (`now() < expires_at`); (b) presented `fencing_token ≥ artifacts.accepted_fencing_token`. Design C.2; the live-claim check closes the slow-claimant-after-sweep window. |
| R3 | The sweep destroys a `baselined` artifact a rolling consumer already pulled. | HIGH | The sweep **never un-baselines** — it reverts to the *last `baselined` version* (or `draft` if never baselined); `accepted_fencing_token` is left unchanged (monotonic). Revert **appends** an `artifact_versions` row — never in-place. Design C.4. |
| R4 | `PUT /artifacts/:id` — the derived id contains colons; route matching / id parsing. | MED | Colon is path-segment-legal; `:id` matches one segment. The service re-derives `(topic, task, slot)` by splitting on `:` and validates against the `artifacts` row. |
| R5 | MCP tool output crashes the client via the `z.discriminatedUnion` SDK bug. | MED | All MCP `outputSchema` are flat `z.object` (DEFERRED-007) — carried from 15.1. |
| R6 | Cross-project topic/task scope not enforced. | LOW | Inherited DEFERRED-009 (out of 15.2 scope; same posture as 15.1). |
| R7 | `closeTopic` left atomic — a closed topic with live claims. | LOW | The abandoned-claim sweep expires them; the topic being `closed` blocks further progress via the seal. Documented out-of-scope (the drain). |

## Acceptance criteria (Sprint 15.2)

- [ ] **AC1** — Migration 0054 applies cleanly + idempotently; `tasks`, `artifacts`,
  `artifact_versions`, `claims` tables + `coordination_fencing_seq` + `claims_active_uniq`.
- [ ] **AC2** — `post_task` / `POST /topics/:id/tasks` inserts a `tasks` row (`status='posted'`)
  **and** its one `artifacts` row (`state='draft'`, derived id), emits `task.posted` +
  `artifact.created`, in one transaction.
- [ ] **AC3** — `list_board` / `GET /topics/:id/board` lists `posted` (claimable) tasks.
- [ ] **AC4** — `claim_task` / `POST /tasks/:id/claim` → `{claim_id, fencing_token, expires_at}`;
  task `→ claimed`; emits `claim.granted` + `task.claimed`. `fencing_token` strictly increases.
- [ ] **AC5** — Concurrent `claim_task` on one task: exactly one `claimed`, the rest `conflict`
  (with the incumbent), no 500 — covered by a `Promise.all` unit test.
- [ ] **AC6** — `write_artifact` / `PUT /artifacts/:id` versions the artifact (`version++`,
  appends an `artifact_versions` row, `state draft→working`) **only** when the presented
  `claim_id` is a live claim **and** `fencing_token ≥ accepted_fencing_token`; otherwise
  `conflict`. On success `accepted_fencing_token` is set to the presented token.
- [ ] **AC7** — A `PUT` with a *fencing token below* `accepted_fencing_token`, or with an
  *expired* `claim_id`, is rejected `conflict` (the stale-holder cases — R2).
- [ ] **AC8** — `baseline_artifact` / `POST /artifacts/:id/baseline` moves a `working`/`draft`
  artifact `→ baselined`; emits `artifact.state_changed`.
- [ ] **AC9** — `complete_task` / `POST /tasks/:id/complete`: task `→ completed`, artifact
  `→ for_review`, the claim is **released**; emits `task.completed` + `artifact.state_changed`.
- [ ] **AC10** — `release_task` / `POST /tasks/:id/release` by the claim holder: claim removed,
  task `→ posted`; emits `task.released`.
- [ ] **AC11** — The abandoned-claim sweep detects a `claims` row past `expires_at` and
  recovers it. *[r2-fix — scoped open-vs-closed]* **On an open topic:** emits `claim.expired`,
  returns the task `→ posted` + emits `task.released`, and reverts the output artifact to its
  **last `baselined` version** (or `draft` if never baselined) by **appending** an
  `artifact_versions` row — never un-baselining, never touching `accepted_fencing_token`.
  **On a `closed` topic:** the sweep drops only the dangling `claims` row — **no revert, no
  task-status change, no events**. Rationale: the recovery's purpose — a re-claimable task
  and a clean artifact for the *next* claimant — is moot on a closed topic (its tasks cannot
  be re-claimed; `claimTask`'s `appendEvent` is rejected by the seal), and a closed topic is
  a sealed, self-consistent record — mutating its artifact would desync it from the sealed
  event log. (A later sprint's `closing`-drain force-lapses claims *during* `closing`, before
  the log seals, so they never reach this path.)
- [ ] **AC12** — Artifact identity is **derived** `<topic>:<task>:<slot>`; an actor cannot
  supply or diverge on it.
- [ ] **AC13** — REST mirrors the 7 MCP tools 1:1; one envelope `{status, data?, error?}`.
- [ ] **AC14** — `tsc --noEmit` clean; new unit tests pass; the existing suite stays green.
- [ ] **AC15** — Live smoke on the deployed stack: charter+join a topic → post a task → claim
  → write → baseline → complete; then a second scenario where a claim is left to expire and the
  sweep returns the task to the board.

## Pre-emptive self-review (for the cold-start Adversary)

1. **Fencing + claim-liveness must be one atomic check.** `PUT /artifacts/:id` must verify the
   live claim AND the fencing token in a single statement (a `UPDATE artifacts … WHERE …
   RETURNING`, joined against `claims`), or a TOCTOU reopens. DESIGN must show the one statement.
2. **The sweep's revert is the riskiest path** (Phase 13 leasing had no artifact revert). It
   must never un-baseline and must append (not mutate) `artifact_versions`. Needs explicit tests
   for: revert-to-baseline, revert-to-draft (never baselined), and never-un-baseline.
3. **`completeTask` releases the claim** — so a completed task's artifact (`for_review`) has no
   live claim; the sweep must never act on a `for_review` artifact (it only sweeps live claims,
   and `complete` deleted the claim). DESIGN must make this explicit.
4. **The derived-id round-trip** — the service derives the id on create and must re-derive /
   validate it on `PUT`; an actor passing a hand-built id that does not match an `artifacts`
   row gets `not_found`, never a silent create.
5. **`closeTopic` left atomic** — confirm a closed topic's live claims are genuinely recovered
   by the sweep and nothing wedges (the re-deferral of the drain, vs 15.1-D4).
