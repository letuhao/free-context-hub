## Intent

Build **the collective-decision primitive** — the voting half of the Phase 15 governance model
(master design B.6). The Request-Approval primitive's "approver" generalizes from one office
to a **decision body** governed by a voting rule; a single sign-off is the degenerate body of
one — the same mechanism.

- A **decision body** is a project-scoped electorate of weighted members (`decision_bodies` +
  `body_members`). Vote weight is attached to `(actor, body)` and is **orthogonal to level** —
  level is chain-of-command authority, vote weight is ballot power in a body.
- A **motion** is a topic-scoped proposition put to a body:
  `proposed → seconded → balloting → carried | failed | lapsed | vetoed`.
- Members cast weighted **votes** (`for` / `against` / `abstain`). A **tally** applies a
  **quorum** (minimum participating weight) and a **threshold** (fraction needed to carry).
- A **veto** is a first-class unilateral act — a `veto_holders` member overrides an
  otherwise-carried motion. Veto and tally are mutually exclusive on the motion row.

It instantiates corporate governance + parliamentary procedure (Robert's Rules) — the design
principle "inherit, do not invent."

## Why now

15.1 (substrate), 15.2 (the Board), 15.3 (Request-Approval) are shipped. 15.4 is the
**Collective decision** row of the master sprint sketch — primitive A generalized from a
single approver to a voting body. It is a self-contained new primitive (its own tables, its
own service surface) and it unblocks 15.5: a *tribunal* dispute resolution is a
collective-procedure decision and needs the motion machinery to exist first.

## Open questions — decided at CLARIFY (human-in-loop, 2026-05-18)

| # | Question | Decision |

| Q1 | 15.4 scope — standalone vs. + collective request-step integration | **Standalone motion primitive.** `decision_bodies`/`motions`/`votes`/tally/veto as its own primitive; the `procedure='collective'` request-step wiring is a follow-on. *(User.)* |
| Q2 | DEFERRED-013 — fold in or re-defer | **Re-defer to 15.5.** 15.4 does not touch `requests.ts`. *(User.)* |
| Q3 | Motion lifecycle — `proposed → seconded → balloting` | `proposeMotion` → `proposed`; `secondMotion` → **`balloting`** (the second opens the floor — master C.2 has no separate "open balloting" act). `motion.seconded` event records the second. The `seconded` status value ships in the CHECK enum but is **unused** in 15.4 — the 15.1 D4 precedent (`closing` shipped unused). |
| Q4 | Who may do what | **Propose:** a **topic participant** (puts a motion on this topic's log) — the body must exist. **Second:** a **body member ≠ the proposer** (Robert's Rules — a second is a distinct member). **Vote:** a **body member** (the electorate = the body). **Veto:** an actor in `decision_bodies.veto_holders`. |
| Q5 | Body ⋈ topic linkage | A motion carries both `body_id` (the electorate) and `topic_id` (the log it lives on). A decision body is **project-scoped** (`decision_bodies.project_id`, no `topic_id`). Body members vote regardless of topic participation (the "shareholders" model — vote power is orthogonal to topic membership, B.6). |
| Q6 | Quorum & threshold semantics | `quorum` = an **absolute participating-weight floor** — the sum of `votes.weight` over **all** principal rows (`for + against + abstain`). `threshold` = a **fraction in `(0,1]`** of the **decisive base** (`for + against`; abstentions count to quorum, **not** to the threshold base — B.6). **Carried** iff quorum met **AND** `for_weight ≥ threshold · (for_weight + against_weight)`, computed integer-safe (cross-multiply, no float equality). Quorum unmet at tally → **`lapsed`**; quorum met, threshold unmet → **`failed`**. |
| Q7 | Veto | A `veto_holders` member exercises veto **only while `balloting`** (master B.6 rev 4 — the active decision window). `vetoMotion` and `tallyMotion` each **first** run an atomic `UPDATE motions SET status=… WHERE status='balloting'` — the loser of any race sees 0 rows affected and aborts cleanly (master C.2). Veto → terminal `vetoed`, overriding a would-be `carried`. |
| Q8 | Proxy | `castVote` accepts an optional `proxy_for` (the proxy *holder* who physically cast the ballot). The `votes` row is **principal-keyed** (`votes.actor_id` = the principal whose vote it is); `proxy_for` is recorded for audit. A holder casting their own ballot + a proxied ballot = **two distinct rows**. The tally counts principal rows once each; `proxy_for` never affects the count. Proxy-*grant* verification is out of scope (above). |
| Q9 | The expired-motion sweep | A **third swept entity** in `coordinationSweep.ts` — one scheduler cycle now sweeps abandoned claims, stalled steps, and expired motions. A `balloting` motion past `deadline` is auto-resolved (tally → `carried`/`failed`, or `lapsed` if quorum unmet); a `proposed` motion past `deadline` (never seconded) → `lapsed`. Closed-topic branch: **skip** — no mutation, no events (the 15.3 stalled-step precedent). |
| Q10 | Timeouts | A motion's `deadline` is **caller-supplied at propose time** — a deliberation window is a governance choice, unlike a step's fixed 60-min SLA. Validated to `[now + MOTION_MIN, now + MOTION_MAX]`; **default 1440 min (24 h)** if omitted, min 5 min, max 30 days. Sweep interval: the existing 5-min cadence. |
| Q11 | Migration | **Additive** — `decision_bodies`/`body_members`/`motions`/`votes` are new tables; no ALTER to any existing table. |
| Q12 | Re-vote | A second `castVote` by the same principal on the same motion is **rejected** (`already_voted`) — one immutable ballot per member (`votes` PK `(motion_id, actor_id)`, `ON CONFLICT DO NOTHING`). One `vote.cast` event per voter; an unambiguous tally. (Change-your-mind UPSERT semantics are noted as the alternative — not chosen, to keep the surface bounded.) |

## Approaches considered

### Approach A — two services + sweep extension (RECOMMENDED)

`decisionBodies.ts` (the project-scoped body/membership **config** — plain table writes, no
coordination events, no topic), `motions.ts` (the topic-scoped motion **lifecycle** —
propose/second/vote/veto/tally, every write emitting an event via `appendEvent`), and the
expired-motion sweep **added to the existing `coordinationSweep.ts`** (one scheduler, now three
swept entities).
- **Pros:** mirrors 15.3's proven 3-concern split (`doaMatrix.ts` + `requests.ts` + the sweep);
  the body-config concern (no events/topic/locks) and the motion-lifecycle concern (events,
  topic, row locks, the tally) are cleanly separable and independently unit-testable; the sweep
  reuses the proven scheduler.
- **This is the recommendation.**

### Approach B — one `collectiveDecision.ts` covering bodies + motions + votes

Rejected — one oversized module conflating two genuinely separable concerns (the 15.3
Approach-B rejection reason).

### Approach C — motions as a `procedure='collective'` variant inside `requests.ts`, no standalone tables

Rejected — the master design models motions as their own primitive
(`decision_bodies`/`motions`/`votes`); conflating them with the request route loses the
body / quorum / threshold / veto model and the orthogonal-to-level vote weight. (Also the
user's Q1 decision.)

## Acceptance criteria (Sprint 15.4)

- [ ] **AC1** — Migration 0057 applies cleanly + idempotently; `decision_bodies`,
  `body_members`, `motions`, `votes` tables + indexes.
- [ ] **AC2** — `create_decision_body` / `POST /decision-bodies` creates a project-scoped body
  with `quorum`, `threshold`, `veto_holders`; `add_body_member` adds a weighted member. An
  invalid `quorum`/`threshold`/`vote_weight` → a clean `BAD_REQUEST`, never a 500.
- [ ] **AC3** — `propose_motion` / `POST /topics/:id/motions` creates a `proposed` motion bound
  to a `(body, topic)` by a topic participant; emits `motion.proposed`. A non-participant
  proposer / an unknown body → a clean status; a closed topic → `topic_closed`.
- [ ] **AC4** — `second_motion` by a body member ≠ the proposer moves the motion
  `proposed → balloting` and emits `motion.seconded`. The proposer seconding their own motion →
  rejected; a non-member → rejected; a non-`proposed` motion → conflict.
- [ ] **AC5** — `cast_vote` by a body member while `balloting` (before `deadline`) records a
  weighted `for`/`against`/`abstain` ballot and emits `vote.cast`. A vote on a non-`balloting`
  or past-`deadline` motion → rejected; a non-member → rejected; a re-cast → `already_voted`.
- [ ] **AC6** — `tally_motion` atomically closes the ballot (`WHERE status='balloting'`) and
  applies quorum + threshold: quorum unmet → `lapsed`; quorum met + threshold met → `carried`;
  quorum met + threshold unmet → `failed`. Emits `motion.tallied`. Abstentions count toward
  quorum, **not** toward the threshold base.
- [ ] **AC7** — `veto_motion` by a `veto_holders` member while `balloting` atomically sends the
  motion → `vetoed` and emits `motion.vetoed`; a non-veto-holder → rejected. A veto and a tally
  are mutually exclusive on the motion row — the race loser aborts cleanly.
- [ ] **AC8** — Proxy: `cast_vote` with `proxy_for` records a principal-keyed ballot with
  `proxy_for` set; a proxy holder may cast their own ballot + a proxied ballot as two distinct
  rows; the tally counts each principal once.
- [ ] **AC9** — The expired-motion sweep detects a `balloting` motion past `deadline`,
  auto-resolves it (tally or `lapsed`), and emits the resolution event; a `proposed` motion
  past `deadline` → `lapsed`. A motion on a closed topic is skipped (no mutation, no event).
- [ ] **AC10** — Every state change emits a `coordination_events` row; nothing is destructively
  deleted; the motion lifecycle is fully reconstructable from the log.
- [ ] **AC11** — REST mirrors the MCP tools 1:1; one envelope `{status, data?, error?}`. Writes
  use `requireRole('writer')`; **GET routes use `requireRole('reader')` from the start** (the
  15.3.1 F4 posture — role-gated reads built in, not retrofitted).
- [ ] **AC12** — `tsc --noEmit` clean; new unit tests pass; the existing 429-test suite stays
  green.
- [ ] **AC13** — Live smoke on the deployed stack: create a body (3 weighted members) → propose
  a motion → second → 3 members vote → tally → `carried`; a second motion vetoed by a
  `veto_holders` member; a third motion left to expire → the sweep `lapsed`/auto-tallies it.
- [ ] **AC14** — POST-REVIEW: a security-framed cold-start Adversary confirms the
  body/motion/vote authorization model is **honestly scoped** — the self-declared-body residual
  is owned by a deferred item, not overstated as closed (guardrail `5c0b7b25`).

## Pre-emptive self-review (for REVIEW-DESIGN's adversarial pass)

1. **The tally math is the novel core.** DESIGN must give the explicit integer-safe algorithm:
   quorum (participating-weight floor, incl. abstain), threshold (fraction of `for+against`,
   abstain excluded), and the `carried`/`failed`/`lapsed` branch — with the exact comparison
   expression and the boundary cases (quorum exactly met; threshold exactly met).
2. **Vote / tally / veto must be mutually exclusive on the motion row.** The atomic
   `WHERE status='balloting'` pattern (master C.2) must be spelled out for both `tallyMotion`
   and `vetoMotion`, and `castVote` must re-check `status` + `deadline` in the **same txn** as
   the insert. No TOCTOU.
3. **The expired-motion sweep is a third concurrent sweep.** The §10 lock-order table must be
   derived in DESIGN, covering the new `motion`/`vote` transaction alongside the claims +
   stalled-step sweeps. No ABBA.
4. **Self-declared body authority.** DESIGN must state honestly what 15.4 does **not** close —
   body / membership / veto-power creation is coordinator-trusted; the residual is owned by a
   deferred item (the DEFERRED-015 family). No "complete" overclaim (the 15.3.1 design-rev-1
   BLOCK class).
5. **Vote-weight snapshot.** `castVote` must copy `body_members.vote_weight` onto `votes.weight`;
   the tally must sum `votes.weight`, never re-join `body_members`. A mid-motion membership edit
   must not re-weight a cast ballot. Explicit test.
6. **Proxy double-count.** A proxy holder casting its own ballot + N proxied ballots must be
   N+1 distinct principal-keyed rows; the tally must count each principal once. `proxy_for`
   must not affect the count.
7. **All-abstain ballot.** If every cast ballot is `abstain`, the threshold base
   (`for+against`) is 0 — the carried test must not divide by zero; DESIGN must define an
   all-abstain motion as quorum-met-but-`failed` (no decisive support).
8. **Closed-topic motion operations.** propose / second / vote / veto / tally on a closed
   topic — the 15.3 `topic_closed` plain-read pattern + `appendEvent`'s seal as the
   authoritative guard for a mid-transaction close race.
