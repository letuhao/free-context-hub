**Rev 1 → rev 2.** REVIEW-DESIGN round 1 (main-session adversarial self-review,

- **BLOCK-1 — early-tally forgery.** rev 1's `tallyMotion` had no deadline gate — any `writer`
  could tally a `balloting` motion at any instant, so a proposer could tally right after their
  own `for` vote, before the opposition voted, manufacturing a `carried`. Fixed: `tallyMotion`
  rejects `now() < deadline` (`balloting_open`); a ballot is tallied **only post-deadline**
  (on-demand or by the sweep). The window model is now explicit (§3.5, §11 inv. 2).
- **WARN-1 — the electorate is not frozen.** §0.5 + §11 inv. 11 now state honestly that a
  `body_members` change during `balloting` affects the in-flight motion (the electorate is
  live; `vote_weight` is snapshotted per-cast, membership is not snapshotted per-motion).
- **WARN-2 — threshold tie semantics.** §4 now documents the inclusive-threshold rule
  (`for ≥ threshold·base`; a tie carries at `threshold=0.5`; `threshold=1` is unanimity).

## 0. Instantiation decisions (CLARIFY → formalized)

| D | Decision |
|---|---|
| D1 | **`body_id` / `motion_id` are `UUID DEFAULT gen_random_uuid()`** — the codebase UUID-PK convention (`request_id`, `motion_id` in master C.1 is "indicative TEXT"; the 15.1 D-precedent settles PK shape per sprint). |
| D2 | **`decision_bodies` + `body_members` are project-scoped config — plain table writes, NO coordination events.** A body is not tied to a topic; there is no topic log to append to. Mirrors `doa_matrix` (15.3's config table — resolved by a plain read, no event). |
| D3 | **A motion is topic-scoped AND body-scoped** (`motions.topic_id` + `motions.body_id`). All five motion operations (propose / second / vote / veto / tally) emit `coordination_events` on `topic_id`. |
| D4 | **`secondMotion` transitions `proposed → balloting` directly** (CLARIFY Q3). The `seconded` status value ships in the CHECK enum but is **unused** in 15.4 (the 15.1 D4 `closing`-unused precedent). The `motion.seconded` event + `motions.seconded_by` record the act. |
| D5 | **One `deadline`, caller-supplied at propose, fixed for the motion's lifetime.** It governs the whole lifecycle — a motion must be seconded *and* resolved by it. Not reset on `second` (master C.1 has a single `motions.deadline`; a second timeout param is avoided). A motion seconded near its deadline gets a short ballot — correct governance, the proposer owns the window. |
| D6 | **The veto window is `balloting` only** (master B.6 rev 4). A `proposed` or terminal motion cannot be vetoed. |
| D7 | **Vote weight is snapshotted at cast.** `castVote` copies `body_members.vote_weight` onto `votes.weight`; the tally sums `votes.weight` and never re-joins `body_members` (CLARIFY A4 — the B.7 "snapshot the rules" discipline). |
| D8 | **`veto_holders` is independent of `body_members`.** A veto holder need not be a weighted voting member (a golden-share holder may hold veto without an ordinary ballot). `vetoMotion` checks `actor ∈ decision_bodies.veto_holders`; it does not require a `body_members` row. |
| D9 | **All non-veto terminal transitions emit `motion.tallied`** carrying `outcome ∈ {carried,failed,lapsed}`; a veto emits `motion.vetoed`. The master C.3 catalog has no `motion.lapsed` — a lapse is a tally outcome (`tally` produces `carried|failed|lapsed`, master C.4). A `proposed`-never-seconded lapse also emits `motion.tallied{outcome:'lapsed'}`. |
| D10 | **Closed-topic handling** — every motion op plain-reads `topics.status` → a clean `topic_closed` (the 15.3 B3 pattern); `appendEvent`'s close-seal is the authoritative mid-transaction guard. The sweep skips closed-topic motions (DEFERRED-012's `closing`-drain will force-lapse them). |

### 0.1 Transaction & connection contract

Verbatim the Phase 13 / 15.1 / 15.2 / 15.3 pattern: every transactional fn does
`const c = await pool.connect(); try { BEGIN … COMMIT } catch (e) { await
c.query('ROLLBACK').catch(()=>{}); logger.error(...); throw e } finally { c.release() }`,
with an explicit `ROLLBACK` before every early return. The motion sweep's per-motion loop uses
the **§0.1-loop** variant — the catch logs and `continue`s, never rethrows.
**`createBody` / `addBodyMember` are single-statement** (one `INSERT` / `INSERT … ON CONFLICT`)
— atomic without an explicit transaction; `getBody` / `listBodies` / `getMotion` /
`listMotions` are plain reads.

### 0.2 Canonical lock order

15.4 extends the global canonical row-lock order to:

> **`task → claim → request → request_step → artifact → motion → vote → topics`**

`decision_bodies` / `body_members` are **config tables — plain reads, never row-locked** (like
`doa_matrix`); they do not enter the lock order. Every 15.4 transaction acquires its row locks
as a **prefix-consistent subsequence**; 15.4's set `{motion, vote, topics}` is **disjoint from**
`{task, claim, request, request_step, artifact}` — the only shared lock is `topics`, always
last (via `appendEvent`). Derived exhaustively in §10.

## 1. Migration `0057_collective_decision.sql`

All `CREATE … IF NOT EXISTS`; additive — no ALTER, no seed (a decision body is a named set of
named actors — there is nothing to seed; `createBody` is real API).

```sql
-- 0057_collective_decision.sql — Phase 15 Sprint 15.4
-- Collective decision: decision bodies + weighted membership + motions + votes.

-- Decision bodies — a project-scoped electorate governed by a voting rule (B.6).
CREATE TABLE IF NOT EXISTS decision_bodies (
  body_id      UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id   TEXT        NOT NULL,
  name         TEXT        NOT NULL,
  quorum       NUMERIC     NOT NULL DEFAULT 0,        -- absolute participating-weight floor
  threshold    NUMERIC     NOT NULL,                  -- fraction in (0,1] of the for+against base
  veto_holders TEXT[]      NOT NULL DEFAULT '{}',
  created_by   TEXT        NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  CHECK (quorum >= 0),
  CHECK (threshold > 0 AND threshold <= 1)
);
CREATE INDEX IF NOT EXISTS decision_bodies_project_idx ON decision_bodies (project_id, created_at DESC);

-- Body membership — weighted (B.6: vote weight is orthogonal to chain-of-command level).
CREATE TABLE IF NOT EXISTS body_members (
  body_id     UUID        NOT NULL REFERENCES decision_bodies(body_id),
  actor_id    TEXT        NOT NULL,
  vote_weight NUMERIC     NOT NULL,
  added_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (body_id, actor_id),
  CHECK (vote_weight > 0)
);

-- Motions — a topic-scoped proposition put to a body.
CREATE TABLE IF NOT EXISTS motions (
  motion_id   UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  body_id     UUID        NOT NULL REFERENCES decision_bodies(body_id),
  topic_id    TEXT        NOT NULL REFERENCES topics(topic_id),
  subject_ref TEXT        NOT NULL,                   -- the proposition reference (free text, ≤256)
  status      TEXT        NOT NULL DEFAULT 'proposed'
                CHECK (status IN ('proposed','seconded','balloting','carried','failed','lapsed','vetoed')),
  proposed_by TEXT        NOT NULL,
  seconded_by TEXT,
  deadline    TIMESTAMPTZ NOT NULL,
  tally       JSONB,                                  -- the frozen count result; NULL until a count happens
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS motions_topic_status_idx ON motions (topic_id, status);
CREATE INDEX IF NOT EXISTS motions_sweep_idx ON motions (status, deadline);

-- Votes — principal-keyed: one row per member; a proxy ballot IS the principal's row.
CREATE TABLE IF NOT EXISTS votes (
  motion_id UUID        NOT NULL REFERENCES motions(motion_id),
  actor_id  TEXT        NOT NULL,                     -- ALWAYS the principal whose vote it is
  choice    TEXT        NOT NULL CHECK (choice IN ('for','against','abstain')),
  weight    NUMERIC     NOT NULL,                     -- snapshotted from body_members at cast time (D7)
  proxy_for TEXT,                                     -- non-null = cast by proxy-holder <proxy_for>
  cast_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (motion_id, actor_id),
  CHECK (weight > 0)
);
```

`motions`/`votes`/`body_members` reference their parents with **no `ON DELETE`** (the
topics-permanence invariant carried from 15.1/15.2/15.3 — coordination rows are never
orphaned). `decision_bodies.project_id` is a loose reference (no FK — consistent with
`doa_matrix.project_id`).

**No new event type or subject type.** 15.1 pre-provisioned the catalog —
`coordinationConstants.ts` `EVENT_TYPES` already lists `motion.proposed` / `motion.seconded` /
`vote.cast` / `motion.tallied` / `motion.vetoed`, and `SUBJECT_TYPES` has `motion`. 0057 is
purely four `CREATE TABLE`s.

## 2. `src/services/decisionBodies.ts` (new) — body + membership config

### 2.1 `createBody({ project_id?, name, quorum, threshold, veto_holders?, created_by }) → BodyRecord`

Validate: `name`, `created_by` non-empty; `name` ≤ 256 chars (the 15.3.1 F7 length-cap
lesson); `quorum` a finite number `≥ 0`; `threshold` a finite number in `(0, 1]`;
`veto_holders` (default `[]`) an array of non-empty strings. Any miss →
`ContextHubError('BAD_REQUEST', …)` — never an unhandled `22003`/500. `project_id` via
`resolveProjectIdOrThrow` (the 15.1 `charterTopic` pattern). A single
`INSERT … RETURNING *` — no transaction, no event (D2). Returns the body record.

### 2.2 `addBodyMember({ body_id, actor_id, vote_weight }) → AddMemberResult`

Validate `body_id`, `actor_id` non-empty; `vote_weight` a finite number `> 0`.
`SELECT 1 FROM decision_bodies WHERE body_id=$1` — none → `{status:'body_not_found'}`. Then
`INSERT INTO body_members … ON CONFLICT (body_id, actor_id) DO UPDATE SET vote_weight =
EXCLUDED.vote_weight` — idempotent; re-adding a member **sets** their weight (a weight edit is
safe — D7 snapshots weight at cast, so it only affects future ballots). Returns
`{status:'ok', body_id, actor_id, vote_weight}`.

### 2.3 `getBody({ body_id }) → BodyRecord | null` / `listBodies({ project_id }) → { bodies }`

`getBody` — **one query**, body + members via `json_agg` (the 15.1 `fetchTopicWithRoster`
snapshot pattern); `null` → the route maps `NOT_FOUND`. `listBodies` — bodies for a project,
each with its members.

## 3. `src/services/motions.ts` (new) — the motion lifecycle

`MOTION_DEADLINE_DEFAULT_MINUTES = 1440` (24 h); `MOTION_DEADLINE_MIN_MINUTES = 5`;
`MOTION_DEADLINE_MAX_MINUTES = 43200` (30 d).

### 3.1 `proposeMotion({ topic_id, body_id, subject_ref, proposed_by, deadline_minutes? }) → ProposeResult`

Validate: `topic_id`, `body_id`, `subject_ref`, `proposed_by` non-empty; `subject_ref` ≤ 256
(F7); `deadline_minutes`, if provided, an integer in `[MIN, MAX]` else `BAD_REQUEST` — default
`1440`. Pre-`BEGIN` plain reads:
```
(project_id, topic_status) ← SELECT project_id, status FROM topics WHERE topic_id=$1
                              none → NOT_FOUND ; status='closed' → return {topic_closed}
body                       ← SELECT project_id FROM decision_bodies WHERE body_id=$1
                              none OR body.project_id ≠ topic.project_id → return {body_not_found}
                              -- cross-project body rejected; one status for "missing" and
                              -- "in another project" (id-probing defense — the 15.3.1 F3a pattern)
participant                ← SELECT 1 FROM topic_participants WHERE topic_id=$1 AND actor_id=$proposed_by
                              none → return {not_participant}
```
```
BEGIN
  INSERT INTO motions (motion_id, body_id, topic_id, subject_ref, status, proposed_by, deadline)
    VALUES ($m, $body, $topic, $ref, 'proposed', $by, now() + $deadline_minutes * interval '1 minute')
  appendEvent(client, { topic_id, actor_id:$by, type:'motion.proposed',
                        subject_type:'motion', subject_id:$m,
                        payload:{ body_id:$body, subject_ref:$ref, deadline } })   -- (topics)
COMMIT ; return { status:'proposed', motion_id:$m, deadline }
```
Lock order: **`topics` only** (the INSERT is a fresh row; the pre-checks are unlocked reads).

### 3.2 `secondMotion({ motion_id, actor_id }) → SecondResult`

Lock order **`motion → topics`**:
```
BEGIN
  SELECT body_id, topic_id, status, proposed_by FROM motions WHERE motion_id=$1 FOR UPDATE
    none                     → ROLLBACK {not_found}
    status ≠ 'proposed'      → ROLLBACK {conflict}            -- already seconded/balloting/terminal
  SELECT status FROM topics WHERE topic_id=$topic             -- plain read
    'closed'                 → ROLLBACK {topic_closed}
  actor_id = proposed_by     → ROLLBACK {self_second_forbidden}  -- a second is a DISTINCT member (D4)
  SELECT 1 FROM body_members WHERE body_id=$body AND actor_id=$actor
    none                     → ROLLBACK {not_member}
  UPDATE motions SET status='balloting', seconded_by=$actor WHERE motion_id=$1   -- D4
  appendEvent(client, { topic_id, actor_id:$actor, type:'motion.seconded',
                        subject_type:'motion', subject_id:$1, payload:{ seconded_by:$actor } })
COMMIT ; return { status:'seconded', motion_status:'balloting' }
```

### 3.3 `castVote({ motion_id, actor_id, choice, proxy_for? }) → VoteResult` — *core*

`choice ∈ {for, against, abstain}`; `actor_id` is the **principal**. If `proxy_for` is provided
and equals `actor_id` → `BAD_REQUEST` (a self-proxy is a direct vote — store `proxy_for` only
when it differs). Lock order **`motion → vote → topics`**:
```
BEGIN
  SELECT body_id, topic_id, status, deadline FROM motions WHERE motion_id=$1 FOR UPDATE
    none                            → ROLLBACK {not_found}
    status ≠ 'balloting'            → ROLLBACK {not_balloting}
    now() ≥ deadline                → ROLLBACK {balloting_closed}   -- same-txn deadline check (master C.2)
  SELECT status FROM topics WHERE topic_id=$topic                   -- plain read
    'closed'                        → ROLLBACK {topic_closed}
  SELECT vote_weight FROM body_members WHERE body_id=$body AND actor_id=$actor   -- the PRINCIPAL must be a member
    none                            → ROLLBACK {not_member}
  INSERT INTO votes (motion_id, actor_id, choice, weight, proxy_for)
    VALUES ($1, $actor, $choice, $vote_weight, $proxy_for)
    ON CONFLICT (motion_id, actor_id) DO NOTHING
    rowCount = 0                    → ROLLBACK {already_voted}      -- one immutable ballot per principal (Q12)
  appendEvent(client, { topic_id, actor_id:(proxy_for ?? actor_id), type:'vote.cast',
                        subject_type:'motion', subject_id:$1,
                        payload:{ principal:$actor, choice:$choice, weight:$vote_weight, proxy_for } })
COMMIT ; return { status:'vote_recorded' }
```
`votes.weight` is the **principal's** `body_members.vote_weight`, snapshotted (D7). The event
`actor_id` is *who cast it* — the proxy holder if proxied, else the principal; the payload
carries `principal`. The proxy *holder* is recorded, not verified (CLARIFY Q8 / §0.5).

### 3.4 `vetoMotion({ motion_id, actor_id }) → VetoResult`

Lock order **`motion → topics`**:
```
BEGIN
  SELECT body_id, topic_id, status FROM motions WHERE motion_id=$1 FOR UPDATE   -- serializes vs tally
    none                  → ROLLBACK {not_found}
    status ≠ 'balloting'  → ROLLBACK {not_balloting}        -- D6: veto only during balloting
  SELECT status FROM topics WHERE topic_id=$topic
    'closed'              → ROLLBACK {topic_closed}
  SELECT veto_holders FROM decision_bodies WHERE body_id=$body
    $actor ∉ veto_holders → ROLLBACK {not_veto_holder}      -- D8
  UPDATE motions SET status='vetoed' WHERE motion_id=$1
  appendEvent(client, { topic_id, actor_id:$actor, type:'motion.vetoed',
                        subject_type:'motion', subject_id:$1, payload:{ vetoed_by:$actor } })
COMMIT ; return { status:'vetoed' }
```
**Veto / tally mutual exclusion** — both take `motion … FOR UPDATE` *first*; the loser blocks
on the row lock, then sees `status ≠ 'balloting'` and returns `{not_balloting}`. This is
**stronger** than master C.2's lock-free `WHERE status='balloting'` CAS — the `FOR UPDATE`
row lock serializes the two; the post-lock status branch is the guard (the same way 15.1 D1's
in-txn `next_seq` bump is stronger than the master's gap-tolerant floor).

### 3.5 `tallyMotion({ motion_id }) → TallyResult`

Lock order **`motion → topics`**:
```
BEGIN
  SELECT body_id, topic_id, status, deadline FROM motions WHERE motion_id=$1 FOR UPDATE
    none                  → ROLLBACK {not_found}
    status ≠ 'balloting'  → ROLLBACK {not_balloting}        -- already tallied/vetoed/lapsed
    now() < deadline      → ROLLBACK {balloting_open}       -- BLOCK-1: a ballot is tallied only post-deadline
  SELECT status FROM topics WHERE topic_id=$topic
    'closed'              → ROLLBACK {topic_closed}
  (quorum, threshold)     ← SELECT quorum, threshold FROM decision_bodies WHERE body_id=$body
  result                  ← <the §4 aggregate-and-decide query over votes WHERE motion_id=$1>
  UPDATE motions SET status=result.outcome, tally=result.tally_json WHERE motion_id=$1
  appendEvent(client, { topic_id, actor_id:'system:tally', type:'motion.tallied',
                        subject_type:'motion', subject_id:$1,
                        payload:{ outcome:result.outcome, ...result.tally_json } })
COMMIT ; return { status:result.outcome, tally:result.tally_json }
```
The votes are SELECT-aggregated (no row lock on `votes`) — the `motion … FOR UPDATE` lock
already serializes against `castVote`, so the aggregate sees a stable vote set. `actor_id` on
the `motion.tallied` event is the literal `'system:tally'` (an on-demand tally has no single
actor — consistent with the sweep's `'system:sweep'`).

**Tally is post-deadline only (BLOCK-1).** `tallyMotion` rejects `now() < deadline` — a ballot
is counted only after its window has closed, by an on-demand call (immediacy: the result *now*
rather than waiting up to 5 min for the sweep) or by the sweep (automatic) — master C.2 "tally
— or automatic at deadline". This closes the early-tally forgery: a proposer can no longer
snapshot a transient `for` lead before the opposition votes. The three motion operations have
**disjoint-or-lock-serialized windows**:

| Operation | Allowed window |
|---|---|
| `castVote` | `status='balloting'` ∧ `now() < deadline` |
| `vetoMotion` | `status='balloting'` (the whole live window — master B.6 "while balloting") |
| `tallyMotion` / sweep | `status='balloting'` ∧ `now() ≥ deadline` |

Veto and tally overlap only post-deadline, where the `motion … FOR UPDATE` lock + the
`status='balloting'` re-check serialize them (master C.2 — mutually exclusive). A live-smoke /
unit tally therefore needs a past-`deadline` motion: the test sets `deadline` into the past via
direct SQL (the 15.3 escalation-smoke precedent).

### 3.6 `getMotion({ motion_id })` / `listMotions({ topic_id, status? })`

`getMotion` — one query, the motion + its votes via `json_agg`; `null` → route `NOT_FOUND`.
`listMotions` — **first** `SELECT 1 FROM topics WHERE topic_id=$1` (none → `NOT_FOUND`; the
DEFERRED-014(a) lesson — a 404 vs an empty list, applied forward), then the motions for that
topic (+ optional `status` filter), each with its votes.

## 4. The tally algorithm — exact, computed in Postgres `NUMERIC`

The decision is computed in **one SQL statement** so all arithmetic is Postgres exact
`NUMERIC` — never JavaScript float (no `==` drift at a `2/3` boundary). CLARIFY Q6:

```sql
WITH agg AS (
  SELECT
    COALESCE(SUM(weight) FILTER (WHERE choice = 'for'),     0) AS f,
    COALESCE(SUM(weight) FILTER (WHERE choice = 'against'), 0) AS a,
    COALESCE(SUM(weight) FILTER (WHERE choice = 'abstain'), 0) AS ab
  FROM votes WHERE motion_id = $1
)
SELECT f, a, ab,
       (f + a + ab)               AS participating,
       (f + a)                    AS base,
       ((f + a + ab) >= $quorum)  AS quorum_met,
       ((f + a) > 0 AND f >= $threshold * (f + a)) AS carried
FROM agg
```

Outcome mapping (the service reads the booleans):

| `quorum_met` | `base = f+a` | `carried` | → `outcome` |
|---|---|---|---|
| `false` | — | — | **`lapsed`** — quorum not met (turnout too low) |
| `true` | `0` (all-abstain) | `false` | **`failed`** — no decisive support (the `base>0` guard makes `carried` false; no divide-by-zero) |
| `true` | `> 0` | `true` | **`carried`** — `for ≥ threshold · (for+against)` |
| `true` | `> 0` | `false` | **`failed`** — threshold not met |

`tally_json = { for:f, against:a, abstain:ab, participating, base, quorum, threshold,
quorum_met }` — frozen onto `motions.tally` and carried in the `motion.tallied` payload.
A motion with **no votes at all** aggregates to `f=a=ab=0` → `participating=0`; if
`quorum > 0` → `lapsed`; if `quorum = 0` → `quorum_met=true`, `base=0` → `failed`. Both
terminating, both correct.

**Inclusive-threshold semantics (WARN-2).** `carried` iff `for ≥ threshold · base` —
`threshold` is the **inclusive minimum** `for`-fraction of the decisive base. The `≥` is
required, not incidental: `threshold = 1` is **unanimity** and resolves exactly
(`for ≥ 1·base` ⇔ `against = 0`), which a strict `>` could never reach. The consequence at the
low end: at `threshold = 0.5` an exact tie (`for == against`) **carries** (`5 ≥ 0.5·10`). A
body wanting a strict supermajority sets `threshold` accordingly (e.g. `0.667` for two-thirds);
15.4 has no chair's casting vote (CLARIFY Q6) — the `≥`-threshold rule is the sole,
deterministic decision.

## 5. `src/services/coordinationSweep.ts` (modified) — `sweepExpiredMotions`

A **third swept entity** alongside `sweepAbandonedClaims` + `sweepStalledSteps`. Scan (no
lock):
```sql
SELECT motion_id, topic_id FROM motions
 WHERE status IN ('proposed','balloting')
   AND deadline < now() - make_interval(mins => $grace)
```
Each motion is processed in its **own §0.1-loop transaction** (catch → log → `continue`).
Lock order **`motion → topics`**:
```
BEGIN
  SELECT body_id, status, deadline FROM motions WHERE motion_id=$ FOR UPDATE
    rowCount=0 OR status NOT IN ('proposed','balloting') OR deadline ≥ now() → ROLLBACK; continue
  SELECT status FROM topics WHERE topic_id=$ FOR UPDATE                      -- serializes closeTopic
    status='closed' → ROLLBACK; continue                                    -- skip (frozen; DEFERRED-012)
  if motion.status = 'proposed':                                            -- never seconded
    UPDATE motions SET status='lapsed', tally=NULL WHERE motion_id=$
    appendEvent(motion.tallied, payload:{ outcome:'lapsed', reason:'not_seconded' })   -- D9
  else:  -- 'balloting'
    (quorum, threshold) ← SELECT … FROM decision_bodies WHERE body_id=$body
    result ← <the §4 query>
    UPDATE motions SET status=result.outcome, tally=result.tally_json WHERE motion_id=$
    appendEvent(motion.tallied, payload:{ outcome:result.outcome, auto:true, ...tally_json })
  COMMIT ; resolved++
```
`ExpiredMotionsSweepResult = { resolved: number; swept_at: string }`. **Convergence** — an
expired motion is moved to a terminal status in one tick; terminal statuses are excluded by
the scan predicate, so it cannot re-enter the swept set.

## 10. Lock-order derivation — *front-loaded (calibration note)*

Global canonical order: **`task → claim → request → request_step → artifact → motion → vote →
topics`**. `decision_bodies` / `body_members` are config — **plain reads, never row-locked**
— so they do not appear. Each row-lock-acquiring 15.4 transaction, in acquisition order:

| Transaction | Lock acquisition sequence | Prefix-consistent? |
|---|---|---|
| `createBody` / `addBodyMember` | — (single-statement; no txn) | ✓ trivially |
| `proposeMotion` | `topics` (appendEvent) | ✓ single lock |
| `secondMotion` | `motion` → `topics` | ✓ |
| `castVote` | `motion` → `vote` (the `votes` INSERT) → `topics` | ✓ |
| `vetoMotion` | `motion` → `topics` | ✓ |
| `tallyMotion` | `motion` → `topics` (the `votes` aggregate is unlocked — the `motion` `FOR UPDATE` serializes `castVote`) | ✓ |
| `sweepExpiredMotions` (per motion) | `motion` → `topics` | ✓ |

**Cross-primitive check:** every 15.4 transaction locks only within `{motion, vote, topics}` —
**disjoint from** 15.2's `{task, claim, artifact}` and 15.3's `{request, request_step,
artifact}`. The single shared lock is `topics`, acquired **last** by `appendEvent` in every
transaction across all three primitives. No two transactions acquire a shared lock pair in
opposed order ⇒ **no ABBA cycle**, in-primitive or cross-primitive. The three sweeps run
**sequentially** in one scheduler cycle (never concurrent with each other); each is
independently prefix-consistent. The closed-topic plain-reads in §3 take **no** row lock — they
neither appear in nor perturb this table.

## 0.5 Authorization scope — *what Sprint 15.4 does NOT close* (honest statement)

*(The 15.3.1 RETRO lesson `c26217e2` — a governance design must state its scope honestly; an
overstated "complete" claim is the design-rev-1 BLOCK class. The POST-REVIEW security Adversary
verifies this section, AC14.)*

15.4's authorization model is **coordinator-trusted** under the current `MCP_AUTH_ENABLED=false`
single-operator dev posture. 15.4 explicitly does **NOT** close:

- **Body-creation authority.** Any `writer`-role caller can `createBody` with itself as the
  sole member, weight 100, and itself listed in `veto_holders` — then propose, (self-)second
  is blocked, but with one confederate it drives any motion to `carried`. There is no gate on
  *who* may create a body, grant veto power, or assign a vote weight. This is the **same
  self-declared-authority class as DEFERRED-015** (self-declared participant `level`) and
  **DEFERRED-016** (api-key multiplicity). 15.4 records `decision_bodies.created_by` for
  provenance; it does not *authorize* the act.
- **Membership authority.** `addBodyMember` is ungated — anyone may add anyone (incl.
  themselves) at any weight.
- **The electorate is live, not frozen (WARN-1).** A `body_members` change *during* a motion's
  `balloting` affects that in-flight motion — a member added after `second` can `castVote` on
  it. `vote_weight` is snapshotted per-*cast* (D7), but membership is **not** snapshotted
  per-*motion*. Under the coordinator-trusted posture this is consistent (ungated membership is
  already in this residual); a per-motion electorate freeze is a future enhancement, not a 15.4
  deliverable.
- **Proxy-grant verification.** `castVote(proxy_for=X)` records that X cast principal P's
  ballot; it does **not** verify P granted X that proxy. No `proxies` grant table exists
  (none in master C.1).

What 15.4 **does** guarantee: the *mechanism* is sound — quorum / threshold / veto are computed
exactly (§4) and the ballot state machine is atomically enforced (§3.3–3.5); the cross-project
integrity guard holds (§3.1, inv. 8); the event log is complete and the lifecycle is
replayable. Under a single trusted operator (the dev posture) the model is sound. The
authorization residual is owned by a **new deferred item filed at SESSION** — HARD-trigger
family with DEFERRED-015/016: it MUST be resolved before `MCP_AUTH_ENABLED=true` multi-actor
use, Sprint 15.6 (GUI self-serve), or any production / multi-tenant deployment.

## 11. Key invariants & contracts (for the REVIEW-DESIGN adversarial pass)

1. **Tally exactness** — quorum + threshold are computed in one SQL statement in Postgres
   `NUMERIC` (§4); no JavaScript float, no `==` boundary drift. All-abstain (base 0) →
   `failed`, never a divide-by-zero.
2. **Ballot mutual exclusion + the window model** — `castVote`, `tallyMotion`, `vetoMotion` and
   the sweep each take `motion … FOR UPDATE` first; they are serialized on the motion row.
   Windows: `castVote` = `balloting ∧ now()<deadline`; `vetoMotion` = `balloting` (any time);
   `tallyMotion`/sweep = `balloting ∧ now()≥deadline` (BLOCK-1 — a ballot is tallied only
   post-deadline; no early tally). A vote, a tally and a veto are mutually exclusive
   (master C.2 — overlapping windows are lock-serialized on the motion row).
3. **Vote-weight snapshot** — `castVote` writes `body_members.vote_weight` onto `votes.weight`;
   §4 sums `votes.weight` and never re-joins `body_members`. A mid-motion membership-weight
   edit cannot re-weight a cast ballot (D7).
4. **Principal-keyed ballots** — `votes` PK `(motion_id, actor_id)` with `actor_id` = the
   principal; a proxy ballot is the principal's row with `proxy_for` set; a holder casting its
   own + N proxied ballots = N+1 distinct rows. §4 counts each principal once; `proxy_for`
   never enters the count.
5. **Lock order** — every 15.4 transaction is a prefix-consistent subsequence of `… → motion →
   vote → topics` (§10); no ABBA, in- or cross-primitive; `appendEvent`'s `topics` lock is
   always last.
6. **Sealed-topic safety** — motion ops plain-read `topics.status` → a clean `topic_closed`;
   the sweep skips closed-topic motions; `appendEvent`'s close-seal is the authoritative
   mid-transaction guard (the 15.3 rev-6 MED-2 pattern).
7. **Sweep convergence** — an expired motion is resolved to a terminal status in one tick;
   terminal statuses are outside the scan predicate, so it cannot re-enter the swept set or
   loop.
8. **Cross-project integrity** — `proposeMotion` requires `decision_bodies.project_id ==
   topics.project_id`; a cross-project body → `body_not_found` (the 15.3.1 F3a id-probing
   defense applied to the body↔topic link).
9. **Append-only log** — every motion state change emits a `coordination_events` row; nothing
   is destructively deleted; the motion lifecycle is fully reconstructable from the log.
10. **Authorization is coordinator-trusted** — §0.5 states honestly what 15.4 does not close;
    no "complete" overclaim.
11. **The electorate is live, not frozen** — a `body_members` change during `balloting` affects
    the in-flight motion (WARN-1); `vote_weight` is snapshotted per-cast (D7), membership is
    not snapshotted per-motion. Within the §0.5 coordinator-trusted residual.

## 12. Pre-emptive self-review — anticipated adversarial challenges

1. **A motion seconded near its deadline gets a near-zero ballot window** (D5 — one fixed
   `deadline`). Accepted: correct governance — the proposer owns the window; the outcome is a
   correct, terminating `lapsed`/`failed` (the sweep catches it). Not a bug.
2. **`tallyMotion` on a closed topic returns `topic_closed` and does not tally** — the motion
   stays `balloting` in the sealed log, frozen mid-ballot. Consistent with 15.3's closed-topic
   stalled step and with DEFERRED-012 (the `closing` drain, still open) — the sealed log
   faithfully shows the motion mid-flight.
3. **`createBody` self-dealing** — fully owned by §0.5; the mechanism is sound, the
   authorization residual is deferred (HARD trigger). No overclaim.
4. **The third sweep** — `sweepExpiredMotions` is structurally identical to `sweepStalledSteps`
   (§0.1-loop, per-row txn, closed-topic skip, crash isolation); it adds `motion → topics` to
   the §10 table — disjoint from the other two sweeps' lock sets; the three run sequentially.
5. **All-abstain / no-votes tally** — §4's `(f+a) > 0` guard makes `carried` false without a
   divide; `participating >= quorum` decides `lapsed` vs `failed`. Both terminating.
6. **Proxy double-count** — the principal-keyed PK makes a principal's ballot exactly one row;
   §4 sums rows; `proxy_for` is audit-only. A holder cannot inflate a principal's weight.
7. **`secondMotion` self-second** — rejected (`self_second_forbidden`, D4 — Robert's Rules: a
   second is a distinct member). A lone member cannot open a ballot single-handed — though
   §0.5's confederate caveat still holds for the *authorization* layer.
8. **Early tally** — closed in rev 2 (BLOCK-1). `tallyMotion` rejects `now() < deadline`; a
   ballot is counted only post-deadline (on-demand or by the sweep). A proposer can no longer
   manufacture a `carried` by tallying a transient `for` lead before the opposition votes.
