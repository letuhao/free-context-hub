## Scope

### §1 Topic closing drain (DEFERRED-012)

**Current state:** `closeTopic` is atomic — `chartered|active → closed` in one step. Topics with
in-flight claims, requests, motions, disputes, or intake items are closed with those items
abandoned in-place; post-close sweep branches silently drop them.

**Target state:** `closeTopic` becomes two-phase:
1. `active → closing` (freezes the Board — no new task/request/motion may be posted)
2. Drain — force-lapse all remaining in-flight items atomically
3. `closing → closed` — seals the event log

**Implementation design:**

All three phases happen within a **single long transaction** on one pool connection. This is the
safest approach for Phase 15's scale (topics will not have hundreds of concurrent in-flight items).
The `appendEvent` seal (`WHERE status <> 'closed'`) allows appends while status is `'closing'`;
only after the final UPDATE to `'closed'` does the seal take effect.

**Force-lapse semantics per item type:**

| Type | Source table | Terminal state | Event emitted |
|------|-------------|----------------|---------------|
| Active claims | `claims` | claim deleted, task → `abandoned` | `claim.force_lapsed` |
| Open requests | `requests` | status → `rejected`; pending steps → `rejected` | `request.force_closed` |
| Active motions | `motions` | status → `lapsed` (proposed/seconded/balloting) | `motion.force_lapsed` |
| Open disputes | `disputes` | status → `resolved` (forced) | `dispute.force_closed` |
| Received intake items | `intake_items` | status → `dismissed` | `intake.force_dismissed` |

**Idempotency / partial-failure recovery:**
- If topic is already `closed` → return `already_closed: true` unchanged
- If topic is `closing` (partial failure of a prior call) → skip phase 1, resume drain from phase 2

**Return type extension:**
```typescript
type CloseResult = {
  topic_id: string;
  status: 'closed';
  already_closed: boolean;
  force_lapsed: {
    claims: number;
    requests: number;
    motions: number;
    disputes: number;
    intake_items: number;
  };
};
```

**New event types (Sprint 15.6 additions to `coordinationConstants.ts`):**
- `topic.closing` — emitted when topic enters drain state
- `claim.force_lapsed` — payload: `{ task_id, actor_id }`
- `request.force_closed` — payload: `{ request_id, steps_rejected }`
- `motion.force_lapsed` — payload: `{ motion_id, prior_status }`
- `dispute.force_closed` — payload: `{ dispute_id }`
- `intake.force_dismissed` — payload: `{ intake_id }`

**Board freeze during `closing`:** `postTask`, `submitRequest`, `proposeMotion`, `submitIntake`
(with topic_id) must all reject with `TOPIC_NOT_ACTIVE` when topic status is `'closing'`.
Currently they check `status !== 'active'` — `'closing'` will be caught by the same guard
automatically since `closeTopic` already sets `'closing'` before those calls run.

**Verification:** after calling `closeTopic`, the topic must be `'closed'` with no in-flight
items in any of the 5 tables.

### §2 Distinct-endorser enforcement (DEFERRED-013)

**Problem:** `decideStep` allows a single actor to endorse multiple steps of the same
`counter_sign` request — collapsing the multi-party guarantee when the escalation sweep has
climbed a step to a level a later step also targets.

**Fix in `decideStep`:** if `request.route_shape = 'counter_sign'`, query all prior
`request_steps.decided_by` values for this request and reject with `BAD_REQUEST` if the
current `actorId` appears in that set.

```typescript
// After existing actor/level checks, before the step UPDATE:
if (routeShape === 'counter_sign') {
  const prior = await client.query<{ decided_by: string }>(
    `SELECT decided_by FROM request_steps
      WHERE request_id=$1 AND step_index<$2 AND decided_by IS NOT NULL`,
    [requestId, stepIndex],
  );
  if (prior.rows.some(r => r.decided_by === actorId)) {
    throw new ContextHubError('BAD_REQUEST',
      `actor ${actorId} already decided a prior step of this counter-sign request`);
  }
}
```

**Scope:** this check only applies to `counter_sign` routes. `escalate_to_authority` routes have
one step at the final level — no repeat-endorser risk.

### §3 Request consistency residuals (DEFERRED-014)

Four S-sized fixes to `requests.ts` and its route:

**(a) `listRequests` topic-existence check:**
Add `SELECT 1 FROM topics WHERE topic_id=$1` before the main query; throw `NOT_FOUND` if absent.
Aligns with `listBoard` (board.ts) behavior.

**(b) `request.resolved` event payload uniformity:**
`rejected` path (in `decideStep`) and `escalation_exhausted` path (in `sweepStalledSteps`)
currently omit `artifact_advanced`. Add `artifact_advanced: false` to both so all 4 outcome
payloads are uniform: `approved` → `true`, `returned|rejected|escalation_exhausted` → `false`.

**(c) Route-layer fractional step-index guard:**
`routes/requests.ts` parses `req.params.n` via `parseInt` — a fractional segment like `1.5`
truncates to `1`. Add an explicit check: if `String(parsed) !== req.params.n.trim()` (or
`!/^\d+$/.test(req.params.n)`) throw 400 before calling `decideStep`. (Currently the truncated
index fails safely at `not_current_step`; this makes it an honest 400.)

**(d) `submitted_by` / `actor_id` length cap:**
Add `if (submittedBy.length > 256)` check to `submitRequest` and `decideStep` matching the
Sprint 15.3.1 F7 cap on `kind` / `subject_id`.

## Open design questions

**Q1 — Should `closeTopic` be one big transaction or two separate smaller ones?**
Recommendation: **one big transaction.** Atomicity guarantee is worth the longer hold time for
Phase 15 scale. If topics become large, a two-transaction approach can be adopted later.
Risk: a network timeout during the long transaction leaves the topic in `'closing'` — recovered by
the re-entry path (see idempotency above).

**Q2 — Force-lapsed disputes: `resolved` or a new status?**
Recommendation: **`resolved` with `forced: true` in the event payload.** The `disputes` table
CHECK constraint is `('open','under_resolution','resolved')` — adding a new status would require
a migration. Using `resolved` preserves the invariant that a dispute is either open or resolved.

**Q3 — Should `sweepExpiredMotions` / `sweepStalledSteps` skip `closing` topics (as they
currently skip `closed` topics)?**
Recommendation: **Yes — the sweeps skip `closing` topics.** During the drain, `closeTopic` handles
all in-flight items atomically; the sweep's `closing`-branch is the same as the `closed`-branch
(skip, no mutation). This avoids a race where the sweep partially lapses a motion that `closeTopic`
is also draining.
