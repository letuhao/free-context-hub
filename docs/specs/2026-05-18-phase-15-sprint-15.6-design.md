## §0 Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | `closeTopic` is **three-phase** with **separate short transactions per phase**, not one big long transaction | Eliminates deadlock with concurrent `decideStep` / `appendEvent` callers (see §1.3) |
| D2 | Force-lapse item UPDATEs use **no FOR UPDATE lock** on item rows (optimistic); if rowCount=0 skip + ROLLBACK | Avoids deadlock; item already resolved by concurrent transaction → correct outcome either way |
| D3 | Force-lapsed disputes transition to `resolved` with `force_closed: true` in payload | No new dispute status needed; `disputes` CHECK constraint stays unchanged |
| D4 | All three sweep functions skip `closing` topics (same branch as `closed`) | The drain handles in-flight items; sweep must not race with closeTopic's force-lapse |
| D5 | Distinct-endorser check (DEFERRED-013) queries prior `request_steps.decided_by` rows when `route_shape='counter_sign'` | Only counter-sign routes need the multi-party guarantee; escalate_to_authority has one step |
| D6 | No migration required | `topics.status` CHECK already includes `'closing'`; `coordinationConstants.ts` already has it in `TOPIC_STATUSES` |

## §1 Topic closing drain (DEFERRED-012)

### §1.1 Three-phase `closeTopic`

**Phase 1 — Freeze** (brief transaction, `pool.connect`):
```sql
BEGIN
SELECT status FROM topics WHERE topic_id=$1 FOR UPDATE
  → closed: ROLLBACK → return already_closed
  → closing: ROLLBACK → skip to Phase 2 (recovery from partial failure)
  → active|chartered:
    UPDATE topics SET status='closing' WHERE topic_id=$1
    appendEvent(topic.closing, actor_id)   ← UPDATE topics.next_seq (same lock, safe)
COMMIT
```

**Phase 2 — Drain** (one short transaction per in-flight item):
```
For each item type: SELECT items WHERE topic_id=? AND status IN (<in-flight states>)
For each row:
  BEGIN
    UPDATE <table> SET status=<terminal> WHERE id=? AND status IN (<in-flight>)
    -- rowCount=0 → concurrent transaction already resolved this item; ROLLBACK + skip
    -- rowCount>0 → our update succeeded; emit event and COMMIT
    appendEvent(<force-lapse event>)
  COMMIT
  count++
```

Item types in order (no inter-table dependency risk):
1. claims → DELETE from claims; UPDATE tasks SET status='abandoned'
2. requests → UPDATE request_steps SET status='rejected' WHERE status='pending'; UPDATE requests SET status='rejected' WHERE status='open'
3. motions → UPDATE motions SET status='lapsed' WHERE status IN ('proposed','seconded','balloting')
4. disputes → UPDATE disputes SET status='resolved' WHERE status='open'
5. intake_items → UPDATE intake_items SET status='dismissed' WHERE status='received' AND topic_id=$1

**Phase 3 — Seal** (brief transaction, `pool.connect`):
```sql
BEGIN
appendEvent(topic.closed, actor_id)   ← the last event; status still 'closing' → seal fires
UPDATE topics SET status='closed' WHERE topic_id=$1
COMMIT
```
After Phase 3 COMMIT the seal is in effect: `appendEvent` `WHERE status <> 'closed'` returns 0 rows for any further attempt.

### §1.2 Deadlock analysis (D1/D2)

`decideStep` lock order: `requests` → `request_steps` → `topics.next_seq` (via appendEvent).
Force-lapse lock order: `requests` (optimistic UPDATE without FOR UPDATE, brief hold) → `topics.next_seq` (via appendEvent).

**No-deadlock proof:**
- If decideStep holds requests before force-lapse UPDATE runs: force-lapse UPDATE blocks, waits for decideStep's COMMIT; decideStep proceeds to appendEvent (acquires topics briefly), COMMITs; force-lapse UPDATE sees 0 rows → ROLLBACK. No cycle.
- If force-lapse UPDATE succeeds first: decideStep's SELECT FOR UPDATE blocks, waits; force-lapse COMMITs (status now 'rejected'); decideStep proceeds, req.status='rejected' → `already_resolved`. No cycle.
- Phase 1 and Phase 3 both acquire the topics lock (via appendEvent UPDATE) briefly; no concurrent operation holds the topics lock first (the topics `SELECT FOR UPDATE` in Phase 1 serializes with any concurrent appendEvent, which is the existing invariant).

## §2 Distinct-endorser enforcement (DEFERRED-013)

**Location:** `decideStep` in `requests.ts`, after the level-authorization check, before the step UPDATE.

**Guard — only for `counter_sign` routes:**
```typescript
// Read route_shape from the requests row (add to the SELECT)
// After level check:
if (routeShape === 'counter_sign') {
  const prior = await client.query<{ decided_by: string }>(
    `SELECT decided_by FROM request_steps
      WHERE request_id=$1 AND step_index<$2 AND decided_by IS NOT NULL`,
    [requestId, stepIndex],
  );
  if (prior.rows.some(r => r.decided_by === actorId)) {
    await client.query('ROLLBACK');
    return { status: 'repeat_endorser' };
  }
}
```

**`route_shape` field must be added to the existing `SELECT` on the `requests` row** in `decideStep`.

**New `DecideResult` variant:** `{ status: 'repeat_endorser' }` — the route returns 409 at the
REST layer with code `REPEAT_ENDORSER`.

**New `ContextHubError` code:** `'REPEAT_ENDORSER'` added to the union in `errors.ts`.

**REST route** (`routes/requests.ts`): map `repeat_endorser` → 409 in `STATUS_MAP`.

## §5 Lock table (§0.2 ordering invariant)

Existing lock order: `dispute → topics → request → request_step → artifact → topics.next_seq`.

Sprint 15.6 `closeTopic` lock order:
- Phase 1: `topics` (SELECT FOR UPDATE) → `topics.next_seq` (appendEvent UPDATE) — same as existing `closeTopic`
- Phase 2: `requests` (UPDATE without FOR UPDATE, brief) → `topics.next_seq` (appendEvent) 
           `motions` (UPDATE) → `topics.next_seq`; `disputes` (UPDATE) → `topics.next_seq`; etc.
- Phase 3: `topics.next_seq` (appendEvent) → `topics` (UPDATE status)

`decideStep` extended: reads `route_shape` (already on the locked `requests` row — no new lock).

No new lock-order violations introduced.
