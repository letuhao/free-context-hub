# Sprint 13.5 — F3 core (Domain Taxonomy Extension)

## What this sprint ships

1. **Migration 0050** — `taxonomy_profiles` + `project_taxonomy_profiles` tables (idempotent per Sprint 13.2 pattern).
2. **`src/constants/lessonTypes.ts`** — `BUILTIN_LESSON_TYPES` (5 values) + `GUARDRAIL_LESSON_TYPES` (`'guardrail'` + `'codex-guardrail'`).
3. **`src/services/taxonomyService.ts`** — CRUD: `listProfiles`, `getProfile`, `getProfileBySlug`, `createProfile`, `getActiveProfile`, `activateProfile`, `deactivateProfile`, `getValidLessonTypes(project_id)`, `getLessonTypeLabel(project_id, type)`. Service-layer validation: no shadowing of built-in types, `is_builtin` immutable via API.
4. **Bootstrap seeding** — on startup, read `config/taxonomy-profiles/*.json` and upsert each as `is_builtin=true, owner_project_id=NULL`. Slug is upsert key.
5. **Bundled built-in profile** — `config/taxonomy-profiles/dlf-phase0.json` per master design L454-494.
6. **Guardrail engine extension** — `src/services/lessons.ts:290` extends `if (payload.lesson_type === 'guardrail')` to `if (GUARDRAIL_LESSON_TYPES.includes(payload.lesson_type))`. The guardrails-table insertion now fires for `'codex-guardrail'` too. The existing JOIN-on-rule_id query in `guardrails.ts` continues to work because the guardrails table now holds rule rows for both types.
7. **KG edge mapping** — `src/kg/linker.ts:7` extends `if (t === 'guardrail') return 'CONSTRAINS'` to `if (GUARDRAIL_LESSON_TYPES.includes(t)) return 'CONSTRAINS'`.
8. **MCP enum site updates** — `src/mcp/index.ts` lines ~959, ~1051: read-path filters change from `z.enum([5-types])` to `z.string().optional()` (no validation, pass-through). The `add_lesson` write path keeps `z.string()` input + adds runtime validation via `taxonomyService.getValidLessonTypes(project_id)`.
9. **NEW MCP tools** (avoid `z.discriminatedUnion` per DEFERRED-007 — use `z.object` with optional fields or nullable status):
   - `list_taxonomy_profiles` — list active project's profile + all available profiles
   - `activate_taxonomy_profile` — activate a profile by slug for a project
   - `deactivate_taxonomy_profile` — clear active profile
   - `get_active_taxonomy_profile` — fetch the active profile for a project (or null)
10. **REST routes** — `src/api/routes/taxonomy.ts`:
    - `GET /api/taxonomy-profiles` — list (filter: owner_project_id, is_builtin)
    - `GET /api/taxonomy-profiles/:slug` — detail by slug
    - `POST /api/taxonomy-profiles` — create custom (is_builtin forced false, type shadowing validated)
    - `GET /api/projects/:id/taxonomy-profile` — active profile (or null)
    - `POST /api/projects/:id/taxonomy-profile/activate` — `{slug}`
    - `DELETE /api/projects/:id/taxonomy-profile` — deactivate
11. **Unit tests** — `src/services/taxonomyService.test.ts` covering: CRUD, shadowing rejection, is_builtin immutable, activate/deactivate, getValidLessonTypes (with + without active profile), codex-guardrail edge mapping path (integration-style).

## ACs (from master design §"Feature 3 complete when:")

(Master design doesn't have a numbered AC checklist for F3 explicitly. Deriving from the §Concept + §Validation + §Search-and-add_lesson sections.)

- **F3-AC1:** `add_lesson` with a lesson_type that's NOT in built-ins AND NOT in active profile → rejected with informative error listing valid types.
- **F3-AC2:** `add_lesson` with lesson_type in the active profile → accepted (lesson row persisted with that type string verbatim).
- **F3-AC3:** Custom profile with a lesson_type that shadows a built-in (`decision`, `preference`, etc.) → rejected at create-time.
- **F3-AC4:** `POST /api/taxonomy-profiles` ignores `is_builtin: true` in the payload — always stores `is_builtin = false` for API-created profiles.
- **F3-AC5:** `check_guardrails` engine matches lessons with `lesson_type = 'codex-guardrail'` as guardrail rules (per decision D2).
- **F3-AC6:** KG edge mapping returns `'CONSTRAINS'` for `codex-guardrail` lesson type.
- **F3-AC7:** Deactivating a profile does NOT change existing lesson `lesson_type` strings. Lessons with deactivated profile types are still stored and retrievable.
- **F3-AC8:** Bundled built-in profile `dlf-phase0` is present after server startup with its 5 lesson types.

## Locked assumptions

**A1.** Migration 0050 follows Sprint 13.2 idempotent pattern (DO block with set-equality / IF NOT EXISTS for CREATE TABLE).

**A2.** Service-layer validation NOT zod-layer for lesson_type (the type is now project-conditional, can't be a static enum). The read paths (`list_lessons`, `search_lessons`) accept any string — no validation, pass-through to SQL. The write paths (`add_lesson`, `suggest_lessons_from_commits`) call `taxonomyService.getValidLessonTypes(project_id)` and reject with `ContextHubError('BAD_REQUEST', ...)` if invalid.

**A3.** `GUARDRAIL_LESSON_TYPES` constant used in 2 places: `lessons.ts:290` (guardrails-table insertion trigger) and `kg/linker.ts:7` (KG edge mapping). The constant is the single source of truth — adding a third guardrail-class type later requires only updating the constant.

**A4.** Bundled profile JSON files live in `config/taxonomy-profiles/` (a NEW directory). The bootstrap step is in a new file `src/services/taxonomyBootstrap.ts` called from `src/index.ts` after `applyMigrations()`.

**A5.** Slug uniqueness: built-ins use `(slug, owner_project_id) UNIQUE NULLS NOT DISTINCT` → built-in slugs globally unique (since owner_project_id is NULL). Custom profiles use the same constraint with their owner's project_id → custom slugs scoped per-project. This is the cleanest schema; matches master design L517-519.

**A6.** Activation race: `project_taxonomy_profiles` PRIMARY KEY is `project_id` (one active profile per project). UPSERT semantics. No race window — at most one row per project.

**A7.** Custom profile creation by writer+; activation/deactivation by writer+; built-in management is admin-only but in practice only happens via startup bootstrap (no API endpoint to mutate is_builtin).

**A8.** GUI work (Taxonomy tab on Project Settings, deactivation dialog) is OUT OF SCOPE for Sprint 13.5 → Sprint 13.6 owns it.

## Risks

- **R-med:** changing `lessons.ts:290` from `===` to `INCLUDES` is small but it's the line that triggers guardrails-table INSERT. Test must verify a `codex-guardrail` lesson with a `guardrail` payload creates BOTH the lesson row + the guardrails row.
- **R-med:** `check_guardrails` engine query in `guardrails.ts` joins on `guardrails.rule_id = lessons.lesson_id` — it does NOT filter on `lesson_type` directly. So the extension is solely via the INSERT path widening (more rule rows in guardrails table). The query continues to work unchanged. Verify via smoke.
- **R-low:** Per Sprint 13.2 calibration, idempotent CHECK constraints work; reapplying the same migration pattern.
- **R-low:** `GUARDRAIL_LESSON_TYPES.includes(payload.lesson_type)` is a strict-typed string comparison; `payload.lesson_type` is typed as `LessonType` (now a static string union). Need `GUARDRAIL_LESSON_TYPES as readonly string[]).includes(payload.lesson_type)` cast or extend LessonType. Use string-cast for minimal blast radius.
- **R-med (DEFERRED-007 interaction):** new MCP tools MUST NOT use `z.discriminatedUnion` in outputSchema. Use `z.object` with optional fields (`status: z.enum([...]).optional()` or `status: z.string()`) for now.
