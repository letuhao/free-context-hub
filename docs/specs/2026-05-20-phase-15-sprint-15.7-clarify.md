## Background

Phase 15 master design (`docs/phase-15-design.md` §C.4):

> **Chaining:** a resolved Request or a carried Motion emits an event whose handler **posts
> a new board task** ("execute the approved/carried outcome") — **unless the topic is
> `closing` or `closed`**, in which case the handler does **not** post; it emits
> `task.deferred` (recording the would-be task in the sealed trail) so a draining topic is
> never re-filled. The two primitives interlock — an approval result becomes claimable work
> — but never against a topic that is terminating.

Sprint 15.6 shipped the drain mechanics (writer-block on `closing`, three-phase
`closeTopic`) and pre-provisioned the `task.deferred` event in `EVENT_TYPES`. Neither 15.3
nor 15.4 implemented the chaining handler itself.

For DEFERRED-011: `closeTopic` is already idempotent on `closing` re-entry — line 318 of
`topics.ts` documents "already-closing → re-runs Phase 2+3". What's missing is an
**automatic** sweep that picks up a topic stuck in `closing` (e.g., DB crash mid-Phase-2)
without operator intervention.

## Acceptance Criteria

### Chaining (DEFERRED-019 — with Q1 execution_task blob)

- **AC1** — `submitRequest` accepts an optional `execution_task` field (JSONB shape
  `{title?: string, topology?: 'parallel'|'sequential'|'rolling', slot?: string, kind?:
  string, depends_on?: string[], raci?: object}`). Validates per-field at submit time
  using the same constraints as `postTask` (SLOT_REGEX, TOPOLOGIES, length caps). Stored
  in `requests.execution_task`. NULL is allowed.

- **AC2** — `proposeMotion` accepts an optional `execution_task` field with the same shape
  and validation. Stored in `motions.execution_task`.

- **AC3** — A request resolves with `outcome='approved'`. In the **same transaction**, the
  chaining handler reads `requests.execution_task`:
  - If non-null: use the blob as `postTask` params (fields override the derived defaults).
  - If null: use derived defaults (`title="Execute approved request: <kind>"`, `topology=
    'parallel'`, `slot="exec-<16hex of request_id>"`, `kind=requests.kind`).
  Calls `postTask` and returns `chained_task_id` in the `request.resolved` payload.

- **AC4** — A motion resolves with `outcome='carried'` via `tallyMotion` (user-driven
  path). Same logic, with motion-derived defaults (`title="Execute carried motion:
  <subject>"`, `kind=motions.subject_type`).

- **AC5** — A motion resolves with `outcome='carried'` via `sweepExpiredMotions` auto-tally.
  Same chaining logic. `created_by='system'` for sweep-driven path; for user-driven (AC3,
  AC4) `created_by` = the resolving actor.

- **AC6** — Negative cases — no chaining and no new task:
  - request `outcome='returned'` / `'rejected'` / `'escalation_exhausted'`
  - motion `outcome='failed'` / `'lapsed'` (both from `tallyMotion` and from
    `sweepExpiredMotions` not_seconded/deadline-lapsed) / `'vetoed'`

- **AC7** — Topic is `'closing'` when chaining fires (race: `closeTopic` interleaves with
  `decideStep`/`tallyMotion`/sweep). The handler **does not** call `postTask`; instead it
  emits a `task.deferred` event with payload `{source_request_id?: uuid, source_motion_id?:
  uuid, would_be_task: <postTask params>}`. Returns `chained_task_deferred: true` in the
  `request.resolved`/`motion.tallied` payload. Same for `'closed'` (theoretical for user
  paths post-15.6 writer-block; reachable from sweep).

- **AC8** — Atomicity — postTask failure rolls back the source `request.resolved`/
  `motion.tallied` emission. Single transaction.

- **AC9** — `execution_task` validation negatives — submit_request/propose_motion rejects
  with HTTP 400 / `BAD_REQUEST` when execution_task contains invalid topology, slot
  regex violation, slot > 64 chars, depends_on with non-UUID elements, depends_on cross-
  topic refs (validated at chain time — submit-time check is structural only).

- **AC10** — `depends_on` in `execution_task` at chain time is validated against the topic's
  tasks (same check as `postTask`). If a referenced task is not in the topic (e.g., it was
  on a different topic or never existed), chaining fails atomically — the source event
  rolls back, the request/motion stays at its prior state, and the operation returns
  `chained_task_dependency_invalid` to the caller.

### Stuck-closing sweep (DEFERRED-011a — Q2 event-log-based)

- **AC11** — A topic in `'closing'` state whose most-recent `topic.closing` event in
  `coordination_events` is >5 minutes old is picked up by `sweepStuckClosingTopics()`. The
  sweep calls `closeTopic(topic_id)` — idempotent re-entry runs Phase 2+3 and seals it.

- **AC12** — A topic in `'closing'` state for **less than 5 minutes** is **not** picked up
  (an active in-flight `closeTopic` call gets time to complete).

- **AC13** — The recovery sweep runs in the same advisory-lock hold as the existing 3
  sweeps (`startClaimsSweepScheduler` cycle), 4th in order, after the motion sweep.

- **AC14** — A failure in `sweepStuckClosingTopics()` does not abort the other 3 sweeps
  in the cycle (§0.1-loop, per-topic try/catch, log+continue).

### Topology enforcement on `claimTask` (DEFERRED-011b — Q4)

- **AC15** — `claimTask` on a task with `topology='sequential'`: rejects with `BAD_REQUEST` /
  `unmet_dependencies` if **any** task in `depends_on` has `status != 'completed'`. Returns
  the list of incomplete predecessor task_ids in the error detail.

- **AC16** — `claimTask` on a task with `topology='sequential'` and **all** `depends_on`
  predecessors `status='completed'` succeeds (positive case).

- **AC17** — `claimTask` on a task with `topology='rolling'`: rejects with `BAD_REQUEST` /
  `upstream_not_baselined` if **any** upstream-by-depends_on task's **output artifact** is
  not `state='baselined'` (the rolling consumer gates on a baselined producer). The
  upstream task itself does NOT need `status='completed'` — `'rolling'` is the parallel-
  with-handoff topology; consumer can start as soon as the producer baselines.

- **AC18** — `claimTask` on a task with `topology='parallel'` (the common case) has **no**
  predecessor check (regression test — 15.2 behavior preserved).

- **AC19** — Empty `depends_on` on `sequential`/`rolling` is permitted and yields a normal
  claim — the topology metadata records intent; the rule only fires on non-empty
  `depends_on`.

## Chained task params resolution (with Q1 execution_task blob)

At chain time the handler builds postTask params by **merging submitter blob on top of
defaults**:

| Field | Default | Submitter override |
|---|---|---|
| `topic_id` | inherited from source primitive | (always inherited; not overridable) |
| `title` | request: `"Execute approved request: <kind>"`<br>motion: `"Execute carried motion: <subject>"` | `execution_task.title` |
| `topology` | `'parallel'` | `execution_task.topology` (must validate against TOPOLOGIES) |
| `slot` | `"exec-<16hex>"` (first 16 hex chars of source_id) | `execution_task.slot` (must validate SLOT_REGEX, ≤64 chars) |
| `kind` | request: `requests.kind` · motion: `motions.subject_type` | `execution_task.kind` |
| `created_by` | resolving actor for user paths; `'system'` for sweep | (always system-set; not overridable) |
| `depends_on` | `[]` | `execution_task.depends_on` (validated at chain time, same as postTask) |
| `raci` | `{source_request: <id>}` or `{source_motion: <id>}` | merge: submitter `raci` ∪ source-link key (source-link key wins on conflict) |

Validation:
- Submit-time validation: structural only (types, regex on slot, length caps, topology
  enum). NOT validated: `depends_on` task existence (because at submit time, the
  predecessor tasks may not yet exist on the topic — they could be created between submit
  and chain).
- Chain-time validation: `depends_on` task existence + same-topic check, mirroring
  `postTask` lines 192–205. A chain-time failure rolls back the source resolution event
  atomically.

Slot collision rationale: 16 hex chars = 64 bits of collision space, well below the
slot 64-char cap. Birthday-collision probability is negligible for any realistic topic
(~ 2^-32 at 65k tasks/topic). On the unlikely full collision, the artifact PK insert in
`postTask` raises 23505 and the source event rolls back; the user can retry with an
explicit `execution_task.slot`.

## Decisions (resolved with the user 2026-05-20)

**Q1 — Chained task params: Submitter-specified `execution_task` blob** (with derived
defaults as fallback when blob is null).

  - Adds `execution_task JSONB NULL` column to `requests` and `motions` via migration 0060.
  - `submit_request` / `propose_motion` (REST + MCP) accept the optional field.
  - Structural validation at submit time; semantic validation (`depends_on` existence) at
    chain time.
  - Backwards-compat: NULL `execution_task` → chain uses the derived defaults shown in
    §Chained task params resolution above.

**Q2 — Sweep staleness detection: Event-log-based** (no migration for sweep).

  - `sweepStuckClosingTopics` joins `coordination_events` with `topics`, selecting
    `topic.status='closing'` AND `max(coordination_events.created_at) FILTER (WHERE
    type='topic.closing') < now() - interval '5 minutes'`.
  - Each picked topic enters its own §0.1-loop transaction and calls `closeTopic` (which
    is idempotent on `closing`).

**Q3 — Security review scope: Skip security-framed cold-start adversarial review.**

  - Standard v2.2 self-review at REVIEW-DESIGN + REVIEW-CODE.
  - Rationale recorded: chaining wires existing authorized primitives; topology
    enforcement strengthens (not adds) authorization. No new authority surface introduced.

**Q4 — Scope: Expand to include 011 topology enforcement on `claimTask`.**

  - `sequential` topology: all `depends_on` predecessor tasks must be `status='completed'`.
  - `rolling` topology: upstream task's output artifact must be `state='baselined'`.
  - `parallel` is unchanged (no predecessor check).
  - Sprint size revised from M → L.

## Risks

1. **Closing-window race nuance** — between `topics.status` read and `postTask` execution,
   a concurrent `closeTopic` could flip the topic. Mitigation: read `topics.status` with
   `FOR UPDATE` inside the same transaction. The existing `decideStep`/`tallyMotion`
   transactions already lock the request/motion row; we add a topic-status lock at the
   chaining point.

2. **Sweep idempotency** — `sweepStuckClosingTopics` calls `closeTopic` again on the same
   topic. `closeTopic` is documented idempotent (line 318), but Phase 2 may re-run the
   drain on a now-empty in-flight set. Verify: Phase 2 drain on a `closing` topic with no
   in-flight items returns `force_lapsed.claims = 0` etc. (already tested as AC8 in 15.6).

3. **Chained-task slot collision** — derived slot `"exec-<8 hex>"`: collision space is 2^32,
   per-topic. Two carried motions on the same topic with the same 8-hex prefix would
   collide on the `(topic_id, task_id, slot)` artifact PK. Probability: ~negligible for
   realistic topic sizes, but accept defensively: on collision, fall back to a full 36-char
   slot `"exec-<full-uuid>"` — but slot is capped at 64 chars, so we'd have to truncate or
   use 12-hex (2^48 space) by default. **Decision: use 16-hex (8 bytes / 32 bits is too
   small; 16-hex / 64 bits is safe).** Slot becomes `"exec-<16 hex>"` (21 chars, well under
   64).

4. **Backwards compatibility of `request.resolved` payload** — adding `chained_task_id` or
   `chained_task_deferred` to existing payload. Replay consumers must handle missing field
   as before (`approved` outcomes from pre-15.7 events). The 15.7 chaining only applies to
   newly-emitted events, so historic event replay is unaffected.
