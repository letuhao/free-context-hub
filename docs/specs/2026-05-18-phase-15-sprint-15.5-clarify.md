## 2. Scope (clear in-scope)

### 2.1 Intake mailbox — `intake_items`

An inbound channel for items that belong to no current task (design B.8).

**Table (from C.1):**
```sql
CREATE TABLE intake_items (
  intake_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id  TEXT NOT NULL,
  topic_id    TEXT,                 -- null until triaged
  kind        TEXT NOT NULL CHECK (kind IN ('violation_report','suggestion','request')),
  body        TEXT NOT NULL,
  submitted_by TEXT NOT NULL,
  status      TEXT NOT NULL DEFAULT 'received'
                CHECK (status IN ('received','triaged','dismissed')),
  routed_to   TEXT                  -- ID of the task/request/motion/dispute it became
);
```

**Service functions (`src/services/intake.ts`):**
- `submitIntake(params)` → inserts row, emits `intake.received`
- `triageIntake(intake_id, route)` → updates status/routed_to, emits `intake.triaged`
- `dismissIntake(intake_id)` → status = 'dismissed', no event
- `getIntake(intake_id)` → row
- `listIntake(project_id, opts?)` → rows

**Triage routing (design decision D1 — see §4.1):**
- `route_kind: 'task'|'request'|'motion'` → link-only: caller passes `routed_to` (existing ID)
- `route_kind: 'dispute'` → create-on-triage: triage body includes dispute params
  (`topic_id`, `subject_ref`, `parties`, `procedure`) → triage creates the dispute and links it

**REST routes (`src/api/routes/intake.ts`):**
```
POST /api/intake                   submitIntake
GET  /api/intake/:id               getIntake
POST /api/intake/:id/triage        triageIntake
POST /api/intake/:id/dismiss       dismissIntake
GET  /api/projects/:id/intake      listIntake (query filter: kind, status)
```

**MCP tools (5):** `submit_intake`, `get_intake`, `triage_intake`, `dismiss_intake`, `list_intake`

### 2.2 Dispute resolution — `disputes`

A dispute is a Request-Approval item routed to an arbiter (unilateral) or tribunal (collective)
(design B.8). The dispute creates a resolution request in `requests`; its lifecycle follows that
request.

**Table (from C.1):**
```sql
CREATE TABLE disputes (
  dispute_id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id              TEXT NOT NULL,
  subject_ref           TEXT NOT NULL,
  parties               TEXT[] NOT NULL,
  status                TEXT NOT NULL DEFAULT 'open'
                          CHECK (status IN ('open','under_resolution','resolved')),
  resolution_request_id UUID          -- → requests(request_id)
);
```

**Service functions (`src/services/disputes.ts`):**
- `openDispute(params)` → inserts row, calls `submitRequest` to create resolution request,
  updates `resolution_request_id`, emits `dispute.opened`
- `resolveDispute(dispute_id)` → verifies resolution request is terminal (approved/returned/rejected),
  sets status = 'resolved', emits `dispute.resolved`
- `getDispute(dispute_id)` → row + resolution request summary
- `listDisputes(topic_id)` → rows

**Dispute status transitions:**
- `open` → dispute created, resolution request submitted
- `under_resolution` → set automatically when the first `request.step_decided` event is emitted
  for `resolution_request_id` (chaining hook in `disputes.ts`, called by REST decide route)
- `resolved` → explicit `POST /disputes/:id/resolve` after resolution request is terminal

**Coupling strategy:** `disputes.ts` queries `requests` table directly — no circular import.
The REST decide route for requests calls `maybeAdvanceDisputeStatus(request_id)` after step decision.
This function lives in `disputes.ts` and updates `open → under_resolution` if needed.

**REST routes (`src/api/routes/disputes.ts`):**
```
POST /api/disputes                 openDispute
GET  /api/disputes/:id             getDispute
POST /api/disputes/:id/resolve     resolveDispute
GET  /api/topics/:id/disputes      listDisputes
```

**MCP tools (4):** `open_dispute`, `get_dispute`, `resolve_dispute`, `list_disputes`

### 3.1 DEFERRED-013 — Counter-sign distinct-endorser enforcement (Q1)

**What:** `decideStep` does not track which actors decided earlier steps — one officeholder can
endorse multiple steps of a counter-sign route, collapsing the distinct-endorser guarantee.

**Estimated size:** S–M (add `decided_by` set tracking in `decideStep`; 1-2 tests)

**Scope options:**
- **(A) Fold into 15.5:** touches `requests.ts` (just hardened in 15.3.1). S sub-item inside an L sprint.
- **(B) Re-defer to 15.6:** governance path only (post-escalation counter-sign), LOW priority,
  safe to defer one more sprint.

**Recommendation:** (B) — re-defer. Touching `requests.ts` increases the 15.5 security review
surface without a direct feature need. The dispute primitive (the trigger reason) doesn't require
it — disputes route to arbiter/tribunal, not counter-sign.

### 3.2 DEFERRED-012 — Topic `closing`-drain (Q2)

**What:** `closeTopic` is currently atomic (`active → closed` in one step, no drain). The design
specifies a two-phase `closing` drain state: freeze the board, force-lapse all in-flight items
(claims, requests, motions, disputes), then seal to `closed`. Sprint 15.5 is the trigger because
all in-flight item types now exist.

**Estimated size:** M–L (two-phase `closeTopic`; force-lapse logic for 4 item types; tests)

**Scope options:**
- **(A) Fold into 15.5:** trigger is now met; full in-flight set exists (15.2 + 15.3 + 15.4 + 15.5).
  This is the right time to build it once over the complete set. Expands sprint to XL.
- **(B) Defer to 15.6:** 15.6 is the GUI sprint; the GUI needs a correct topic close UX.
  Deferring keeps 15.5 at size L (intake + dispute) and ensures 15.6 has a concrete backend task.

**Recommendation:** (B) — defer to 15.6. Sprint 15.5 is already L (intake + dispute). A closing-drain
adds M–L work including 4-primitive force-lapse logic that deserves its own careful design.
15.6 is the GUI sprint — having the drain ready for the topic-close GUI interaction is cleaner than
splitting the design across two sprints.

## 4. Design decisions (settled in CLARIFY)

### D1 — Triage routing model
Triage for `task/request/motion` is **link-only** (caller passes the already-existing `routed_to` ID).
Triage for `dispute` is **create-on-triage** (dispute params embedded in the triage body; triage
creates the dispute and links it). Rationale: disputes are semantically the result of an intake
escalation — it is natural to create them at triage time rather than requiring a 3-step flow.
Standalone `POST /api/disputes` also exists for non-intake dispute creation.

### D2 — Dispute `under_resolution` transition
Automatic via `maybeAdvanceDisputeStatus()` called from the requests REST route after a step is
decided. Keeps the state machine correct without requiring an explicit caller action.

### D3 — Dispute `resolveDispute` precondition
`resolveDispute` checks that `resolution_request_id` status is in
`{'approved','returned','rejected'}` — any terminal request state resolves the dispute.
`returned` resolves as settled (the dispute outcome is recorded; the parties can re-escalate
via a new dispute if needed).

### D4 — `routed_to` format
`routed_to` stores the raw ID string of the target entity. For disputes this is `dispute_id`
(UUID). For tasks/requests/motions the caller passes the appropriate UUID/string ID.
