# Design — Phase 12 Sprint 12.1e2

Resolves the 4 DESIGN-phase questions left open in the spec.

## 1. R5 — Access-log drift between sweep runs (RESOLVED)

**Problem.** Each baseline run writes ~1200 `consideration-search` rows to `lesson_access_log`. Run 1's writes drift the salience state visible to run 2, confounding the "is half-life X better than half-life Y" question — run 2 measures `hl=Y on log-state-after-hl=X-run`, not `hl=Y on clean-baseline`.

**Resolution — R5-H (snapshot + reset before each run):**

1. At sprint start: capture `T0 = MAX(accessed_at) FROM lesson_access_log`. Save as a literal timestamp string in the plan.
2. Before EACH of the 4 baseline runs: `DELETE FROM lesson_access_log WHERE accessed_at > '<T0>'`.
3. Every run therefore measures its half-life against the SAME pre-sprint access-log state.

**Why not truncate.** Truncating destroys the Sprint 12.1c audit-bootstrap (90 rows) + accumulated legitimate consumption rows. R5-H only removes this-sprint's own writes.

**Why not snapshot/restore.** `pg_dump` of one table per run adds 4 dump/restore cycles — infra overhead for no benefit over R5-H.

**Why not timestamp-filter in salience code.** Adds a new `LESSONS_SALIENCE_LOG_CUTOFF` env var + code change in `src/services/salience.ts`. Scope creep; R5-H achieves the same isolation without touching code.

**Risk with R5-H:** if another process (user, background job) writes to the log during the sweep, those writes are also deleted. Mitigation: verify no other writers are active (only the dev-stack baseline runs + possibly the worker, which doesn't write to `lesson_access_log`). Log the `T0` timestamp + a final tally of total rows pre-delete vs post-delete so the reset is auditable.

**SQL helper:**
```bash
# One-time: capture T0
docker compose exec -T db psql -U contexthub -d contexthub -t -c \
  "SELECT MAX(accessed_at) FROM lesson_access_log;" | xargs -I{} echo "T0={}"
# Expected output: T0=<ISO timestamp>

# Per-run reset:
docker compose exec -T db psql -U contexthub -d contexthub -c \
  "DELETE FROM lesson_access_log WHERE accessed_at > '<T0>';"
```

## 2. Docker env-var dance — how to change half-life per run

**Current state.** `LESSONS_SALIENCE_HALF_LIFE_DAYS` is NOT listed in `docker-compose.yml`'s mcp service `environment:` block. It's only read via `src/env.ts` at process startup from `.env` (which doesn't set it) → falls back to default 7.

**Resolution: add ONE line to `docker-compose.yml`** for the mcp service's environment block:

```yaml
# Phase 12 Sprint 12.1e2 — expose for sweep-time overrides.
# Default stays at 7 per Sprint 12.1c.
LESSONS_SALIENCE_HALF_LIFE_DAYS: ${LESSONS_SALIENCE_HALF_LIFE_DAYS:-7}
LESSONS_SALIENCE_ALPHA: ${LESSONS_SALIENCE_ALPHA:-0.10}
LESSONS_SALIENCE_DISABLED: ${LESSONS_SALIENCE_DISABLED:-false}
```

With this, per-run command becomes:
```bash
LESSONS_SALIENCE_HALF_LIFE_DAYS=<N> docker compose up -d --force-recreate mcp
# Wait for ready
npm run qc:baseline -- --tag sprint-12.1e2-hl<N> --samples 3 --surfaces lessons
```

**This IS a code change (docker-compose.yml is infra-as-code).** Bumps sprint's logic-count to 1. Still XL by file count. The change is minimal and arguably should have been there from 12.1c — Phase 12's salience env knobs should be overridable at compose-time. Documented as 12.1e2 also addressing the 12.1c infra gap.

**Verification after each recreate:** `docker compose exec -T mcp node -e "console.log('HL:', process.env.LESSONS_SALIENCE_HALF_LIFE_DAYS)"` should print the expected value before the baseline run.

## Additional design decisions

### D-ADD-1 — Reuse of 12.1e1 noise_floor

The 12.1e1 archive's `noise_floor` field (`docs/qc/baselines/2026-04-19-sprint-12.1e1.json`) captures within-session jitter for the same goldenset + same system state. 12.1e2's 4 runs are sequential on the same goldenset — noise floor applies unchanged. **Do NOT re-establish noise floor per run.** This is why `--samples 3` without `--control` is sufficient (per spec D3).

### D-ADD-2 — Order of the 4 runs

Run order: `3d → 7d → 14d → 30d` (ascending). Rationale: each run is independent by R5-H (access log reset before each), so order shouldn't matter mechanically. Ascending reads cleanly in the summary table.

### D-ADD-3 — Salience `α` stays at 0.10

Per spec D1, α is fixed at 0.10 for the INITIAL sweep. If escalation is triggered and we switch to α axis, we'll document then.

### D-ADD-4 — What happens to the access log at sprint close

After all runs, the access log will have been reset to its pre-sprint state multiple times. The final state equals the pre-sprint state (no residual writes from this sprint). Future sprints start from a clean handoff.

### D-ADD-5 — Script automation

Given 4 sequential runs + reset + env change + verification per run, write a BUILD-time bash script `scripts/sprint-12.1e2-sweep.sh` that loops `{3, 7, 14, 30}` and:
1. DELETE access-log rows after T0
2. Recreate mcp container with new env
3. Verify env propagated
4. Run baseline
5. Tag archive

This is throwaway — used once per sprint. Do NOT commit the script if the sweep runs clean on the first try; if it's needed for troubleshooting, commit to `scripts/` but mark as sprint-scoped.

## Risks introduced at design time

**R-D1 — docker-compose change is infra.** One-line addition, but infra changes have higher blast radius than data changes. Mitigation: change is backward-compatible (default 7 preserved); verify the env var propagates BEFORE running baseline; if something goes wrong, revert is a one-line git-reset.

**R-D2 — access-log reset SQL is destructive.** `DELETE FROM lesson_access_log WHERE accessed_at > '<T0>'` could accidentally delete rows if T0 is mis-computed. Mitigation: T0 captured BEFORE any sweep action, logged to the plan + BUILD output; DELETE run against a read-only dev stack where no other writer is active; verify row-count delta before vs after each DELETE is bounded.

**R-D3 — summary doc format has a "fill in numbers" shape.** I could forget to fill a column or mis-paste. Mitigation: node script produces the tables programmatically from the 4 JSON files; I only hand-write the top-line recommendation + observations.
