## Part A — DEFERRED-009 tenant-scope authz

### Current state
`bearerAuth` sets `req.apiKeyScope = keyEntry.project_scope` (a project_id, or null =
global). The existing `requireScope(paramName)` middleware compares `apiKeyScope` to a
URL param **that is itself a project_id** — it does NOT do a DB lookup. Topic routes
have `:id` = a **topic_id**, so `requireScope` can't be reused directly.

### Proposed: `requireTopicScope(paramName='id')` middleware
A DB-lookup variant:
1. `attachedScope = req.apiKeyScope`. If `undefined` (auth-off / env-token) → `next()`
   (unrestricted — same fallback as `requireScope`, preserves the dev posture).
2. If `null` (global scope) → `next()`.
3. Load `SELECT project_id FROM topics WHERE topic_id = req.params[paramName]`.
4. If not found → **404 NOT_FOUND** (topic doesn't exist).
5. If `project_id !== attachedScope` → **404 NOT_FOUND** (NOT 403 — a cross-tenant
   topic_id must be indistinguishable from a non-existent one, to avoid id-probing /
   topic-existence disclosure across tenants).
6. Else → `next()`.

### Where applied (Q1 — minimal vs complete)
The `/api/topics/:id/*` routes where `:id` is the topic_id:
- topics.ts: `GET /:id`, `POST /:id/join`, `POST /:id/close`, `POST /:id/grant-level`,
  `GET /:id/events`, `GET /:id/stream`.
- The topic-nested families (board/requests/motions/disputes/intake) mounted under
  `/api/topics/:id/...` — `POST /topics/:id/requests`, `POST /topics/:id/motions`,
  `GET /topics/:id/requests`, etc.

**Routes keyed on a non-topic id** (`/api/requests/:id/steps/:n/decide`,
`/api/motions/:id/...`) reach the topic INDIRECTLY (via the request/motion row). These
are a **secondary surface**; Q1 decides whether 15.12 covers them (needs a derived
lookup) or documents them as a follow-up residual.

### Enforcement posture
Keyed on `req.apiKeyScope` → only meaningful **auth-on**. Auth-off (`apiKeyScope`
undefined) → unrestricted (the trusted single-operator dev posture, consistent with
15.11 Q2 + the existing `requireScope`). The MCP path (workspace_token, single shared
secret, no per-project scope) is governed by `assertWorkspaceToken` — out of scope for
tenant-isolation (a single-operator credential).

## Part B — DEFERRED-010 induction-pack tail mode

### Current state
`replayEvents({topic_id, since_seq})` returns events `WHERE seq > since_seq ORDER BY seq
ASC LIMIT 1000` + `has_more` + `next_cursor`. `joinTopic` calls it with `since_seq=0`
for a fresh joiner → the OLDEST 1000 events; on a large topic the joiner's own
`topic.actor_joined` (at the tail) is omitted, and `your_cursor` is the high-water of
that oldest-1000 prefix.

### Proposed: tail mode
Add `replayEvents({topic_id, tail: true})` (or a `tail_limit`): returns the most-recent
N events (`ORDER BY seq DESC LIMIT N`, re-sorted ASC) + `has_more` (true if older events
exist before the window) + `your_cursor` = max seq.

`joinTopic`'s induction pack uses tail mode so a fresh joiner ALWAYS gets recent context
including their own `topic.actor_joined`. For a small topic (< N events) tail mode
returns everything (no behavior change). For a large topic, the joiner gets recent
context + `has_more=true` (they can page BACKWARD for history if needed — a future
forward/backward pagination, not in 15.12).

**Coherence note:** the existing §9.8 cursor-continuation invariant said a fresh joiner
re-primes forward from `your_cursor`. With tail mode, `your_cursor` = max seq (fully
primed to HEAD); older history is reachable by an explicit backward replay. This is
strictly better for the re-prime use case (the deferred's stated problem).

## Acceptance Criteria

### A — tenant-scope (009)
- **AC1** — auth-on, a key scoped to project A calling `POST /api/topics/<B-topic>/close`
  → 404 NOT_FOUND (not 403, not 200).
- **AC2** — auth-on, a key scoped to A calling `GET /api/topics/<B-topic>` → 404.
- **AC3** — auth-on, a global-scope (null) key → allowed for any project's topic.
- **AC4** — auth-on, a key scoped to A on its OWN topic → allowed.
- **AC5** — auth-off (no apiKeyScope) → unrestricted (existing behavior preserved).
- **AC6** — unknown topic_id → 404 (indistinguishable from cross-tenant).
- **AC7** — applied to grant-level (the new 15.11 destructive-ish path) + join + close +
  get + events.
- **AC8** — Q1 disposition: the topic-nested route families (requests/motions/etc) are
  covered OR documented as a residual.

### B — induction-pack tail (010)
- **AC9** — `replayEvents({tail:true})` returns the most-recent N events, seq-ASC.
- **AC10** — a fresh `joinTopic` on a topic with > N events: the induction pack `events`
  includes the joiner's own `topic.actor_joined`; `your_cursor` = max seq; `has_more`
  = true.
- **AC11** — a `joinTopic` on a SMALL topic (< N events): pack contains all events
  (no behavior change; existing tests pass).
- **AC12** — backward compat: `replayEvents` without `tail` is unchanged (forward
  cursor from since_seq).

   shim, like the existing `x-test-key-role` / `x-test-key-name` pattern) to exercise
   the scoped path.

   recent-1000 alters what a fresh joiner sees first. Existing joinTopic tests assert
   on small topics (< 1000 events) where tail == full, so they're unaffected. Verify
   no test asserts the oldest-prefix behavior on a large topic (none exist — the cap
   is latent per DEFERRED-010).
