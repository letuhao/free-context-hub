## §0 — Honest scope

Sprint 15.5 builds two entry/routing patterns over the existing coordination primitives
(master design B.8): the **intake mailbox** (`intake_items`) and **dispute resolution**
(`disputes`). Neither adds a new primitive — both produce tasks / requests / motions / disputes
that route into already-shipped services.

**Not in scope:**
- DEFERRED-012 (topic `closing`-drain) — deferred to Sprint 15.6.
- DEFERRED-013 (distinct-endorser enforcement) — re-deferred to Sprint 15.6.
- DEFERRED-017/015/016 (authorization residuals) — HARD pre-production triggers unchanged.
- GUI (Sprint 15.6).

## §1 — Migration

**File:** `migrations/0058_intake_dispute.sql`

```sql
-- Intake mailbox: inbound channel for items belonging to no current task
CREATE TABLE intake_items (
  intake_id    UUID    PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id   TEXT    NOT NULL,
  topic_id     TEXT,                          -- null until triaged; FK topics(topic_id) optional
  kind         TEXT    NOT NULL
                 CHECK (kind IN ('violation_report','suggestion','request')),
  body         TEXT    NOT NULL,
  submitted_by TEXT    NOT NULL,
  status       TEXT    NOT NULL DEFAULT 'received'
                 CHECK (status IN ('received','triaged','dismissed')),
  routed_to    TEXT,                          -- ID of the task/request/motion/dispute it became
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX intake_items_project_id_idx ON intake_items (project_id, status);

-- Dispute resolution: a Request-Approval item routed to an arbiter or tribunal
CREATE TABLE disputes (
  dispute_id            UUID  PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id              TEXT  NOT NULL,
  subject_ref           TEXT  NOT NULL,
  parties               TEXT[] NOT NULL,
  status                TEXT  NOT NULL DEFAULT 'open'
                          CHECK (status IN ('open','under_resolution','resolved')),
  resolution_request_id UUID,               -- → requests(request_id)
  created_at            TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX disputes_topic_id_idx ON disputes (topic_id, status);
```

**No ALTER to existing tables.** `coordinationConstants.ts` is unchanged — Sprint 15.1
pre-provisioned `'intake'`/`'dispute'` subject types and `intake.received`/`intake.triaged`/
`dispute.opened`/`dispute.resolved` event types.

## §2 — Service API

### §2.1 `src/services/intake.ts`

```typescript
// All functions throw a typed AppError on validation failure.

submitIntake(params: {
  project_id: string;
  topic_id?: string;           // null = pre-triage, no topic yet
  kind: 'violation_report' | 'suggestion' | 'request';
  body: string;
  submitted_by: string;
}) → Promise<IntakeItem>
// • Validates: project exists; topic_id (if provided) exists + is active.
// • Inserts intake_items row.
// • Emits intake.received to topic_id (only if topic_id provided).
// • Returns the full intake_items row.

triageIntake(
  intake_id: string,
  route: TriageRoute
) → Promise<TriageResult>
// TriageRoute (discriminated union — WARN-1 fix: actor_id added to all variants):
//   | { route_kind: 'task' | 'request' | 'motion';
//       actor_id: string; topic_id: string; routed_to: string }
//   | { route_kind: 'dispute'; actor_id: string; topic_id: string; subject_ref: string;
//       parties: string[]; procedure: 'unilateral' | 'collective';
//       submitted_by: string; kind?: string; weight?: number }
//
// WARN-2 fix: begin with SELECT ... FOR UPDATE on intake item to serialize concurrent triage calls:
// • BEGIN
// • SELECT intake_items ... FOR UPDATE WHERE intake_id = ... AND status = 'received'
//     → NOT_FOUND or INTAKE_ALREADY_TRIAGED if status != 'received'
// • For 'dispute': calls openDispute(...) → { dispute_id, resolution_request_id };
//     routed_to = dispute_id.
// • For others: routed_to = route.routed_to (caller-supplied existing ID).
// • UPDATE intake_items SET status='triaged', topic_id=..., routed_to=... WHERE intake_id=...
// • COMMIT (releases the FOR UPDATE lock)
// • Emits intake.triaged to topic_id (outside transaction, consistent with other services).
// • TriageResult = { intake_id, status, routed_to } | { ..., dispute_id, resolution_request_id }

dismissIntake(intake_id: string) → Promise<IntakeItem>
// • Checks status === 'received'.
// • UPDATE status='dismissed'.
// • No event emitted.

getIntake(intake_id: string) → Promise<IntakeItem>
// Simple SELECT; NOT_FOUND if missing.

listIntake(
  project_id: string,
  opts?: { kind?: string; status?: string; limit?: number; offset?: number }
) → Promise<{ items: IntakeItem[]; total: number }>
```

**Atomicity note:** `triageIntake` for `'dispute'` is sequential: `openDispute` first (its own
transaction), then the `intake_items` UPDATE. If the UPDATE fails after `openDispute` succeeds,
the intake item remains `'received'` but the dispute + resolution request exist. The coordinator
can re-triage by calling `triageIntake` again with `route_kind: 'task'` and `routed_to: <dispute_id>`
(link-only) — no duplicate dispute is created. This partial-failure window is acceptable for Sprint 15.5.

### §2.2 `src/services/disputes.ts`

```typescript
openDispute(params: {
  topic_id: string;
  subject_ref: string;          // free-text ref to the disputed artifact/task/request
  parties: string[];            // actor_ids of disputing parties (≥2)
  procedure: 'unilateral' | 'collective';
  submitted_by: string;         // the actor opening the dispute (becomes request submitter)
  kind?: string;                // DoA matrix lookup kind (default: 'dispute_resolution')
  weight?: number;              // DoA matrix lookup weight (default: 1)
}) → Promise<{ dispute: Dispute; resolution_request_id: string }>
// • Validates: topic exists + status === 'active'; parties.length >= 2.
// • INSERT disputes (status='open', resolution_request_id=null).
// • Calls submitRequest({ topic_id, subject_type:'dispute', subject_id: dispute_id,
//     procedure, submitted_by, kind: params.kind ?? 'dispute_resolution',
//     weight: params.weight ?? 1 }).
// • UPDATE disputes SET resolution_request_id = <request_id>.
// • Emits dispute.opened.
// • Returns { dispute, resolution_request_id }.

resolveDispute(dispute_id: string) → Promise<Dispute>
// • SELECT ... FOR UPDATE — locks dispute row.
// • Checks status !== 'resolved' (409 ALREADY_RESOLVED if it is).
// • SELECT requests WHERE request_id = resolution_request_id.
// • Checks request.status IN ('approved','returned','rejected') — else 409 RESOLUTION_PENDING.
// • UPDATE disputes SET status='resolved'.
// • Emits dispute.resolved.
// • Returns updated dispute row.

getDispute(dispute_id: string) → Promise<DisputeDetail>
// DisputeDetail = disputes row + { resolution_request: { request_id, status, steps: RequestStep[] } }
// Steps ordered by step_index.

listDisputes(
  topic_id: string,
  opts?: { status?: string; limit?: number; offset?: number }
) → Promise<{ disputes: Dispute[]; total: number }>
```

**`under_resolution` state (D5):** Not used in Sprint 15.5. The schema column exists; transition
logic is deferred. `open → resolved` is the effective lifecycle for this sprint. The `under_resolution`
value remains in the CHECK constraint for future use.

**`openDispute` DoA matrix:** `kind` defaults to `'dispute_resolution'`. The migration 0056 seed
has a `__default__` row that catches any unrecognized kind — a dispute resolution request falls
back to `escalate_to_authority`. Project operators can override by inserting a `dispute_resolution`
row into `doa_matrix`.

## §3 — REST routes

### `src/api/routes/intake.ts` — `intakeRouter`

```
POST   /api/intake                 → submitIntake          200 { intake }
GET    /api/intake/:id             → getIntake             200 { intake } | 404
POST   /api/intake/:id/triage      → triageIntake          200 { intake, dispute? } | 409
POST   /api/intake/:id/dismiss     → dismissIntake         200 { intake } | 409
GET    /api/projects/:id/intake    → listIntake            200 { items, total }
```

Role gates: all endpoints require `requireRole('reader')` for GET, `requireRole('writer')` for POST.

Error codes:
- `INTAKE_NOT_FOUND` 404
- `INTAKE_ALREADY_TRIAGED` 409 — triage called on a non-received item
- `INTAKE_ALREADY_DISMISSED` 409 — dismiss called on a non-received item
- `PROJECT_NOT_FOUND` 404 — project does not exist
- `TOPIC_NOT_ACTIVE` 409 — topic_id provided but not active

### `src/api/routes/disputes.ts` — `disputesRouter`

```
POST   /api/disputes               → openDispute           200 { dispute, resolution_request_id }
GET    /api/disputes/:id           → getDispute            200 { dispute, resolution_request } | 404
POST   /api/disputes/:id/resolve   → resolveDispute        200 { dispute } | 409
GET    /api/topics/:id/disputes    → listDisputes          200 { disputes, total }
```

Role gates: GET requires `requireRole('reader')`; POST requires `requireRole('writer')`.

Error codes:
- `DISPUTE_NOT_FOUND` 404
- `ALREADY_RESOLVED` 409 — resolve called on already-resolved dispute
- `RESOLUTION_PENDING` 409 — resolution request is not yet terminal
- `TOPIC_NOT_ACTIVE` 409 — topic not active when opening

| Event type | Subject type | Subject ID | Payload |
|---|---|---|---|
| `intake.received` | `intake` | intake_id | `{ project_id, kind, submitted_by }` |

## §6 — Design decisions

| ID | Decision |
|----|---------|
| D1 | Triage for `task/request/motion`: link-only (caller passes existing `routed_to` ID). Triage for `dispute`: create-on-triage (sequential — `openDispute` first, then update intake). |
| D2 | Dispute `under_resolution` state exists in schema but is not actively used in Sprint 15.5. Effective lifecycle: `open → resolved`. |
| D3 | `resolveDispute` precondition: resolution request status must be in `{'approved','returned','rejected'}`. `'returned'` counts as resolved — the dispute is settled; re-escalation requires a new dispute. |
| D4 | `openDispute` uses `kind='dispute_resolution'` (or caller-supplied) for DoA matrix lookup. Falls through to `__default__` → `escalate_to_authority`. No migration-0056 seed change needed. |
| D5 | No auto-advance of dispute status from REST decide route. `maybeAdvanceDisputeStatus` is not implemented — avoids coupling `routes/requests.ts` to `disputes.ts`. `under_resolution` is a future-sprint concern. |
| D6 | `intake.received` is NOT emitted when `topic_id` is null at submission — coordination events are always topic-scoped. The event is emitted at triage time (`intake.triaged`) when a topic is assigned. |
| D7 | `parties` minimum length: 2. Validated in `openDispute`. |
| D8 | `actor_id: string` added to all `TriageRoute` variants (WARN-1 fix). For link-only routes, `actor_id` is the triager's identity for the `intake.triaged` event. For dispute routes, `actor_id` = `submitted_by` (the dispute creator) may differ from the triager — both fields are required. |
| D9 | `triageIntake` uses `SELECT … FOR UPDATE` on the intake item at the start of a transaction to serialize concurrent triage calls (WARN-2 fix). The `openDispute` call happens within this transaction window. The transaction commits before `appendEvent` (same out-of-transaction pattern as all other services). |
| D10 | WARN-3 accepted: `dismissIntake` emits no event. Dismiss is an administrative action not in the master design C.3 event catalog. The `intake_items` row is the source of truth; no topic coordination is needed. |

## §7 — Lock table

New lock surfaces introduced by Sprint 15.5:

| Operation | Locks acquired (in order) |
|---|---|
| `openDispute` | INSERT `disputes` (no lock) → `submitRequest` (inherits 15.3 lock order: `request → request_step → topics via appendEvent`) |
| `resolveDispute` | `SELECT disputes FOR UPDATE` → `SELECT requests` (read-only) → `appendEvent` (topics advisory lock) |
| `triageIntake (dispute)` | `openDispute` locks (above) → `UPDATE intake_items` (PK update, no topic lock) → `appendEvent` (topics advisory lock) |

Full canonical order (updated): `task → claim → request → request_step → artifact → motion → vote → dispute → topics`

`disputes` joins the order before `topics` (appendEvent always last). No ABBA with the existing primitives — `resolveDispute` reads `requests` without locking them; `openDispute` creates a new `disputes` row (no concurrent lock contention on a new PK).
