## Scope

Resolve the **Phase 15 authorization model** ‚Äî the three interlocking HARD pre-prod
triggers:

- **DEFERRED-015** ‚Äî participant `level` is self-asserted at `joinTopic`. A caller
  joins as `authority` and then decides `authority`-target request steps. The
  officeholder check (`decideStep`: `topic_participants.level === target_office`) is
  only as trustworthy as a self-declared level.
- **DEFERRED-016** ‚Äî api-key multiplicity is unbounded. One operator who can mint keys
  obtains N distinct coordination identities and can drive a multi-level approval
  single-handed (each key = a distinct `actor_id` via 15.3.1 F1 token-binding).
- **DEFERRED-017** ‚Äî `createBody` / `addBodyMember` / `veto_holders` are ungated beyond
  the route layer's `requireRole('writer')`. Anyone with writer role mints a body with
  itself as sole weighted member + sole veto holder, and `castVote.proxy_for` is
  recorded but unverified.

All three are the **same subsystem**: who is authorized to hold a level, a credential,
and a vote. They were deferred together (DEFERRED-015 ¬ß) to be "built once as a coherent
piece."

## Current state (honest baseline)

| Layer | Mechanism | Gap |
|---|---|---|
| API-key auth | `bearerAuth` ‚Üí role (reader/writer/admin) via `api_keys` table OR env token (admin). Skipped when `MCP_AUTH_ENABLED=false`. | Dev runs auth-off ‚Üí no identity at all. |
| Identity binding | 15.3.1 F1 ‚Äî when auth-on, the acting `actor_id` is bound to `apiKeyName`. | One operator can mint N keys = N actor_ids (016). |
| Topic level | `joinTopic` inserts `topic_participants(level)` with whatever the caller passes. No gate. | Self-asserted (015). |
| Officeholder check | `decideStep` authorizes by `level === target_office` + `actor ‚âÝ submitted_by`. | Trusts the self-asserted level. |
| Body config | `createBody`/`addBodyMember` gated only by `requireRole('writer')`. | No "who may create/grant" rule (017). |
| Topic ownership | `charterTopic` records `created_by` but it confers no authority. | No formal owner role to bootstrap the level-grant chain. |

**The bootstrap problem:** if `joinTopic` no longer lets a caller self-assert
`authority`, WHO grants the first authority level? Without an answer, no topic can ever
have an authority. The natural answer: the **topic creator** (`charterTopic.created_by`)
is the topic **owner** and implicitly holds `authority` on their own topic ‚Äî the root
of the grant chain.

## Proposed authorization model

### A. Topic ownership + level-grant chain (DEFERRED-015)

1. **Owner = `charterTopic.created_by`.** The owner implicitly holds `authority` on
   their topic. (No migration needed if we treat `created_by` as owner; a
   `topics.owner_id` alias is optional clarity.)
2. **`joinTopic` defaults to `execution`.** A joiner can no longer self-assert
   `coordination`/`authority`. The `level` param is either removed or accepted only
   for the OWNER's own first join (which matches their implicit authority) ‚Äî TBD by Q1.
3. **`grantLevel(topic_id, actor_id, level, granted_by)`** ‚Äî a new operation:
   - Only the **owner** OR an **existing `authority`** participant may grant a level.
   - Sets/raises (or lowers) a participant's level; records `granted_by`; emits
     `topic.level_granted`.
   - **Self-grant forbidden** (an actor cannot raise their own level) ‚Äî except the
     owner, who is authority by charter.
4. **`decideStep` is unchanged** ‚Äî it already checks `level === target_office`. Now the
   level is authoritative because it was granted, not self-asserted.

### B. Decision-body authorization (DEFERRED-017)

1. **`createBody`** requires `admin` role (project-config operation) ‚Äî raised from
   `writer`. Bodies are project-scoped config (like `doa_matrix`), so project-admin is
   the right gate.
2. **`addBodyMember`** + **veto_holders management** require `admin` role.
3. **`castVote.proxy_for`**: either (a) verify the proxy grant exists (needs a
   `proxies` table ‚Äî bigger) or (b) keep `proxy_for` as audit-only metadata and
   document that it does NOT confer voting power (the principal's own vote is what
   counts) ‚Äî TBD by Q3.
4. **veto_holders length cap** (15.4 REVIEW-CODE LOW-3) ‚Äî bound array length +
   element length. (Small, fold in.)

### C. API-key multiplicity (DEFERRED-016)

The honest framing: **whoever can mint api keys is the trust root.** A single admin
operator minting N keys is not an attack ‚Äî it's the trust model. The real residual is:
can a NON-admin obtain multiple coordination identities? Fix:
1. **Actor-identity uniqueness**: one `actor_id` maps to exactly one `apiKeyName`
   (already bound by 15.3.1 F1). Add: an `actor_id` cannot be claimed by two different
   key names within a project (a uniqueness guard at actor-registration / decide time).
2. **Document** that key-minting is an `admin` capability and the multi-key-per-operator
   residual is accepted under the admin-trust-root model (the same operator who can mint
   keys can already do anything).
3. (Optional, Q4) a per-operator key limit ‚Äî a provisioning-side rule. Bigger; may
   defer the hard limit and ship only uniqueness + documentation.

## Enforcement posture (the crux ‚Äî Q2)

The current dev posture is `MCP_AUTH_ENABLED=false` ‚Üí no identity ‚Üí no enforcement
possible for api-key-keyed rules. But the **level-grant chain (A)** is keyed on
`actor_id` (a coordination-layer concept), which exists even auth-off.

- **Level-grant (A)**: enforce ALWAYS (auth-on and auth-off) ‚Äî it's keyed on actor_id,
  present in every coordination call. This makes the model sound regardless of auth.
- **Body authz (B)**: gated by `requireRole('admin')` ‚Äî only meaningful auth-on
  (auth-off has no role, `requireRole` allows all). Accept: auth-off is the trusted
  single-operator dev posture; the gate activates with auth.
- **Key multiplicity (C)**: only meaningful auth-on. Same acceptance.

So: **A is always-on; B + C activate with auth.** This resolves all three triggers (the
model is SOUND when auth is turned on) while preserving the auth-off dev experience for
A only where it's cheap.

## Acceptance criteria (proposed ‚Äî refine post-Q)

- **AC1** ‚Äî `joinTopic` defaults a new participant to `execution` (per Q1 disposition
  for the owner's first join).
- **AC2** ‚Äî The topic owner (`created_by`) is treated as `authority` for grant-chain
  purposes even without an explicit participant row at that level (or: their first
  join is auto-`authority`).
- **AC3** ‚Äî `grantLevel` by the owner succeeds; sets the target participant's level;
  emits `topic.level_granted` with `granted_by`.
- **AC4** ‚Äî `grantLevel` by an existing `authority` participant succeeds.
- **AC5** ‚Äî `grantLevel` by a `coordination`/`execution` participant ‚Üí rejected
  (`not_authorized`).
- **AC6** ‚Äî Self-grant (actor raising own level) ‚Üí rejected (`self_grant_forbidden`),
  except the owner.
- **AC7** ‚Äî `decideStep` on an `authority`-target step by an actor who SELF-asserted
  authority at join (pre-15.11 behavior) is now impossible ‚Äî joinTopic no longer
  accepts a non-execution self-assertion.
- **AC8** ‚Äî `createBody` requires `admin` role (403 for writer when auth-on).
- **AC9** ‚Äî `addBodyMember` requires `admin` role.
- **AC10** ‚Äî `veto_holders` array length + element length capped (LOW-3).
- **AC11** ‚Äî Actor-identity uniqueness: an `actor_id` already bound to key-name X cannot
  act under key-name Y (auth-on). (Per Q4 disposition.)
- **AC12** ‚Äî Backward compat: existing topics/participants (pre-15.11) ‚Äî migration
  backfills `granted_by` (e.g., to `created_by` or a sentinel); existing levels are
  preserved (not retroactively reset to execution).
- **AC13** ‚Äî Security scenario suite: a hostile actor cannot (a) self-grant authority,
  (b) decide a step above their granted level, (c) create a body to rubber-stamp their
  own request, (d) drive a multi-level approval via multiple identities.

## Open Questions

**Q1 ‚Äî joinTopic level param disposition:**
- (a) Remove the `level` param entirely; ALL joiners (including owner) default to
  `execution`; the owner is recognized as authority via `created_by` separately;
  owner must `grantLevel` themselves up (or is implicitly authority without a row).
- (b) Keep `level` but honor it ONLY when `actor_id === created_by` (owner's first
  join sets their own authority); all others forced to `execution`. **(recommended ‚Äî
  smoother bootstrap)**
- (c) Keep `level`, default `execution`, ignore any higher self-assertion silently
  (vs rejecting). Less honest (silent downgrade).

**Q2 ‚Äî Enforcement posture:**
- (a) **Level-grant always-on; body authz + key uniqueness activate with
  MCP_AUTH_ENABLED=true** (recommended ‚Äî sound when auth-on, cheap always-on for A).
- (b) Everything activates only with auth-on (simpler, but level self-assertion remains
  exploitable in any future auth-off multi-actor run ‚Äî weaker).

**Q3 ‚Äî proxy_for verification (017 sub-item):**
- (a) Keep `proxy_for` as audit-only metadata; document it does NOT confer voting
  power (the principal row is what's counted ‚Äî already true per 15.4 tally). No
  `proxies` table. **(recommended ‚Äî the tally already ignores proxy_for)**
- (b) Build a `proxies` grant table + verification (bigger; its own design).

**Q4 ‚Äî API-key multiplicity (016) depth:**
- (a) Ship actor-identity uniqueness (one actor_id ‚Üî one key-name) + document
  key-minting as admin trust-root. Defer a hard per-operator key limit.
  **(recommended ‚Äî closes the forgery vector; the hard limit is a provisioning policy)**
- (b) Also implement a per-operator key-count limit (provisioning-side; bigger surface,
  touches the api-keys admin endpoints).

**Q5 ‚Äî Sprint scope realism:**
- (a) Ship all three (015 + 016-per-Q4a + 017) this sprint as the coherent model.
  **(recommended ‚Äî they interlock; the user chose all three)**
- (b) Split: ship 015 (level-grant) + 017 (body authz) now; defer 016 (key
  multiplicity) ‚Äî if 016 proves to need the provisioning-endpoint surface.

## Risks

1. **Bootstrap chicken-and-egg**: resolved by owner = created_by = implicit authority.
   But every existing test that calls `joinTopic(level: 'authority')` for a
   non-owner will break. Migration + test updates needed (AC12).

2. **Massive test surface**: requests/motions/board/topics tests all rely on
   self-asserted levels in their setup. Most need a `grantLevel` step inserted, OR the
   test helper `mkTopicWithParticipants` must be updated to grant levels via the owner.
   This is the bulk of the work.

3. **Security review cost**: a security-framed cold-start adversarial review at
   POST-REVIEW (guardrail 5c0b7b25) ‚Äî ~30 min + sub-agent tokens. Mandatory for this
   sprint (it adds the authz primitive).

4. **Scope creep into provisioning**: 016's hard key limit pulls in the api-keys admin
   surface (DEFERRED-004 territory). Q4a bounds this out.

## Decisions (2026-05-21)

- **Q1 ‚Äî Honor level only for owner's first join.** joinTopic keeps the `level`
  param but honors it ONLY when `actor_id === created_by`; all other joiners are
  forced to `execution`. The owner's first join may set `authority` (bootstrap).
- **Q2 ‚Äî Level-grant always-on; body authz + key rules activate with auth-on.**
- **Q3 ‚Äî Build a `proxies` grant table + verification (EXPANDED).** A real proxy
  grant (`proxies(topic_id|body_id, principal, proxy, granted_at)`); `castVote`
  with `proxy_for` verifies the grant exists before recording the ballot on behalf
  of the principal. New design surface (table + grant/revoke + verification).
- **Q4 ‚Äî Per-operator api-key-count limit (EXPANDED).** Beyond actor-identity
  uniqueness, add a provisioning-side per-operator key-count limit. Touches the
  api-keys admin surface (DEFERRED-004 territory).
- **Q5 (implied) ‚Äî Ship all three** (015 + 016-full + 017-full) as the coherent
  authorization model. Sprint size: **XL.**
