## 4. F1 — the identity-binding contract (the substantive analysis)

CLARIFY investigation of the two auth surfaces:

**REST** (`src/api/middleware/auth.ts`): under `MCP_AUTH_ENABLED=true`, a DB-issued api key
attaches `req.apiKeyName` — a per-caller authenticated identity. The env-var fast-path
(`CONTEXT_HUB_WORKSPACE_TOKEN` → admin) and auth-disabled mode set **no** `apiKeyName`. The
Phase-13 pattern is `reviewRequests.ts:reviewerIdentity()` — `apiKeyName` if present, else a
generic label.

**MCP** (`src/mcp/index.ts`): every request tool authenticates only via
`assertWorkspaceToken(workspace_token)` — a single shared secret. There is **no per-caller
identity** on the MCP transport. A caller holding the workspace token is a single trusted
principal — identical to REST's env-token admin.

**The threat F1 closes** (audit Finding 1): a single REST `writer`-role **DB key** forging
multiple `actor_id`/`submitted_by` strings to drive a multi-level approval single-handed. DB
keys are REST-only (`api_keys` table + `validateApiKey`), can be many, and are mutually
distrusting.

**F1 fix — REST** (`routes/requests.ts`): the acting identity (`submitted_by` for submit,
`actor_id` for decide) is bound to `req.apiKeyName`:

- `apiKeyName` **present** (DB-keyed caller) → acting identity = `apiKeyName`. A body
  `submitted_by` / `actor_id` naming a *different* actor → **403** (`identity_mismatch`)
  [open Q1]. A body value equal to `apiKeyName`, or omitted, is fine.
- `apiKeyName` **absent** (env-token admin OR `MCP_AUTH_ENABLED=false`) → the body value
  stands. This is a single-trusted-operator posture: the operator legitimately names which
  coordination actor it is acting as. Preserves the dev posture and every existing test.

**F1 fix — MCP: SCOPED OUT, by design.** The MCP transport has no per-caller identity to
bind to; a workspace-token holder is already a single trusted principal — the same trust
level as REST's env-admin, for whom F1 *also* lets the body stand. The MCP request tools
genuinely need the caller to name *which* coordination actor it acts as (`alice` vs `bob`
are coordination participants, not auth principals) — body-supplied identity is *necessary*
for the feature under the current MCP auth model. Real MCP per-caller identity is the
Phase 15 authorization model — the same surface as DEFERRED-009 / DEFERRED-015. 15.3.1 adds
a doc comment on the MCP request tools recording this; no behavioral change.

> **This MCP scope decision is the key item for the REVIEW-DESIGN Adversary to challenge.**
> The set of principals who can forge via MCP = holders of `CONTEXT_HUB_WORKSPACE_TOKEN` =
> the trusted super-admin set (who are already full admins on REST via the env fast-path).
> The set F1 must constrain = holders of DB `writer` keys = REST-only. So the REST-only fix
> is *complete* for the stated threat model — but the Adversary should verify that claim.

## 6. Acceptance criteria

- **AC1** — F1: with `MCP_AUTH_ENABLED=true` + a DB `writer` key, `submitRequest`'s
  `submitted_by` and `decideStep`'s `actor_id` are forced to the key's `apiKeyName`; a body
  value naming a different actor → 403.
- **AC2** — F1: with auth disabled (or the env-token), the body `submitted_by` / `actor_id`
  stands — the dev posture and all existing 15.3 tests/smoke still pass.
- **AC3** — F3a: `submitRequest` rejects an artifact whose `topic_id` ≠ the request's topic
  (status per open Q2).
- **AC4** — F3a: `resolveArtifact` emits `artifact.versioned` / `artifact.state_changed` on
  the artifact's own (derived) topic, not a passed-in topic.
- **AC5** — F4: `GET /api/topics/:id/requests` and `GET /api/requests/:id` require ≥ `reader`
  (403 for an under-rolled key; pass when auth disabled).
- **AC6** — F5: `decideStep` rejects a non-integer or negative `step_index` → `BAD_REQUEST`.
- **AC7** — F7: `submitRequest` rejects a `kind` or `subject_id` longer than 256 chars →
  `BAD_REQUEST`.
- **AC8** — regression: `tsc` exit 0; full `npm test` green (≥ 414 + the new tests); the live
  smoke of the 14 prior 15.3 paths still passes.
- **AC9** *(reworded — REVIEW-DESIGN r1 Finding 2; see DESIGN rev 2 §0.5/§10)* — the
  cold-start security re-review confirms F1 closes the **body-string identity-forgery
  vector for the `MCP_AUTH_ENABLED=true` DB-key path** (a DB-keyed caller cannot forge a
  second actor by editing the request body) and F3a closes the cross-topic artifact
  scenario. The re-review **explicitly acknowledges** the forgery remains reachable under
  `MCP_AUTH_ENABLED=false` (no auth — dev posture) and via the deferred F2 self-declared
  `level` (DEFERRED-015) — these are accepted scope boundaries, not regressions. The full
  audit CRITICAL is closed only by F1 + F2 together.

## 7. Open questions for DESIGN

- **Q1 — F1 mismatch handling.** `apiKeyName` present and a *different* body
  `submitted_by`/`actor_id` present → 403 `identity_mismatch`, or silently override with
  `apiKeyName`? *Lean: 403.* The audit fix text says "derived from **or verified against**
  the token"; 403 = verified, explicit, debuggable, no silent surprise.
- **Q2 — F3a reject status.** Cross-topic artifact → reuse `NOT_FOUND` ("artifact not found

  requires a non-empty body `submitted_by`/`actor_id` (today's behavior — `submitRequest`
