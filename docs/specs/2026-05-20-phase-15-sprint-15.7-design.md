## §0 Invariants & contracts (carried from Phase 15)

- §0.1-loop: each per-item sweep operation runs in its own short transaction with
  try/catch that logs and continues. One bad item never aborts the cycle.
- §0.2 lock order (canonical): `task → claim → artifact → request → request_step →
  motion → topics`. The chaining additions extend this — see §3.
- §0.3 event log: `appendEvent` rejects on topic `status='closed'` (the seal). Accepts on
  `'active'` and `'closing'`. The chain handler must serialize with `closeTopic` Phase 3 to
  avoid emitting after seal.
- §0.4 closing-window writer-block (15.6): `postTask`, `submitRequest`, `proposeMotion`,
  `claimTask` reject on `topic.status='closing'`. **The chaining handler bypasses this
  block at chain-time because the source event has already been authorized — chaining
  emits `task.deferred` instead of postTask on a closing/closed topic.**

## §2 New service module — `src/services/chaining.ts`

Public API:

```ts
// Pure helper — structural validation at submit time.
// Throws ContextHubError('BAD_REQUEST', ...) on shape violation.
export function validateExecutionTask(blob: unknown): ExecutionTaskBlob | null;

export type ExecutionTaskBlob = {
  title?: string;     // ≤512 chars
  topology?: 'parallel' | 'sequential' | 'rolling';
  slot?: string;      // SLOT_REGEX, ≤64 chars
  kind?: string;      // non-empty, ≤64 chars
  depends_on?: string[];  // each UUID
  raci?: Record<string, unknown>;  // recorded as-is, ≤8 KB JSON
};

// Pure helper — merge blob over derived defaults to produce postTask params.
// No DB access.
export function buildChainedTaskParams(args: {
  source: 'request' | 'motion';
  source_id: string;
  topic_id: string;
  kind: string;       // request.kind or motion.subject_type
  blob: ExecutionTaskBlob | null;
  acting_actor: string;  // resolving actor's id or 'system'
}): PostTaskParams;

export type PostTaskParams = {
  topic_id: string;
  title: string;
  topology: 'parallel' | 'sequential' | 'rolling';
  slot: string;
  kind: string;
  depends_on: string[];
  raci: Record<string, unknown>;
  created_by: string;
};

// Transactional helper — invoked inside the source-event txn.
// Locks topic.status FOR UPDATE, branches to inline-postTask or task.deferred.
// Returns chain result for the caller to merge into source-event payload.
export async function emitChain(
  client: PoolClient,
  args: {
    topic_id: string;
    source_event: { type: 'request.resolved' | 'motion.tallied'; source_id: string };
    params: PostTaskParams;
  }
): Promise<ChainResult>;

export type ChainResult =
  | { kind: 'posted'; task_id: string; artifact_id: string }
  | { kind: 'deferred'; reason: 'topic_closing' | 'topic_closed' };
  // rev 2 (F2): invalid_depends_on is no longer a 'deferred' kind — it throws
  // ContextHubError so the source transaction rolls back. See §2 flow step 2.
```

**`emitChain` transactional flow (rev 2 — F1+F2 fixes):**

```
1. SELECT status FROM topics WHERE topic_id=$1 FOR UPDATE
   → if 'closed':  appendEvent task.deferred (reason='topic_closed') + return
                   {kind:'deferred', reason:'topic_closed'}.
                   Note: appendEvent rejects on closed — see §6.2; caller-side
                   ROLLBACK handles this race correctly.
   → if 'closing': appendEvent task.deferred (reason='topic_closing') + return
                   {kind:'deferred', reason:'topic_closing'}.
   → if 'active': continue to step 2.

2. Validate depends_on at chain time:
   SELECT count(*) FROM tasks WHERE task_id = ANY($1::uuid[]) AND topic_id = $2
   → if count < depends_on.length:
       *** rev 2 (F2 fix) *** — THROW ContextHubError('CHAINED_TASK_DEPENDENCY_INVALID', ...)
       The caller's outer try/catch ROLLBACKs the entire transaction, undoing the
       source-event UPDATEs. The source resolution is NOT recorded. The caller
       returns status='chained_task_dependency_invalid' to its caller (matches
       CLARIFY AC10). No task.deferred event is emitted for this case — the
       failure is not a "deferral" (race), it is a submitter blob bug.

3. INSERT tasks row (status='posted')
   INSERT artifacts row (state='draft', version=1)
   INSERT artifact_versions row (version=1, state='draft')
   appendEvent task.posted
   appendEvent artifact.created
   Return {kind:'posted', task_id, artifact_id}.
```

**`task.deferred` event payload (new in rev 2 — F1 fix):**
```jsonc
{
  "source_event_type": "request.resolved" | "motion.tallied",
  "source_id": "<request_id or motion_id>",
  "reason": "topic_closing" | "topic_closed",
  "would_be_task": {
    "title": "...",
    "topology": "parallel" | "sequential" | "rolling",
    "slot": "...",
    "kind": "...",
    "depends_on": ["..."],
    "raci": {...}
  }
}
```

**`subject_type='topic'`, `subject_id=topic_id`** *(rev 2 refinement — initial draft used
`subject_type='task'` with a fresh phantom UUID; revised because no real task exists, and
a topic-scoped event-type filter ("events about this topic") cleanly includes the
deferral)*. `actor_id` = the resolving actor (or `'system:tally'` / `'system:sweep'` for
auto-resolution paths).

**Race-safety:** the topic FOR UPDATE serializes against `closeTopic` Phase 1 (which
acquires the same lock to flip active→closing) and Phase 3 (which acquires it to flip
closing→closed). Phase 2 (drain) holds no topic lock. So at the FOR UPDATE point we see a
status that is stable for the duration of our transaction.

## §3 Touchpoint changes

### §3.1 `decideStep` (src/services/requests.ts) — approve branch

Current order: `request → request_step → topics (plain) → request_step UPDATE → request
UPDATE → resolveArtifact → appendEvent request.resolved → COMMIT`.

New order (approve, the only chaining branch):
```
... existing ...
→ resolveArtifact (existing)
→ NEW: chainResult = await emitChain(client, {topic_id, source_event: {type:'request.resolved', source_id: request_id}, params: buildChainedTaskParams({source:'request', source_id:request_id, topic_id, kind: req.kind, blob: req.execution_task, acting_actor: actorId})})
→ appendEvent request.resolved with payload {outcome:'approved', artifact_advanced, chain: chainResult}
→ COMMIT
```

Lock order extension: the chain handler's `topics FOR UPDATE` is acquired AFTER
`request_step` lock and AFTER the source `artifact` write — fits within the `request →
request_step → artifact → topics` canonical order (topics last). No new lock-order cycle.

Return shape change:
```ts
type DecideResult =
  | { status: 'approved', chain: ChainResult }
  | ...other cases...
```

Other branches (return, reject) — no chaining; chain field absent.

### §3.2 `tallyMotion` (src/services/motions.ts) — carried branch

Current order: `motion → topics (plain) → decision_bodies plain → tally → motion UPDATE
→ appendEvent motion.tallied → COMMIT`.

New order: same up to motion.tallied, but BEFORE appendEvent:
```
→ tally + UPDATE motion (existing)
→ if outcome === 'carried':
    chainResult = await emitChain(client, {topic_id: motion.topic_id, source_event: {type:'motion.tallied', source_id: motionId}, params: buildChainedTaskParams({source:'motion', source_id:motionId, topic_id, kind: motion.subject_type, blob: motion.execution_task, acting_actor: 'system:tally'})})
→ appendEvent motion.tallied with payload {outcome, ...tally, chain: chainResult?}
→ COMMIT
```

Other outcomes (failed/lapsed) — no chain, chain field absent from payload.

Note: `tallyMotion` currently uses `actor_id: 'system:tally'` in its appendEvent (line
705). The chain handler will reuse `'system:tally'` as `acting_actor` for symmetry.

### §3.3 `sweepExpiredMotions` (src/services/coordinationSweep.ts) — carried branch

The auto-tally path (line ~482) currently emits motion.tallied. Same chaining logic as
tallyMotion, but `acting_actor='system:sweep'`. Per-motion §0.1-loop transaction; a
chain failure logs+continues to next motion.

The `not_seconded` lapsed path (~461) — outcome='lapsed' — does NOT chain.

### §3.4 `claimTask` topology enforcement (src/services/board.ts)

Current flow at line 358-426: lock task, check topic status, lazy-cleanup expired
claims, live-claim check, INSERT claim, UPDATE task, emit claim.granted.

New step inserted **after** the topic-status check (line 382) and **before** the
lazy-cleanup (line 392):

```
→ Read tasks.topology + tasks.depends_on for THIS task (already locked at FOR UPDATE).
→ If topology === 'sequential' AND depends_on.length > 0:
    SELECT task_id, status FROM tasks WHERE task_id = ANY($1::uuid[])
    Reject with UNMET_DEPENDENCIES if any row has status != 'completed' or row missing.
    Error payload: {missing: [task_ids], incomplete: [{task_id, status}]}.
→ If topology === 'rolling' AND depends_on.length > 0:
    SELECT a.task_id, a.state FROM artifacts a WHERE a.task_id = ANY($1::uuid[])
    Reject with UPSTREAM_NOT_BASELINED if any row has state != 'baselined' or row missing.
    Error payload: {not_baselined: [{task_id, state}]}.
→ If topology === 'parallel' OR depends_on.length === 0: skip (no check).
```

Lock order extension: the dependency tables (`tasks`, `artifacts`) are queried with plain
SELECT (no FOR UPDATE). They sit at the front of the canonical order (`task → claim →
artifact → ...`); a plain read of preceding-order rows is safe inside any later transaction.

Return shape extends `ClaimResult`:
```ts
| { status: 'unmet_dependencies', missing?: string[], incomplete?: {task_id, status}[] }
| { status: 'upstream_not_baselined', not_baselined: {task_id, state}[] }
```

### §3.5 `sweepStuckClosingTopics` (new, src/services/coordinationSweep.ts)

Query — pure SQL, no service-layer joins:
```sql
SELECT t.topic_id, max_ce.most_recent_closing_at
  FROM topics t
  JOIN LATERAL (
    SELECT max(created_at) AS most_recent_closing_at
      FROM coordination_events
     WHERE topic_id = t.topic_id AND type = 'topic.closing'
  ) max_ce ON true
 WHERE t.status = 'closing'
   AND max_ce.most_recent_closing_at < now() - interval '5 minutes';
```

Per-topic recovery (§0.1-loop) — **rev 2 (F3 fix): per-call statement_timeout cap**:
```
for each topic_id in scan:
  try:
    // F3 fix — bound each closeTopic call so a single slow topic cannot block
    // the shared advisory-lock hold (which also gates the 3 other sweeps).
    // closeTopic accepts statementTimeoutMs; sets `SET statement_timeout` on
    // every per-phase / per-item connection it allocates internally.
    await closeTopic({
      topic_id,
      actor_id: 'system:closing-recovery',
      statementTimeoutMs: 60_000,  // rev 2 refinement
    });
  except (err):
    if err is statement_timeout error (pg code '57014'):
      logger.warn({event:'stuck_closing_timeout', topic_id}, 'closeTopic recovery exceeded 60s; will retry next cycle');
    else:
      logger.error({event:'stuck_closing_failed', err, topic_id}, '...');
    continue;
```

**Note on `closeTopic` signature change** *(rev 2 refinement — initial draft passed an
external `PoolClient`; revised to a simple optional `statementTimeoutMs` param because
closeTopic's 3-phase internal structure already allocates fresh connections per phase
and per-item — a single per-call timeout config is cleaner than threading a borrowed
client through all phases)*:

```ts
// rev 2 signature
export async function closeTopic(params: {
  topic_id: string;
  actor_id: string;
  statementTimeoutMs?: number;  // NEW — if set, apply `SET statement_timeout = '<ms>ms'`
                                // on every internal connection right after pool.connect().
}): Promise<CloseResult>
```

When `statementTimeoutMs` is undefined: behavior unchanged for all existing 15.6 callers.
When set: every internal `pool.connect()` is followed by a `SET statement_timeout`
statement. A timed-out query throws pg error code `57014` which the §0.1 inner try/catch
already catches and continues from — Phase 2 drain skips that item; Phase 3 seal may or
may not run depending on which phase timed out. The sweep's caller then logs a `warn`
for the timed-out topic, and the next sweep cycle re-attempts.

Scheduler integration — added 4th in `startClaimsSweepScheduler` cycle, after the
3 existing sweeps:
```
const stuck = await _sweepStuckClosingTopics();
logger.info({event: 'stuck_closing_sweep_done', recovered: stuck.recovered}, '...');
```

`SWEEP_STALE_CLOSING_MINUTES = 5` — fixed constant.
`SWEEP_STUCK_CLOSING_STATEMENT_TIMEOUT_S = 60` — fixed. Future-tunability deferred.

## §4 Error codes & MCP mapping

`src/core/errors.ts` — extend `ContextHubError.code` union with:
- `UNMET_DEPENDENCIES` → HTTP 400, MCP `invalid_request`
- `UPSTREAM_NOT_BASELINED` → HTTP 400, MCP `invalid_request`
- `CHAINED_TASK_DEPENDENCY_INVALID` → HTTP 400, MCP `invalid_request` *(rev 2 F2)*

Service-level result status codes (not thrown errors):
- `claimTask` → `'unmet_dependencies'`, `'upstream_not_baselined'`
- `decideStep` (approve branch) → `'chained_task_dependency_invalid'` *(rev 2 F2 —
  source rolls back, request remains 'open')*
- `tallyMotion` (carried branch) → `'chained_task_dependency_invalid'` *(rev 2 F2 —
  source rolls back, motion remains 'balloting')*

These are sibling cases to the existing `'topic_closed'`, `'conflict'`. The route layer
maps them to HTTP 400 (consistent with 15.2 conventions).

The chain `kind: 'deferred'` is NOT an error — it's a normal result on a closing/closed
topic. Mapped into the source-event payload's `chain.kind` field AND emitted as a
separate `task.deferred` event *(rev 2 F1)*.

## §5 Event payloads (rev 2 — F1 fix: dual-emit on deferral)

`request.resolved` payload extends:
```jsonc
{
  "outcome": "approved" | "returned" | "rejected" | "escalation_exhausted",
  "artifact_advanced": true | false,
  // NEW — only present on `outcome='approved'`:
  "chain": {
    "kind": "posted",
    "task_id": "<uuid>",
    "artifact_id": "<derived-id>"
  } | {
    "kind": "deferred",
    "reason": "topic_closing" | "topic_closed",
    "deferred_event_id": "<uuid of the task.deferred event for cross-ref>"
  }
}
```
(`invalid_depends_on` is no longer a deferred kind — F2 fix means the source event
rolls back; the chain field is never reached in that case.)

`motion.tallied` payload extends — same `chain` field, only on `outcome='carried'`.

**`task.deferred` event** is emitted alongside the source event whenever the chain
result is `kind:'deferred'`. Shape per §2:
```jsonc
{
  "topic_id": "<source topic>",
  "actor_id": "<resolving actor or system:tally|system:sweep>",
  "type": "task.deferred",
  "subject_type": "task",
  "subject_id": "<fresh uuid — the would-be-task handle>",
  "payload": {
    "source_event_type": "request.resolved" | "motion.tallied",
    "source_id": "<request_id or motion_id>",
    "reason": "topic_closing" | "topic_closed",
    "would_be_task": { ... postTask-shaped params ... }
  }
}
```

Rationale (rev 2): the master design §C.4 contract demands a discrete `task.deferred`
event in the sealed trail. The `chain` field in the source event remains for
caller-side convenience (a single read of `request.resolved` tells the caller "approved
and chain-deferred") AND the separate `task.deferred` event enables type-filterable
discovery (`WHERE type='task.deferred'`).

**Cross-reference:** `chain.deferred_event_id` in the source event = the `task.deferred`
event's `event_id`. A consumer reading either event can hop to the other.

## §6 Race analyses

### §6.1 Closing-during-chain race

Scenario: decideStep approves a request; topic is `'active'` at all prior reads. Between
the request-step UPDATE and the chain handler's `topics FOR UPDATE`, `closeTopic` Phase 1
flips status to `'closing'`.

Resolution: `topics FOR UPDATE` in chain handler blocks until Phase 1 commits → reads
status='closing' → emits task.deferred(reason='topic_closing'). Source event
`request.resolved` commits with chain.kind='deferred'. closeTopic Phase 2 drain may then
pick up the request (no — request is already 'approved'/terminal; sweep skips it).
closeTopic Phase 3 acquires the topics FOR UPDATE lock after we release on COMMIT and
seals.

### §6.2 Phase-3-during-chain race (rev 2 — clarified call order)

Scenario: chain handler's `topics FOR UPDATE` waits while closeTopic Phase 3 is sealing.
After Phase 3 commits (status='closed'), chain reads 'closed'. **The chain handler is
called BEFORE the source appendEvent** (per §3.1 the order is: build chain params →
emitChain → appendEvent source). So:

- Chain reads status='closed' under FOR UPDATE.
- Chain calls appendEvent for task.deferred → rejects on closed.
- Caller's outer try/catch sees the error, ROLLBACK, returns `topic_closed` status to
  the caller.

Net: the source event never commits. The request remains 'open'. The decider's call
returns `topic_closed`. The decider observes the topic is sealed and acts accordingly.

This is the only design path where the chain handler "fails" by appendEvent rejection
— and it's the right behavior: a closed topic refuses ALL new events, including
task.deferred. Symmetric with the existing appendEvent-rejects-on-closed behavior.

### §6.3 Concurrent decideStep + closeTopic Phase 2 drain

Scenario: decideStep holds request row lock. closeTopic Phase 2 tries to UPDATE the
same request to status='rejected' (force-lapse).

Resolution: Phase 2's UPDATE blocks on the row lock. decideStep proceeds to chain (which
takes topics FOR UPDATE). No deadlock — Phase 2 doesn't hold topics. decideStep commits.
Phase 2 unblocks, sees status now 'approved' (not 'open'), the per-row UPDATE clause
`WHERE status='open'` filters it out (0 rows updated). Phase 2 logs and continues per
§0.1-loop. No double-write.

### §6.4 claimTask topology check race

Scenario: claimTask reads task A's `depends_on` includes task B. Task B is `status=
'in_progress'` at the read. Between the read and the claim INSERT, task B is completed
by another actor (completeTask).

Result: our topology check rejects with UNMET_DEPENDENCIES even though B is now
completed. The caller retries claimTask; the second attempt sees B as 'completed' and
succeeds. Eventually-consistent and acceptable — the rejection is a transient race, not
data corruption.

Reverse scenario: read sees B='completed', between read and INSERT B reverts to 'posted'
via abandoned-claim sweep. Our claim INSERT proceeds. Now task A is claimed but its
dependency B is no longer completed. This violates AC15 momentarily but: B's revert
implies its claim expired without completion; the sweep then posts B back. A's owner
sees this in the event log. The next time A is claimed (if released), the check fires
properly. We accept the transient window — completeTask + claim release are racey at
the dependency level, but no data loss.

Mitigation if stronger guarantee desired: `SELECT ... FOR UPDATE` on the dependency
task rows. Defer to a future sprint — current correctness is "eventually consistent
with sweep visibility", aligning with the §0.1-loop philosophy.

## §7 Backwards compatibility

- **NULL `execution_task`**: existing requests/motions chain via derived defaults. No
  migration backfill.
- **Replay consumers**: pre-15.7 `request.resolved`/`motion.tallied` events have no
  `chain` field. Consumers must treat missing `chain` as "no chain happened" (legacy
  behavior). Documented in `docs/coordination-events.md` (update at SESSION phase).
- **No new event types**: `task.deferred` is pre-provisioned but unused; no schema or
  enum change beyond migration 0060.
- **Existing tests**: 602 tests in 15.6 will continue to pass. Any test that asserts on
  `request.resolved` payload structure must be updated if it currently checks payload
  keys exactly — most assert specific fields (`outcome`, `artifact_advanced`), so adding
  a new optional `chain` field should not break them.

## §8 Idempotency

- **`closeTopic` re-entry on `'closing'`**: documented (topics.ts line 318). The new
  sweep relies on this. No re-design.
- **`emitChain` re-entry**: not idempotent — calling twice would create two tasks.
  Caller (decideStep/tallyMotion/sweepExpiredMotions) guarantees one call per source
  event commit. Source event is appended once per transaction; chain is one call per
  transaction.
- **`sweepStuckClosingTopics`**: each per-topic call invokes `closeTopic` which is
  idempotent. Two concurrent sweep cycles (rare — advisory lock prevents) would both
  call closeTopic; the second sees `status='closed'` and returns `already_closed: true`
  per existing line 349.
