# Spec — Phase 12 Sprint 12.1e3 — `LESSONS_SALIENCE_NO_WRITE` gate + friction class

Measurement-infrastructure hygiene sprint. Closes the `goldenset-pollution` friction pattern discovered during Sprint 12.1e2's POST-REVIEW investigation.

## Problem

Per Sprint 12.1e2's POST-REVIEW findings: every `qc:baseline` run accumulates `consideration-search` rows in `lesson_access_log` (each of 40 queries × up to K=10 top hits × samples = 400-1200 rows per run). Over 20+ baseline runs across 12.1c-12.1e1, fixture lessons (duplicate-trap targets) accumulated hundreds of rows → inflated salience → distorted subsequent ranking measurements. Sprint 12.1e2 diagnosed this as a measurement artifact and manually truncated the pollution before running clean-state A/Bs.

Without a code-level mitigation, every future salience-sensitive sprint faces the same hazard. Manual truncate-before-sprint is risky (destructive, error-prone) and doesn't scale.

## Scope

**IN SCOPE:**
1. **Add `LESSONS_SALIENCE_NO_WRITE` env var** to `src/env.ts`, boolean, default `false`. When `true`: `logLessonAccess` becomes a no-op (salience reads still work normally).
2. **Gate the write path at ONE site** — at the top of `logLessonAccess` in `src/services/salience.ts`. Checking at the function entry guarantees all 6 call sites respect the flag without touching each.
3. **Expose via docker-compose.yml** — add 4th salience knob alongside the existing three.
4. **Unit test** in `src/services/salience.test.ts` — confirms: flag=false → write happens; flag=true → no SQL call.
5. **Validation baseline run** — execute `LESSONS_SALIENCE_NO_WRITE=true npm run qc:baseline -- --tag sprint-12.1e3-validate --samples 1 --surfaces lessons` and assert `lesson_access_log` count is unchanged before vs after. Archive the validation baseline for reproducibility.
6. **Friction-class entry** in `docs/qc/friction-classes.md` — new class `goldenset-pollution`: definition, cause, diagnostic signal, Sprint-12.1e2 example, mitigation paths (including the new flag).

**OUT OF SCOPE:**
- Changing salience read-path behavior (math, blend, ranking).
- Removing/modifying the existing `LESSONS_SALIENCE_DISABLED` umbrella (which kills reads + writes; this new knob is write-only).
- CLI flag on `qc:baseline` runner (the env-var pattern is established; CLI wrapper can ship later if painful).
- Truncating any existing pollution (12.1e2 already cleaned the access log to 90 rows).
- α sweep or any other ranking tuning.

## Acceptance criteria

## CLARIFY decisions locked

**D1 — Gate location:** `logLessonAccess` function entry. Single point of enforcement; no site-by-site changes needed.

**D2 — Gate is independent from `LESSONS_SALIENCE_DISABLED`:** the two are composable — DISABLED kills reads AND writes (umbrella kill-switch); NO_WRITE kills only writes (measurement mode).

**D3 — Default: `false`.** Backward-compat; existing production deployments unaffected.

**D4 — Validation approach:** single baseline run with flag=true + pre/post COUNT. Don't need a full sweep — one run is enough to prove the flag works.

**D5 — Validation uses `--samples 1`** to cut runtime to ~5min. Retrieval is deterministic; samples count doesn't affect the write-path behavior we're testing.

**D6 — Friction class documentation is part of THIS sprint**, not deferred. The pattern is fresh; document while it's clear.

## Risks + mitigations

**R1 — Gate bypassed by direct SQL.** If some future code writes to `lesson_access_log` without calling `logLessonAccess`, the flag doesn't apply. Mitigation: document the intent in the function-level docstring. Future site authors should go through `logLessonAccess`.

**R2 — Gate accidentally breaks kill-switch semantics.** If `LESSONS_SALIENCE_DISABLED=true` is checked at each call site BEFORE invoking `logLessonAccess`, the new gate doesn't affect that path. But we should verify — a future refactor might move the DISABLED check INTO `logLessonAccess`, in which case ordering matters. Mitigation: document the independence; unit test covers both flags.

**R3 — Validation baseline could write BEFORE the gate takes effect.** If the mcp container isn't recreated between setting the env var and running the baseline, the old env value persists. Mitigation: the validation step includes `docker compose up -d --force-recreate mcp` with the flag set, verify via `docker compose exec mcp node -e "console.log(process.env.LESSONS_SALIENCE_NO_WRITE)"` before running baseline.

**R4 — Friction-class doc quality drift.** I could write a shallow entry that doesn't actually help future sprints. Mitigation: follow the existing friction-class format (definition, why, signal, example, mitigation, cross-ref) that other classes use — structure forces thoroughness.
