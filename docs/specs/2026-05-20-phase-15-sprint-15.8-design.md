## §0 Invariants & contracts (carried)

- §0.2 lock order: `request → request_step → motion → topics`. Sprint 15.8 extends:
  when motion-tally triggers a step-update, the motion lock is already held (caller),
  and the step row is then locked FOR UPDATE — same canonical direction (request_step
  is downstream of motion in the new chain).
- §0.3 event log: `appendEvent` rejects on `'closed'`. Motion-step updates that touch
  closing/closed topics ROLLBACK via existing 15.6/15.7 guards.
- Sprint 15.7 chaining contract: on a request step's "final endorse → approved" path,
  the chain handler still fires. Sprint 15.8 only changes WHAT triggers the endorse —
  unilateral `decideStep` OR a `motion.tallied` carry both reach the same chain code
  path.

## §1 Migration 0061

```sql
-- migrations/0061_collective_step.sql
-- Phase 15 Sprint 15.8 — wire procedure='collective' into Request-Approval
-- (DEFERRED-018). A request step may be decided by a motion's tally instead of
-- a unilateral officeholder's decideStep.

-- DoA matrix gains per-row procedure + body_id.
-- procedure='unilateral' (default) → step decided by decideStep (15.3 path).
-- procedure='collective' → step decided by a motion under body_id (15.4+15.8 path).
-- body_id MUST be non-null when procedure='collective' (CHECK below).
ALTER TABLE doa_matrix
  ADD COLUMN procedure TEXT NOT NULL DEFAULT 'unilateral'
    CHECK (procedure IN ('unilateral','collective')),
  ADD COLUMN body_id   UUID NULL REFERENCES decision_bodies(body_id),
  ADD CONSTRAINT doa_matrix_collective_body_ck
    CHECK (procedure = 'unilateral' OR body_id IS NOT NULL);

-- request_steps gains body_id + motion_id snapshots (frozen at submission, like
-- target_office + doa_snapshot). procedure column already exists with default
-- 'unilateral'. Extend status enum to include 'motion_proposed' — collective
-- step has a motion in flight (sweep-skip predicate already excludes
-- non-'pending', so the escalation sweep does not double-handle).
ALTER TABLE request_steps
  ADD COLUMN body_id   UUID NULL REFERENCES decision_bodies(body_id),
  ADD COLUMN motion_id UUID NULL REFERENCES motions(motion_id);

ALTER TABLE request_steps
  DROP CONSTRAINT request_steps_status_check,
  ADD CONSTRAINT request_steps_status_check
    CHECK (status IN (
      'pending', 'motion_proposed', 'endorsed',
      'returned', 'rejected', 'escalated'
    ));

-- Lookup index — tallyMotion + sweepExpiredMotions need to find the step linked
-- to a motion in O(1). Sparse partial index (only collective steps).
CREATE INDEX request_steps_motion_lookup_idx
  ON request_steps (motion_id)
  WHERE motion_id IS NOT NULL;

COMMENT ON COLUMN doa_matrix.procedure IS
  'Sprint 15.8 (DEFERRED-018) — per-matrix-row decision procedure. unilateral = decideStep; collective = body-based motion tally.';
COMMENT ON COLUMN doa_matrix.body_id IS
  'Sprint 15.8 — decision body authoritative when procedure=collective. NOT NULL enforced by CHECK.';
COMMENT ON COLUMN request_steps.body_id IS
  'Sprint 15.8 — snapshotted from doa_matrix.body_id at submission. Frozen for the request lifetime.';
COMMENT ON COLUMN request_steps.motion_id IS
  'Sprint 15.8 — link to the in-flight motion (set at step activation, NULL once the step terminates).';
```

### §2.3 `requests.ts:proposeStepMotion` (new helper)

```ts
async function proposeStepMotion(
  client: PoolClient,
  args: {
    request_id: string;
    step_index: number;
    body_id: string;
    topic_id: string;
    deadline_minutes: number;
  },
): Promise<{ motion_id: string }>;
```

Flow (inside caller's transaction):
1. INSERT a `motions` row with `proposed_by='system:request-step-proposer'`,
   `subject_ref='request_step:<request_id>:<step_index>'`, `status='proposed'`,
   `deadline=now() + deadline_minutes`.
2. appendEvent `motion.proposed` with payload `{body_id, subject_ref, deadline,
   source: 'request_step', request_id, step_index}`.
3. UPDATE `request_steps SET status='motion_proposed', motion_id=<new id>` for the
   step.
4. Return `{motion_id}`.

**Authorization choice (Q1):** the system-proposer bypasses the topic-participant check
of `proposeMotion`. Per design CLARIFY Q1: this is *correct* — the system is acting on
behalf of the body, not as a topic participant.

**Body-membership check:** the motion's `proposed_by='system:request-step-proposer'`
will not be a body member; the body's members second + vote per existing 15.4 rules.

### §2.8 `requests.ts:applyMotionToStep` (new exported helper)

```ts
export async function applyMotionToStep(
  client: PoolClient,
  args: {
    motion_id: string;
    request_id: string;
    step_index: number;
    target_office: string;
    outcome: 'carried' | 'failed' | 'lapsed' | 'vetoed';
    topic_id: string;
  },
): Promise<void>;
```

Logic (all inside the motion-lock'd transaction):

```
case 'carried':
  Read request (FOR UPDATE) — kind, subject_id, execution_task.
  UPDATE request_steps SET status='endorsed', decided_by=<motion_id>, decided_at=now() WHERE request_id=$1 AND step_index=$2.
  appendEvent request.step_decided (payload: {step_index, decision:'endorse', decided_by:'motion:<motion_id>'}).
  Check if next step exists:
    - If yes: UPDATE next step.deadline = now()+STEP_DEADLINE_MINUTES, UPDATE requests.current_step. If next step is collective, call proposeStepMotion(client, ...).
    - If no: UPDATE requests SET status='approved'. resolveArtifact('approve'). Build chain params + emitChain. appendEvent request.resolved with chain.
  Done.

case 'failed':
  UPDATE request_steps SET status='returned', decided_by=<motion_id>, decided_at=now().
  UPDATE requests SET status='returned'.
  resolveArtifact('return') — moves artifact for_review→working.
  appendEvent request.step_decided (decision:'return', decided_by:'motion:<motion_id>').
  appendEvent request.resolved (outcome:'returned', artifact_advanced:<bool>).
  Done.

case 'lapsed':
  Apply the existing 15.3 escalation logic — same as a pending-past-deadline step
  hitting sweepStalledSteps — with a **rev 2 (F1 fix) degrade-to-unilateral rule**:
    - Compute next level up (LEVEL_RANK[target_office] + 1)
    - If at authority (already top): UPDATE request_steps SET status='escalated', appendEvent step_escalated. UPDATE requests SET status='escalation_exhausted', resolveArtifact unchanged, appendEvent request.resolved (outcome:'escalation_exhausted', artifact_advanced:false).
    - Else: **degrade to unilateral** at the escalated level:
        UPDATE the current step row: SET status='pending', target_office=<newLevel>,
        procedure='unilateral', body_id=NULL, motion_id=NULL,
        deadline = now() + STEP_DEADLINE_MINUTES;
        appendEvent request.step_escalated with payload {step_index, from_office, to_office, reason: 'motion_lapsed', degraded_to: 'unilateral'}.

  **Rationale (rev 2 F1 fix):** the matrix row carries one (procedure, body_id) per
  (kind, weight) range. Re-resolving the matrix at the new level for the same kind/
  weight is non-trivial (it requires `required_level >= newLevel` matching). The
  simplest correct contract is: "if the body couldn't decide, fall back to a single
  officeholder at the higher tier". This collapses the unbounded matrix-cascade
  question into a deterministic rule. Multi-tier collective (a fresh body at every
  escalated level) is deferred to a future sprint introducing per-level body
  assignment.

case 'vetoed':
  UPDATE request_steps SET status='rejected', decided_by=<motion_id>, decided_at=now().
  UPDATE requests SET status='rejected'.
  Artifact untouched (matches existing reject behavior).
  appendEvent request.step_decided (decision:'reject', decided_by:'motion:<motion_id>'); also notes 'reason':'vetoed'.
  appendEvent request.resolved (outcome:'rejected'). No chain.
```

### §2.9 `coordinationSweep.ts:sweepExpiredMotions`

After motion-tallied appendEvent, same step-update logic (extracted helper). The
auto-tally branch handles the `lapsed` (most common) case, where the motion didn't
reach quorum by deadline. Carried/failed auto-resolutions are theoretical (the
sweep tallies the motion if quorum was met before deadline expiry; usually that
happens via user-driven tallyMotion).

### §2.10 `coordinationSweep.ts:sweepStalledSteps`

Already filters `WHERE status='pending'`. Collective steps with `status='motion_
proposed'` are excluded by predicate — no change needed. The motion sweep handles
their timeout.

## §3 Event payloads

`motion.proposed` payload extends (only for system-proposed motions):
```jsonc
{
  "body_id": "...",
  "subject_ref": "request_step:<request_id>:<step_index>",
  "deadline": "...",
  "source": "request_step",            // NEW field — distinguishes from user-proposed
  "request_id": "<uuid>",              // NEW
  "step_index": 0                      // NEW
}
```

`request.step_decided` payload extends when triggered by motion-tally:
```jsonc
{
  "step_index": N,
  "decision": "endorse" | "return" | "reject",
  "decided_by": "motion:<motion_id>",   // NEW — distinguishes from actor_id
  "reason": "vetoed"                    // OPTIONAL — only for vetoed motions
}
```

`request.step_escalated` payload extends with `reason: 'motion_lapsed'` when
escalation is triggered by a lapsed collective motion (vs `reason: 'deadline'` for
unilateral).

## §4 Error codes & MCP mapping

- `procedure_is_collective` (HTTP 422) — from `decideStep` when target step is
  collective. Sibling to existing `not_authorized`, `topic_closed`. Not a thrown
  ContextHubError — service-level result status.

- `BAD_REQUEST` `collective_no_route` — at submitRequest when DoA matrix row says
  collective but body_id is null (shouldn't happen given the CHECK constraint, but
  defensive).

- `BAD_REQUEST` `counter_sign_collective_not_supported` — 15.8 limitation per §2.2.

No new ContextHubError codes; reuse `BAD_REQUEST`.

## §5 Race analyses

### §5.1 Concurrent castVote + tallyMotion + sweepExpiredMotions
The motion FOR UPDATE serializes all three (existing 15.4 invariant). When tallyMotion
or sweepExpiredMotions reaches the step-update branch, no other writer is touching
the motion. The new `request_steps FOR UPDATE` then serializes with concurrent
`decideStep` (which also locks request_steps FOR UPDATE) — but `decideStep` already
rejects collective steps with `procedure_is_collective`, so contention is real only
between two callers of tallyMotion on the same motion — already prevented by motion
lock.

### §5.2 Concurrent submission + body-deletion
The CLARIFY notes this — no body-delete API exists. If one is added, it must check
for `request_steps` with non-null `body_id`. Future concern.

### §5.3 Closing-topic + collective step
- During submit: 15.6 writer-block rejects new collective requests on closing topics.
- During motion lifecycle: motion appendEvent rejects on closed; appendEvent for
  step_decided rejects on closed. Existing 15.6 + 15.7 race-handling covers this:
  the entire tallyMotion txn rolls back on a closed topic.
- During lapsed escalation: same — re-targeting a step requires appendEvent
  step_escalated; closed topic → ROLLBACK.

### §5.4 Lapsed motion at authority tier
The handler must NOT re-propose a new motion on the same body (infinite loop). Instead,
mark `status='escalation_exhausted'` on the request and emit `request.resolved` —
mirrors unilateral 15.3 behavior at the top tier.

## §6 Backwards compatibility

- Existing DoA matrix rows default `procedure='unilateral'`, `body_id=null` — existing
  unilateral requests unchanged.
- Existing requests in DB have no `body_id`/`motion_id` on their steps — NULL, treated
  as unilateral by all read paths.
- `decideStep` behavior unchanged for unilateral steps.
- 15.7 chain handler triggers on both unilateral approve AND collective carry → final
  step (no special-casing; chain reads request.execution_task as before).
