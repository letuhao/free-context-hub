# Spec — Phase 12 Sprint 12.1e2 — Half-life sweep (observation-led)

Second half of the split 12.1e arc. 12.1e1 broadened the lessons goldenset to 40 queries and established a reference baseline; this sprint runs the half-life sweep against that goldenset and surfaces the data for a human decision.

## Problem

`LESSONS_SALIENCE_HALF_LIFE_DAYS` defaults to 7 (per Sprint 12.1c). The access-frequency salience feature ranks a time-decayed lesson-access signal into retrieval with half-life applied to the decay. The 7-day default was picked by instinct; no empirical evidence has been gathered to confirm it's optimal. Sprint 12.1c's /review-impl LOW-2 explicitly noted that audit-bootstrap data decays within ~3-4 weeks, suggesting longer half-life could retain useful signal further.

## Scope

**IN SCOPE (this sprint):**
1. **4 baseline runs** against the 40-query goldenset (`qc/lessons-queries.json`) with:
   - `LESSONS_SALIENCE_HALF_LIFE_DAYS` ∈ {3, 7, 14, 30}
   - `LESSONS_SALIENCE_ALPHA` fixed at 0.10 (current default)
   - `--samples 3` (match 12.1e1 for statistical comparability)
   - NO `--control` flag (reuse 12.1e1 `noise_floor` since same goldenset + same system)
2. **Per-group + per-query analysis** of the 4 archives:
   - Aggregate metrics (recall, MRR, nDCG@5, nDCG@10, dup@10)
   - Per-group nDCG@10 breakdown (ambig, paraphrase, confident-hit, duplicate-trap, cross-topic, adversarial-miss)
   - Per-query `found_ranks` distribution for ambig group (which targets moved, how far)
3. **Surface a summary diff doc** (`docs/qc/baselines/2026-04-19-sprint-12.1e2-summary.md`) presenting the 4 runs side-by-side with commentary. No auto-decision; this is a decision-support document.
4. **Adaptive escalation** (conditional on null result): if all 4 values land within 12.1e1's `noise_floor` (ambig-group nDCG@10 ±0.013), widen — either add {1, 45, 60}d or add α axis {0.05, 0.20}. Document the widen and run additional baselines as needed.
5. **Human decides default** via POST-REVIEW. If a new half-life is chosen, update `src/env.ts` default in a follow-up commit. If 7d stays, document why.

**OUT OF SCOPE:**
- α tuning beyond the escalation path above.
- Changing salience math, write paths, or any code in `src/services/salience.ts`.
- Broadening goldensets for chunks/code/global (separate sprint).
- Production runtime default changes to `LESSONS_SALIENCE_DISABLED` umbrella.

## CLARIFY decisions locked

**D1 — Sweep scope:** Adaptive. Start with half-life ∈ {3, 7, 14, 30}d @ α=0.10. Widen conditionally if null result.

**D2 — Decision rule:** Observation-led. I run all 4, present full per-group tables + per-query shifts, human decides default by eye. No auto-decision algorithm.

**D3 — Sample count + control:** `--samples 3` per run, no `--control` flag (reuse 12.1e1 noise_floor from `docs/qc/baselines/2026-04-19-sprint-12.1e1.json`).

**D4 — Escalation trigger:** If all 4 values land within 12.1e1's noise_floor (ambig-group nDCG@10 ±0.013) AND no clear per-query found_ranks pattern emerges, widen with outer half-life values {1, 45} or α axis {0.05, 0.20}. Document in summary doc.

**D5 — How env gets set per run:** `LESSONS_SALIENCE_HALF_LIFE_DAYS=<N> npm run qc:baseline -- --tag sprint-12.1e2-hl<N> --samples 3 --surfaces lessons`. The env var is already read by [src/services/salience.ts](src/services/salience.ts) per Sprint 12.1c.

## Risks + mitigations

**R1 — Docker compose env var resetting between runs.** The MCP container reads env at startup. If we change `LESSONS_SALIENCE_HALF_LIFE_DAYS` and run baseline, the container may still be using the old value. Mitigation: `docker compose up -d --force-recreate mcp` between runs, OR verify the env var is actually picked up (log inspection) before committing the archive. Add verification step in BUILD.

**R2 — Analytical paralysis.** Observation-led means I might write a long summary doc without a clear recommendation. Mitigation: structure the summary doc with a "top-line recommendation" section BEFORE the data tables, forcing myself to commit to an opinion.

**R3 — Adaptive escalation scope creep.** If null, widening could keep going. Mitigation: cap escalation at ONE additional round (either outer half-life OR α, not both) unless user approves more.

**R4 — Cross-session latency jitter.** 12.1e1's 31-minute run was cold-cache; 12.1e2 runs are sequential so cache warms up. Latency comparison across runs is unreliable. Mitigation: ignore latency deltas in the summary; only quality metrics matter.

**R5 — Salience data drift between sweep runs.** Every baseline run writes `consideration-search` rows to `lesson_access_log`. The salience of each lesson changes between run 1 and run 4 because of the run-1 writes. Mitigation: either (a) `TRUNCATE lesson_access_log` before each run to isolate half-life effect from cumulative-writes effect, (b) accept the drift and note it in summary. Decision deferred to DESIGN phase.
