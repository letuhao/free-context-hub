# Sprint 13.4 — F2 GUI: Submitted for Review tab

## What this sprint ships

GUI tab + actions for human review of agent-submitted lessons:

1. Add 4 methods to `gui/src/lib/api.ts`: `listReviewRequests`, `getReviewRequest`, `approveReviewRequest`, `returnReviewRequest`.
2. Modify `gui/src/app/review/page.tsx` to:
   - Add visible Tabs UI with two tabs: "Auto-Generated" (existing draft-only flow) + "Submitted for Review" (new, uses review_requests API).
   - Badge count = sum of draft count + pending-review count (already supported by existing filter logic, just need to display the SUM).
   - Tab 2 cards render: lesson title + type badge + pending-review badge + submitter agent + relative time + intended reviewer + note + View Full Lesson + Approve → Active + Return to Draft.
   - Approve action: prompt for optional resolution_note → call `api.approveReviewRequest(projectId, requestId, { resolved_by, resolution_note? })` → toast + refresh.
   - Return action: REQUIRED resolution_note prompt → call `api.returnReviewRequest(projectId, requestId, { resolved_by, resolution_note })` → toast + refresh.
   - `resolved_by` derived from current `/api/me` identity (key_source disambiguates env_token vs db_key for the audit trail).

## ACs (mapped from master design §"GUI: Review Inbox update" L414-430)

- **GUI-AC5:** Return button → modal for required note → REST POST /return → lesson goes to draft → toast + list refreshes.

## Assumptions

**A1.** `resolved_by` comes from `/api/me`: prefer `key_source === 'db_key'` returning the role + scope; for `env_token` / `no_auth` paths, use a sensible fallback like "env-admin" or "anonymous-dev". The existing `getCurrentUser()` from Sprint 13.2 returns role + project_scope + auth_enabled + key_source.

**A2.** Tabs UI follows the existing pattern from agents/page.tsx audit-log tabs (lines 178-193 of that file). Active-tab styling: blue underline.

**A3.** Multi-project mode: `listReviewRequests` is called per project_id in `effectiveProjectIds` (fan-out, similar to ActiveWorkPanel in agents/page.tsx). Each row carries its project_id for the approve/return calls.

**A4.** Card layout follows the existing draft-lesson card pattern. Add explicit `<Badge>` for status="pending-review" using a yellow/amber color.

**A5.** Confirmation: Approve goes through with a single click but shows a toast; Return REQUIRES a modal for the resolution_note (parity with the REST API contract).

## Risks

- **R-low:** existing review page uses `filter` state for tabs internally but the master design wants a separate visible tab strip. Adding the tab UI without rewriting the filter logic is fine — they coexist.
- **R-low:** `effectiveProjectIds` already memoized in context (verified during Sprint 13.2 r3 cycle), so fetchClaims-style auto-refresh is safe.
- **R-med:** badge count for sidebar — depends on how the existing sidebar reads pending counts. Skip if existing sidebar already handles the pending-review status correctly via lessons.status; only update if there's a separate counter that needs review_requests integration.
