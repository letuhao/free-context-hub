## 1. Migration `0054_coordination_board.sql`

All `CREATE … IF NOT EXISTS` (idempotent — additive, no existing constraint altered).

```sql
-- 0054_coordination_board.sql — Phase 15 Sprint 15.2
-- The Board: tasks, derived-identity artifacts + versioning, claims + fencing.
-- See docs/specs/2026-05-16-phase-15-sprint-15.2-design.md.

-- Global monotonic fencing-token source (design C.2). nextval is non-transactional
-- and strictly increasing — any later claim's token strictly exceeds any earlier one's.
CREATE SEQUENCE IF NOT EXISTS coordination_fencing_seq AS BIGINT START 1;

-- FK note [r3-fix §0.3]: tasks/artifacts/claims reference topics with NO ON DELETE clause
-- by design — a topics row is permanent (closeTopic is a status flip; Phase 15 has no
-- delete-topic operation). If a hard-delete path is ever added, these FKs + postTask's
-- existence check (§2.1) must be revisited.
CREATE TABLE IF NOT EXISTS tasks (
  task_id    UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id   TEXT        NOT NULL REFERENCES topics(topic_id),
  title      TEXT        NOT NULL,
  topology   TEXT        NOT NULL CHECK (topology IN ('parallel','sequential','rolling')),
  depends_on UUID[]      NOT NULL DEFAULT '{}',
  raci       JSONB       NOT NULL DEFAULT '{}',
  status     TEXT        NOT NULL DEFAULT 'posted'
               CHECK (status IN ('posted','claimed','in_progress','completed','disputed')),
  created_by TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS tasks_topic_status_idx ON tasks (topic_id, status, created_at);

-- artifact_id is DERIVED <topic_id>:<task_id>:<slot> — never free-text (closes Run 1 #1).
CREATE TABLE IF NOT EXISTS artifacts (
  artifact_id            TEXT        PRIMARY KEY,
  topic_id               TEXT        NOT NULL REFERENCES topics(topic_id),
  task_id                UUID        NOT NULL REFERENCES tasks(task_id),
  slot                   TEXT        NOT NULL,
  kind                   TEXT        NOT NULL,
  state                  TEXT        NOT NULL DEFAULT 'draft'
                           CHECK (state IN ('draft','working','baselined','for_review','final','superseded')),
  version                INT         NOT NULL DEFAULT 1,
  accepted_fencing_token BIGINT      NOT NULL DEFAULT 0,
  content_ref            TEXT,
  created_at             TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS artifacts_task_idx ON artifacts (task_id);

-- Append-only artifact history — one row per version (writes, baselines, sweep reverts).
CREATE TABLE IF NOT EXISTS artifact_versions (
  artifact_id   TEXT        NOT NULL REFERENCES artifacts(artifact_id),
  version       INT         NOT NULL,
  state         TEXT        NOT NULL,
  content_ref   TEXT,
  fencing_token BIGINT,                       -- the write's token; NULL for create / baseline / revert
  note          TEXT        NOT NULL,         -- 'created' | 'write' | 'baselined' | 'reverted to vN' | …
  created_by    TEXT        NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (artifact_id, version)
);

-- claims — evolves Phase 13 artifact_leases. Rows are ephemeral (deleted on
-- release / complete / expiry-sweep); the lifecycle is in the event log.
CREATE TABLE IF NOT EXISTS claims (
  claim_id      UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  topic_id      TEXT        NOT NULL,
  task_id       UUID        NOT NULL REFERENCES tasks(task_id),
  artifact_id   TEXT        NOT NULL REFERENCES artifacts(artifact_id),
  actor_id      TEXT        NOT NULL,
  fencing_token BIGINT      NOT NULL,
  expires_at    TIMESTAMPTZ NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- PLAIN unique index (NOT partial — `WHERE now()` is invalid, migration 0048 documents this).
-- An INTEGRITY INVARIANT — "one claim row per artifact". claimTask never relies on it as a
-- race arbiter (the task-row FOR UPDATE lock is the serializer — §2.3); it stands to catch a
-- future code path that inserted a claim without that lock.
CREATE UNIQUE INDEX IF NOT EXISTS claims_active_uniq ON claims (artifact_id);
CREATE INDEX IF NOT EXISTS claims_expires_idx ON claims (expires_at);
```

## 2. `src/services/board.ts` (new) — tasks + the claim lifecycle

Constants: `DEFAULT_TTL_MINUTES = 30`, `MAX_TTL_MINUTES = 240`, `SLOT_REGEX = /^[a-z0-9][a-z0-9-]*$/`.

### 2.1 `postTask({ topic_id, title, topology, depends_on?, raci?, slot, kind, created_by }) → TaskRecord`

Validate: `title`/`kind`/`created_by` non-empty, `topology ∈ {parallel,sequential,rolling}`,
`slot` matches `SLOT_REGEX`. `task_id = randomUUID()`; `artifact_id = topic_id+':'+task_id+':'+slot`.
```
-- [r2-fix F3] explicit existence check FIRST — a MISSING topic must be a clean 404, not the
-- raw 23503 FK violation the tasks INSERT would otherwise raise (→ unclassified 500).
SELECT 1 FROM topics WHERE topic_id=$topic
  none → throw ContextHubError('NOT_FOUND', 'topic <id> not found')        -- → 404
BEGIN
  INSERT INTO tasks (task_id, topic_id, title, topology, depends_on, raci, status, created_by)
    VALUES ($task, $topic, $title, $topology, $depends_on, $raci, 'posted', $created_by)
  INSERT INTO artifacts (artifact_id, topic_id, task_id, slot, kind, state, version, accepted_fencing_token)
    VALUES ($artifact, $topic, $task, $slot, $kind, 'draft', 1, 0)
  INSERT INTO artifact_versions (artifact_id, version, state, content_ref, fencing_token, note, created_by)
    VALUES ($artifact, 1, 'draft', NULL, NULL, 'created', $created_by)
  appendEvent(client, { topic_id, actor_id: created_by, type:'task.posted',
                        subject_type:'task', subject_id: task_id, payload:{ title, topology, slot } })
  appendEvent(client, { topic_id, actor_id: created_by, type:'artifact.created',
                        subject_type:'artifact', subject_id: artifact_id, payload:{ task_id, slot, kind } })
COMMIT
```
*[r2-fix F3]* A **missing** topic is caught by the explicit existence check (→ `NOT_FOUND` /
404). A **closed** (existing) topic passes the check, the `INSERT`s succeed, and the first
`appendEvent`'s seal then throws `BAD_REQUEST` (→ 400) — the §0.1 catch rolls back. *[r3-fix
F3]* The check is a plain `SELECT` (no `FOR UPDATE`) and has no check-to-`INSERT` TOCTOU
*because of §0.3* — a `topics` row is permanent, so it cannot vanish between the check and the
`tasks` `INSERT`. (§0.3 is the invariant the `tasks.topic_id` FK + this check both rely on.)

### 2.2 `listBoard({ topic_id, status? }) → { tasks: TaskSummary[] }`

`SELECT t.*, a.artifact_id, a.state AS artifact_state FROM tasks t
 JOIN artifacts a ON a.task_id = t.task_id
 WHERE t.topic_id=$1 [AND t.status=$2] ORDER BY t.created_at`. Default lists all; pass
`status:'posted'` for the claimable set.

### 2.3 `claimTask({ task_id, actor_id, ttl_minutes? }) → ClaimResult` — *[r1-fix F1]*

The Phase 13 `artifactLeases.ts` claim *structure* (lazy-cleanup → check-live → INSERT),
**serialized by the task-row `FOR UPDATE` lock**. Lock order **task → claim → topics** (§0.2):
```
BEGIN
  SELECT topic_id, status FROM tasks WHERE task_id=$1 FOR UPDATE          -- (task) THE serializer
    none               → ROLLBACK; return { status:'not_found' }
    status='completed' → ROLLBACK; return { status:'conflict', reason:'task_completed' }
  artifact_id ← SELECT artifact_id FROM artifacts WHERE task_id=$1        -- plain read (NO lock)
  DELETE FROM claims WHERE artifact_id=$artifact AND expires_at <= now()  -- (claim) lazy cleanup
  SELECT actor_id, expires_at FROM claims WHERE artifact_id=$artifact AND expires_at > now()
    found → ROLLBACK; return { status:'conflict', incumbent_actor_id, expires_at }
  fencing_token ← nextval('coordination_fencing_seq')
  expires_at    ← now() + ttl                                            -- ttl clamped to [1, MAX_TTL]
  INSERT INTO claims (claim_id, topic_id, task_id, artifact_id, actor_id, fencing_token, expires_at) …  -- (claim)
  UPDATE tasks SET status='claimed' WHERE task_id=$1
  appendEvent(client, { type:'claim.granted', subject_type:'artifact', subject_id:artifact_id, … })  -- (topics)
  appendEvent(client, { type:'task.claimed',  subject_type:'task',     subject_id:task_id, … })       -- (topics)
COMMIT
return { status:'claimed', claim_id, fencing_token, expires_at, artifact_id }
```
**Why no 23505 handler / retry loop** *[r1-fix F1]*: a task has exactly one artifact and its
`artifact_id` embeds `task_id`, so the *only* path that inserts a claim for a given
`artifact_id` is `claimTask` for that one task. Two such calls both block on
`SELECT tasks … FOR UPDATE`; the loser, once it acquires the lock, runs `DELETE`-expired then
the live-claim `SELECT`, finds the winner's committed claim, and returns `conflict` (with a
**real** incumbent) — it never reaches the INSERT. A `23505` on `claims_active_uniq` is
*structurally impossible* in `claimTask`. The index is a pure integrity invariant; were it
ever violated, the §0.1 `catch` surfaces a loud `500`, not a fabricated `conflict`.

### 2.4 `releaseTask({ task_id, actor_id }) → { status }` — *[r1-fix F2]*

A holder voluntarily gives up a **live** claim. Lock order **task → claim → topics** (§0.2):
```
BEGIN
  SELECT 1 FROM tasks WHERE task_id=$1 FOR UPDATE                         -- (task)
    none → ROLLBACK; return { status:'not_found' }
  artifact_id ← SELECT artifact_id FROM artifacts WHERE task_id=$1        -- plain read
  SELECT claim_id, actor_id FROM claims
    WHERE artifact_id=$artifact AND expires_at > now() FOR UPDATE         -- (claim) liveness filter [r1-fix F2]
    none                     → ROLLBACK; return { status:'claim_expired' }   -- the sweep owns it
    claim.actor_id ≠ actor_id → ROLLBACK; return { status:'not_owner' }
  DELETE FROM claims WHERE claim_id=$claim_id
  UPDATE tasks SET status='posted' WHERE task_id=$1
  appendEvent(client, { type:'task.released', subject_type:'task', subject_id:task_id, payload:{ actor_id } })  -- (topics)
COMMIT
return { status:'released' }
```
*[r1-fix F2]* The `expires_at > now()` filter is mandatory — an *expired* claim is the
**sweep's exclusive domain** (only the sweep retires an expired claim and reverts the
artifact, §4). An expired claim ⇒ `claim_expired` (no-op). A voluntary release of a **live**
claim does **not** revert the artifact — the work is an intentional hand-off, not abandonment.

### 2.5 `completeTask({ task_id, actor_id }) → { status }` — *[r3-fix F2]*

Lock order **task → claim → artifact → topics** (§0.2):
```
BEGIN
  SELECT status FROM tasks WHERE task_id=$1 FOR UPDATE                        -- (task)
    none → ROLLBACK; return { status:'not_found' }
    'completed' → ROLLBACK; return { status:'already_completed' }
  artifact_id ← SELECT artifact_id FROM artifacts WHERE task_id=$1            -- plain read
  SELECT actor_id FROM claims
    WHERE artifact_id=$ AND expires_at > now() FOR UPDATE                     -- (claim) [r3-fix F2] FOR UPDATE
    none → ROLLBACK; return { status:'no_live_claim' }
    claim.actor_id ≠ actor_id → ROLLBACK; return { status:'not_owner' }
  UPDATE artifacts SET state='for_review'                                     -- (artifact)
    WHERE artifact_id=$ AND state IN ('draft','working','baselined')
    0 rows → ROLLBACK; return { status:'bad_artifact_state' }
  DELETE FROM claims WHERE artifact_id=$            -- claim released at for_review (design B.5)
  UPDATE tasks SET status='completed' WHERE task_id=$1
  appendEvent(task.completed, subject task, payload:{ actor_id })             -- (topics)
  appendEvent(artifact.state_changed, subject artifact, payload:{ from:<prev>, to:'for_review' })
COMMIT
```
*[r3-fix F2]* The claim SELECT carries `FOR UPDATE` — the claim row is write-locked at first
touch (not first at the `DELETE`), so §9 invariant 7 is genuinely lock-enforced and the lock
is acquired *before* the artifact `UPDATE` (the canonical `task → claim → artifact → topics`).

*[code-r1 F1]* `<prev>` — the pre-transition state carried in the `artifact.state_changed`
event — is read by an explicit `SELECT state FROM artifacts WHERE artifact_id=$ FOR UPDATE`
placed immediately before the `UPDATE`. That `SELECT` *is* the `(artifact)` row-lock
acquisition of the canonical order; reading `<prev>` from the row this transaction has locked
is the only verifiable pre-image (a `WITH prev` CTE read was not — it could observe a stale
snapshot value under READ COMMITTED).

## 3. `src/services/artifacts.ts` (new) — versioned writes

### 3.1 `writeArtifact({ artifact_id, claim_id, fencing_token, content_ref, actor_id }) → WriteResult`

The stale-holder defense (design C.2, CLARIFY R2). The claim-liveness **and** fencing checks
are **one atomic statement** — a guarded `UPDATE`. Lock order **artifact → topics** (§0.2 —
the `EXISTS` subquery on `claims` is a plain read, no row lock):
```
BEGIN
  UPDATE artifacts
     SET version = version + 1,
         state = 'working',
         content_ref = $content_ref,
         accepted_fencing_token = $fencing_token
   WHERE artifact_id = $1
     AND state IN ('draft','working','baselined')          -- writable states only
     AND accepted_fencing_token <= $fencing_token          -- fencing: presented ≥ accepted
     AND EXISTS (SELECT 1 FROM claims c                     -- claim-liveness (plain read — no lock)
                  WHERE c.claim_id = $claim_id
                    AND c.artifact_id = $1
                    AND c.expires_at > now())
   RETURNING version, state
  0 rows → ROLLBACK; classify via a re-SELECT (artifact missing? state not writable?
           token below accepted? claim not live?) → return { status:'conflict', reason }
  INSERT INTO artifact_versions (artifact_id, version, state, content_ref, fencing_token, note, created_by)
    VALUES ($1, $newVersion, 'working', $content_ref, $fencing_token, 'write', $actor_id)
  appendEvent(artifact.versioned,      subject artifact, payload:{ version:$newVersion, fencing_token })
  appendEvent(artifact.state_changed,  subject artifact, payload:{ from:<prev>, to:'working' })   -- only if state changed
COMMIT
return { status:'ok', version:$newVersion, state:'working' }
```
*Why one statement:* a fencing token guards against a *superseding* holder; the live-claim
`EXISTS` guards against *your own claim having expired with no successor* (the
slow-claimant-after-sweep window). Splitting them reopens a TOCTOU. `accepted_fencing_token`
is set to the presented token on success — a later, lower-token write is then rejected.

*[code-r1 F1]* `<prev>` (the pre-transition state, for the conditional `artifact.state_changed`)
is read by an explicit `SELECT state FROM artifacts WHERE artifact_id=$1 FOR UPDATE` immediately
before the guarded `UPDATE` — that `SELECT` is the `(artifact)` row-lock acquisition. The
guarded `UPDATE` then runs on the already-locked row; its three checks remain one atomic
statement (the pre-`SELECT` only locks and reads — it checks nothing).

### 3.2 `baselineArtifact({ artifact_id, claim_id, fencing_token, actor_id }) → BaselineResult`

Same guarded shape — a holder marks a checkpoint (`draft|working → baselined`):
```
UPDATE artifacts SET state='baselined', version=version+1, accepted_fencing_token=$fencing_token
 WHERE artifact_id=$1 AND state IN ('draft','working')
   AND accepted_fencing_token <= $fencing_token
   AND EXISTS (SELECT 1 FROM claims c WHERE c.claim_id=$claim_id AND c.artifact_id=$1 AND c.expires_at>now())
 RETURNING version
```
0 rows → `conflict` (classify). On success: append an `artifact_versions` row (`note:'baselined'`,
same `content_ref` as the prior version); `appendEvent(artifact.state_changed, to:'baselined')`.

*[code-r1 F1]* As in §3.1, the pre-`baselined` `state` and the `content_ref` to carry forward
are read by a `SELECT state, content_ref FROM artifacts WHERE artifact_id=$1 FOR UPDATE` before
the guarded `UPDATE`; that `SELECT` is the `(artifact)` row-lock acquisition.

### 3.3 internal — `revertArtifact(client, artifact_id, actor_id)` (used by the sweep, §4)

Find the artifact's **last `baselined`** `artifact_versions` row. If found → revert to it:
`UPDATE artifacts SET state='baselined', content_ref=<that row's content_ref>, version=version+1`
+ append a version row (`note:'reverted to v<K>'`). If **no** baselined version →
`UPDATE artifacts SET state='draft', content_ref=NULL, version=version+1` + append
(`note:'reverted to draft'`). **Never un-baselines; never touches `accepted_fencing_token`**
(monotonic). The revert is an *append*, never an in-place content edit. The caller (the
sweep) already holds the artifact row lock (§4.1 step 3).

## 4. `src/services/coordinationSweep.ts` (new) — abandoned-claim recovery

### 4.1 `sweepAbandonedClaims({ grace_minutes? }) → { recovered: number, swept_at }` — *[r1-fix F3; r2-fix; r3-fix]*

`grace_minutes` default 0 (claims carry a real TTL; configurable). Each expired claim is
recovered **in its own transaction under the §0.1-loop contract** — a `catch` that rolls
back, **logs, and `continue`s the loop (never re-throws)**, inside a `finally` that releases
the client.
```
expired ← SELECT claim_id, topic_id, task_id, artifact_id, actor_id
            FROM claims WHERE expires_at < now() - make_interval(mins => $grace)
for each expired claim:
  client = await pool.connect()
  try {
    BEGIN
      -- canonical lock order task → claim → artifact → topics (§0.2 / §10 — matches completeTask)
      SELECT 1 FROM tasks     WHERE task_id=$     FOR UPDATE     -- (1) task row
      SELECT 1 FROM claims    WHERE claim_id=$    FOR UPDATE     -- (2) claim row — still present?
        gone (released/completed since the scan) → ROLLBACK; continue
      SELECT 1 FROM artifacts WHERE artifact_id=$ FOR UPDATE     -- (3) artifact row  [r3-fix F1]
      SELECT status FROM topics WHERE topic_id=$ FOR UPDATE     -- (4) topics row — closeTopic serialized
        status='closed' → DELETE FROM claims WHERE claim_id=$;  -- [r2-fix F2] closed topic: drop the
                           COMMIT; continue                     -- dangling claim ONLY — no revert, no
                                                                -- task change, no events (CLARIFY AC11)
      -- open topic — full recovery (closeTopic cannot race in: we hold the topics-row lock)
      DELETE FROM claims WHERE claim_id=$
      UPDATE tasks SET status='posted' WHERE task_id=$ AND status IN ('claimed','in_progress')
      revertArtifact(client, artifact_id, 'system:sweep')        -- §3.3 — artifact row already locked at (3)
      appendEvent(client, claim.expired       — subject artifact, payload:{ claim_id, actor_id })
      appendEvent(client, task.released       — subject task, payload:{ reason:'claim_expired' })
      appendEvent(client, artifact.versioned + artifact.state_changed — the revert)
    COMMIT
  } catch (err) {
    await client.query('ROLLBACK').catch(()=>{}); logger.error({err}, 'sweep claim recovery failed');
    /* §0.1-loop: CONTINUE — do NOT rethrow */
  } finally { client.release() }
```
*[r3-fix F1]* The sweep locks rows in the canonical **task → claim → artifact → topics**
order (§0.2 / §10) — claim at (2), artifact at (3) — identical to `completeTask`, so no ABBA
cycle. The topic status is read under `SELECT … FOR UPDATE`, serializing `closeTopic`. The
per-claim `catch` is the backstop — one bad claim never aborts the cycle.

*[r2-fix F2]* **Closed-topic branch — drop the claim only.** On a `closed` topic the sweep
deletes the dangling ephemeral `claims` row and stops — no `revertArtifact`, no task-status
change, no events. The recovery's purpose (a re-claimable task + a clean artifact for the
*next* claimant) is moot on a closed topic (its tasks cannot be re-claimed: `claimTask`'s
`appendEvent` is rejected by the seal); a closed topic is a sealed self-consistent record —
a revert would desync the artifact from the sealed event log. CLARIFY AC11 is scoped to
match. (A later sprint's `closing`-drain force-lapses claims *during* `closing`, before the
log seals, so they never reach this path.)

### 4.2 `startClaimsSweepScheduler(opts) → { stop() }`

The `sweepScheduler.ts` *scheduling structure* — a `setTimeout`-chained cycle, an advisory
lock around the work (key `sha256('phase-15.claims-sweep')` → signed int64), an outer
`try/finally` that always re-schedules, test-injectable deps, a `stop()` handle. **The cycle
calls `sweepAbandonedClaims()` directly** (no `enqueueJob`). `SWEEP_INTERVAL_MS = 5 * 60_000`.
Started from `src/index.ts` alongside `startSweepScheduler()`.

## 9. Key invariants & contracts (for the cold-start Adversary)

1. **One active claim per artifact** *[r1-fix]* — every claim on a task is serialized by
   `claimTask`'s task-row `FOR UPDATE` lock (§2.3); concurrent `claimTask` → exactly one
   winner, the rest `conflict` with a **real** incumbent. `claims_active_uniq` is a pure
   integrity invariant — structurally unreachable as a race given the lock.
2. **Stale-holder write rejected** — `writeArtifact`/`baselineArtifact` verify claim-liveness
   **and** `fencing_token ≥ accepted_fencing_token` in **one** guarded `UPDATE`; no TOCTOU.
3. **Fencing monotonic** — `coordination_fencing_seq` is a non-transactional sequence; every
   claim's token strictly exceeds every earlier one's.
4. **Derived identity** — `artifact_id = topic:task:slot`, service-derived (closes Run 1 #1).
5. **The sweep never destroys a baseline** — `revertArtifact` reverts to the *last baselined*
   version (or `draft`), **appends** an `artifact_versions` row, never un-baselines, never
   touches `accepted_fencing_token`.
6. **No deadlock — derived, not asserted** *[r3-fix F1]* — every transaction acquires row
   locks as a prefix-consistent subsequence of the canonical **`task → claim → artifact →
   topics`**; §10 *derives* each transaction's order from its pseudocode, including
   `appendEvent`'s `UPDATE topics` lock. No two transactions acquire a shared lock pair in
   opposed order ⇒ no ABBA cycle.
7. **The sweep is crash-isolated and close-race-safe** *[r1-fix; r2-fix]* — each expired
   claim is recovered in its own txn under the §0.1-loop contract (`catch` logs and
   `continue`s); topic status read under `FOR UPDATE` serializes `closeTopic`. On a `closed`
   topic the sweep drops only the dangling claim (no revert, no events).
8. **Expired claims have one consumer** *[r1-fix; r3-fix F2]* — only the sweep retires an
   expired claim; `releaseTask` and `completeTask` act on **live** claims only — both lock
   the claim row `FOR UPDATE` at the liveness `SELECT` (first touch), so the guarantee is
   lock-enforced, not incidental.
9. **Event/state atomicity** — every state change + its event commit in one txn via
   `appendEvent(client,…)`; a closed topic is rejected by the seal, a missing topic by
   `postTask`'s explicit existence check.
10. **Transaction cleanliness** — every transactional fn has the §0.1 `catch`→`ROLLBACK`→`finally
    release` guard; the sweep loop uses the §0.1-loop variant (`catch`→log→`continue`).
11. **Topics are permanent** *[r3-fix F3]* — §0.3: a `topics` row is never hard-deleted;
    Phase 15 has no delete-topic operation. The `tasks`/`artifacts`/`claims` → `topics` FKs
    declare no `ON DELETE`, and `postTask`'s plain-`SELECT` existence check relies on it.

## 10. Lock-order derivation — *[r3-fix F1]*

`appendEvent` (Sprint 15.1, `coordinationEvents.ts`) executes `UPDATE topics SET next_seq =
next_seq + 1 WHERE topic_id=$1 …` — it **locks the `topics` row**, held to COMMIT. Every
`appendEvent(...)` call in §2/§3/§4 is therefore a `topics`-row lock, acquired last.

Canonical order **`task → claim → artifact → topics`**. Each transaction's contended
row-lock acquisition sequence, derived from its pseudocode:

| Transaction | locks (in acquisition order) | subsequence of `task→claim→artifact→topics`? |
|---|---|---|
| `postTask` | `topics` (`appendEvent`) — the `tasks`/`artifacts` `INSERT`s are brand-new, uncontended rows | ✓ (only `topics` is contended) |
| `claimTask` | `task` (`SELECT…FOR UPDATE`) → `claim` (`DELETE`-expired / `INSERT`) → `topics` (`appendEvent`) | ✓ |
| `releaseTask` | `task` (`SELECT…FOR UPDATE`) → `claim` (`SELECT…FOR UPDATE`) → `topics` (`appendEvent`) | ✓ |
| `completeTask` | `task` (`SELECT…FOR UPDATE`) → `claim` (`SELECT…FOR UPDATE` *[r3-fix F2]*) → `artifact` (`UPDATE`) → `topics` (`appendEvent`) | ✓ |
| `writeArtifact` | `artifact` (guarded `UPDATE`) → `topics` (`appendEvent`) — the `claims` `EXISTS` is a plain read | ✓ |
| `baselineArtifact` | `artifact` (guarded `UPDATE`) → `topics` (`appendEvent`) | ✓ |
| sweep (per claim) | `task` (`SELECT…FOR UPDATE`) → `claim` (`SELECT…FOR UPDATE`) → `artifact` (`SELECT…FOR UPDATE`) → `topics` (`SELECT status FOR UPDATE`) | ✓ |

Every transaction is a prefix-consistent subsequence of the canonical order. No two
transactions acquire a shared pair of locks in opposed order, so no ABBA cycle can form
(invariant 6). The two transactions that do **not** start at `task` — `writeArtifact` /
`baselineArtifact` — lock only `artifact` then `topics`; they can never hold a lock that
precedes `artifact` in the canonical order, so they cannot be the "back" edge of a cycle.
`postTask` acquires only one contended lock (`topics`), so it cannot be in a cycle either.
