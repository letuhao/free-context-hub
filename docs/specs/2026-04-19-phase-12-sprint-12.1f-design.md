## 2. docker-compose RERANK_* exposure

Currently `RERANK_TYPE` and `RERANK_MODEL` are NOT listed in `docker-compose.yml` mcp env block. They're read from `.env` (which doesn't set them → env.ts defaults apply). To override per-run via shell env, add one-line exposures:

```yaml
# After the existing DISTILLATION_ENABLED line in the mcp environment block:
RERANK_TYPE: ${RERANK_TYPE:-generative}
RERANK_MODEL: ${RERANK_MODEL:-}
```

`RERANK_MODEL` default empty string → null-ish → cross-encoder returns unchanged order (no-op) if RERANK_TYPE=cross-encoder but RERANK_MODEL empty. Same semantic as current (unset). No behavior change for production.

**This docker-compose edit is the one logic change in this sprint** (spec A7 alluded). Small, additive, backward-compat.

## 3. Winner criterion

**Primary metric: aggregate nDCG@10.**
**Secondary tiebreaker: aggregate MRR.**

Comparison targets:
- Generative-ref (run 1) is the "current production" reference.
- No-rerank (reuse 12.1e4 `hl7-a010-norerank.json`: MRR=0.9198, nDCG@10=0.9100) is the "skip rerank" floor.

**Decision matrix:**

| Cross-encoder result | Decision |
|---|---|
| nDCG@10 ≥ generative-ref AND deterministic | **Strong win** — recommend changing `RERANK_TYPE` default to cross-encoder (and set default `RERANK_MODEL`). |
| nDCG@10 ∈ [generative−0.013, generative] AND deterministic | **Within-noise match** — cross-encoder as good as generative without the drift. Recommend switching for measurement+production. |
| nDCG@10 < generative−0.013 AND ≥ no-rerank+0.013 | **Partial win** — cross-encoder better than no-rerank but loses to generative. Keep generative for prod, use cross-encoder for QC measurement (trades some quality for determinism). |
| nDCG@10 < no-rerank+0.013 | **Cross-encoder loses** — use no-rerank for measurement (already established); keep generative for prod. |
| Non-deterministic (winner-repeat shows query diffs) | **Reject** — same drift problem as generative. Move to next-best. |

## 4. Timing budget

- Generative rerank: ~5 min (100× LLM queries)
- Cross-encoder rerank: uncertain — batch embedding of 60 candidates per query should be ~1-2 sec per query. Could be 40×2s = 80s per run. Much faster than generative but slower than no-rerank (which was 3s).
- Container recreate: ~10s each.
- 5 runs × (recreate + baseline) = ~10s + 80-300s per run. Total: 10-25 min.

## 7. Non-obvious detail — rerank budget

The rerank budget is computed per-query from total lesson count: `totalLessons < 50 ? 0 : totalLessons < 200 ? 15 : totalLessons < 500 ? 25 : 30` (src/services/lessons.ts:804-808). We have 106 active lessons → rerankBudget=15. So ~15 candidates go through the reranker (generative or cross-encoder).

**Implication:** cross-encoder sees 15 candidates per query, not 30 (the SQL fetch limit is `max(limit, rerankBudget*2) = 30`). The SQL returns 30 candidates; rerank operates on the top-15 post-salience. Same for generative — apples-to-apples.

## Risks introduced at design time

**R-D1 — Env precedence confusion.** Docker-compose's `environment:` takes precedence over `env_file: .env`. But if `.env` has `RERANK_MODEL=foo` uncommented, that overrides unset shell env via substitution `${RERANK_MODEL:-}` → "foo". Currently `.env` has RERANK_MODEL commented out (we checked). Safe. But verify pre-flight.

**R-D2 — `content_snippet` quality for cross-encoder scoring.** Cross-encoder embeds `${title}. ${snippet}` per candidate. Snippets are truncated to ~280 chars. The snippet quality might differ from what generative sees (generative gets the same string). No action needed; both reranker types operate on the same candidate representations.
