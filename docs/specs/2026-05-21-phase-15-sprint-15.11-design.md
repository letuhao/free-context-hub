## §2 A — Level-grant chain (DEFERRED-015)

### §2.1 `topics.ts:joinTopic` — owner-only level honored

After loading the topic (already SELECTs project_id, status; add `created_by`):
```ts
const isOwner = actorId === topicRow.created_by;
const effectiveLevel = isOwner ? level : 'execution';
```
Insert the participant with `effectiveLevel` + `granted_by = isOwner ? actorId : null`.
A non-owner who passes `level: 'authority'` is silently seated at `execution` — **NO,
per Q1 we honor-only-for-owner; a non-owner's explicit non-execution request should
be REJECTED**, not silently downgraded (honest failure). Decision:
- If `!isOwner && level !== 'execution'` → `BAD_REQUEST` (`level_grant_required`:
  "non-owner joiners are seated at execution; request a grant from the topic owner or
  an authority").
- If `isOwner` → honor `level` (bootstrap; typically `authority`).

### §2.2 `topics.ts:grantLevel` (new)

```ts
export async function grantLevel(params: {
  topic_id: string;
  actor_id: string;       // target participant
  level: string;          // new level
  granted_by: string;     // the grantor
}): Promise<GrantLevelResult>;
```
Flow (single transaction):
1. Validate `level ∈ LEVELS`.
2. Load topic (`created_by`, `status`). Reject if closed/closing → `topic_closed`.
3. **Self-grant guard**: `granted_by === actor_id` → `self_grant_forbidden`.
4. **Authorize the grantor**: grantor is authorized iff
   - `granted_by === topic.created_by` (owner), OR
   - grantor's `topic_participants.level === 'authority'`.
   Else → `not_authorized`.
5. Target must be an existing participant (else `target_not_participant`).
6. UPDATE `topic_participants SET level=$level, granted_by=$granted_by` for the target.
7. appendEvent `topic.level_granted` payload `{actor_id, level, granted_by, prior_level}`.
8. Return `{status: 'granted', level}`.

Result statuses: `granted | topic_not_found | topic_closed | self_grant_forbidden |
not_authorized | target_not_participant`.

**rev 2 (F3) — owner-permanence + authority semantics (made explicit):**
- The **owner** (`topic.created_by`) is a **permanent grant root**. Authorization in
  step 4 keys on `granted_by === created_by`, INDEPENDENT of the owner's participant
  `level`. So even if a granted authority demotes the owner's participant row to
  `execution`, the owner RETAINS grant power. Intentional — the owner is the
  bootstrap anchor / recovery root (like a repository owner). It cannot be stripped
  by demotion.
- `grantLevel` may LOWER a level (demotion), not only raise. `authority` is a
  **mutually-trusted role**: an authority granted by the owner may demote a peer
  authority. This is a management capability within the trusted set, not a
  vulnerability — all authorities were vouched for by the owner. If authorities
  deadlock (mutual demotion), the **owner is the tiebreaker** (always authorized,
  can re-grant). Mutual demotion among authorities is therefore an accepted
  availability trade-off, bounded by the owner-root.
- The security review (§10) checks this model holds: a non-owner cannot strip the
  owner's grant power; the owner can always recover.

### §2.3 REST + MCP

export async function grantProxy(params: {

- `proxies` is body-scoped config (no event log — mirrors body_members).

### §3.4 `motions.ts:castVote` — verify proxy grant (rev 2 F1 — gated behind auth-on)

Per Q2, body authz (incl. proxy verification) activates with `MCP_AUTH_ENABLED=true`.
When `proxy_for` is non-null AND auth is enabled:
```ts
// proxy_for is the agent physically casting; actor_id is the principal/member.
// rev 2 (F1) — verification gated behind auth-on (Q2 posture). Auth-off preserves
// the 15.4 behavior: proxy_for recorded unverified (trusted single-operator dev).
if (proxyFor && getEnv().MCP_AUTH_ENABLED) {
  const grant = await client.query(
    `SELECT 1 FROM proxies WHERE body_id=$1 AND principal=$2 AND proxy=$3`,
    [motion.body_id, actorId, proxyFor],
  );
  if (grant.rowCount === 0) {
    await client.query('ROLLBACK');
    return { status: 'proxy_not_granted' };
  }
}
```
Placed after the `not_member` check (principal must be a member AND, auth-on, must
have granted the proxy). Auth-off: `proxy_for` recorded unverified (15.4 behavior
preserved — existing auth-off proxy tests unaffected).

## §7 Backward compatibility & the test-helper problem

**The biggest practical risk.** Existing tests across requests/motions/board/topics
call `joinTopic({level: 'authority'})` or `{level: 'coordination'}` for NON-OWNER
actors in their setup helpers (e.g., `mkTopicWithParticipants`). Under §2.1 these now
either reject (`level_grant_required`) or seat at execution.

**Migration of test helpers:** the canonical helper must change to:
1. Charter the topic with `created_by = <owner>` (an authority actor).
2. Owner joins first with `level: 'authority'` (honored — owner bootstrap).
3. Owner calls `grantLevel` for each other participant to set their coordination/
   authority/execution level.

This is mechanical but touches MANY test files. The helper `mkTopicWithParticipants`
(used widely) absorbs most of it — update it once and most tests pass. Tests that
inline `joinTopic` with non-execution levels need individual updates.

**DB backfill:** existing `topic_participants` rows keep their levels (`granted_by`
backfills to NULL = "legacy/ungranted"). We do NOT retroactively reset existing
participants to execution (would break live topics). Only NEW joins are gated.

## §8 Race analyses

### §8.1 Concurrent grantLevel + decideStep
grantLevel UPDATEs `topic_participants.level`; decideStep reads it (plain SELECT, no
lock). A grant landing mid-decide: decideStep sees either old or new level — both are
valid authorizations at their instant. No corruption. The grant is monotonic-ish
(owner/authority-driven); a concurrent decide using the old level is still a legitimate
decision by a then-authorized officeholder.

### §8.2 Self-grant via two identities (the 016 × 015 interaction)
Attack: operator mints 2 keys (Alice=owner/authority, Bob). Alice grants Bob authority.
Bob decides a step Alice submitted. This is DEFEATED by:
- 015: Bob's authority is real (granted by owner Alice) — legitimate.
- The self-decision guard (`actor ≠ submitted_by`) — Bob ≠ Alice, so it passes.
- BUT: if the SAME human controls both Alice and Bob, this is the "single operator
  drives multi-level" residual. 016's key-count limit BOUNDS how many identities one
  operator can mint; the actor-identity uniqueness prevents identity ambiguity. The
  RESIDUAL (one human, two legitimately-distinct keys, colluding) is fundamentally a
  human-trust problem — documented as the accepted boundary (you cannot
  cryptographically prevent one human from controlling two authorized principals; the
  key-count limit caps the blast radius).

This boundary MUST be stated explicitly in the security review.

### §8.3 Proxy grant race
grantProxy INSERT + castVote SELECT on proxies. A proxy granted mid-vote: castVote sees
it or not (committed-read). If not yet committed, vote rejects `proxy_not_granted` —
caller retries. No corruption.

## §10 Security-review checklist (for POST-REVIEW adversarial pass)

The cold-start security Adversary at POST-REVIEW must verify:
1. A non-owner cannot self-assert authority at join.
2. A non-owner/non-authority cannot grantLevel.
3. No actor can self-grant (raise own level).
4. decideStep cannot be driven above an actor's granted level.
5. A non-member cannot vote; a proxy without a grant cannot cast.
6. createBody/addBodyMember require admin (auth-on).
7. Actor-identity uniqueness holds; key-count limit enforced.
8. The one-human-two-keys residual is documented + bounded, not silently open.
