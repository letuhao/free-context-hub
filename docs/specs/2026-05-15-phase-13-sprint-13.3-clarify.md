## Locked assumptions

**A1.** `LessonStatus` type in `src/services/lessons.ts:26` is the source of truth for the TypeScript type. Extend it to `'draft' | 'active' | 'pending-review' | 'superseded' | 'archived'`. All callers of `LessonStatus` automatically get the extended union.

**A2.** Use `audit_logs` table for review-event audit (existing pattern from Phase 8 agent audit). Action types: `lesson.submitted-for-review`, `lesson.review-approved`, `lesson.review-returned`. Look at `src/services/audit.ts` for the existing emission helper if it exists; otherwise inline `INSERT INTO audit_logs` calls matching the existing schema.

**A3.** Approve/return endpoints accept optional `resolution_note` (per master design L405-406). REST `approve` body: `{ resolution_note?: string }`; `return` body: `{ resolution_note: string }` (required for return — explains what's wrong).

**A4.** `submit_for_review` requires `agent_id` parameter (NOT derived from API key). Consistent with `claim_artifact` decision D1 in master design.

**A5.** The partial unique index `review_requests_lesson_pending_uniq WHERE status = 'pending'` is the enforcement layer for AC2 ("no existing pending request"). Service-layer pre-check + race-condition catch on 23505 unique violation. Pattern matches Sprint 13.1 `artifact_leases_active_uniq`.

**A6.** Re-submission after `returned`: per AC6, the lesson is back to `draft`; agent calls `submit_for_review` again; new `review_requests` row created (old `returned` row has `status != 'pending'`, partial index allows it). Documented in service-module comment.

**A7.** Audit log emits should NOT block the main action on failure (fire-and-forget pattern from existing audit code). Errors logged via `logger.warn` and swallowed.

**A8.** REST authorization: GET list/detail = reader+; POST approve/return = writer+. No admin requirement at this layer. Tenant-scope (DEFERRED-004) is a separate broader concern and NOT in scope for this sprint.

**A9.** MCP tool inputs use the standard `discriminatedUnion` output schema pattern established in Sprint 13.1 (so `submit_for_review` has separate shapes for success vs error cases).

## Open questions (decided inline)

**Q1.** Should `submit_for_review` auto-release any active artifact-lease the submitting agent holds on the lesson? The master design §"Inter-feature integration F1+F2" says: "When an agent calls submit_for_review, it should also call release_artifact for any active lease on the same lesson — the work is done. Recommended in documentation; not enforced." Default: documented in tool description, NOT auto-executed. Keeps service modules decoupled.

**Q2.** Should approve/return emit a separate audit-log row per side effect (lesson status change + review_request status change), or a single combined row? Default: single combined row per user action — the action_type captures the intent ("lesson.review-approved"), `details` JSONB captures both side effects.

**Q3.** Should `list_review_requests` join `lessons` to return `lesson_title` + `lesson_type` inline, or just IDs (forcing two MCP calls)? Default: inline join — saves an MCP roundtrip for the common UX case (GUI listing review queue with previews). Adds one JOIN, no extra round-trip cost. Per master design L388-389 output spec.

**Q4.** Approval message audit detail: include the resolver's identity? Default: `resolved_by` field in `review_requests` row is set from the auth context (`apiKeyRole` info if available; else 'anonymous'). Audit row stores it too.

**Q5.** Race on simultaneous approve/return on same request? Default: handle via DB-level check — `UPDATE review_requests SET status='approved' WHERE request_id=$1 AND status='pending' RETURNING *` returning 0 rows means another resolver already moved it. Surface as 409 Conflict in REST.

Likely findings:
1. **Migration idempotency:** apply the Sprint 13.2 pattern (pre/post set-equality assert). The `lessons_status_check` constraint is similarly enum-shaped.
2. **`pending-review` slug:** the literal string is dashed (`pending-review`), not snake_cased. Verify consistency across all SQL constants, TS type, MCP schema, REST handlers.
3. **`audit_logs` schema:** does it exist? what columns? Verify before writing emissions.
4. **`submit_for_review` race window:** between pre-check (no pending row) and INSERT (creates pending row), another agent could insert. Partial unique index catches; service-layer catches 23505.
5. **Cross-tenant**: a writer agent on project-A submits a lesson_id that belongs to project-B. Service must filter by project_id throughout; do NOT trust client's project_id. Use the URL/`apiKeyScope` for authorization.
6. **REST approve/return double-fire:** simultaneous calls. UPDATE … WHERE status='pending' RETURNING * is the guard. 409 on 0 rows.
7. **Order in mcp/index.ts edits:** the 4 enum sites have different signatures. Verify each replacement individually.
