## Intent

Build the **substrate** of the Phase 15 Multi-Actor Coordination Protocol: the durable
append-only event log plus the Topic, Actor, and participant model that every later sprint
sits on. After this sprint an actor can **charter** a topic, **join** it (receiving an
induction pack), **read** its state, **close** it (sealing the log), **replay** the event
log from a cursor, and a human GUI can **subscribe** to a live event stream.

This sprint ships **no** Board, Request-Approval, voting, intake, or dispute machinery —
those are sprints 15.2–15.5. It ships only the spine: *coordination across time*.

## Why now

Phase 15 design (rev 4) is BUILD-ready and names Sprint 15.1 as the entry point. Run 1's
multi-agent contention test (`docs/audit/phase-13-review.md`) established that Phase 13's
blackboard is correct but has *no history* (a released lease leaves zero trace) and *no
notification* (actors must poll). The event log is the structural fix for both, and every
later Phase 15 entity "is state that also emits events" (design C.1) — so the log, the
Topic that scopes it, and the Actor that authors events must exist first.

## Open questions — decided inline at CLARIFY

| # | Question | Decision |
|---|---|---|
| Q1 | `seq` allocation under concurrency (design Open Q2) | **`topics.next_seq` counter column**, bumped via `UPDATE topics SET next_seq = next_seq + 1 WHERE topic_id=$1 AND status<>'closed' RETURNING next_seq` **in the same transaction** as the event INSERT. The row-level lock on the `topics` row serializes per-topic appends; same-txn increment means an aborted txn rolls the number back too — so 15.1's seq is in fact **gap-free** (a stronger guarantee than the design's "gap-tolerant" floor, fully compatible with it). The `status<>'closed'` predicate makes the same statement **enforce the seal**. |
| Q2 | `chartered` vs `active` — what flips it? | Charter creates the topic `chartered`. The **first `join`** flips it `chartered → active` (one guarded `UPDATE ... WHERE status='chartered'`). No separate event — the `topic.actor_joined` event already records the cause. A topic with zero participants stays `chartered`. |
| Q3 | SSE delivery mechanism (design Open Q5) | **Postgres `LISTEN/NOTIFY`**: `appendEvent` issues `pg_notify('coordination_events', <topic_id>)`; the SSE handler holds one dedicated `LISTEN` connection (outside the pool) and, on notify, queries `replayEvents` past the client's cursor and pushes. Falls back to a bounded poll if `LISTEN` setup fails. Final mechanism choice is re-validated in DESIGN — polling-only is the fallback if `LISTEN` adds disproportionate connection-lifecycle risk. |
| Q4 | Induction-pack replay bound | `join` accepts optional `since_seq` (default 0 = full replay). A re-priming ephemeral agent passes its last cursor. 15.1 topics are new and small; event-count pagination is noted as a future concern (R5), not built now. |
| Q5 | `level` assignment | The **joining actor declares its `level`** in the join payload (`authority`/`coordination`/`execution`). Coordinator-assigned levels are a later concern; 15.1's substrate just records what the joiner declares. |
| Q6 | Actor `type`/`display_name` on re-join | Actor row is upserted `ON CONFLICT (project_id, actor_id) DO UPDATE SET display_name = EXCLUDED.display_name` — re-join refreshes the display name but **`type` is immutable after first registration** (a human cannot silently become an ai); a conflicting `type` on re-join is rejected with a clear error. |

## Approaches considered

### Approach A — Two service modules + a txn-joining event helper (RECOMMENDED)

`coordinationEvents.ts` owns the log: `appendEvent(client, evt)` takes an **already-open
pg client** so it executes inside the caller's transaction (the state change and its event
commit atomically or not at all); `replayEvents(params)` is a standalone cursor read.
`topics.ts` owns charter/join/get/close and calls `appendEvent` within its transactions.
MCP + REST handlers are thin wrappers over the same service — no logic in handlers.

- **Pros:** event/state atomicity is structural (helper requires a client); one business
  path for MCP + REST (no drift); mirrors the proven Phase 13 `artifactLeases.ts` pattern;
  splitting by concern scales — Phase 15 grows to ~14 tables and 5 more service modules.
- **Cons:** `appendEvent`'s "must be called inside a transaction" contract is a convention
  the type system only partly enforces (it takes a `PoolClient`, not a `Pool`).
- **This is the recommendation.**

### Approach B — One `coordination.ts` service module

All Phase 15 logic in one file.
- **Cons:** Phase 15 reaches ~14 tables; one module becomes unmaintainable by 15.3.
  **Rejected.**

### Approach C — Pure event-sourcing (no state tables; derive Topic/roster by replay)

Keep only `coordination_events`; compute topic state and roster by folding the log.
- **Cons:** the design (C.1) is explicit — "the event log is the spine; **everything else
  is state that also emits events**" — i.e. blackboard *and* log, not log alone. `GET
  /topics/:id` would replay the whole log on every call; uniqueness/`CHECK` constraints on
  topic/participant state would be impossible. **Rejected** — contradicts the design.

## Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Concurrent `appendEvent` to one topic races on `seq`. | HIGH | `UPDATE topics SET next_seq=next_seq+1 ... RETURNING` takes a row lock — appends to one topic serialize. Concurrent-append unit test asserts distinct increasing seq, no duplicate-PK 500. |
| R2 | SSE connections leak (client disconnects, server holds the response + `LISTEN` conn). | HIGH | SSE handler registers `req.on('close')` cleanup that releases the dedicated connection and clears the heartbeat timer. Test covers disconnect cleanup. |
| R3 | Charter emits `topic.chartered` referencing a topic row created in the same call — if the event INSERT fails, the topic exists without its genesis event. | MED | Charter wraps topic INSERT + `appendEvent` in **one transaction**; either both land or neither. |
| R4 | Event log not actually sealed after `close` — a late `appendEvent` lands on a `closed` topic. | HIGH | `appendEvent`'s seq-allocation `UPDATE` carries `WHERE status<>'closed'`; 0 rows ⇒ throw `TOPIC_CLOSED`. The seal is enforced by the same statement that allocates seq — un-bypassable. |
| R5 | Induction-pack replay returns the entire log — unbounded for a long-lived topic. | LOW | 15.1 topics are new/small; `since_seq` cursor already bounds re-priming agents. Pagination noted as a future sprint concern, not built now. |
| R6 | MCP tool output crashes the client via the `z.discriminatedUnion` SDK bug (DEFERRED-007). | MED | All 5 new MCP tools use **flat `z.object` with a `z.enum` status field** — never `z.discriminatedUnion` in `outputSchema`. Carried forward from the Phase 13 13.7 fix. |
| R7 | `LISTEN/NOTIFY` needs a dedicated connection held outside the pool; mishandling exhausts connections. | MED | One `LISTEN` connection per live SSE client, explicitly released on disconnect; a server-wide cap is noted. Polling fallback if DESIGN finds the lifecycle risk disproportionate. |
| R8 | `actor_id` is caller-supplied free text (design D1, inherited from Phase 13). | LOW | Accepted per design B.1 — identity is project-scoped `(project_id, actor_id)`, not authenticated. Same risk posture as Phase 13 `agent_id`. |

## Acceptance criteria (Sprint 15.1 only)

- [ ] **AC1** — Migration 0053 applies cleanly and is idempotent on re-run; creates the 4
  tables with their indexes; `coordination_events` PK is `(topic_id, seq)`.
- [ ] **AC2** — `charter_topic` / `POST /topics` creates a `chartered` topic and emits
  `topic.chartered` as `seq=1` **in the same transaction**; failure of either rolls back both.
- [ ] **AC3** — `join_topic` / `POST /topics/:id/join` auto-registers the actor
  (project-scoped, upsert), inserts a `topic_participants` row at the declared level, emits
  `topic.actor_joined`, flips a `chartered` topic to `active`, and returns an induction
  pack `{ topic, roster, events }`.
- [ ] **AC4** — Re-joining an already-joined actor is idempotent: no duplicate participant
  row, no duplicate `topic.actor_joined` event; the call still returns a fresh induction
  pack. A re-join with a conflicting `type` is rejected with a clear error.
- [ ] **AC5** — `get_topic` / `GET /topics/:id` returns the topic record plus the full
  participant roster.
- [ ] **AC6** — `close_topic` / `POST /topics/:id/close` transitions `chartered|active →
  closed`, emits `topic.closed` as the final event; after close, any `appendEvent` is
  rejected (`TOPIC_CLOSED`) — the log is sealed.
- [ ] **AC7** — `replay_topic_events` / `GET /topics/:id/events?since=:seq` returns events
  with `seq > cursor` ordered by `seq` ascending; the cursor is treated as a high-water
  mark (a missing seq is never waited on).
- [ ] **AC8** — `seq` is monotonic per topic and transactionally allocated: a
  concurrent-append unit test produces distinct, increasing seq values and zero 500s.
- [ ] **AC9** — `GET /topics/:id/stream` (SSE) pushes newly-appended events to a connected
  client and cleans up its connection on client disconnect.
- [ ] **AC10** — Actor identity is project-scoped: the same `actor_id` joining topics in
  two different projects yields two distinct `actors` rows.
- [ ] **AC11** — REST endpoints mirror the 5 MCP tools 1:1 (charter/join/get/close/replay);
  all responses use one envelope `{ status, data?, error? }`; SSE is GUI-only.
- [ ] **AC12** — `tsc --noEmit` clean (backend); all new unit tests pass; existing test
  suite still green.
- [ ] **AC13** — Live smoke against the running Docker stack: charter → join → get →
  replay → close, with the event log showing `topic.chartered`, `topic.actor_joined`,
  `topic.closed` in seq order.

## Pre-emptive self-review (for the cold-start Adversary)

Anticipated challenge points, surfaced now so DESIGN addresses them:
1. **`appendEvent` transaction contract** — it *must* run inside a caller transaction; a
   caller that passes a raw `Pool` instead of a `PoolClient`, or forgets `BEGIN`, breaks
   event/state atomicity. DESIGN must make the signature force a `PoolClient`.
2. **Seal vs. seq race** — `close` emits `topic.closed` (status still `active`) then flips
   to `closed`; an `appendEvent` interleaving between the two must still be ordered before
   `topic.closed` or rejected. Both run under the topics-row lock, so they serialize —
   DESIGN must show the lock ordering explicitly.
3. **Idempotent join** — the "no duplicate event" guarantee needs the participant INSERT
   and the event emission to be conditional on the same `ON CONFLICT` outcome, in one txn.
4. **SSE backpressure / missed notifies** — `LISTEN/NOTIFY` can coalesce or drop notifies
   under load; the SSE handler must re-query from its cursor on every notify (not assume
   one notify == one event) and must not rely on `NOTIFY` payload completeness.
5. **`project_id` provenance on join** — derived from the topic, never caller-supplied, so
   an actor cannot join a topic into the wrong project's actor namespace.
