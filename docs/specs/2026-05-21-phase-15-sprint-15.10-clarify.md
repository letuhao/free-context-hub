## Scope

Resolve **DEFERRED-022** — multi-tier collective request-step routing. Sprint 15.8
shipped collective on single-step routes only; multi-step counter_sign+collective
was hard-rejected at `submitRequest` because the DoA matrix carries ONE `body_id`
per row, so all steps in a route would share one body — collapsing the distinct-
endorser guarantee.

This sprint enables the realistic governance pattern: "coordination committee
endorses, then authority board endorses" — TWO collective steps with TWO DIFFERENT
bodies.

Out-of-scope:
- DEFERRED-015/016/017 (HARD pre-prod authz triggers) — same posture as before.
- DEFERRED-009/010 (topic-scope authz, induction-pack pagination) — defer.

## Acceptance Criteria

- **AC1** — Per-level body source via Q1 design decision (see Open Questions). Each
  collective matrix row carries a body_id per level the route may traverse.
- **AC2** — `submitRequest` for a multi-step counter_sign+collective route (e.g.,
  levels=[coordination, authority]) succeeds when DISTINCT bodies are assigned per
  level. Persists `request_steps.body_id` per step from the map.
- **AC3** — Multi-step counter_sign+collective with SAME body for 2+ steps → submit
  rejects with `BAD_REQUEST` (`distinct_body_required_for_counter_sign_collective`).
- **AC4** — `submitRequest` for a multi-step collective route where ANY level lacks
  a body assignment → `BAD_REQUEST` (`missing_collective_body`).
- **AC5** — At step 0 activation, auto-propose motion under step 0's body. At step
  1 carries → advance to step 2 (if exists), auto-propose under step 2's body.
  Each step gets its own motion under its own body.
- **AC6** — Lapsed motion at an intermediate collective step:
  - Q2 (a) — **re-propose under the higher level's body** if one is configured in
    the map (preserves collective semantics at each level).
  - Q2 (b) — degrade-to-unilateral (15.8 behavior, simpler).
  - At top tier (authority lapsed): escalation_exhausted (unchanged).
- **AC7** — Existing single-step collective (Sprint 15.8) routes continue to work.
  The 15.8 matrix rows (one body_id per row) are backward-compatible.
- **AC8** — Existing unilateral routes unaffected. AC10 from 15.8 still holds.
- **AC9** — Distinct-body guard on counter_sign+collective routes: 15.10 enforces
  that no two steps in the route share a body_id (rejects at submission with
  `distinct_body_required`). This is the STRONGEST 15.10 guarantee — analogous to
  DEFERRED-013 "distinct endorser per step" mapped to the body level. **Per-actor
  cross-body collusion** (one actor being a member of MULTIPLE distinct bodies on
  the same route, voting in each) is **out of scope** for 15.10 — interlocks with
  DEFERRED-015 (Phase 15 participant-level authority model). Documented as a
  semantic gap; not a regression because no equivalent property held before 15.10.
  (REVIEW-DESIGN r1 F3 — accept-with-doc.)
- **AC10** — `decideStep` early-reject collective unchanged (15.8 path).
- **AC11** — Live smoke confirms: submit multi-step counter_sign+collective →
  motion 1 carries → motion 2 auto-proposed → motion 2 carries → request approved
  + 15.7 chain fires (1 chained task).
- **AC12** — Backwards compatibility: 15.8 matrix rows with `body_id` populated and
  no per-level entries continue to route as single-step collective (same behavior).

## Open Questions (Q1–Q3)

### Q1 — Per-level body source mechanism

How does the matrix express "level → body" for collective routes?

- **(a) New table `doa_matrix_levels`** (recommended): `(matrix_id, level, body_id)`
  rows; one matrix row → multiple level-body rows. Proper relational model;
  queryable; honest about cardinality.
- **(b) JSON column `doa_matrix.body_by_level JSONB`**: `{"coordination": "uuid",
  "authority": "uuid"}`. Simpler schema; less queryable; requires JSON validation
  for level enum values.
- **(c) Per-submission `collective_bodies` blob at submitRequest**: submitter
  passes `{step_index → body_id}`. Couples submitter to admin-only DoA matrix
  concerns; bypasses "snapshot the rules" discipline.

**Recommendation: (a) new table.** Cleanest architecturally; mirrors Phase 15's
"matrix is records-management" stance (rules frozen at submission). Migration 0062
adds the table + index; doaMatrix.ts joins it in resolveMatrixRow.

### Q2 — Lapsed-escalation at intermediate collective level

Sprint 15.8 degrades to unilateral on lapsed (single-step constraint forced
this). With per-level bodies available, lapsed could re-propose under the
NEXT level's body if configured.

- **(a) Re-propose under next level's body if configured; else degrade to
  unilateral** (recommended). Preserves collective semantics where possible;
  graceful fallback when matrix is incomplete.
- (b) Always degrade to unilateral (keep 15.8 simple behavior across the board).
- (c) Always require next level's body (reject if missing).

**Recommendation: (a)** — graceful. Operators who want strict collective
escalation populate the full map; operators using 15.8-style single-tier get the
existing degrade behavior.

### Q3 — Security-framed cold-start adversarial review?

Per guardrail 5c0b7b25, sprints adding governance primitives require security
review.

- **(a) Skip — standard self-review** (recommended). DEFERRED-022 extends an
  existing primitive (15.8 collective wiring) by adding per-level body
  resolution. No new authority surface; same authz model (admin-controlled
  DoA matrix; body memberships already authoritative for second/vote).
- (b) Run security-framed review. ~30 min extra; conservative.

**Recommendation: (a) skip.** Same logic as 15.8 Q3 (which was also skipped).

## Plan preview (Phase 4 — final decomposition TBD)

T1. Migration 0062: `doa_matrix_levels (matrix_id UUID, level TEXT, body_id UUID,
   PRIMARY KEY (matrix_id, level))` with FK to doa_matrix + decision_bodies + CHECK
   level enum. Backward-compat: existing single-step collective rows continue to
   use `doa_matrix.body_id` (single source); the new table is for multi-step.

   alongside the existing `body_id` (legacy). LEFT JOIN doa_matrix_levels on
   matrix_id. If the table has entries, populate the map; else fall back to
   `{required_level → body_id}` (single-step backward compat).

   - Remove 15.8 multi-step reject.

     reject `missing_collective_body`.
   - Distinct-body check on multi-step counter_sign+collective: enumerate per-step
     body_ids; reject if any duplicate.
   - INSERT each step's body_id from the resolved map.

T4. `requests.ts:applyMotionToStep` lapsed branch — Q2 (a):
   - Read next level's body_id from matrix (via fresh resolveMatrixRow call OR
     stored snapshot — see design).
   - If body exists → re-propose under it (collective at new level).
   - Else degrade to unilateral (15.8 behavior).

T5. Tests:
   - requests.test.ts: AC2/AC3/AC4 (positive + distinct-body neg + missing-body neg).
   - motions.test.ts: AC5/AC11 (multi-step carries → next motion → final approve + chain).
   - coordinationSweep.test.ts: AC6 (lapsed re-propose vs degrade per Q2).
   - Sprint 15.8's AC1-neg test ("multi-step counter_sign+collective rejected") is
     UPDATED to test the new path (no longer auto-rejects; checks distinct-body).

T6. VERIFY + live smoke + SESSION + COMMIT + RETRO.

## Risks

1. **Lapsed-escalation matrix re-resolution**: applyMotionToStep needs the new
   level's body_id. Either re-call resolveMatrixRow (matrix lookup is plain SELECT,
   acceptable) OR persist the full body_by_level map onto the request at submission.
   The latter aligns with "snapshot the rules" but bloats request_steps. Decide in
   DESIGN.

2. **Single-step backward compat**: Sprint 15.8 matrix rows with `procedure=
   'collective'` + `body_id=<X>` + NO entries in doa_matrix_levels MUST continue
   to work as before. The fallback in resolveMatrixRow (Q1 (a)) handles this:
   empty map → use single body_id for the required_level.

3. **Distinct-body enforcement**: counter_sign+collective with intentionally same
   body (operator misconfiguration) → reject at submitRequest. Defensive; aligns
   with DEFERRED-013 distinct-endorser principle.
