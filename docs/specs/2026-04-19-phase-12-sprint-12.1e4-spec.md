# Spec — Phase 12 Sprint 12.1e4 — α × HL grid sweep under NOWRITE

Second salience-tuning sprint. First (12.1e2) tuned half-life and got a drift-artifact-driven result; 12.1e3 reverted that and shipped the NO_WRITE gate. Now we use the clean measurement infra to properly sweep α.

## Problem

`LESSONS_SALIENCE_ALPHA` defaults to 0.10 (set in Sprint 12.1c by instinct). Two open questions:

1. **Is α=0.10 the right default at the current HL=7?** Analytically, at HL=7 NOWRITE, bootstrap-only salience is ~0.012 per guardrail lesson; even α=0.50 × 0.012 × 0.7 ≈ 1.004 boost — indistinguishable from α=0. Expected: null across the α dimension at HL=7.
2. **Does α matter at HL=30?** Under HL=30 NOWRITE, bootstrap-sum saturates salience at ~1.0 for guardrail lessons. α then directly controls boost magnitude (10%-50% final-score lift for maximally-salient lessons). Expected: real α-sensitive behavior at HL=30.

The 2×2 analysis in 12.1e3 showed that HL alone doesn't shift aggregate metrics under NOWRITE. This sprint tests whether α does. If α provides measurable lift at HL=30 without hurting HL=7, that's an argument for future tuning. If α is null at both HLs, the salience feature — under clean measurement on this goldenset — has no measurable production effect, and we should document that finding for future sprints.

## Scope

**IN SCOPE (this sprint):**
1. **8 baseline runs** against the 40q goldenset:
   - `α ∈ {0.05, 0.10, 0.20, 0.50}` × `HL ∈ {7, 30}` = 8 combinations
   - `LESSONS_SALIENCE_NO_WRITE=true` (reuse 12.1e3 gate)
   - `--samples 1` per run (retrieval is deterministic; samples-for-latency not needed)
2. **Analysis:** aggregate metrics + per-group nDCG@10 per run, 4×2 matrix presentation.
3. **Summary doc** `docs/qc/baselines/2026-04-19-sprint-12.1e4-summary.md` leading with top-line recommendation BEFORE data tables (per 12.1e2 R2 mitigation).
4. **Human decides default change** at POST-REVIEW. If α=X beats α=0.10 meaningfully at HL=7 or HL=30, update `src/env.ts` default accordingly. If null, keep α=0.10 and document.

**OUT OF SCOPE:**
- Other salience code changes (blend math, write paths, etc.)
- Changes to HL default (reverted to 7 in 12.1e3; no reason to touch again)
- Other parameter sweeps (half-life window >= 1yr, or cross-project tuning)
- New goldenset queries

## CLARIFY decisions locked

**D1 — Grid shape:** 4 × 2 = 8 runs. α values chosen to span from "tiny boost" (0.05) to "maximal boost" (0.50) with the current default (0.10) and a middle value (0.20). HL values: current default (7) and the 12.1e2 historical pick (30). Not including α=0 or HL=0 — α=0 is equivalent to NO_WRITE+α=N (no boost regardless), and HL=0 is invalid per the env.ts min=1 constraint.

**D2 — Sample count:** `--samples 1`. Retrieval is deterministic given fixed state. Sample count only affects latency p95 stability, which isn't a sprint metric.

**D3 — NOWRITE always on:** every run has `LESSONS_SALIENCE_NO_WRITE=true`. The 12.1e3 flag and docker-compose infra make this the default protocol for salience-sensitive A/B.

**D4 — Run order:** `HL=7 × [α=0.05, 0.10, 0.20, 0.50]`, then `HL=30 × [α=0.05, 0.10, 0.20, 0.50]`. Sequential. Each run is independent (NOWRITE keeps state identical).

**D5 — Decision rule for default change:** observation-led (like 12.1e2). Present 4×2 matrix; human decides. Candidate default change only if some (α, HL) pair beats (α=0.10, HL=7) on aggregate nDCG@10 by ≥ noise floor (0.013) WITHOUT regressing any per-group beyond noise floor.

**D6 — Filename convention:** `2026-04-19-sprint-12.1e4-hl{HL}-a{α×100, 3-digit zero-padded}.{json,md}`. E.g., `hl7-a010` for HL=7, α=0.10. Matches 12.1e2's hl{3,7,14,30} pattern.

## Risks + mitigations

**R1 — Null result everywhere.** Per analytical prediction, α at HL=7 has no effect. If HL=30 ALSO shows null, the salience feature is effectively inert under NOWRITE on this goldenset. Mitigation: that's a finding, not a failure. Document it as a candidate friction class ("salience is a no-op under NOWRITE; production drift is what makes it work"). Sprint still delivers evidence-based answer.

**R2 — Between-run access-log pollution.** NOWRITE flag must stay ON every run. Mitigation: verify env inside container before each baseline run; if ever shown false, stop + debug.

**R3 — Goldenset saturation on confident-hit/duplicate-trap groups.** These are already at 1.0; no signal available. Any α effect has to come from cross-topic / ambig / paraphrase. Small effect-size space. Mitigation: primary metric is per-group nDCG@10 on the non-saturated groups.

**R4 — Docker build step might be needed before first run.** 12.1e3's NO_WRITE gate is now in the mcp image (per commit ac5c556 + rebuild). But if anything changed since, rebuild. Verify at T0.

**R5 — 8 baselines × 5min = 40min baseline time + analysis. User cadence might not support this without breaks.** Mitigation: kick off each run via `run_in_background`, use Monitor for progress events. User doesn't need to babysit.
