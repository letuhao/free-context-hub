# Design — Phase 12 Sprint 12.1e4

Resolves 3 CLARIFY-deferred questions.

## 1. Baseline-run loop

**Inline execution, one run at a time** (matches 12.1e2/12.1e3 pattern). For each of the 8 combinations:

1. `LESSONS_SALIENCE_HALF_LIFE_DAYS=<HL> LESSONS_SALIENCE_ALPHA=<α> LESSONS_SALIENCE_NO_WRITE=true docker compose up -d --force-recreate mcp`
2. Wait for ready: `until curl -sf ...`
3. Verify env: `docker compose exec -T mcp node -e "console.log(process.env.LESSONS_SALIENCE_ALPHA, process.env.LESSONS_SALIENCE_HALF_LIFE_DAYS, process.env.LESSONS_SALIENCE_NO_WRITE)"` → matches expected
4. Run baseline: `npm run qc:baseline -- --tag sprint-12.1e4-hl<HL>-a<α×100 padded> --samples 1 --surfaces lessons` (background + Monitor)
5. Verify archive exists + COUNT unchanged
6. Move to next combination

**Not using a shell-loop script** because (a) the 8 runs are ~40min total; babysitting via Monitor is fine, (b) shell-loop adds failure modes (one bad run hangs the chain), (c) explicit per-run verification keeps ACCEPTABLE latency and trust.

**No docker build needed at sprint start** — 12.1e3's NO_WRITE gate is already baked in (commit ac5c556 + rebuild). Verify at T0 with a quick `docker compose exec mcp node -e "console.log(typeof isSalienceWriteDisabled)"`... actually just run a quick baseline with flag=true and check COUNT — if COUNT stays 90, the gate is in the image.

## 3. Commit granularity

- **C1** — spec + design + plan (docs-only). This commit happens at plan completion.
- **C2** — all 8 baseline archives + summary doc. One commit since they're produced together.
- **C3** (conditional) — `src/env.ts` + `docker-compose.yml` default change, if human chooses to update the default at POST-REVIEW.

Minimum 3 commits (C1 + C2 + C4). Maximum 4 (if default changes).

## Additional design decisions

### D-ADD-1 — Between-run access-log verification

Before each run, confirm `COUNT(*) FROM lesson_access_log = 90`. If ever higher, something wrote — either NOWRITE isn't in effect or the image is stale. Halt + debug. Adds ~2 seconds per run (1 SQL query).

### D-ADD-2 — Run order sequencing

Order: `hl7-a005 → hl7-a010 → hl7-a020 → hl7-a050 → hl30-a005 → hl30-a010 → hl30-a020 → hl30-a050`. Varying α inner (more frequent env changes), HL outer. Rationale: each HL section is internally consistent; compare α within-HL first before crossing to HL=30.

### D-ADD-3 — Container recreate cost

Each run needs a container recreate (env change) + wait for ready (~10s). 8 recreates = ~80s total overhead. Acceptable.

### D-ADD-4 — α default change thresholds

Candidate default change only if (matching spec D5):
- Aggregate nDCG@10 improvement ≥ noise floor (0.013)
- AND no per-group regression beyond noise floor
- AND the improvement is CONSISTENT across both HLs (or HL=7 specifically, since that's the current default)

If these aren't met, keep α=0.10.

## Risks introduced at design time

**R-D1 — Env change without container recreate.** If I accidentally run baseline without recreate-after-env-change, the previous run's env persists. Mitigation: verification step (D-ADD-1 + env exec print) catches this.

**R-D2 — Aggregate summary-doc data wrong.** 8 archives × 6 metrics = 48 cells to potentially mis-paste. Mitigation: produce tables programmatically via node script reading all 8 JSONs; human only writes top-line recommendation + per-query observations.
