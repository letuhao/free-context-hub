## Scope

Resolve **DEFERRED-018** — wire `procedure='collective'` into the Request-Approval
lifecycle so a request step may be decided by a **motion's tally** (15.4 collective-
decision primitive) instead of by a single officeholder's `decideStep`.

The Phase 15 master design §B.7 already specifies: *"At each step an approver may
endorse-and-forward, return, or reject. A step's decision procedure may itself be
unilateral or collective (B.6)."* Sprint 15.3 implemented unilateral steps and rejected
`procedure='collective'` outright; Sprint 15.4 built the standalone motion primitive
but did not wire it into request steps. This sprint closes that loop.

## Acceptance criteria (proposed — to refine post-CLARIFY answers)

- **AC1** — `submitRequest` accepts `procedure='collective'` (no longer hard-rejects).
  At least one step in the route must have `body_id` populated when procedure is
  collective; otherwise return `BAD_REQUEST` (`collective_body_required`).
- **AC2** — Per-step procedure is sourced from the DoA matrix (`doa_matrix.procedure`)
  via `resolveMatrixRow`. If `procedure='collective'`, the matrix row's `body_id` must
  be non-null (otherwise `no_route` — same response as missing matrix).
- **AC3** — A request submitted with a collective step persists `request_steps.procedure
  ='collective'` and `request_steps.body_id=<body>` (frozen from matrix at submission,
  same snapshot discipline as `target_office` + `doa_snapshot`).
- **AC4** — At step activation (transition from inactive → current), if
  `procedure='collective'`, the system **auto-proposes a motion** under the step's
  `body_id`. The motion's `subject_ref` is `request_step:<request_id>:<step_index>`;
  proposer is `'system:request-step-proposer'`; deadline = `now() + STEP_DEADLINE_
  MINUTES`. Sets `request_steps.motion_id = <new motion_id>` and `request_steps.status
  = 'motion_proposed'`.
- **AC5** — `decideStep` on a step with `procedure='collective'` returns
  `{status: 'procedure_is_collective'}` (HTTP 422). Collective steps are decided by
  the motion, not by `decideStep`.
- **AC6** — When the linked motion is tallied (`tallyMotion` user-driven OR
  `sweepExpiredMotions` auto-tally), the step is updated atomically in the same
  transaction:
  - `carried` → step `'endorsed'`, advance request (next step or approve final).
  - `failed` → step `'returned'`, request `'returned'` (terminal).
  - `lapsed` → step `'escalated'` (same as unilateral pending-past-deadline path:
    re-target up one level; if already at authority, request → `escalation_exhausted`).
  - `vetoed` → step `'rejected'`, request `'rejected'` (terminal).
- **AC7** — The collective-step approve path (last step `'endorsed'` via carried motion)
  fires the existing 15.7 chain handler — chain task posted on the topic (or task.deferred
  on closing).
- **AC8** — **REMOVED in rev 2 (post-DESIGN F2 reconciliation).** The previous
  AC8 required a distinct-body guard for multi-step collective counter_sign routes.
  DESIGN §2.2 hard-rejects multi-step counter_sign+collective entirely for 15.8
  because the DoA matrix carries one body per row (collapse to single endorser).
  Multi-tier collective routing (different body per level) is deferred to a future
  sprint that introduces per-level body assignment in the DoA matrix. 15.8 supports:
  (a) single-step counter_sign collective, or (b) escalate_to_authority collective.
- **AC9** — `proposeMotion` on a body that is currently assigned to an active
  `request_step` (i.e., the body is mid-deliberation on a step's auto-proposed motion)
  is **not blocked** — the body's calendar is the application's concern, not the
  primitive's. We may revisit if multi-motion contention surfaces in practice.
- **AC10** — Existing tests still pass. Specifically: existing unilateral request tests
  must continue to work (DoA matrix migrations default to `procedure='unilateral'` for
  all existing rows; `request_steps.body_id` defaults NULL).

## Open questions (Q1–Q3 — most consequential)

**Q1 — Motion proposal mechanism:** when a collective step becomes active, how is the
motion proposed?

  - **(a) Auto-propose by system actor at step activation (recommended).** The system
    acts as proposer; the body's members second + vote. Clean — no race between submit
    and activation; the motion always exists when the step is active. Requires
    `'system:request-step-proposer'` as a valid `proposed_by` (motion service does not
    currently restrict the proposer to a body member; only `secondMotion` and `castVote`
    require body membership — so this works without changes).
  - (b) Explicit API — caller must `proposeStepMotion(step_id, proposer)` after
    submission. Race: between submission and propose, the step has no motion;
    `decideStep` and the escalation sweep need to handle this gap. Bigger surface.
  - (c) Submitter declares the motion at submission (extended `execution_task`-style
    blob). Coupling of motion params to request submission — design smell.

**Q2 — `lapsed` motion → step outcome:** a motion that doesn't reach quorum by deadline
auto-tallies to `lapsed` (sweepExpiredMotions).

  - **(a) Escalate the step up one level (recommended).** A lapsed motion = the body
    didn't have enough participation; treat it as "this office couldn't decide" and
    escalate to the next authority — same as the unilateral pending-past-deadline path.
    At the authority tier, → dispute (existing `escalation_exhausted` semantics).
    If the new escalated level is collective too, auto-propose a new motion under that
    level's body (which may differ from the original).
  - (b) Returned — treat lapsed = "no decision; revisit". Simpler; consistent with
    `failed` (both mean "didn't carry"). But loses the temporal-escalation semantics.
  - (c) Stay open + extend deadline — keep proposing fresh motions on the same body
    until one carries. Risk: infinite loop on a permanently-non-quorate body.

**Q3 — Security-framed adversarial review required?**

  - **(a) Skip — standard self-review (recommended).** Wiring two existing authorized
    primitives. The motion's authz (body membership for second/vote, veto_holders for
    veto) is unchanged. The request's authz (level-targeting, self-decision guard) is
    untouched. No new authority surface; the cross-primitive contract is purely
    behavioral. v2.2 default self-review at REVIEW-DESIGN + REVIEW-CODE is sufficient.
  - (b) Run security-framed cold-start adversarial review. Conservative — treat the
    new motion-step linkage as an exploitable surface (e.g., can a malicious actor
    propose a body that decides their own request? But submitRequest validates
    body_id comes from the DoA matrix, not caller-supplied).

## Risks

1. **Status-string proliferation:** new `request_steps.status='motion_proposed'` joins
   the existing `pending|endorsed|returned|rejected|escalated`. The existing
   sweep-skip predicate (`WHERE status='pending'`) already excludes motion_proposed,
   so the escalation sweep won't misfire. But every status-check site must be reviewed
   for completeness.

2. **Concurrency between tallyMotion (user-driven) and sweepExpiredMotions (auto):**
   the motion is locked FOR UPDATE in both paths (existing 15.4 invariant), so they
   serialize. The step update inside our new branch runs inside the motion-lock, so
   it's safe.

3. **Repeat-body guard (AC8) on counter_sign routes:** per-request distinct body —
   enforced at submit-time. But the DoA matrix is the source of bodies; an admin
   configuring two same-body matrix rows would surface this as a submitRequest
   rejection. Operationally clean.

4. **Body lifecycle vs request lifecycle:** if a body is deleted (no delete API exists
   today, but FK is `body_members.body_id REFERENCES decision_bodies(body_id)` — no
   cascade), the linked request_step's motion would have a dangling reference. Today
   no body-delete code path exists; surface as a future concern.
