# Spec — Phase 12 Sprint 12.1g — TEI external rerank integration

## Problem

Sprint 12.1f found that LM Studio's current rerank path (`/v1/embeddings` + cosine) only works for bi-encoder-compatible models like `gte-reranker-modernbert-base`. True cross-encoders like `bge-reranker-v2-m3` and `jina-reranker-v3` need a `/v1/rerank` endpoint that scores `(query, doc)` pairs together. LM Studio does not expose any such endpoint — confirmed by probing (`/v1/rerank`, `/rerank`, `/v1/reranking`, etc. all return "Unexpected endpoint").

CLARIFY experiment (option 4) validated that HuggingFace's `text-embeddings-inference` (TEI) Docker image, loading `BAAI/bge-reranker-v2-m3`, supports `POST /rerank` out of the box:
- Takes `{query, texts: [...]}` input.
- Returns `[{index, score}, ...]` sorted descending.
- Correctly ranks the pg UUID query's target at rank-1 with 0.0736 score (vs 0.00002 for unrelated lessons — clear separation).
- CPU-only performance: ~1s per query with 15 candidates → ~40s for the 40q goldenset.

## Scope

**IN SCOPE:**
1. **Add `tei-rerank` service** to `docker-compose.yml`. CPU, `ghcr.io/huggingface/text-embeddings-inference:cpu-latest` image, loads `BAAI/bge-reranker-v2-m3`. Named volume `tei_model_cache` for model persistence.
2. **New env knob `RERANK_TYPE=api`** that triggers a new code path calling an external `/rerank` endpoint (Cohere/TEI-compatible shape).
3. **Implement `rerankExternalApi(query, candidates)`** in `src/services/lessons.ts` alongside existing `rerankCrossEncoder` and `rerankGenerative`. POSTs to `${RERANK_BASE_URL}/rerank` with `{query, texts: [title + snippet, ...]}`. Parses Cohere/TEI-style `[{index, score}]` response. Returns the new index order.
4. **Updated `rerankLessons` dispatch** in `src/services/lessons.ts` to handle the new `api` type.
5. **Env config:** `RERANK_BASE_URL=http://tei-rerank:80` points MCP at the new service (via Docker network).
6. **Baseline run** with the new rerank path against the 40q goldenset. Compare to:
   - Sprint 12.1f's generative baseline (MRR=1.0, nDCG@10=0.9724) — current prod
   - gte (MRR=0.9538, nDCG@10=0.9237) — current "measurement winner"
   - no-rerank (MRR=0.9198, nDCG@10=0.9100) — lower bound
7. **Determinism check:** same-config repeat, verify 0/40 query diffs.
8. **Decision:** if TEI+bge beats or matches generative (MRR ≥ 0.98, nDCG@10 ≥ 0.96) AND is deterministic, recommend as new production default.

**OUT OF SCOPE:**
- Multiple reranker models on TEI (one model per TEI instance).
- GPU-based TEI (avoids competition with LM Studio for VRAM).
- Infinity or other rerank servers (TEI is chosen; if TEI fails architecturally, document and defer).
- Salience math changes.

## CLARIFY decisions locked

**D1 — TEI over Infinity.** Both supported by upstream cross-encoders. TEI is HF-official, mature, slightly better docs. Pick TEI.

**D2 — CPU-only TEI.** Avoids GPU contention with LM Studio. ~1s per query on bge-reranker-v2-m3 is acceptable for 40q baselines (~40s total). User can switch to GPU image later if needed.

**D3 — Model: `BAAI/bge-reranker-v2-m3`.** Strong baseline, known-good on our goldenset via experimental check. 560MB download on first TEI startup; cached afterwards via named volume.

**D4 — Env value: `RERANK_TYPE=api`.** Vague enough to support TEI, Infinity, Cohere, or any `/rerank` compat endpoint. Schema: `z.enum(['generative', 'cross-encoder', 'api'])`.

**D5 — Request body shape: `{query, texts}`.** Matches TEI native format. `model` field NOT sent (TEI serves only the model it was started with). If we later target Cohere directly, we may need a `{model, query, documents}` variant; defer until needed.

**D6 — Response parsing: `[{index, score}]` array.** TEI native format. Returns indices into the `texts` array, sorted by score descending.

**D7 — Naming: `rerankExternalApi`.** Distinguishes from the two LM-Studio-bound reranker functions. Clear intent.

## Risks + mitigations

**R1 — TEI model download on first `docker compose up` takes ~4min.** One-time cost per volume. Mitigation: use named volume `tei_model_cache` for persistence across compose up/down cycles.

**R2 — CPU-only TEI might be too slow for real usage.** At ~1s/query, 40q baseline = ~40s (fine). For production, each user query adds ~1s latency (noticeable but acceptable). Mitigation: document latency; user can switch to GPU image if needed.

**R3 — TEI container runs out of memory.** bge-reranker-v2-m3 is ~2GB resident. Mitigation: document RAM requirement (~2.5GB for TEI container alone).

**R4 — Network unreachable from mcp to tei-rerank.** Docker network misconfig. Mitigation: health-check + startup logging + verification step before first baseline run.

**R5 — Quality might be WORSE than generative.** Possible — generative LLM rerank has "semantic reasoning" that bge-v2-m3 lacks. Mitigation: data-driven decision per A9; if worse, keep generative for prod; still document determinism.

**R6 — `RERANK_TYPE=api` code path could accidentally trigger when RERANK_BASE_URL isn't set.** Mitigation: guard function early-return if `RERANK_BASE_URL` is empty.
