# Sprint 13.2 — CLARIFY: F1 TTL sweep job + Active Work GUI

## What this sprint ships

Two complementary deliverables that close F1 (Artifact Ownership / Leasing):

1. **Background TTL sweep job** — periodically deletes leases whose `expires_at` is more than 1 hour in the past. Triggered by an in-process `setTimeout` scheduler in `src/index.ts` that enqueues a `leases.sweep` job every 15 minutes. The job runs via the existing job queue (RabbitMQ or Postgres backend, whichever is enabled).

2. **Active Work GUI panel** — a new section on the existing Agents page (`/agents`) that lists all currently-active artifact leases for the selected project. Shows artifact id/type, agent, task description, time remaining. 10-second auto-refresh. Force-release action (visible to admin role) calls the existing `DELETE /:leaseId/force` REST endpoint.

These two together complete F1's remaining ACs (7, 8) — Sprint 13.1 shipped ACs 1–6 plus extras (attempt-rate limit, id-format validation, etc.).

## Design assumptions (locked unless reviewed otherwise)

**A1.** Scheduler lives in `src/index.ts` (server process), not the worker. Rationale: server is the entry point that always starts; worker may be disabled in low-resource deployments. The `setTimeout` enqueues a job — actual execution happens in the worker. If the worker is offline, jobs queue up in Postgres harmlessly (the partial unique index in `artifact_leases_active_uniq` already excludes expired leases from uniqueness checks, so stale rows do not cause functional bugs — only DB bloat).

**A2.** Sweep cadence: 15 minutes (per design doc §"TTL enforcement"). First fire happens 15 min after startup, not immediately — avoids a redundant sweep right after the lazy cleanup that ran at startup. Subsequent fires use `setTimeout` chained (not `setInterval`) to prevent overlap if a sweep ever takes >15 min.

**A3.** Grace period: deletes leases where `expires_at < now() - interval '1 hour'`. The grace window keeps recently-expired leases briefly as audit/debug evidence. Pulled directly from design doc.

**A4.** Single-process timer: this codebase runs one MCP+API process in docker-compose. If multi-replica deployment ever happens, the timer becomes 1-per-replica = N redundant sweeps every 15 min. Documented as known limitation; deferral path: convert to a DB-lock-guarded distributed timer (similar to `leases.sweep` itself becoming idempotent via the WHERE clause). Tracked as in-line comment, not a DEFERRED entry yet — premature.

**A5.** Sweep job is fire-and-forget by the timer (`enqueueJob` returns immediately). If RabbitMQ is down, the Postgres-backed queue is the fallback. If Postgres is also down, the timer logs an error and re-schedules.

**A6.** The `setTimeout` handle is held in module scope but not exported for cleanup. SIGINT handler in `src/index.ts:80` calls `process.exit(0)` which clears all timers — no leak risk in current shape. If hot-reload tooling is added, the handle needs explicit `clearTimeout` — flagged for future work, not in scope.

**A7.** GUI: "Active Work" section is **additive** to the existing agents page. Inserted in a new card/section below the agent-trust table that currently exists. The 10-second auto-refresh uses `setInterval` inside a `useEffect`; cleanup on unmount. Manual refresh button + "Last updated X ago" label per design doc.

**A8.** Force-release UX: button visible on every row in the table to all users. On click, calls `DELETE /:leaseId/force`. If the backend returns 403 (non-admin), surface a toast "Admin role required" and don't update the table. If 200, optimistic-remove the row + show success toast + trigger refresh. Rationale: there is no `/api/me/role` endpoint in the codebase today; adding one is out of scope and the 403-handling UX is acceptable since the action is rare. (Alternative noted under §Open Questions.)

**A9.** Active Work panel respects the project context (`useProject()`): when "All Projects" is selected, the panel shows leases across all effective projects (multi-project list call). When a specific project is selected, single-project call. To avoid a new MULTI endpoint this sprint, "All Projects" mode iterates client-side over `effectiveProjectIds` and concatenates results (N small fetches in parallel). Acceptable because N is small (typically 1–3 projects per session) and the operation is read-only with 10s cadence.

## Open questions (decided inline rather than blocking)

**Q1.** Should the Active Work panel poll only when visible on screen? Default: yes — pause auto-refresh when the tab is hidden (`document.visibilityState !== 'visible'`). Trivial guard, easy win for resource use.

**Q2.** Should force-release require a confirm dialog? Default: yes — destructive admin action. Reuses existing toast/modal pattern from elsewhere in the GUI.

**Q3.** Should the timer run when `QUEUE_ENABLED=false`? In that case the postgres-backed queue is the only path and the worker is the only consumer. If the worker isn't running, `leases.sweep` jobs sit queued forever. Default: run the timer regardless. Bloated `async_jobs` table is recoverable; missed sweeps are not. The worker is the user's operational concern.

**Q4.** What happens if two sweeps overlap due to slow DB? Sweep is `DELETE … WHERE expires_at < now() - interval '1 hour'` — idempotent and concurrency-safe. No mutex needed.

**Q5.** Should sweep emit metrics (rows-deleted count)? Yes — logger.info({ rows_deleted: N }) at completion. Useful for ops, costs nothing.

## Out of scope this sprint

- Distributed scheduler (single-replica is fine for now)
- Multi-project `listActiveLeases` REST endpoint (client-side fan-out is good enough)
- Custom GUI styling for the "Active Work" panel beyond what the existing Agents page styling provides
- Rate-limiting the force-release endpoint (admin action, trust the role)
- Long-term retention of expired leases for audit (the grace window is enough)
- DLF-specific lesson types (covered by F3 in sprints 13.5+)

## Approach options considered

**Option A (selected): single in-process setTimeout in src/index.ts**
- Pros: minimal moving parts, matches design doc verbatim, easy to reason about
- Cons: single-replica only

**Option B (rejected): cron-based external scheduler (e.g., shell cron)**
- Pros: standard ops pattern
- Cons: adds external dependency the design doc explicitly says to avoid; harder to test

**Option C (rejected): DB-triggered cleanup via Postgres pg_cron**
- Pros: zero application code
- Cons: not all deployments have pg_cron; adds an extension dependency; the design doc explicitly went with setTimeout

Option A wins on simplicity and design alignment.

## Risk register

- **R-low**: timer drift over long uptime — `setTimeout` chaining keeps cadence within ~ms tolerance for 15-min intervals; acceptable.
- **R-low**: worker offline → backlog of sweep jobs — sweep is idempotent, draining backlog is harmless.
- **R-med**: Active Work panel + 10s polling could load the API at scale (many users open the page) — N requests/10s × M users. At M=10 users, F=1 req/s — fine. Larger scale not in scope.
- **R-low**: stale UI flash during refresh — use stale-while-revalidate render pattern (don't blank the table on refetch).
