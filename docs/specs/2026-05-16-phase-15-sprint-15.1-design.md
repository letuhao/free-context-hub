## 0. Instantiation decisions — deviations from the design doc, resolved at DESIGN

The Phase 15 design doc states (Part E) its DDL is "indicative … Final column lists are
settled per migration" and lists Open Questions to settle in the sprints. This sprint
settles five, each a conscious, documented choice:

| # | Design doc | Sprint 15.1 decision | Rationale |
|---|---|---|---|
| D1 | Open Q2: `seq` allocation undecided | **`topics.next_seq` BIGINT counter**, bumped `UPDATE … RETURNING` in the append txn | Row-lock on the topics row serializes per-topic appends; same-txn increment ⇒ seq is **gap-free** in 15.1 — strictly stronger than the design's gap-tolerant floor, fully compatible (a gap-tolerant replay trivially accepts gap-free input). |
| D2 | Open Q5: SSE shape undecided; 15.1 line says "SSE" | **SSE shipped, poll-based** (not `LISTEN/NOTIFY`) | No backend SSE/`LISTEN` precedent exists; `LISTEN/NOTIFY` is greenfield connection-lifecycle risk. | D3 | C.1 `coordination_events.type` has no CHECK | **No DB CHECK on `type`; service-layer validation** against the `EVENT_TYPES` catalog | The event-type set grows every sprint; a DB CHECK would force a constraint migration per sprint (the Phase 13 0049/0051 pain). `subject_type` is a small stable 7-value set ⇒ it *does* get a DB CHECK. |

## 1. Migration `0053_coordination_substrate.sql`

Four new tables. All `CREATE … IF NOT EXISTS` (idempotent — no existing constraint to
alter, so no DO-blocks; cf. 0048's pattern, not 0049's).

```sql
-- 0053_coordination_substrate.sql — Phase 15 Sprint 15.1
-- Coordination substrate: the append-only event log + Topic/Actor/participant model.
-- See docs/specs/2026-05-16-phase-15-sprint-15.1-design.md.

CREATE EXTENSION IF NOT EXISTS pgcrypto;   -- gen_random_uuid()

-- Topics — chartered initiatives. topic_id is a global TEXT PK (service-generated UUID).
CREATE TABLE IF NOT EXISTS topics (
  topic_id   TEXT        PRIMARY KEY,
  project_id TEXT        NOT NULL,
  name       TEXT        NOT NULL,
  charter    TEXT        NOT NULL,
  status     TEXT        NOT NULL DEFAULT 'chartered'
                CHECK (status IN ('chartered','active','closing','closed')),
  next_seq   BIGINT      NOT NULL DEFAULT 0,   -- per-topic event-seq counter (D1)
  created_by TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS topics_project_idx ON topics (project_id, created_at DESC);

-- Actors — project-scoped identity. Auto-registered on first join.
CREATE TABLE IF NOT EXISTS actors (
  project_id   TEXT        NOT NULL,
  actor_id     TEXT        NOT NULL,
  type         TEXT        NOT NULL CHECK (type IN ('human','ai')),
  display_name TEXT        NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (project_id, actor_id)
);

-- Topic participants — actor ⋈ topic, with a chain-of-command level.
CREATE TABLE IF NOT EXISTS topic_participants (
  topic_id  TEXT        NOT NULL REFERENCES topics(topic_id) ON DELETE CASCADE,
  actor_id  TEXT        NOT NULL,
  level     TEXT        NOT NULL CHECK (level IN ('authority','coordination','execution')),
  joined_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (topic_id, actor_id)
);

-- The event log — append-only spine. Replay cursor is (topic_id, seq).
-- topic_id REFERENCES topics WITHOUT cascade: topics are never deleted (closed != deleted);
-- a delete attempt on a topic that has events MUST fail loud — the log is permanent.
-- `type` has NO CHECK (D3) — validated service-side against EVENT_TYPES.
CREATE TABLE IF NOT EXISTS coordination_events (
  topic_id     TEXT        NOT NULL REFERENCES topics(topic_id),
  seq          BIGINT      NOT NULL,
  event_id     UUID        NOT NULL DEFAULT gen_random_uuid(),
  ts           TIMESTAMPTZ NOT NULL DEFAULT now(),
  actor_id     TEXT        NOT NULL,
  type         TEXT        NOT NULL,
  subject_type TEXT        NOT NULL
                 CHECK (subject_type IN ('task','artifact','request','motion','dispute','intake','topic')),
  subject_id   TEXT        NOT NULL,
  payload      JSONB       NOT NULL DEFAULT '{}',
  PRIMARY KEY (topic_id, seq)
);
-- event_id is globally unique for cross-topic dedup (e.g. future A2A bridge / export).
CREATE UNIQUE INDEX IF NOT EXISTS coordination_events_event_id_uniq ON coordination_events (event_id);
```

## 3. `src/services/coordinationEvents.ts` (new) — the event log

```ts
import type { PoolClient, Pool } from 'pg';

export type CoordinationEventInput = {
  topic_id: string;
  actor_id: string;
  type: string;          // validated against EVENT_TYPE_SET
  subject_type: string;  // validated against SUBJECT_TYPE_SET
  subject_id: string;
  payload?: Record<string, unknown>;
};

export type CoordinationEvent = {
  topic_id: string; seq: number; event_id: string; ts: string;
  actor_id: string; type: string; subject_type: string; subject_id: string;
  payload: Record<string, unknown>;
};

export type ReplayResult = { topic_id: string; events: CoordinationEvent[]; next_cursor: number };
```

### 3.1 `appendEvent(client, evt) → { seq, event_id, ts }`

**Contract: takes a `PoolClient`, never a `Pool`** — it *must* run inside the caller's
open transaction so the state change and its event commit atomically. Algorithm:

1. Validate `evt.type ∈ EVENT_TYPE_SET` and `evt.subject_type ∈ SUBJECT_TYPE_SET`;
   else `throw ContextHubError('BAD_REQUEST', …)`.
2. **Allocate seq + enforce the seal, in one statement:**
   ```sql
   UPDATE topics SET next_seq = next_seq + 1
   WHERE topic_id = $1 AND status <> 'closed'
   RETURNING next_seq
   ```
   `rowCount === 0` ⇒ topic is missing **or** closed ⇒
   `throw ContextHubError('BAD_REQUEST', 'topic <id> is closed or does not exist')`.
   This `UPDATE` takes the `topics`-row lock — **the per-topic append serializer** —
   held to end-of-txn.
3. `INSERT INTO coordination_events (topic_id, seq, actor_id, type, subject_type,
   subject_id, payload) VALUES (…) RETURNING event_id, ts`.
4. Return `{ seq, event_id, ts: ts.toISOString() }`.

### 3.2 `replayEvents({ topic_id, since_seq?, limit? }, executor?) → ReplayResult`

`executor` is an optional `PoolClient | Pool` (default: the pool). Passing a `PoolClient`
lets the read **join an open transaction** — `joinTopic` uses this for its post-commit
snapshot read (§4.2). `since_seq` default `0`; `limit` default `1000` (safety cap — 15.1
topics are small; pagination beyond the cap is a noted future concern).

1. `SELECT 1 FROM topics WHERE topic_id=$1` — none ⇒ `throw ContextHubError('NOT_FOUND', …)`.
2. `SELECT seq,event_id,ts,actor_id,type,subject_type,subject_id,payload
    FROM coordination_events WHERE topic_id=$1 AND seq > $2 ORDER BY seq ASC LIMIT $3`.
3. `next_cursor` = last row's `seq`, or `since_seq` if no rows (high-water mark — never
   waits on a missing seq).

## 4. `src/services/topics.ts` (new) — Topic lifecycle

### 4.0 Transaction & connection-management contract — *[r3-fix]*

Every transactional service function below (§4.1 `charterTopic`, §4.2 `joinTopic`, §4.4
`closeTopic`) follows the **verbatim pattern of Phase 13 `src/services/artifactLeases.ts`
`_claimArtifactOnce` / `renewArtifact`** — a `catch` that rolls back and re-throws, inside
a `finally` that always releases:

```ts
const client = await pool.connect();
try {
  // BEGIN … SQL … COMMIT      (an early `return` path must run ROLLBACK before returning)
} catch (err) {
  await client.query('ROLLBACK').catch(() => {});   // unconditional, best-effort;
  logger.error({ err: String(err) }, '<fn> failed'); // a no-op if no txn is open
  throw err;
} finally {
  client.release();                                  // ALWAYS — a pooled client is never
}                                                    // returned with an open/aborted txn
```

This `catch` is **mandatory and is the real rollback guard.** An *unanticipated* error —
a deadlock `40P01`, a `23505` race on the actor or participant `INSERT`, a
`statement_timeout`, a transient socket error, or any throw inside `joinTopic`'s txn-2
reads — must not escape to `release()` with a transaction still open: `pg`'s
`client.release()` does **not** roll back, so a dirty client would poison the next caller
(`current transaction is aborted` / `SET TRANSACTION ISOLATION LEVEL must be called
before any query`). The single unconditional `ROLLBACK` aborts whichever transaction is
open at throw time — for `joinTopic`, txn 1 *or* txn 2 — and is a harmless no-op if the
error arrived between txn 1's `COMMIT` and txn 2's `BEGIN`.

### 4.1 `charterTopic({ project_id, name, charter, created_by }) → TopicRecord`

Validate all fields non-empty (`BAD_REQUEST`). `topic_id = randomUUID()`. Body wrapped in
the §4.0 `try/catch/finally`:
```
BEGIN
  INSERT INTO topics (topic_id, project_id, name, charter, status, next_seq, created_by)
    VALUES ($1,$2,$3,$4,'chartered',0,$5)
  appendEvent(client, { topic_id, actor_id: created_by, type:'topic.chartered',
                        subject_type:'topic', subject_id: topic_id,
                        payload:{ name, project_id } })   -- ⇒ seq 1
COMMIT
return { topic_id, project_id, name, charter, status:'chartered', created_by, created_at }
```

### 4.2 `joinTopic({ topic_id, actor_id, actor_type, display_name, level, since_seq? }) → InductionPack`

Validate `level ∈ LEVEL_SET`, `actor_type ∈ ACTOR_TYPE_SET`, ids non-empty (`BAD_REQUEST`).
**Two transactions on one pooled client** — txn 1 does the join writes and releases the
topics-row lock at `COMMIT`; txn 2 is a `READ ONLY REPEATABLE READ` snapshot that builds
the pack **holding no write lock**. Both transactions sit inside the **one §4.0
`try/catch/finally`** — the single `catch` is what guarantees a rollback on every
unanticipated error in either txn *[r3-fix]*:
```
client = await pool.connect()
try {
  -- ── txn 1: the join writes ──
  BEGIN
    SELECT project_id, status FROM topics WHERE topic_id=$1 FOR UPDATE
      none      → throw NOT_FOUND                       (§4.0 catch ROLLBACKs txn 1)
      'closed'  → throw BAD_REQUEST 'topic is closed'
    -- upsert the project-scoped actor; RETURNING type is atomic (no TOCTOU)
    INSERT INTO actors (project_id, actor_id, type, display_name) VALUES (project_id,$2,$3,$4)
      ON CONFLICT (project_id, actor_id) DO UPDATE SET display_name = EXCLUDED.display_name
      RETURNING type
    if returned.type <> actor_type → throw BAD_REQUEST
          'actor <id> already registered as <returned.type>'
    -- idempotent participant insert
    pRes = INSERT INTO topic_participants (topic_id, actor_id, level) VALUES ($1,$2,$5)
      ON CONFLICT (topic_id, actor_id) DO NOTHING RETURNING actor_id
    if pRes.rowCount > 0:                               -- first join for this actor
      UPDATE topics SET status='active' WHERE topic_id=$1 AND status='chartered'
      appendEvent(client, { topic_id, actor_id, type:'topic.actor_joined',
                            subject_type:'topic', subject_id: topic_id,
                            payload:{ level, actor_type } })
    -- pRes.rowCount == 0 ⇒ re-join: no participant row added, NO duplicate event (AC4)
  COMMIT                                                -- releases the topics-row lock
  -- ── txn 2: coherent induction-pack read — snapshot isolation, NO write lock ──
  BEGIN ISOLATION LEVEL REPEATABLE READ READ ONLY
    tr = fetchTopicWithRoster(client, topic_id)                     -- §4.5 — one JOIN query
    ev = replayEvents({ topic_id, since_seq: since_seq ?? 0 }, client)
  COMMIT
  return { topic: tr.topic, roster: tr.roster, events: ev.events, your_cursor: ev.next_cursor }
} catch (err) {
  await client.query('ROLLBACK').catch(() => {})        -- §4.0 — aborts txn 1 or txn 2, whichever is open
  logger.error({ err: String(err) }, 'joinTopic failed')
  throw err
} finally { client.release() }
```
**Idempotent re-join:** the `ON CONFLICT DO NOTHING RETURNING` makes the event emission
conditional on the *same* outcome as the participant insert — one txn, no duplicate row,
no duplicate event. A re-join still returns a fresh pack.

**Txn-2 failure — honest recovery story** *[r3-fix F2]*: if txn 2 fails after txn 1
committed, `joinTopic` throws — but the join (txn 1) is **durably committed and not
lost**. txn 2 is `READ ONLY` (only `SELECT`s) so it cannot suffer a serialization
conflict; its only failure modes are infrastructure faults (lost connection,
`statement_timeout`). A *transient* fault recovers on a `joinTopic` retry (the idempotent
re-join repeats txn 1 as a no-op and re-reads the pack). A *persistent* fault is **not**
something an unbounded `joinTopic` retry resolves — the caller surfaces the thrown error,
and once the fault clears it obtains the pack directly via `get_topic` +
`replay_topic_events` (the join is already on record). The design does **not** promise
that retrying `joinTopic` always yields the pack.

`InductionPack = { topic: TopicRecord, roster: Participant[], events: CoordinationEvent[],
your_cursor: number }`. The pack is read in **one `REPEATABLE READ` snapshot transaction**
(txn 2) — all four fields are a single coherent snapshot. The exact coherence guarantee:
- `your_cursor` is the max `seq` in `events`, or `since_seq` when `events` is empty.
- `roster` is the **complete** current participant set as of the snapshot.
- *[r-code-fix]* `events` is the `seq > since_seq` slice **capped at the replay limit**
  (`DEFAULT_REPLAY_LIMIT`, §3.2); `your_cursor` is its high-water mark, and the agent
  continues via `replay_topic_events` from `your_cursor` if more events remain. When the
  slice since `since_seq` fits within the replay limit (the 15.1 norm — topics are small),
  every roster actor whose `topic.actor_joined` has `seq > since_seq` appears in `events`
  — so for a fresh joiner (`since_seq=0`) every roster actor's join event is present; for
  a re-priming caller (`since_seq>0`, CLARIFY Q4) earlier join events are intentionally
  *outside* `events` (the caller already holds them). On a topic that exceeds the replay
  limit the first pack is a *prefix* — the joiner's own join event is reached by
  continuing from `your_cursor` (a real paginated-pack API — DEFERRED-010).
- `topic.status` reflects the txn-2 snapshot: if a concurrent actor closes the topic
  *between* the join commit and the pack read, the pack honestly reports `status:'closed'`
  — the join itself succeeded and is durably recorded (this is correct, not incoherent:
  the four fields still agree with one another).

(15.1 pack carries no `board`/`open_requests` — those tables arrive in 15.2/15.3.)

### 4.3 `getTopic({ topic_id }) → { topic, roster }`

`getTopic` calls `fetchTopicWithRoster(pool, topic_id)` (§4.5) — **one query**, so topic +
roster are a single snapshot. `null` (no topic row) ⇒ `throw ContextHubError('NOT_FOUND', …)`.

### 4.4 `closeTopic({ topic_id, actor_id }) → { topic_id, status:'closed', already_closed }`

Body wrapped in the §4.0 `try/catch/finally`. The `already_closed` path **`return`s**, so
it runs an explicit `ROLLBACK` first (a `return` skips the `catch`; the txn must be closed
before `finally` releases the client):
```
BEGIN
  SELECT status FROM topics WHERE topic_id=$1 FOR UPDATE
    none     → throw NOT_FOUND                          (§4.0 catch ROLLBACKs)
    'closed' → ROLLBACK; return { topic_id, status:'closed', already_closed:true }  -- idempotent
  appendEvent(client, { topic_id, actor_id, type:'topic.closed',
                        subject_type:'topic', subject_id: topic_id, payload:{} })
  -- status is still chartered|active here ⇒ appendEvent's WHERE status<>'closed' passes;
  -- topic.closed is therefore guaranteed the LAST event (the txn holds the row lock)
  UPDATE topics SET status='closed' WHERE topic_id=$1
COMMIT
return { topic_id, status:'closed', already_closed:false }
```
*15.1 close is open to any `writer` caller; level-based authorization of `close` (Authority
tier only) is deferred — the substrate records who closed via the event `actor_id`.*

### 4.5 `fetchTopicWithRoster(executor, topic_id) → { topic, roster } | null`

Internal helper (`executor: PoolClient | Pool`). **One query** — topic + participant
roster in a single snapshot:
```sql
SELECT t.topic_id, t.project_id, t.name, t.charter, t.status, t.created_by, t.created_at,
  COALESCE(json_agg(json_build_object(
      'actor_id', tp.actor_id, 'level', tp.level, 'joined_at', tp.joined_at,
      'type', a.type, 'display_name', a.display_name
    ) ORDER BY tp.joined_at) FILTER (WHERE tp.actor_id IS NOT NULL), '[]'::json) AS roster
FROM topics t
LEFT JOIN topic_participants tp ON tp.topic_id = t.topic_id
LEFT JOIN actors a ON a.project_id = t.project_id AND a.actor_id = tp.actor_id
WHERE t.topic_id = $1
GROUP BY t.topic_id
```
`GROUP BY t.topic_id` is the PK ⇒ every `t.*` column is selectable un-aggregated; a topic
with no participants yields `roster = []`. Returns `null` when no topic row matches.
Used by `getTopic` (on the pool) and `joinTopic` (on its txn-2 `REPEATABLE READ` client).

### 5.1 SSE handler — `GET /api/topics/:id/stream` (poll-based, D2) — *[r1-fix F1+F2; r2-fix F1; r3-fix]*

`POLL_MS = 2000`; `MAX_STREAM_MS = 1_800_000` (30 min) *[r3-fix]*. The handler is `async`
inside the route's `try/catch`. Lifecycle teardown (`cleanup` + `req.on('close')`) is
wired **first — before any `await`**. The first `replayEvents` runs before any header is
written (existence check, F1). The poll loop is self-scheduling, never `setInterval` (F2).
**A `MAX_STREAM_MS` cap bounds a half-open-socket zombie loop** *[r3-fix F3]*; the resume
cursor is read from `?since=` or, on `EventSource` auto-reconnect, the `Last-Event-ID`
header.

```
async (req, res, next) => {
  const topic_id = req.params.id
  const sinceSeq = Number(req.query.since ?? req.headers['last-event-id'] ?? 0)  // [r3-fix] resume
  let closed = false
  let timer: NodeJS.Timeout | undefined
  const cleanup = () => {                          // idempotent; ends the response
    if (closed) return
    closed = true
    if (timer) clearTimeout(timer)
    if (!res.writableEnded) res.end()
  }
  req.on('close', cleanup)                          // [r2-fix F1] wired BEFORE any await
  try {
    // [r1-fix F1] PRE-FLIGHT — before headers. NOT_FOUND ⇒ catch ⇒ next(e) ⇒ real 404 JSON.
    const first = await replayEvents({ topic_id, since_seq: sinceSeq })
    if (closed || req.destroyed) { cleanup(); return }   // [r2-fix F1] disconnected mid-pre-flight
    res.setHeader('Content-Type','text/event-stream'); res.setHeader('Cache-Control','no-cache')
    res.setHeader('Connection','keep-alive'); res.setHeader('X-Accel-Buffering','no')
    res.flushHeaders()
    let cursor = first.next_cursor
    const streamDeadline = Date.now() + MAX_STREAM_MS                // [r3-fix F3]
    const writeEvents = (evts) => { for (const e of evts)
      res.write(`id: ${e.seq}\nevent: ${e.type}\ndata: ${JSON.stringify(e)}\n\n`) }
    const endStream = () => { res.write('event: stream_end\ndata: {}\n\n'); cleanup() }
    const drainIfClosed = (evts) =>                                  // topic.closed ⇒ end
      evts.some(e => e.type === 'topic.closed') ? (endStream(), true) : false
    writeEvents(first.events); res.write(`: ping ${Date.now()}\n\n`)
    if (drainIfClosed(first.events)) return
    // [r1-fix F2] self-scheduling loop — NOT setInterval. tick re-arms the timer only
    // after it fully settles ⇒ at most one tick ever live ⇒ single-writer `cursor`.
    const tick = async () => {
      if (closed || res.writableEnded) return
      if (Date.now() > streamDeadline) { endStream(); return }       // [r3-fix F3] max-age cap
      const r = await replayEvents({ topic_id, since_seq: cursor })
      if (closed || res.writableEnded) return
      writeEvents(r.events); cursor = r.next_cursor; res.write(`: ping ${Date.now()}\n\n`)
      if (drainIfClosed(r.events)) return
      timer = setTimeout(() => { tick().catch(cleanup) }, POLL_MS)
    }
    timer = setTimeout(() => { tick().catch(cleanup) }, POLL_MS)
  } catch (e) {
    if (closed || req.destroyed) return              // disconnected during pre-flight — nothing to report
    next(e)                                          // pre-flight NOT_FOUND ⇒ 404 JSON
  }
}
```
No DB connection is *held* — each `replayEvents` borrows from the pool and returns it.
`cleanup` is wired before the pre-flight `await`, so a disconnect there is captured and
the handler bails before flushing headers (R2/R7). After headers flush, a *clean*
disconnect fires `req.on('close')` ⇒ `cleanup`. A **silent half-open socket** (laptop
sleep / NAT timeout — `res.write` to a peer-reset socket returns `false` without throwing,
so `'close'` may never fire) is bounded by `MAX_STREAM_MS`: `tick` checks `streamDeadline`
and force-ends the stream at the cap *[r3-fix F3]*. An `EventSource` client reconnects
automatically with `Last-Event-ID`, so the cap is invisible to a live client. Because the
timer is re-armed only inside a settled `tick`, **F2's overlapping-tick race is
structurally impossible** — at most one `tick` is ever live.

## 9. Key invariants & contracts (for the cold-start Adversary)

1. **Event/state atomicity** — every state change and its event commit together;
   `appendEvent` *requires* a `PoolClient` so it physically cannot run outside a txn.
2. **Per-topic seq monotonic & gap-free** — the `UPDATE topics SET next_seq=…` row-lock
   serializes all appends to one topic; same-txn increment ⇒ an aborted append burns no seq.
3. **The seal is un-bypassable** — the *same statement* that allocates seq carries
   `WHERE status<>'closed'`; no append can land on a closed topic, and `topic.closed` is
   provably the final event (its txn holds the row lock through the `status='closed'` flip).
4. **Idempotent join** — participant insert and event emission share one `ON CONFLICT …
   RETURNING` outcome; re-join produces neither a duplicate row nor a duplicate event.
5. **Project-scoped actor identity** — `actors` PK `(project_id, actor_id)`; `project_id`
   on join is derived from the *topic*, never caller-supplied — an actor cannot be
   namespaced into the wrong project.
6. **Replay tolerates gaps** — `replayEvents` treats the cursor as a high-water mark
   (`seq > cursor ORDER BY seq`); it never waits for a missing seq.
7. **No destructive deletes** — `coordination_events` is append-only; `close` is a status
   flip, not a delete; the `topic_id` FK has no cascade so the log cannot be orphaned.
8. **Coherent induction pack** *[r-code-fix]* — `joinTopic` reads the pack in a post-commit
   `REPEATABLE READ` snapshot txn (txn 2), holding **no write lock**: the four fields are
   one consistent snapshot. `roster` is the complete participant set; `events` is the
   `seq > since_seq` slice **capped at the replay limit** (`DEFAULT_REPLAY_LIMIT`, §3.2);
   `your_cursor` is the high-water mark of that slice — **the continuation cursor**: if
   events remain past it, the agent reads on via `replay_topic_events`. When the slice
   since `since_seq` fits the replay limit (the 15.1 norm), `events` is the complete tail
   and every roster actor's `topic.actor_joined` (`seq > since_seq`) is present; on a topic
   exceeding the limit the first pack is a prefix and the joiner reaches its own join event
   by continuing from `your_cursor` (DEFERRED-010). A txn-2 failure throws but never loses
   the (txn-1-committed) join — see §4.2's recovery story.
9. **SSE single-tick** — the stream poll is a self-scheduling `setTimeout`; at most one
   `tick` is ever live, so the delivery `cursor` has no concurrent writer.
10. **SSE leak-bounded lifecycle** *[r3-fix]* — a *clean* disconnect fires
    `req.on('close')` (wired before the pre-flight `await`) ⇒ immediate `cleanup`. A
    *silent half-open socket* that never delivers `'close'` is bounded by `MAX_STREAM_MS`:
    `tick` force-ends the stream at the cap, so a zombie poll loop cannot outlive 30 min.
    A missing topic yields a real `404`, never a hung `200`.
11. **Transactional cleanliness** *[r3-fix]* — every transactional service function (§4.0)
    has a `catch` running an unconditional best-effort `ROLLBACK` before re-throwing,
    inside a `finally` that always `release()`s. **The `catch` is the real rollback
    guard** — no unanticipated error (deadlock, `23505` race, timeout, a txn-2 throw in
    `joinTopic`) can return a pooled client with an open or aborted transaction. This is
    the verbatim Phase 13 `artifactLeases.ts` `_claimArtifactOnce` pattern.

## 10. Pre-emptive self-review — anticipated Adversary challenges

- *[r1-fix / r2-fix / r3-fix]* All findings of design-review rounds 1–3 are resolved — see
  the `[r1-fix]` / `[r2-fix]` / `[r3-fix]` markers. Rev 4 adds no new mechanism: the BLOCK
  fix is the verbatim Phase 13 transaction `catch` pattern (§4.0); the WARNs are an honest
  reword (§4.2 txn-2 recovery) and a `MAX_STREAM_MS` bound (§5.1).
- **`joinTopic` two-txn cleanliness** — both transactions are inside one §4.0
  `try/catch/finally`; the single unconditional `catch`→`ROLLBACK` aborts whichever txn is
  open at throw time and is a harmless no-op between txn 1's COMMIT and txn 2's BEGIN.
- **`joinTopic` txn-2 failure** — `joinTopic` throws; the join is durably committed (txn 1)
  and never lost. txn 2 is `READ ONLY` ⇒ no serialization-failure surface; transient infra
  faults recover on retry, a persistent fault is surfaced as a thrown error and the pack is
  then fetched via `get_topic` + `replay_topic_events`. The design does not over-promise
  retry (§4.2).
- **Pack `topic.status` may read `closed`** — if a concurrent `closeTopic` commits between
  txn 1 and txn 2, the pack honestly reports `status:'closed'`. Correct, not incoherent:
  the join succeeded and the four fields still agree (one txn-2 snapshot).
- **SSE half-open socket** — bounded by `MAX_STREAM_MS`; invariant 10 states the bound and
  the trade-off rather than claiming unconditional leak-freedom.
- **`appendEvent` client contract** — the type signature takes `PoolClient`; a caller
  passing a `Pool` is a compile error. T1/T4 exercise the real txn path.
- **`charter`+`appendEvent` same-txn** — the `topic.chartered` `appendEvent` runs after the
  `INSERT INTO topics` in the *same* txn; it sees the just-inserted row (`status='chartered'`,
  `<>'closed'` ⇒ passes). If the event INSERT fails, the §4.0 `catch` rolls back both.
- **Re-join after close** — `joinTopic` txn 1's `SELECT … FOR UPDATE` rejects a `closed`
  topic before any write; the close-vs-join race resolves on the topics-row lock.
