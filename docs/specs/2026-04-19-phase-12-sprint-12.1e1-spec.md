# Spec — Phase 12 Sprint 12.1e1 — Broaden lessons goldenset + re-baseline

## Problem

The 12.1d A/B showed `MRR = nDCG@5 = 0.9412` for BOTH control and salience-on on the existing 20-query lessons goldenset. Those metrics are **saturated** — at least 18/20 queries find their target at rank 1, leaving no headroom for ranking-signal changes to register. The one metric with non-zero delta (`nDCG@10 = +0.0073`) was within the noise floor.

This means any future ranking-tuning sprint (12.1e half-life, 12.2 sleep consolidation, etc.) would run against a measuring-stick that cannot distinguish good tuning from bad. The 12.1c `popularity-feedback-loop` only surfaced because its effect was large (−0.0373 MRR); smaller real shifts would be invisible.

The fix is to broaden the goldenset so the measured metrics sit away from the 1.0 ceiling. This sprint (12.1e1) does the broadening + re-baseline. A follow-up (12.1e2) runs the half-life sweep against the broader set.

## Scope

**12.1e1 — This sprint (IN SCOPE):**
1. Add **20 new queries** to `qc/lessons-queries.json`, composed as:
   - **12 synthesized adversarial-ambiguous** — multi-target (2-4 plausible hits each), phrasings that stress the embedder + FTS differently. New group: `ambiguous-multi-target`.
   - **5 synthesized semantic-paraphrase** — single-target, rephrased so keyword-FTS underperforms vs. semantic matching (or vice-versa). New group: `semantic-paraphrase`.
   - **3 mined real queries** — pulled from `lesson_access_log WHERE context='consideration-search'`, deduplicated by `metadata->>'query'`, filtered to exclude goldenset-adjacent text. New group: `real-dogfood`.
2. Run the QC baseline against the broader goldenset at **current production defaults** (salience ON, α=0.10, half-life=7d) — this is the new "12.1e1 control" for future comparisons.
3. Archive results + honest diff vs. the 12.1d final baseline (raw metric drop expected; that's fine and is the point).

**Non-goals (12.1e2 — deferred):**
- Half-life sweep {3, 7, 14, 30}d — requires 4 separate baseline runs.
- Change to `LESSONS_SALIENCE_HALF_LIFE_DAYS` default in `src/env.ts`.
- Any code change to salience service, ranking, or env config.
- Tuning α.

**Out of scope (later):**
- Broadening chunks / code / global golden sets.
- Changing the scoring metrics themselves (MRR / nDCG / recall / dup@10).
- Replacing the saturated `confident-hit` queries (kept as regression guards).

## Acceptance criteria

A1. `qc/lessons-queries.json` contains **40 queries** (20 existing unchanged + 20 new).
A2. Every new query has a `target_lesson_ids` array populated with real lesson UUIDs that exist in the `free-context-hub` project (verified via `/api/lessons/:id` or DB lookup).
A3. Every `ambiguous-multi-target` query has **≥ 2 and ≤ 4** target IDs (so it's genuinely ambiguous, not a single-target in disguise).
A4. Every `semantic-paraphrase` query has **exactly 1** target ID (single-target by design).
A5. Every `real-dogfood` query text is **NOT** a substring of any existing goldenset query text, and vice-versa (confirms "novel").
A6. New baseline archive `docs/qc/baselines/2026-04-19-sprint-12.1e1-baseline.{json,md}` exists, produced via `npm run qc:baseline -- --tag sprint-12.1e1-baseline --samples 3`.
A7. Diff doc `docs/qc/baselines/2026-04-19-sprint-12.1e1.diff.md` compares new baseline to `2026-04-18-sprint-12.1d-fix.json`, explicitly calls out that raw MRR/nDCG will drop (because 20 new non-saturated queries dilute the ceiling) — this is EXPECTED, not a regression.
A8. Session patch entry added to `docs/sessions/SESSION_PATCH.md`.
A9. Goldenset reasoning is traceable: each new query has a `reasoning` field explaining why it's included + why those targets were chosen.

## CLARIFY decisions locked

**D1 — Sourcing:** Hybrid (8-10 adversarial + 3-5 mined). Resolved to: 12 ambiguous + 5 paraphrase + 3 mined = 20 new.

**D2 — Target size:** 40 queries total (keep all 20 existing).

**D3 — Structure:** Split sprint. This sprint = 12.1e1 (broaden + re-baseline). Follow-up 12.1e2 = sweep + default change.

**D4 — Mining protocol:** `SELECT DISTINCT metadata->>'query' FROM lesson_access_log WHERE context='consideration-search' AND project_id='free-context-hub' AND accessed_at > NOW() - INTERVAL '180 days'`, then manually filter out query texts that overlap the existing 20 goldenset phrasings (by substring or obvious paraphrase). If mining yields < 3 novel queries, **fall back to synthesizing extras** (e.g., 4 paraphrase + 14 ambiguous + 2 mined, or all synthesized if zero yield). Document the fallback in the session patch.

**D5 — Target-ID selection for synthesized queries:** For each synthesized query, I will:
1. Use the current hybrid search via REST `/api/lessons?q=...` OR direct corpus scan to identify top 5-10 plausible lesson IDs.
2. For `ambiguous-multi-target`: include all lessons that a reasonable reader would consider "on-topic" (2-4 IDs).
3. For `semantic-paraphrase`: pick the ONE best target; rewrite the query in 2-3 phrasings until only that one is the obvious answer.
4. Record the reasoning in each query's `reasoning` field (JSON-only; `GoldenQuery` TS type in `src/qc/goldenTypes.ts` is NOT updated — reasoning is informational, not consumed by runBaseline. Preserves zero-code-change constraint).

**D6 — Baseline run:** `npm run qc:baseline -- --tag sprint-12.1e1-baseline --samples 3`. Salience is ON at current defaults (α=0.10, half-life=7d). This is the NEW reference point for 12.1e2 and beyond.

## Risks + mitigations

**R1 — Target-ID selection drift:** My judgment of "plausible hits" may not match a user's. Mitigation: `reasoning` field makes decisions reviewable; POST-REVIEW is a human-interactive checkpoint, so the user can challenge specific queries.

**R2 — Mining yield too low:** Access log may not have enough novel queries. Mitigation: fallback to synthesis; still hit 20 new total.

**R3 — Baseline run failure:** `qc:baseline` tooling could error on the expanded goldenset. Mitigation: runBaseline.ts iterates `queries.length` dynamically — verified no hardcoded counts. But I'll run a 5-query dry run first.

**R4 — Query-target false positives:** A synthesized "ambiguous" query could accidentally have a very-clear single target (not ambiguous). Mitigation: self-review pass before commit — check that each ambiguous query's rank-1 hit is NOT obviously the "correct" answer (i.e., targets are genuinely plausible alternatives).
