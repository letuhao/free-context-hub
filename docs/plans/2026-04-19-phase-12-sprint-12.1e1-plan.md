## Mining (T1 — 1 commit prep task)

**T1.1 — Execute mining SQL**
- Cmd:
  ```bash
  docker compose exec -T db psql -U contexthub -d contexthub -c "\
  SELECT COUNT(DISTINCT metadata->>'query') AS novel_queries FROM lesson_access_log \
  WHERE context='consideration-search' AND project_id='free-context-hub' \
  AND metadata->>'query' IS NOT NULL AND accessed_at > NOW() - INTERVAL '180 days';"
  ```
- Verify: sanity-check the count is plausible (likely 20-100; if >500 something is wrong; if <10 may be mining-yield problem).

**T1.2 — Fetch candidate query list**
- Cmd: same as the DESIGN doc §4 full query, dump to `/tmp/mined-queries.txt`.
- Verify: file has ≥ 20 rows.

**T1.3 — Apply goldenset-adjacent filter**
- For each candidate query text:
  - Compute substring match vs. existing 20 goldenset queries (both directions).
  - Compute token-overlap (after stopword strip): reject if `|intersection| / |T_tokens| >= 0.7` for any existing.
- Do this as a bash/node one-liner writing survivors to `/tmp/mined-survivors.txt`.
- **Decision point:** if survivors < 3, write down the shortfall count; the synthesis phase (T3-T5) will pick up N extra ambiguous queries to compensate. Document in session patch.

**T1.4 — Select final 3 mined queries**
- From survivors, pick the 3 most topically diverse + highest `hit_count`.
- For each, determine target_lesson_ids by re-running the query via `/api/lessons?q=...` and picking 1-2 genuinely-correct hits (not just the current rank-1, which may be wrong — use content reading).
- Record in notepad; write to JSON in T6.

## Synthesis (T2-T5)

Seed ideas (final list locked during BUILD after corpus scan):
- **Workflow v2.2** — phases, review-impl, post-review, workflow-gate state machine, skip rules
- **pg/UUID conventions** — casing, RETURNING, type pollution, connection pooling

- **Multi-project isolation** — project_id scoping, cross-tenant guard, selector V2

- **Docker / stack lifecycle** — compose up, worker, healthchecks, rebuilds

Exact 12 picked after scanning corpus (T2.2).

**T2.2 — Corpus scan to confirm 2-4 candidates per cluster**
- Cmd: `curl -s 'http://localhost:3001/api/lessons?project_id=free-context-hub&limit=200' | jq '.items[] | {id, title}' | less` (or save to file).
- For each of the 12 clusters, verify 2-4 lessons visibly relate. If a cluster has <2, drop it and pick another from the seed list.

**T2.3 — Draft 12 ambiguous queries (one per cluster)**
- For each cluster, write a query that any of the 2-4 related lessons could plausibly answer.
- Run each draft query through `/api/lessons?q=<query>` and confirm the targets appear in the top 10.
- Per-query: write `id`, `group='ambiguous-multi-target'`, `query`, `target_lesson_ids: [2-4 UUIDs]`, `must_keywords: []` (usually), `reasoning`.
- Save to `/tmp/ambiguous-queries.json` (staging).

**Verify:** `jq '. | length' /tmp/ambiguous-queries.json` → `12`.

### T3.1 — Synthesize 5 semantic-paraphrase queries

- Pick 5 specific lessons (each with distinctive content). For each:
  - Read the lesson content.
  - Write a query that describes the lesson's concept WITHOUT using the lesson's main keywords (forces semantic match).
  - Verify: run the query through `/api/lessons?q=...`. The target MUST appear in top 5 (sanity — not in top 1, because then the paraphrase was too easy). If target is at rank > 10, rewrite the query; the paraphrase is too obscure.
- Per-query: `id`, `group='semantic-paraphrase'`, `query`, `target_lesson_ids: [1 UUID]`, `must_keywords: [1-2 targeted]`, `reasoning`.
- Save to `/tmp/paraphrase-queries.json`.

**Verify:** `jq '. | length' /tmp/paraphrase-queries.json` → `5`.

**T6.3 — Self-check: cardinality rules**
```bash
node -e "const d = JSON.parse(require('fs').readFileSync('qc/lessons-queries.json', 'utf8'));
  let ok = true;
  for (const q of d.queries) {
    if (q.group === 'ambiguous-multi-target' && (q.target_lesson_ids.length < 2 || q.target_lesson_ids.length > 4)) {
      console.error('ambig cardinality violation:', q.id, q.target_lesson_ids.length); ok = false;
    }
    if (q.group === 'semantic-paraphrase' && q.target_lesson_ids.length !== 1) {
      console.error('paraphrase cardinality violation:', q.id, q.target_lesson_ids.length); ok = false;
    }
    if (q.group === 'real-dogfood' && (q.target_lesson_ids.length < 1 || q.target_lesson_ids.length > 2)) {
      console.error('dogfood cardinality violation:', q.id, q.target_lesson_ids.length); ok = false;
    }
  }
  if (ok) console.log('cardinality: OK');
  else process.exit(1);"
```

## Subagent dispatch decision

**NOT using subagent dispatch.** Reasoning:
- All 20 queries require topic-cluster judgment that benefits from single-context reasoning (avoid parallel-agent inconsistency in cluster choice).
- Files touched are all one file (`qc/lessons-queries.json`) + baselines. No independent parallel tasks.
- Total BUILD estimate: 90-150 min. Manageable inline.

Subagent dispatch would help if: goldenset was 200 queries (true parallelism), or queries spanned multiple unrelated surfaces. Neither applies.
