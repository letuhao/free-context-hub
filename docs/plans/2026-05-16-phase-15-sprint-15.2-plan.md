## Dependency notes (why this order)

- `board.ts` imports only `appendEvent` (15.1) + `pool` — it does **not** import `artifacts.ts`
  (`postTask` inserts the `artifacts` row directly; `completeTask` does `UPDATE artifacts`
  directly). So `board.ts` is buildable and testable standalone.
- `artifacts.ts` imports only `appendEvent` + `pool` — standalone. Its **test** uses
  `claimTask` (from `board.ts`) for setup, so the test runs after `board.ts` exists.
- `coordinationSweep.ts` imports `revertArtifact` from `artifacts.ts` — it is built **after**
  `artifacts.ts`. Its test uses `claimTask`/`writeArtifact`/`baselineArtifact` for setup.
- `core/index.ts` exports must land before `routes/board.ts` and `mcp/index.ts` (both import
  the service fns via the barrel).

## TDD discipline

The three service modules are DB-integration code: each test file imports its module, so the
module must compile before its test runs. Order per service: write the service (T2/T4/T6) →
write the test (T3/T5/T7) → run the test; a failing test is the RED signal → fix the service
→ green. The migration (T1) and the wiring tasks (T8/T10/T11/T12) are not TDD — "just build
and verify" (CLAUDE.md Phase 5) — their gate is `tsc` + (for T1) an idempotent re-run.

`board.test.ts` exercises `board.ts` + raw SQL only (no `artifacts.ts` import). `artifacts.test.ts`
and `coordinationSweep.test.ts` use `board.ts` fns for setup — both run after T2.
