## Group B — Scheduler module (new, isolated for testability)

### B1 — sweepScheduler.ts (new module)
- **Files:** `src/services/sweepScheduler.ts` (new)
- **What:** Per design Section 5a. SHA256-derived advisory key computed at module init; `startSweepScheduler(opts?: { intervalMs?: number; graceMinutes?: number }): SweepHandle`. Uses `getDbPool()`, `enqueueJob`, advisory-lock-guarded enqueue. Logs via module logger.
- **Verify:** `npx tsc --noEmit` clean.

### B2 — sweepScheduler tests (5 tests)
- **Files:** `src/services/sweepScheduler.test.ts` (new)
- **What:** Per design Section 7. 5 tests:
  1. `LEASES_SWEEP_ADVISORY_KEY matches SHA256 derivation` — pure assertion against `crypto.createHash('sha256')` output
  2. Lock acquired → enqueue happens once
  3. Lock not acquired → no enqueue, logger logs `leases_sweep_skip_not_leader`
  4. pool.connect rejects → no enqueue, no crash, logger logs `leases_sweep_pool_unavailable`
  5. After enqueue, unlock query issued with same key
- **Approach:** Mock `getDbPool` to return a fake client with configurable responses; mock `enqueueJob` to a Jest-style spy. Use short `intervalMs: 50` for tests so cycles complete quickly; always call `handle.stop()` in `afterEach`.
- **Verify:** `npm test -- src/services/sweepScheduler.test.ts` shows 5 pass.

## Execution sequencing rules

- Tasks within a group can be done in declared order (mostly independent within a group).
- Groups B and C can run in parallel after Group A is fully done (B depends on jobQueue + getDbPool only; C depends only on existing middleware).
- Group D depends on B+C (re-exports both).
- Group E depends on no backend group (independent client work), but the GUI manual smoke during VERIFY/POST-REVIEW requires Group D done (server endpoint live).

Single autonomous session can run all groups inline (no need for sub-agent dispatch). Total estimated logic time: ~30-45 min wall-clock for the agent (file edits + test runs).
