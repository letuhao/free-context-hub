## Three compatibility shims required for ragas 0.4.3 + LM Studio

Documented in `services/ragas-judge/_compat.py` and `config.py`. Each carries a "remove-when" note.

### Shim 1: `langchain_community.chat_models.vertexai` stub
**Why:** ragas 0.4.x has a stale top-level eager import `from langchain_community.chat_models.vertexai import ChatVertexAI`. That submodule was removed when langchain-community split out provider integrations (Vertex moved to `langchain-google-vertexai`). ragas's `setup.py` doesn't pin langchain-community, so pip pulls the latest (0.4.2) which no longer has the path.
**Fix:** register a stub module + dummy class in `sys.modules` BEFORE any ragas import. We don't use Vertex; the stub is never invoked at runtime.
**Remove when:** ragas ships a patch updating the import to `from langchain_google_vertexai import ChatVertexAI`.

### Shim 2: Instructor mode override `Mode.JSON` → `Mode.JSON_SCHEMA`
**Why:** `ragas/llms/base.py` hardcodes `mode=instructor.Mode.JSON` when wrapping the OpenAI client. `Mode.JSON` sends `response_format={"type": "json_object"}` which is OpenAI-specific. LM Studio (and many other OpenAI-compatible endpoints) rejects this with `400 'response_format.type' must be 'json_schema' or 'text'`. `Mode.JSON_SCHEMA` sends `response_format={"type": "json_schema", ...}` which is universally supported.
**Fix:** monkey-patch `instructor.from_openai` in `_compat.py` to swap the mode kwarg when called with `Mode.JSON`.
**Remove when:** ragas exposes mode configuration via `llm_factory` kwargs.

### Shim 3: Temperature default = 0.2 (not 0.0)
**Why:** Gemma 4 26B-A4B at temp=0 degenerates into token-repetition loops when producing schema-constrained output (e.g. emits `"verdict/verdict/verdict/..."` until max_tokens). Temp=0.2 is the lowest value that reliably breaks the loop while keeping outputs near-deterministic.
**Fix:** default `JUDGE_TEMPERATURE=0.2` in `config.py`.
**Remove when:** judge model swaps to one without this failure mode, or a stronger Gemma 4 variant ships.

These three shims are the cost of running ragas 0.4.3 against a local OpenAI-compatible endpoint with a smaller-than-frontier model. The user explicitly preferred the latest ragas to capture 0.3.x + 0.4.x feature work; this is the tradeoff.

## URL normalization fix

`.env` has `EMBEDDINGS_BASE_URL=http://host.docker.internal:1234` (no `/v1`) because the project's existing services hit different paths. The OpenAI Python client requires `/v1` as a base URL suffix. `config.py` now normalizes both `JUDGE_AGENT_BASE_URL` and `EMBEDDINGS_BASE_URL` — appends `/v1` if absent. Other consumers (mcp, worker) are unaffected because they read the env directly.

## Open issues / Known constraints

| ID | Issue | Disposition |
|---|---|---|
| **16.2-OPEN-1** | `context_recall = 0.0` on a row where ground truth IS in context. Pattern observed in smoke #8 (also smoke #5 same score). Likely model-side noise: Gemma 4 is generating low-quality structured output sporadically. | Carry to Sprint 16.3 as a known variance source. Quantify via 5x determinism probe across the 152-row dataset. If consistently noisy, may need stronger judge model OR few-shot examples in ragas's context_recall prompt. |
| **16.2-OPEN-2** | LM Studio auto-unloads idle models. Both Gemma 4 (chat) AND bge-m3 (embeddings) must be loaded simultaneously for /score to succeed. After ~5 min idle LM Studio drops one or both, and `400 'No models loaded'` surfaces on the next call. | **User action required:** disable LM Studio's auto-unload OR pin "always loaded" on both models. Documented in PLAN R-section. |
| **16.2-OPEN-3** | First call after model load takes ~30-90s (LM Studio loads model into VRAM). Subsequent calls ~5-20s per metric. | Acceptable for nightly baseline; not a blocker. Sprint 16.3 budget assumes ~30 min per baseline. |
| 16.2-OPEN-4 | The `prompts_hash` field changes whenever ragas version changes, but doesn't capture user-side prompt customizations (we don't have any yet). Adequate for now. | Revisit if/when we add custom metric prompts beyond `refusal_correctness`. |
