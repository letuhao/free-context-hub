# Alternative rerankers via TEI — Sprint 12.1h

## Recommendation (one line)

**No default changes. Rerank optimization plateaus with self-hostable rerankers.** Sprint 12.1h tried `jinaai/jina-reranker-v2-base-multilingual` (failed architecturally — custom arch without `model_type` field TEI can't parse) and `cross-encoder/ms-marco-MiniLM-L-6-v2` (loaded + strictly-deterministic + fast — 13s vs bge's 239s — but same aggregate quality as bge, below gte). Neither passes the MRR ≥ 0.98 winner threshold. Production stays `RERANK_TYPE=generative`; measurement best is still `gte-reranker-modernbert-base` via LM Studio. Future rerank improvements likely require commercial API or LLM-scale models (Cohere Rerank 3, custom fine-tuning) — out of scope for self-hostable infrastructure.

Standard HF cross-encoder model (~80MB, BERT-MiniLM-6-layer, trained on MS MARCO passage ranking). TEI-safe. Baseline + determinism:

- **recall@10 = 1.0000** (perfect recall — best among non-generative)
- **MRR = 0.9266** (below gte 0.9538; ~tied with bge 0.9279)
- **nDCG@10 = 0.9080** (below gte 0.9237; ~tied with bge 0.9071; slightly below no-rerank 0.9100)
- **elapsed = 13s** (vs bge's 239s — **18× faster**; vs gte's 22s — 40% faster)
- **Strict determinism: 0/40 query diffs** across BOTH tei-rerank AND mcp container recreates (stronger test than 12.1g/12.1f which only tested mcp recreate with fixed TEI/LM-Studio state).

## Recommendation by use case

- **Production (user-facing quality matters):** `RERANK_TYPE=generative` (default, unchanged). Best quality; non-determinism is not a user-visible problem.
- **QC measurement (determinism matters):** `RERANK_TYPE=cross-encoder` + `RERANK_MODEL=gte-reranker-modernbert-base`. Deterministic + better quality than alternatives.
- **Fast QC iteration (speed matters more than top-K precision):** `RERANK_TYPE=api` (now with TEI+minilm). 18× faster than bge, near-identical quality, fully deterministic. Good for rapid iteration when MRR/recall@10 are the primary metrics.

## What we learned across 12.1e4 → 12.1h

This 4-sprint arc investigated rerank behavior exhaustively:
- **12.1e4:** LLM generative rerank non-deterministic across sessions (~0.027 MRR drift per 40q baseline).
- **12.1f:** LM Studio `/v1/embeddings` + cosine works ONLY for bi-encoder-compatible models (gte); bge/jina fail (near-random output).
- **12.1g:** External TEI `/rerank` endpoint unlocks true cross-encoders, but bge-v2-m3 specifically underperforms.
- **12.1h:** Tried jina (architecturally incompatible with TEI) + minilm (functional, fast, same quality as bge).

**Closing finding:** Generative LLM rerank gives ~0.05-0.07 MRR boost over any cross-encoder we've tested. That boost appears to be genuine LLM reasoning (semantic equivalence, indirect references) that ranker-sized models don't replicate. No cross-encoder in our tested set beats it.

## Decision (per spec D4: cap at 2 models)

**Both models lost. Rerank optimization loop closes.**
- No `src/env.ts` default changes.
- `docker-compose.yml` `tei-rerank --model-id` stays at `cross-encoder/ms-marco-MiniLM-L-6-v2` (the second tested, currently loaded + fast + works).
- TEI infra remains deployed for future optional use.

### Notable per-query findings (surfaced by LOW-1 detail)

- **sprint-11-closeout:** minilm hits rank-4 where bge MISSes entirely. That's a meaningful rescue (0 → 0.25 contribution). gte gets rank-8.

- **Pattern:** gte tends to place one target at rank-1 and ignore the rest; minilm and bge spread multiple targets across top-10. For multi-target ambig queries, spread is arguably better (higher recall) but can dilute nDCG if later ranks are graded-0 lessons.
