# Cross-encoder rerank evaluation — Sprint 12.1f

## Recommendation (one line)

**Keep `RERANK_TYPE=generative` as production default. Use `RERANK_TYPE=cross-encoder` + `RERANK_MODEL=gte-reranker-modernbert-base` as an opt-in QC measurement alternative** — gte is deterministic (0/40 diffs across container recreate), 15× faster than generative, and slightly better than no-rerank on aggregate (+0.034 MRR), but quality trails generative by −0.046 MRR / −0.049 nDCG@10 with mixed per-group behavior (wins cross-topic, loses paraphrase). bge and jina rerankers returned near-random results via the `/v1/embeddings` endpoint — incompatible with the current cross-encoder code path.

The cross-encoder is "confused" by abstract/paraphrased queries where it can't find lexical-semantic overlap with the target; the generative reranker handles these better because it understands semantic equivalence via LLM reasoning.

## Why bge and jina failed

Both produced near-random scores:
- bge: MRR=0.1418 (ceiling is 1.0 with perfect retrieval; 0.14 is worse than chance for a ranked list).
- jina: MRR=0.3375 (mildly better than random but still terrible).

**Root cause:** These models are **cross-encoders by design** — they take `(query, doc)` pairs as input and produce a SINGLE score per pair. They are NOT designed to produce independent embeddings that can be compared via cosine similarity (which is what the `rerankCrossEncoder` code in `src/services/lessons.ts:605` does via `/v1/embeddings`). When you ask bge-reranker-v2-m3 or jina-reranker-v3 to emit an embedding for a single text, you get something — but it's not designed for cosine-similarity ranking.

**gte worked** because `gte-reranker-modernbert-base` apparently has a pooling/output layer that produces cosine-compatible embeddings, OR because our queries happen to align with its output space. Some rerankers are more "embedding-like" than others; gte is the best of our three for this code path.

**Future work opportunity:** implement a proper `/v1/rerank` endpoint in the code (LM Studio may support it for some models). That would be a small code change that unlocks bge and jina's actual cross-encoder path. Out of scope for 12.1f.

## Determinism verdict

**gte: 0/40 queries differ across container recreates.** MRR, nDCG@10, all metrics byte-identical. Deterministic to full precision. This is what 12.1e4's NO_WRITE-only measurement couldn't get via generative rerank.

Contrast: generative rerank in 12.1e4 showed 7/40 query shifts across 90min-separated runs at the same config (MRR drifted 0.9784 → 0.9514). gte: zero drift.

## Decision matrix (from design §3)

Winner criterion: aggregate nDCG@10 vs generative and no-rerank with 0.013 noise floor.

- **gte nDCG@10 (0.9237)** vs generative (0.9724): **−0.0487** → does NOT match generative (below noise-floor window of ±0.013).
- **gte nDCG@10 (0.9237)** vs no-rerank (0.9100): **+0.0137** → **just barely** above the noise floor (0.013).

Per the decision matrix: **"Partial win — cross-encoder better than no-rerank but loses to generative. Keep generative for prod, use cross-encoder for QC measurement (trades some quality for determinism)."**

## Production recommendation

**No `src/env.ts` changes.** `RERANK_TYPE=generative` stays the production default.

Rationale:
- Generative wins on aggregate quality (MRR=1.0 vs gte's 0.9538; nDCG@10 0.9724 vs 0.9237).
- Generative's non-determinism is a MEASUREMENT problem, not a user-facing problem. Users get better average rankings even if subject to cross-session variance.
- gte's per-group losses on paraphrase and ambig (−0.13 and −0.02 vs no-rerank!) indicate it's not strictly better than no-rerank for those query classes.

## Measurement recommendation

For QC baseline runs where determinism matters more than peak quality:

```bash
RERANK_TYPE=cross-encoder RERANK_MODEL=gte-reranker-modernbert-base \
LESSONS_SALIENCE_NO_WRITE=true \
docker compose up -d --force-recreate mcp
npm run qc:baseline -- --tag <sprint-tag> --samples 1 --surfaces lessons
```

This gives: deterministic results across recreates + 15× speedup over generative + strict improvement over no-rerank on aggregate (if barely).

**Alternative measurement path:** `DISTILLATION_ENABLED=false` (no-rerank) is 100× faster and also deterministic. Use no-rerank when speed matters more than the ~0.014 nDCG@10 lift from gte.

## Friction class: cross-encoder-via-embeddings-api-mismatch

New finding worth documenting. Added below, should also land in `docs/qc/friction-classes.md`.

**Definition.** The `rerankCrossEncoder` code path uses `/v1/embeddings` to get independent embeddings for query and candidates, then cosine-similarity-ranks them. This works for some rerank-tagged models (gte-reranker-modernbert-base) but fails catastrophically for others (bge-reranker-v2-m3, jina-reranker-v3). Models designed as "pure cross-encoders" (score `(query, doc)` pairs) may produce valid-looking embeddings via `/v1/embeddings` but with outputs that don't correlate well with relevance when cosine-compared.

**Detection.** If a cross-encoder run produces MRR < no-rerank floor (0.92 on our goldenset), suspect this failure mode. Run a smoke test with a known-good query like `"pg uuid casing"` and check if the target lesson is in top-5.

**Mitigation paths.**
1. Test multiple cross-encoder models; pick one that works empirically (gte-reranker-modernbert-base is validated for this code path).
2. Implement `/v1/rerank` endpoint support in `src/services/lessons.ts` — a proper cross-encoder endpoint where query+doc pairs are scored together. Enables bge, jina, and other real cross-encoders. Small code change; could ship in 12.1g or later.
3. Use generative or no-rerank as measurement fallbacks.
