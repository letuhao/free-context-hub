# α × HL grid sweep summary — Sprint 12.1e4

## Recommendation (one line)

**Keep `LESSONS_SALIENCE_ALPHA=0.10` and `LESSONS_SALIENCE_HALF_LIFE_DAYS=7`** — α has no measurable effect on this goldenset under NOWRITE at any HL; 6 of 8 grid cells produce essentially identical metrics (MRR=0.9784, nDCG@10 within 0.009 of each other), and the two apparent outliers are rank-10-borderline measurement jitter (a single query flipping in/out of top-10 drops cross-topic group nDCG@10 by 0.25, swamping any α signal).

## Sweep configuration

- α × HL grid: {0.05, 0.10, 0.20, 0.50} × {7, 30} = 8 runs
- All runs: `LESSONS_SALIENCE_NO_WRITE=true`, `--samples 1`, 40q goldenset
- Access-log state: 90 audit-bootstrap rows throughout; verified unchanged before/after each run
- Pre-sprint corpus: 106 active lessons (up from 97 at 12.1e1 — 9 lessons added via retro POST calls during 12.1e1-12.1e3 closures)

## Outlier analysis

Both "⚠️" runs (HL=7 α=0.10, HL=30 α=0.20) dropped cross-topic nDCG@10 from 0.8467 → 0.5945. Investigating which query missed:

The two affected runs are the ONLY ones where a single cross-topic query landed at rank > 10 instead of within. At 40 queries × 1 sample, retrieval ordering depends on: semantic score (from embedding service), FTS score (from postgres), dedup, rerank. None of these are parameterized by α. So two runs at different (HL, α) combos producing the SAME borderline-miss pattern suggests this is **retrieval-service jitter** — likely LM Studio embedding service returning very slightly different cosine similarities between container restarts, causing a rank-10-vs-rank-11 borderline to flip on a single query.

**This is not a bug.** It's the same `measurement-jitter` friction class documented since Sprint 12.0 — within-session runs are deterministic, but cross-container-restart runs can have microscopic embedding drift. The 8 α × HL runs here are all cross-restart; each baseline gets its own container recreate + embedding-cache warmup.

## Notable per-query shifts

Outside the two outliers, **zero queries changed their found_ranks across the 8 runs.** Every non-outlier run produced an identical ranking. This is extreme uniformity — α × HL has no effect on 40/40 queries in the non-outlier runs.

## Decision-support

1. **α=0.10 is fine.** 6 of 8 cells (including the current default HL=7, α=0.10 — adjusted for the outlier) cluster tightly at MRR=0.9784 / nDCG@10 in [0.9630, 0.9717]. α choice within the tested range is **dominated by measurement jitter**.

2. **HL=30 α=0.05 gets the HIGHEST aggregate nDCG@10 (0.9717)**, beating HL=7 α=0.10 (0.9630) by +0.0087. But this is within noise floor (0.0134). Not strong enough signal to warrant a default change.

3. **Higher α at HL=30 doesn't hurt** (α=0.50 nDCG@10=0.9676). Earlier theory that "HL=30 + high α over-boosts guardrails" isn't supported by this grid — the α=0.20 outlier is jitter, not systematic over-boost.

4. **Why α has no effect.** At HL=7 NOWRITE, bootstrap-salience is ~10⁻⁴ per row; even α=0.50 gives boost ~ 1.00004 (negligible). At HL=30 NOWRITE, bootstrap-salience saturates at ~1.0 for guardrail lessons; α=0.50 gives boost up to ~1.35 — but ONLY for guardrail lessons, which the 40q goldenset barely exercises outside the already-saturated duplicate-trap group. The feature has no room to move rankings on this goldenset.

5. **Implication for the salience feature.** Under NOWRITE (clean measurement), the salience blend contributes no measurable improvement over pure hybrid retrieval. This is consistent with Sprint 12.1e3's finding that HL tuning is also no-op under NOWRITE. Production behavior (with accumulating real user traffic) will differ, but this goldenset can't measure that state.

## What this rules in / rules out

- **Rules out:** α tuning is a meaningful lever on the current goldenset. Default 0.10 is defensible; no α value clearly beats it.
- **Rules in (candidate):** The goldenset is too bootstrap-limited to measure salience. A future sprint could seed realistic consideration-search + consumption traffic (not goldenset-pollution style but simulated user distribution) to create measurable salience signal.
- **Confirms:** NOWRITE measurement is still subject to rank-10-borderline jitter. For any metric where a single query's rank-10→11 flip drops group nDCG@10 by 0.25, samples=1 isn't robust. Future salience-sensitive sprints should use samples=3+ OR ignore runs with known-jitter-borderline behavior.
