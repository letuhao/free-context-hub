## Human-written interpretation (appended 2026-04-19)

### Key finding — the broadening did not dilute aggregate metrics as expected

The pre-sprint hypothesis (spec §1) was that adding 20 harder queries would DROP aggregate MRR / nDCG@10 below ceiling, creating measurement headroom for Sprint 12.1e2's half-life sweep. What actually happened: aggregate MRR rose from 0.9412 → 0.9730 (+3.4%, above noise floor). Two mechanical reasons:

1. **Adversarial-miss dilution:** `adversarial-miss` queries contribute MRR=0. They were 3/20 (15% weight) in 12.1d-fix but only 3/40 (7.5% weight) in 12.1e1. This alone lifts the aggregate.
2. **MRR uses best-ranked target, not all targets:** My 15 ambiguous queries have 2-4 targets each; if ANY target lands at rank-1, MRR for that query = 1.0. Post-hoc per-group analysis below shows 15/15 ambiguous queries hit rank-1 (so each contributes MRR=1.0 despite being "harder").

### Where measurement headroom actually lives

Despite MRR saturation, the 12.1e1 baseline **DOES give 12.1e2 meaningful signal**:

1. **`nDCG@10 = 0.9594`** (not at ceiling 1.0). Unlike MRR, nDCG@10 rewards ranking MORE relevant targets higher. When an ambig query has targets at ranks [1,3,4,6], nDCG@10 < 1.0 and moves if salience lifts rank-3 to rank-2. This is the metric 12.1e2 should optimize.
2. **Per-query `found_ranks` detail.** Ambig queries returned multi-rank hits (e.g. `lesson-ambig-measurement-methodology` → HIT@1,3,4,6). Changes to half-life will shuffle these ranks even if the query's MRR stays at 1.0. 12.1e2 should report per-query `found_ranks` diffs, not just aggregate metrics.
3. **The missed cross-topic query** (`lesson-cross-workflow-gate`) is a real regression vs. 12.1d-fix (where it presumably hit). Possibly data drift (new lessons pushed the target out of top-10). Worth investigating — could be a new friction class.

### Latency regression flagged — is it real?

The diff tool flags `p95_ms +227%` as 🔴. Context: 12.1d-fix ran at 2590ms p95 right after other baseline activity (warm embedding cache). 12.1e1 ran after a ~24h gap + fresh `docker compose up` (cold cache). This is EXACTLY the `measurement-jitter` friction class (`docs/qc/friction-classes.md`): back-to-back runs are deterministic; session-separated runs jitter by embedding-service load. The latency delta is NOT a code regression; quality metrics are unchanged.
