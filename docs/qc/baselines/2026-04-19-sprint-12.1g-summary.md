# TEI external rerank evaluation — Sprint 12.1g

## Recommendation (one line)

**Keep `RERANK_TYPE=generative` as production default. Do NOT adopt TEI+bge-reranker-v2-m3.** Despite architectural success (deterministic, properly routed, no fallback errors), bge via TEI's `/rerank` endpoint underperforms every alternative on aggregate: worse than gte (MRR −0.026), worse than generative (MRR −0.072), and **worse than no-rerank on nDCG@10** (−0.003). Infrastructure deployed (`tei-rerank` docker service + `RERANK_TYPE=api` code path) is useful for future reranker experimentation, but bge-v2-m3 specifically does not earn a default change.

### Per-group reading

- **confident-hit, duplicate-trap:** all rerankers tie; these groups are saturated — no differentiation possible.
- **cross-topic:** TEI+bge is the WORST non-saturated group — 0.6095, essentially same as no-rerank (0.5945). Generative wins decisively here (0.9751); gte is partial (0.8289). bge seems unable to pick the correct target among multi-target cross-topic candidates.
- **ambiguous-multi-target:** TEI+bge is the BEST (+0.003 over generative, within noise but not losing). One of bge's few strengths.
- **semantic-paraphrase:** TEI+bge is WORST (0.8000) alongside gte (0.8712). Both generative and no-rerank get 1.0000 here. bge mis-ranks paraphrased queries — likely a representation / training-domain mismatch.

## Determinism verdict

✅ **TEI+bge is deterministic.** 0/40 queries differ between initial and repeat runs at the same config. Architecturally, the `/rerank` endpoint produces deterministic output across container recreates (unlike LM Studio's generative LLM rerank).

## Why bge-v2-m3 underperformed

A few plausible explanations:

1. **Training domain mismatch.** bge-reranker-v2-m3 is multilingual + general-purpose. Our goldenset is English, highly technical (pg, undici, dedup, salience). A reranker trained on generic QA pairs may not weight the query-document relevance well for our corpus.
2. **Title + snippet representation is short.** We send `"${title}. ${snippet}"` — typically ~300 chars. bge may expect longer document context to score well.
3. **Score calibration.** bge outputs relevance scores in a narrow range (~1e-4 to 1e-1 in our tests). The ranking signal is there (highest scores ARE the right answer for clean queries) but noise at the rank-10 borderline is higher than LM Studio's generative rerank.
4. **Cross-topic / paraphrase queries need semantic reasoning.** Generative LLM can reason that "sprint 11.6c-sec DNS pinning body-stall timeout" → undici-Node version lesson via connecting concepts. bge matches surface features more literally.

The infrastructure works; the model was the wrong choice for this goldenset. Sprint 12.1h candidate: try **jina-reranker-v3** or **qwen3-reranker-8b** via same TEI infra (just change `--model-id` flag).

## Architecture deliverables (shipped regardless of winner verdict)

1. **`tei-rerank` Docker service** — HuggingFace TEI CPU image, loads BAAI/bge-reranker-v2-m3, healthchecked, named volume for model cache. First-run model download ~4 min; subsequent starts <30 s.
2. **`RERANK_TYPE=api` code path** — `rerankExternalApi()` in `src/services/lessons.ts` POSTs `{query, texts}` to `${RERANK_BASE_URL ?? 'http://tei-rerank:80'}/rerank` and parses `[{index, score}]` response. Fails open on HTTP/network/malformed errors.
3. **4 unit tests** covering happy path, HTTP 500, network error, empty response. Full `global.fetch` mock pattern.
4. **docker-compose.yml exposures:** mcp depends_on tei-rerank; tei_model_cache named volume. `RERANK_TYPE` (12.1f) + `RERANK_BASE_URL` via existing env mechanism.

These unlock future swap-in experiments with minimal changes:
```bash
docker compose exec tei-rerank kill 1  # restart TEI container
# Edit docker-compose.yml: change --model-id flag
docker compose up -d --force-recreate tei-rerank  # load new model
RERANK_TYPE=api docker compose up -d --force-recreate mcp
npm run qc:baseline -- --tag sprint-12.1h-<model>
```
