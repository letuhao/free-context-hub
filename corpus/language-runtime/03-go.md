# Go — runtime, concurrency, and selection facts

## Compilation & runtime model
Go is **compiled ahead-of-time to native machine code** (not interpreted, not JIT-compiled at
run time). The toolchain links the Go **runtime** — scheduler, garbage collector, and memory
allocator — directly into every binary, and by default produces a **single statically linked
executable** with no external Go dependencies. The practical payoff is small, self-contained
artifacts with **fast startup and no warm-up phase**, which is why Go is a common choice for
containers, CLIs, and serverless functions where cold-start latency matters. Compile times are
deliberately fast, a stated design goal.

## Concurrency: goroutines and the scheduler
Go's headline feature is the **goroutine** — a function executing concurrently, created with the
`go` keyword. Goroutines are **not OS threads**: the runtime **multiplexes many goroutines onto a
smaller pool of OS threads** (an **M:N scheduler**). When a goroutine blocks — for example on a
blocking system call — the runtime moves the other goroutines on that OS thread to a different,
runnable thread so they are not blocked. Goroutines are cheap because a new one starts with only
**a few kilobytes** of stack (the stack is **resizable**, grown and shrunk automatically), and the
per-call overhead is roughly **a few cheap instructions**. As a result **hundreds of thousands of
goroutines** can coexist in one address space — a scale at which OS threads would exhaust system
resources. Goroutines do run in **true parallelism** across multiple cores (unlike Python's
GIL-bound threads); `GOMAXPROCS` controls how many OS threads execute Go code simultaneously.

## Channels and CSP
Communication between goroutines uses **channels**, derived from Hoare's **Communicating
Sequential Processes (CSP)**. The guiding proverb is **"Do not communicate by sharing memory;
instead, share memory by communicating."** Channels are **first-class values**; a `select`
statement waits on multiple channel operations at once. Channels are the idiomatic synchronization
tool, though Go *also* provides conventional mutexes (`sync.Mutex`) for cases where shared-memory
locking is simpler — channels are preferred, not mandatory.

## Garbage collection
Go is **garbage-collected** using a **concurrent mark-and-sweep** collector that runs in parallel
with the program on separate cores. It is tuned for **low latency**: pause times are often in the
**sub-millisecond range even for large heaps**. This trades some throughput/CPU for predictable
latency — a deliberate fit for network servers. (Contrast Rust, which achieves memory safety with
**no GC at all**.)

## Error handling
Go uses **explicit error values, not exceptions**. The convention is to return a `(result, err)`
pair and check `if err != nil`. The designers argue that coupling errors to `try-catch-finally`
produces convoluted code and tempts programmers to label ordinary failures (like a missing file)
as "exceptional." `panic`/`recover` exist for **truly unrecoverable** situations and resource
teardown — **not** for normal control flow. The trade-off is verbosity (frequent `if err != nil`)
in exchange for explicit, visible error paths.

## Typing and generics
Go is **statically typed** and compiled, with type checking at build time. **Generics (type
parameters) were added in Go 1.18 (2022)**; earlier versions had none, and code relied on
interfaces or `interface{}`/code generation. Generics were deliberately deferred for years to keep
the type system and runtime simple — added only once a design judged "value proportionate to the
complexity" was found. The syntax uses square brackets (`[T any]`).

## Ecosystem and selection fit
Go's design bias is **simplicity and readability over expressive power** (small spec, one
formatting standard via `gofmt`, deliberately few features). Its sweet spot is **cloud-native and
network/back-end infrastructure**: Docker, Kubernetes, and CockroachDB are written in Go. Choose
Go for concurrent network services, CLIs, and ops tooling where fast builds, easy deployment
(single binary), true-parallel concurrency, and team-scale maintainability matter more than raw
single-thread peak performance or a rich type system. It is a weaker fit for hard-real-time or
latency-floor work where GC pauses are unacceptable (→ Rust/C++), or for heavy generic/metaprogramming
and large numeric/ML ecosystems (→ Python/C++).
